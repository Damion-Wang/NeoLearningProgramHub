# spec-私教系统

---

## 一、系统概述

### 1.1 一句话定位

**Leo = 统一前台私教形象。** 学员只看到一个"Leo"——一段关系、一份信任、一个"懂我"的发展伙伴。

后台由两个独立 Persona 支撑，通过 Blackboard 协作：

| Agent | 角色 | 主场 | 特征 |
|-------|------|------|------|
| **CoachPersona**（教练态） | 苏格拉底式提问、引导反思、实践辅导 | 对话框 | 不给答案，引导学员自己找到答案 |
| **LecturerPersona**（教学态） | 结构化教学、知识讲解、验证学习 | 课堂页 | 系统化地教会学员所需知识 |

### 1.2 核心架构关系

```
用户 ──→ Leo ──→ CoachPersona（教练态，默认）
                    ↑↓ 信号驱动切换
                 LecturerPersona（教学态）
                    ↑↓
               Blackboard（共享状态）
```

- **Blackboard**：CoachPersona 和 LecturerPersona 之间的同步协作通道，运行时状态的唯一读写点。
- **教学内容**：通过 tool call 随用随取，context 只放决策信息。

### 1.3 设计原则

| 原则 | 说明 |
|------|------|
| **后台分离，前台统一** | 工程上两个 Persona 独立（各自调优、各自评估、System Prompt 不打架），体验上一个"Leo" |
| **状态在中间，Persona 来读写** | Blackboard Architecture——不靠摘要传递，靠结构化状态对象 |
| **教学内容不进 context** | 通过 tool call 随用随取，context 只放决策信息 |
| **信号驱动，不定时轮询** | 所有主动行为必须有可解释的信号触发 |
| **Leo 是学员的人** | 清晰的立场：私教管 HOW（怎么学、怎么做、怎么提升） |

## 二、Blackboard 状态机

Blackboard 是 Agent 间的同步协作通道——CoachPersona 和 LecturerPersona 不直接对话，而是通过 Blackboard 上的结构化状态完成信息交换。

### 2.1 五类运行时状态

| 状态类 | 内容 | 特征 | 生命周期 |
|--------|------|------|---------|
| **会话状态** | 当前阶段（OPENING / EXPLORING / DEEPENING / INSIGHT / ACTION / CLOSING）、活跃主题、情绪标记 | 实时感知，每轮更新 | 单次会话 |
| **认知状态** | 三向量负荷 `{intrinsic, extraneous, germane}`、当前 Dreyfus 层级、ZPD 区间 | 实时计算 | 单次会话（ZPD 基线除外） |
| **教学状态** | 正在执行的教学脚本 ID、进度百分比、待验证知识点列表、验证结果 | 教学态专属，教练态需读 | 跨会话（未完成教学必须持久化） |
| **人格状态** | 当前模式（MENTOR / STRATEGIST / CHALLENGER / OBSERVER）、五维参数快照（节奏 / 引导度 / 鼓励度 / 信息密度 / 经验调用） | 慢变量，跨轮次稳定 | 持久（归入学员 Profile） |
| **实践状态** | 待执行行动项、脚手架列表、回流时间锚点 | 有截止日期，需持久化 | 跨会话（有时间锚点） |

### 2.2 读写规则

Blackboard 上的状态不是一视同仁的——不同状态类有不同的读写模式：

| 写入模式 | 适用状态 | 规则 |
|---------|---------|------|
| **覆盖写** | 会话状态、认知状态、人格状态 | 只保留最新值，历史值不保留（运行时瞬时状态） |
| **追加写** | 教学状态中的验证记录 | 本次会话的事实日志，不可覆盖 |
| **原子写** | Handoff 状态包 | 整体写入，不允许半写——要么全写，要么全不写 |

### 2.3 Handoff 协议

Agent 间切换时，Blackboard 上的 Handoff 状态包是交接的唯一契约。

**教练态 → 教学态**（CoachPersona 写入）：

| 字段 | 内容 | 用途 |
|------|------|------|
| `triggerReason` | 触发原因（知识缺口 / 概念偏差 / 学员主动求知） | 教学态据此决定教学策略 |
| `topicContext` | 当前话题上下文（正在讨论什么、学员关注什么） | 教学态锚定阶段需要 |
| `emotionSnapshot` | 学员情绪快照 | 教学态调整呈现方式 |
| `scaffoldAttempts` | 已尝试的即时脚手架及其效果 | 避免教学态重复无效策略 |

**教学态 → 教练态**（LecturerPersona 写入）：

| 字段 | 内容 | 用途 |
|------|------|------|
| `taughtContent` | 教了什么（知识点 ID + 验证结果） | 教练态知道学员新学了什么 |
| `masteryAssessment` | 学员掌握度评估 | 教练态决定是否在实践中强化 |
| `unfinishedContent` | 未完成内容（如有中途退出） | 下次教学态继续 |
| `suggestedAction` | 建议后续行动 | 教练态衔接实践桥接 |

### 2.4 Design 细化：Blackboard 字段定义

#### 会话状态 `sessionState`

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `phase` | enum | `OPENING` | OPENING / EXPLORING / DEEPENING / INSIGHT / ACTION / CLOSING |
| `activeTopic` | string \| null | `null` | 当前话题 |
| `topicHistory` | string[] | `[]` | 本次会话话题列表（按时间序） |
| `emotionLabel` | string \| null | `null` | 情绪文本标记（LLM 生成） |
| `emotionCategory` | enum | `neutral` | positive / negative / neutral / mixed（下游分支用） |
| `emotionIntensity` | number(0-1) | `0.5` | 情绪强度 |
| `turnCount` | number | `0` | 会话轮次 |
| `activeMode` | enum | `COACH` | COACH / LECTURER |
| `switchSignal` | SwitchSignal \| null | `null` | 模式切换信号（见 §五 Design 细化） |

#### 认知状态 `cognitiveState`

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `load.intrinsic` | number(0-1) | `0.3` | 内在认知负荷 |
| `load.extraneous` | number(0-1) | `0.2` | 外在认知负荷 |
| `load.germane` | number(0-1) | `0.3` | 有效认知负荷 |
| `dreyfusLevel` | number(1-5) | — | 当前话题 Dreyfus 层级 |
| `zpd.baseline` | [number, number] | — | ZPD 基线区间 |
| `zpd.adjusted` | [number, number] | — | 动态调整后 ZPD |
| `chunksInFlight` | number(0-4) | `0` | 活跃概念块数（Cowan 限制） |

**【外审修正】** 不再有 `composite` 综合指标。硬停条件改为双阈值：`extraneous >= 0.6` 或 `intrinsic >= 0.7`。germane 单独作为 ZPD 推进的正向信号。

**【Stress-test S1】** 优先级链：**硬停 > 心流保护 > 常规演进**。当 intrinsic≥0.7 同时 germane>0.6 时，硬停优先执行，忽略心流判定。

**【外审修正】** 增加对话行为代理指标 `proxyMetrics`：

| 字段 | 类型 | 说明 |
|------|------|------|
| `consecutiveQuestions` | number | 连续追问次数（>3 → extraneous 偏高） |
| `selfCorrectionCount` | number | 自我修正次数 |
| `scaffoldSuccessRate` | number | 脚手架使用成功率 |
| `responseLatencyTrend` | enum | faster / stable / slower |
| `engagementLevel` | enum | active / passive / disengaged |

#### 教学状态 `teachingState`

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `active` | boolean | `false` | 是否在教学周期中 |
| `scriptId` | string \| null | `null` | 教学脚本 ID |
| `lessonId` | string \| null | `null` | 关联课程 ID |
| `stage` | enum \| null | `null` | ANCHORING / DELIVERING / VERIFYING / CLOSING |
| `knowledgeBlocks` | KnowledgeBlock[] | `[]` | 知识块列表 |
| `currentBlockIndex` | number | `0` | 当前知识块索引 |
| `verificationLog` | VerificationEntry[] | `[]` | 验证记录（追加写） |
| `blockRetryCount` | number | `0` | 当前块验证重试计数（≤3） |
| `sessionFailureCount` | number | `0` | 本次会话累计失败（≥4 触发暂停） |

**【外审修正】** 退出条件：单知识块同策略限 1 次，第 2 次切换策略+认知卸载，第 3 次写入断点。会话累计 ≥4 次失败全面暂停。

**【Stress-test S2】** 暂停恢复条件：新会话自动重置 `sessionFailureCount`；同一知识块标记 `deferred` 后 24h 冷却期，冷却后解锁可重新教学。永不"死档"。

#### 人格状态 `personaState`

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `mode` | enum | `MENTOR` | MENTOR / STRATEGIST / CHALLENGER / OBSERVER |
| `params.rhythm` | number(1-10) | `5` | 节奏 |
| `params.guidanceLevel` | number(1-10) | `5` | 引导度 |
| `params.encouragement` | number(1-10) | `5` | 鼓励度 |
| `params.infoDensity` | number(1-10) | `5` | 信息密度 |
| `params.experienceInvoke` | number(1-10) | `5` | 经验调用 |

#### 实践状态 `practiceState`

- `actionItems: ActionItem[]` — 待执行行动项（id, description, dueDate?, status, linkedKnowledge?）
- `scaffolds: Scaffold[]` — 脚手架列表（id, type: checklist/agenda/tips/template, title, content）
- `reflowAnchors: ReflowAnchor[]` — 回流时间锚点（id, triggerDate, reason, linkedAction, status）

### 2.5 Design 细化：Handoff 接口契约

#### 元数据（两个方向共用）

| 字段 | 类型 | 说明 |
|------|------|------|
| `handoffId` | string(UUID) | 幂等键 |
| `version` | number | 乐观并发控制，写入时 +1 |
| `direction` | enum | coach_to_lecturer / lecturer_to_coach |
| `timestamp` | string(ISO 8601) | |
| `initiator` | string | 切换发起方标识 |

写入规则：Supabase `upsert` with `handoff_id` 唯一约束；读取端校验 `version`；同一 `handoffId` 重复写入为幂等操作。

#### 教练态 → 教学态

| 字段 | 说明 |
|------|------|
| `triggerReason` | 信号类型 + 次要信号 + 置信度 + 证据摘要 |
| `topicContext` | 当前话题 + 学员关注点 + **triggerContext**（压缩摘要 + anchorTurnId 定位索引 + 关联记忆 ID，不搬原文） |
| `emotionSnapshot` | category + label + intensity |
| `scaffoldAttempts` | 已尝试的脚手架及效果 |
| `currentStage` | **【外审修正】** 会话阶段（EXPLORING/DEEPENING 等） |
| `dreyfusSnapshot` | **【外审修正】** 当前话题 Dreyfus 层级 |
| `cognitiveLoadSnapshot` | **【外审修正】** 三向量快照 |

**设计决策**：Handoff = 索引 + 压缩，不搬原文。教学态需要更多上下文时，按 `anchorTurnId` 从 AgentContext 对话流中回溯。**【Stress-test S6】** `triggerContext.summary` 作为降级恢复基线——当 anchorTurnId 因 context 清理失效时，仍可从 summary 恢复最小上下文。

#### 教学态 → 教练态

| 字段 | 说明 |
|------|------|
| `taughtContent` | 知识块 ID + 话题 + 验证结果 + mastery |
| `masteryAssessment` | 整体评估（solid / shaky / insufficient）+ 详情 |
| `unfinishedContent` | 未完成块 + 原因（hard_stop/learner_exit/consecutive_failure）+ 断点描述 |
| `suggestedAction` | 后续建议（practice/reflect/revisit_later/none）+ 关联脚手架 |
| `exitReason` | **【外审修正】** completed / learner_exit / hard_stop / consecutive_failure |
| `cognitiveLoadAtExit` | **【外审修正】** 退出时的三向量 |

#### 教学态期间的消息处理

教学态期间消息直接由 LecturerPersona 处理，教学态自主判断：

| 学员行为 | 教学态处理 |
|---------|-----------|
| 顺承/有意义发散 | 接住，融入教学 |
| 主题相关的支撑性延伸 | 接住，标记偏离，适时拉回 |
| 超纲 | 收住（"这个好问题，先搞定手头的"），写入待讨论话题 → Handoff 交回教练态 |

唯一例外：认知负荷硬停由系统强制执行——教学态有"教完的冲动"，自己不会主动停。

## 三、三层记忆架构

### 3.1 三层定义

| 层 | 名称 | 职责 | 生命周期 | 与 Blackboard 关系 |
|---|------|------|---------|---------------------|
| **短期** | AgentContext | 当前对话消息流、即时推理、Blackboard 本身 | 单次会话 | Blackboard 就在这层 |
| **中期** | Session Store | 跨会话学习进度、未完成任务、活跃教学周期、趋势摘要 | 跨会话，可过期/归档 | 会话结束时 Blackboard 沉淀到这层 |
| **长期** | Profile | 能力贝叶斯分布、Dreyfus 层级、行为模式、人格参数、学习风格 | 持久 | 教学验证结果最终聚合到这层 |

### 3.2 写入链（信息沉淀）

信息从短期 → 中期 → 长期逐层沉淀，但不是所有信息都沉淀，也不是每次都沉淀。

#### 短 → 中 沉淀规则

| Blackboard 状态 | 沉淀策略 | 理由 |
|----------------|---------|------|
| 会话状态（阶段、情绪） | **丢弃** | 下次重新感知，历史值无意义 |
| 认知状态（负荷三向量瞬时值） | **丢弃** | 实时计算，历史值交给趋势聚合器 |
| 教学状态（脚本进度、验证结果） | **沉淀到中期** | 未完成教学必须记住，验证结果是事实记录 |
| 人格状态（模式、五维参数） | **沉淀到长期** | 慢变量，归入学员 Profile |
| 实践状态（行动项、脚手架、回流锚点） | **沉淀到中期** | 有时间锚点的必须持久化 |

#### 写入触发点

不只是会话结束才写——关键事件即时沉淀：

| 触发事件 | 写入目标 | 理由 |
|---------|---------|------|
| 教学周期完成 | 立即写中期 | 教学成果不能等到会话结束 |
| 行动项创建 | 立即写中期 | 有截止日期的任务不能丢 |
| 能力验证完成 | 立即写中期；如聚合阈值达标则推长期 | 能力变化是核心数据 |
| 会话正常结束 | 批量沉淀剩余状态 | 兜底机制 |

#### 中 → 长 聚合规则

**不是每次都写长期**——聚合达到置信度才更新：

- 单次验证失败 **不改** Dreyfus 层级。连续三个相关知识点验证失败才触发降级评估。
- 人格参数调整需要跨多次会话的一致性信号（如连续 5 次会话学员都偏好 MENTOR 模式）。
- 能力贝叶斯分布按标准 Beta 更新规则聚合，单次观测仅微调后验分布。

### 3.3 读取链（信息召回）

新会话启动时的加载序列：

```
长期画像加载基线
  → Dreyfus 层级、人格参数、能力分布
    → 中期存储加载活跃任务
      → 未完成教学、待回流行动项、趋势摘要
        → 组装 Blackboard 初始状态
          → 前几轮对话补充实时感知（情绪、负荷）
```

**选择性加载**：

学员开口 → 判断意图 → 决定加载哪些中期记忆 → 路由给对应 Persona。

设计理由：不把中期存储全部灌入 Blackboard，只加载与当前对话相关的。全量加载会导致 context 膨胀和注意力分散。

### 3.4 交叉引用链（教练态 ↔ 教学态间接协作）

两个 Persona 不直接传话，通过记忆层间接感知对方的工作成果。**Blackboard 是同步通道，记忆是异步通道。**

典型协作链路：

```
教练态发现学员反复在"委派任务"上卡住
  → 写入中期存储标记 { type: "knowledgeGap", topic: "委派任务", confidence: 0.7 }
    → 下次会话加载中期存储时看到标记
      → 可主动建议教学（"上次你提到委派，要不要系统学一下？"）
        → 教学态完成教学后写入验证结果
          → 长期画像更新"委派任务"能力分布
            → 教练态下次读到更新后的画像，调整提问策略
```

### 3.5 外审修正项（5 项，已采纳）

#### 修正 1：中期存储 staleness 机制

每个中期条目携带 `lastRelevantAt` 时间戳：

| 条件 | 处理 |
|------|------|
| 超过 N 天未被引用 | 标记 `stale` → 不再自动加载，学员主动提及时可召回 |
| 超过 M 天仍 `stale` | 归档到长期（提取有价值部分）或清除 |

N、M 的具体值待 Design 阶段根据使用频率数据确定（初步估计 N=14，M=60）。

#### 修正 2：趋势聚合器

会话结束时计算并更新中期存储中的趋势摘要：

| 趋势字段 | 计算逻辑 | 沉淀条件 |
|---------|---------|---------|
| `emotionTrend` | 最近 N 次会话的情绪模式（如"DEEPENING 阶段高频焦虑"） | 趋势持续稳定 → 沉淀为长期画像的一部分 |
| `loadTrend` | 最近 N 次的平均负荷分布 | 同上 |
| `engagementTrend` | 最近 N 次的参与度模式（主动提问频率、会话深度） | 同上 |

示例沉淀：`emotionTrend` 连续 5 次显示 DEEPENING 阶段焦虑 → 长期画像新增标记 `"learningStyle.deepeningAnxiety": true` → CoachPersona 在 DEEPENING 阶段主动降速。

#### 修正 3：中期条目 priority + urgency 排序

| 维度 | 定义 | 排序逻辑 |
|------|------|---------|
| `priority` | 重要性 | 能力缺口 > 未完成教学 > 待回流行动 |
| `urgency` | 时效性 | 有时间锚点的 > 无截止日期的 |

加载时：按 `urgency` 排序 → 按 `priority` 筛选 → 最多加载 Top-K 条（K 初始值 = 5，可根据 context window 动态调整）。

#### 修正 4：Handoff 原子性

状态包整体写入，不允许半写状态。实现上使用事务写入（Supabase 的 `rpc` 或前端 batch update）。

半写场景的灾难：教练态写了 `triggerReason` 但没写 `topicContext` → 教学态不知道从哪接话 → 用户体验断裂。

#### 修正 5：ZPD 重分类为混合变量

ZPD 不是纯短期变量，而是混合变量：

| 成分 | 来源 | 生命周期 |
|------|------|---------|
| ZPD 基线 | 从长期画像的 Dreyfus 层级 + 能力分布推算 | 持久，随长期画像更新 |
| ZPD 动态调整 | 当前会话根据表现微调上下界 | 单次会话 |

示例：长期画像显示学员在"委派任务"上是 Dreyfus L2（高级新手）→ ZPD 基线推算为 L2→L3 区间 → 当前会话学员表现超预期 → 动态上调 ZPD 上界至 L3.5。

### 3.6 Design 细化：生命周期管理

#### 中期存储条目结构

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | string | |
| `userId` | string | |
| `type` | enum | teaching_progress / action_item / knowledge_gap / scaffold / reflow_anchor / trend_summary / **persona_observation** |
| `payload` | **Discriminated Union**（按 type 强类型，Zod 校验） | **【外审修正】** 不再是 `Record<string, any>` |
| `createdAt` | string | |
| `lastRelevantAt` | string | 最后被加载/引用的时间 |
| `status` | enum | active / stale / archived |
| `priority` | enum | critical / high / medium / low |
| `urgency` | enum | time_bound / soon / whenever |
| `topicTags` | string[] | 关键词标签（MVP 语义召回） |
| `confidence` | number | **【外审建议】** 条目可靠度 |
| `sourceAgent` | string | **【外审建议】** 写入方 Agent 标识 |

排序：复合衰减分数 `Score = P_weight × U_weight × e^(-λΔt)`，统一排序。**【外审修正】** 替代原来的独立双维度排序。

#### 人格状态沉淀修正

**【外审修正】** 人格状态不再"跳过中期直写长期"。修正流程：
1. 观察到人格偏好变化 → 写入中期 `persona_observation` 类型
2. 灰度验证期（3 次会话）：仅作为读取链的 Soft Hint，不覆盖长期基线
3. 连续 5 次一致且方差低于阈值 → 触发 Mid→Long 聚合

#### 归档有价值判定

**【外审修正】** 超 M 天仍 stale 的条目，量化启发式规则判定"有价值"：
1. 频次阈值：同一 `topicTags` 关联条目 ≥3 次
2. ZPD 影响度：曾触发 `knowledge_gap` 或 `scaffold` 且最终有验证结果
3. 显式反馈：用户明确点赞/纠错

满足 ≥2 条 → 提取聚合到长期；否则软删除。

#### Staleness 参数

| 参数 | 值 | 理由 |
|------|-----|------|
| N（active→stale） | 14 天 | 两周没碰大概率不紧急 |
| M（stale→archive） | 60 天 | 两个月缓冲 |

MVP 写死，预留配置接口。长期改动态衰减（高频用户缩短 N）。

#### 读取链

新会话：加载长期 Profile → 学员第一句话 → 判断意图 → 按 `RelevanceScore` 排序 → Top-K=5 → 组装 Blackboard。后续轮次仅在话题明显转换时增量加载。

**【外审建议】** payload 强制结构化摘要（≤150 tokens），防止 context 膨胀。

#### Dreyfus 降级三层机制

**【外审修正】** 不自动降级：
1. **规则层**：连续 3 次相关知识点验证失败 → 标记 `demotion_candidate`，冻结自动升级
2. **仲裁层**：综合中期趋势（负荷曲线、行动项完成率、ZPD 跨度）+ 用户显式反馈
3. **执行层**：仲裁置信度 > 0.8 时执行降级，否则转为"针对性脚手架干预"

#### 会话 Checkpoint 容错

**【外审修正】** 批量沉淀增加心跳机制：
- 每 N 轮对话或关键节点（验证完成、行动项创建）异步落盘中期 Checkpoint
- 会话正常结束时仅做 Diff 合并
- 会话中途崩溃则从最近 Checkpoint 恢复

## 四、ZPD × 认知负荷耦合策略

### 4.1 核心原则

ZPD 不是固定的"推力"，而是与认知负荷状态耦合的动态策略。

**学习节奏应该是波浪形的：挑战 → 巩固 → 挑战 → 巩固。** 一直推会压垮学员，一直巩固会让学员无聊。正确的策略是在认知负荷允许的窗口期推进 ZPD，在负荷高企时主动巩固。

### 4.2 时机决策矩阵

| 认知负荷状态 | ZPD 策略 | 理由 |
|-------------|---------|------|
| germane 高 + extraneous 低（**心流区**） | 推 ZPD 的**黄金窗口** — 加难度、引入新概念、做挑战性提问 | 深度加工中，工作记忆有余量 |
| intrinsic 高 + germane 中（**啃硬骨头**） | **维持当前难度，不加码** — 给脚手架、拆小步骤 | 概念本身就难，再推会压垮 |
| extraneous 高（**认知噪声**） | **先降噪** — 简化呈现方式、消除干扰信息 | 外在负荷挤占工作记忆，推 ZPD 无效 |
| 三向量都低（**舒适区 / 无聊**） | **温和推入 ZPD** — 提问升维、引入新视角 | 有余量接受挑战 |
| 刚完成高强度学习（**恢复期**） | **巩固 > 挑战** — 回顾、连接已有知识、迁移练习 | 新知识需要"定型"时间 |

### 4.3 波浪节奏监控

趋势聚合器捕捉"最近三次会话全是高强度" → 主动建议一次巩固型会话。

教练态和教学态都使用此机制，但 ZPD 的表现不同：

| 模态 | ZPD 推进的表现 | ZPD 巩固的表现 |
|------|---------------|---------------|
| **教练态** | 提问深度升级、引入更复杂的情境反思、挑战学员假设 | 回顾已有洞察、连接不同场景的共性、低压力对话 |
| **教学态** | 概念难度升级、验证要求提高、练习复杂度增加 | 复习已学内容、做迁移练习、费曼讲解巩固 |

### 4.4 Design 细化：认知负荷计算模型

#### 硬软信号融合

| 信号类型 | 来源 | 擅长测量 | 特征 |
|---------|------|---------|------|
| 硬信号（代理指标） | 对话行为直接提取 | extraneous / germane | 稳定、可复现 |
| 软信号（LLM 判断） | LLM 每轮内部分析 | intrinsic / germane | 语义丰富、方差大 |

融合权重：**MVP 初始 0.3:0.7（软信号主导）**。硬信号在 MVP 阶段不成熟（延迟受多任务干扰、字数受打字习惯影响），以 LLM 语义判断为主，硬信号为趋势参考。

演进路径：硬信号连续 5 会话自身变异系数稳定 → 人工审查 → 手动步进到 0.5:0.5 → 最终 0.6:0.4。

软信号降方差措施：JSON 结构化输出、Few-shot 锚定、Temperature=0.2、每会话 1 次元认知校验提问。

三向量保持**向量解耦**，不压缩为标量。时机决策矩阵直接基于向量组合判断。

**【Stress-test S8】** 超过 20 轮的长会话启用 proxyMetrics 滑动窗口摘要，防止早期指标被 context 截断导致负荷计算失真。

#### 冷启动策略

前 5 次会话为基线采集期：
- 使用同角色/职级群体 P50 值作为初始硬信号基线
- 个人数据权重从 0% 线性增长到第 5 次的 100%
- 冷启动期采用 LLM + 群体基线 + 元认知追问三角校验

#### ZPD 区间计算

从 Dreyfus 层级推算基线 → 根据能力分布均值和近期趋势动态调整：

| Dreyfus | 基线 ZPD | 说明 |
|---------|---------|------|
| L1（新手） | 1.0→2.0 | 推向高级新手 |
| L2（高级新手） | 2.0→3.0 | 推向胜任 |
| L3（胜任） | 3.0→4.0 | 推向熟练 |
| L4（熟练） | 3.5→4.5 | 区间收窄，进步变难 |
| L5（专家） | 4.0→5.0 | 巩固+迁移 |

动态调整：能力均值偏高 → 上调下界；近期连续高表现 → 上调上界；近期挫败 → 收窄区间。

#### 时机决策矩阵补充

**【外审修正】** 新增 intrinsic 高 + extraneous 高的组合：
- 双高是最危险状态 → 先降 extraneous（简化呈现）→ 再评估 intrinsic → 仍高则拆分概念

**【外审修正】** 简化模式具象化：
- intrinsic 高 → Chunking（拆小步骤）
- extraneous 高 → Dual-Channel（图文替代纯文本，移除冗余）
- germane 高 → 维持当前节奏，不干预

#### 波浪节奏：会话内微节奏

**【外审修正】** 每 5 轮检查一次：如果连续高强度，插入低压力回顾/类比/轻松连接。

#### Hysteresis（防微幅震荡）

**【外审修正】** 认知负荷微幅波动不立即触发策略变更。负荷状态需持续 2 轮以上才触发切换，与 §五 的 signal_cooldown 统一。

## 五、模式切换引擎

### 5.1 触发信号（8 类，分两组）

#### "缺"类信号（知识空白）——学员不知道

| # | 信号 | 特征 | 观察窗口 |
|---|------|------|---------|
| 1 | **知识缺口** | 核心概念理解为空白，不是"理解不深"而是"没听过" | 1-2 轮 |
| 2 | **重复卡点** | 同一维度连续多次困难，教练态引导无法突破 | 跨 2-3 轮 |
| 3 | **即时脚手架失败** | 教练态给了即时提示但学员用不了 | 1-2 轮 |
| 4 | **学员主动求知** | 直接问"怎么做"/"能教教我吗"/"这个概念是什么" | 立即 |
| 5 | **实践准备不足** | 即将实践但缺必要知识支撑，做了会出问题 | 1-2 轮 |

#### "偏"类信号（理解偏差）——学员以为自己知道，但理解有误

| # | 信号 | 特征 | 危险度 |
|---|------|------|--------|
| 6 | **错误应用** | 方法用错了但自以为对（如把"教练式提问"理解为"质问"） | 高 |
| 7 | **概念混淆** | 相近概念搞混（如把"委派"和"甩锅"混为一谈） | 中高 |
| 8 | **纠偏后不认同** | 教练态纠正了但学员底层理解有结构性问题，单次纠偏不够 | 极高 |

> **"偏"比"缺"更危险**——缺知识的学员知道自己不知道，偏差的学员以为自己知道，会在错误路径上越走越远。

### 5.2 信号检测机制

信号检测由 LLM 完成，**不硬编码规则**。

CoachPersona 每次回复前做内部分析（不输出给用户）：

```
内部分析（每轮）：
- 是否检测到知识缺口？[是/否 + 证据摘要]
- 是否检测到概念偏差？[是/否 + 证据摘要]
- 建议动作：[继续教练态 / 标记需要教学 / 紧急切换]
```

分析结果写入 Blackboard 的 `switchSignal` 字段，由切换引擎读取后做最终决策。

设计理由：规则硬编码会降智。"学员懂不懂、偏没偏"只有 LLM 才能基于语义判断，正则匹配或关键词检测做不到。

### 5.3 切换二次确认

CoachPersona 标记了信号不等于立即切换——需做二次确认：

| 维度 | 判断逻辑 | 说明 |
|------|---------|------|
| **认知负荷** | 高负荷时仍可切换，但教学态启动时用更简洁的呈现方式降低 extraneous | 不切换 = 学员继续卡着 = 更大的认知挫败 |
| **重复教学检查** | 查中期存储，近期教过同一知识点 → 跳过系统教学，改用快速回顾 | 避免教了又教 |
| **内容可用性** | 是否有对应教学内容 → 无内容则走即时脚手架 | 没有内容的知识点，教学态也教不了 |

### 5.4 灵敏度分级

| 信号类型 | 灵敏度 | 观察期 | 理由 |
|---------|--------|--------|------|
| "偏"类信号（6/7/8） | **快切** | 1-2 轮确认即可 | 带着错误理解继续走，危害更大 |
| "缺"类信号（1/2/3/5） | **缓切** | 观察 2-3 轮，先尝试即时脚手架 | 可能只是一时想不起来，脚手架能解决 |
| 学员主动求知（4） | **立即响应** | 无观察期 | 学员主动性是最强信号，不应让人等 |

### 5.5 切回信号（教学态 → 教练态）

| 触发条件 | 处理 |
|---------|------|
| **教学周期正常完成** | 写入验证结果到 Blackboard → 自然切回教练态，衔接实践 |
| **学员主动跳出** | 尊重学员自主权，记录未完成进度到中期存储 → 切回教练态 |
| **认知负荷硬停**（composite ≥ 0.75） | 暂停教学，写入断点到中期存储 → 切回教练态做情绪接住 |
| **心流保护**（germane > 0.6 且 extraneous < 0.3） | **不切！** 深度学习状态中，继续教学 |

### 5.6 用户体验

**核心原则：不要有明显的 UI 模式切换提示。**

学员感知到的应该是"Leo 自然地开始教我了"，不是"系统切换了模式"。

| 切换方向 | 过渡方式 |
|---------|---------|
| 教练态 → 教学态 | Leo 用自然过渡语衔接："你刚才提到的这个点，我来系统讲一下……" |
| 教学态 → 教练态 | 同样自然："刚才讲的这些，回到你的实际场景——" |

### 5.7 Design 细化：决策流状态机

#### 全局状态流

```
COACH_MODE → 每轮内部分析 → switchSignal
  → SIGNAL_EVALUATION（置信度检查 → 冲突仲裁 → 二次确认）
    → 通过 → HANDOFF → LECTURER_MODE → 退出 → RETURN_HANDOFF → COACH_MODE
    → 拒绝 → SIGNAL_DISMISSED → COACH_MODE
```

#### SwitchSignal 结构

```
{
  primarySignal: SignalType,         // 最高优先级信号
  secondarySignals: SignalType[],    // 次要信号
  confidence: number,                // 0~1
  evidenceSnippets: string[],        // 证据摘要
  suggestedAction: 'continue' | 'flag_for_teaching' | 'urgent_switch'
}
```

`SwitchSignal` 是 CoachPersona 的内部中间表示，由切换引擎消费。`primarySignal` 决定切换方向，`secondarySignals` 作为附加上下文。

信号类型优先级矩阵：

| 优先级 | 信号类型 | 灵敏度 |
|--------|---------|--------|
| 100 | structural_disagreement（纠偏后不认同） | 快切（1轮确认） |
| 90 | wrong_application（错误应用） | 快切 |
| 80 | concept_confusion（概念混淆） | 快切 |
| 70 | learner_request（主动求知） | 立即（0轮） |
| 60 | knowledge_gap（知识缺口） | 缓切（2轮观察） |
| 55 | repeated_stuck（重复卡点） | 缓切 |
| 50 | scaffold_failed（脚手架失败） | 缓切 |
| 45 | practice_unready（实践准备不足） | 缓切 |

冲突仲裁：多信号并发取最高优先级；同级按置信度。特殊规则：learner_request 与偏类信号指向不同话题时，learner_request 优先。

#### 置信度检查 + 澄清探针

- `>= 0.6` → 进入二次确认
- `0.3~0.6` → 触发澄清探针（下一轮插入探测性提问），连续 2 轮低置信度 → 降级规则 fallback
- `< 0.3` → 忽略，记录日志

#### 规则 Fallback

LLM 连续 2 轮低置信度时启用关键词+句法匹配。fallback 置信度：显式请求给 0.6，模糊匹配给 0.45（**【外审修正】** 从 0.7 下调，强制进入二次确认）。标记 `fallback_source: "rule"` 供追踪。

#### 二次确认三维度

| 维度 | 逻辑 |
|------|------|
| 认知负荷 | 高负荷不拒绝，但教学态用简化模式（Chunking/Dual-Channel） |
| 重复教学 | 14 天内同话题教过且通过→拒绝（走快速回顾）；教过未通过→批准（跳过锚定） |
| 教学内容可用性 | 无对应内容→拒绝（走即时脚手架） |

#### 观察期行为

"缺"类信号观察期内，CoachPersona 主动试探（定向脚手架），每观察期限 1 次。脚手架与信号类型绑定：

| 信号类型 | 脚手架策略 |
|---------|-----------|
| knowledge_gap | 概念类比/图示 |
| repeated_stuck | 步骤拆解/工作示例 |
| concept_confusion | 对比矩阵 |

脚手架记录在 Blackboard `scaffold_history` 中，防止重复无效试探。

#### 防震荡机制

**【外审修正】** `signal_cooldown`：切回教练态后 3 轮内抑制同类型/同话题信号。观察期绑定 `topic_context_id`，话题变更时自动重置。Blackboard 增加 `recent_switch_back_reason` 字段供防抖。

**【Stress-test S3】** cooldown 仅作用于模式切换信号（偏/缺/主动求知）。认知负荷硬停（extraneous≥0.6 / intrinsic≥0.7）是安全信号，必须穿透 cooldown 抑制层。

**【Stress-test S4】** 跨会话教学态频率监控：连续 3 次会话都触发教学态 → 自动降低切换灵敏度（缺类观察期从 2 轮延长到 4 轮），避免学员产生"被当差生"的感受。

#### 切回条件

| 条件 | 触发方 | 处理 |
|------|--------|------|
| 教学周期正常完成 | LecturerPersona | Handoff 切回 |
| 学员主动跳出 | LecturerPersona 判断 | 记录未完成 → Handoff |
| 认知负荷硬停 | **系统强制** | extraneous≥0.6 或 intrinsic≥0.7 → 中断+断点 |
| 心流保护 | LecturerPersona | germane>0.6 且 extraneous<0.3 → 不切（但加目标达成度检查） |
| 超纲 | LecturerPersona | 话题记录 → Handoff 交回 |
| **目标已达成** | LecturerPersona | **【外审修正】** 核心知识块全部验证通过 → 自然切回，即使心流中 |

## 六、教学态完整周期

教学态不是一个"讲知识"的黑箱，而是一个结构化的四阶段周期。每个阶段有明确的目标、输入、输出和退出条件。

### 6.1 阶段 1：锚定（目标：30 秒内建立教学起点）

| 动作 | 说明 |
|------|------|
| **接住上下文** | 从 Blackboard 读取教练态写入的 Handoff 状态包（triggerReason + topicContext），**不从零开始** |
| **建立期待** | 告诉学员接下来讲什么、为什么讲这个（与学员当前困境的关联）、大概需要多久 |
| **预激活已有知识** | 一个探测性提问，判断学员在这个知识点上的起点。回答直接更新 Blackboard 认知状态 |
| **快速前测跳级** | 如果学员已有基础 → 从更高起点开始，不浪费时间讲已知内容 |

### 6.2 阶段 2：内容演绎

**Leo 不照着教学脚本念**，而是基于教学内容做个性化演绎。

#### 教学内容原料（通过 API 拉取）

| 组件类型 | 用途 |
|---------|------|
| `ai-script` | 教学脚本——知识的骨架 |
| `case-script` | 案例对话脚本——情境化的知识载体 |
| `tool-template` | 工具模板——可操作的知识输出物 |
| `interaction-design` | 互动点设计——教学过程中的参与节点 |

#### 演绎规则

| 规则 | 说明 |
|------|------|
| **不复读脚本** | 教学内容是骨架，Leo 根据学员画像（Dreyfus 层级、行业背景、学习风格）重新组织语言 |
| **经验调用双路径** | 老手（L3+）：从经验切入再给框架（"你之前遇到过……其实背后的原理是……"）；新手（L1-L2）：先给框架再举例 |
| **认知负荷实时监控** | 每讲完一个知识块评估三向量 → 决定继续 / 降速 / 停下巩固 |
| **Cowan 4 chunks 分块** | 每次最多 3-4 个新概念。多了就停下验证——不是学员笨，是工作记忆有上限 |

#### 内容片段提取策略

| 阶段 | 方案 | 说明 |
|------|------|------|
| MVP（当前） | **方案 B：LLM 定位** | LLM 从完整 `ai-script` 中定位和提取与当前知识缺口相关的片段 |
| 后续优化 | **方案 A：知识点索引** | 内容源增加知识点级别标签和索引，支持精确检索 |

选择先 B 后 A 的理由：LLM 定位在内容量不大时足够好，后续可加索引优化。

### 6.3 阶段 3：验证

见第七章「验证策略选择器」。

每个知识块讲完后**立即验证**，形成"教-验"小循环。不是讲完所有内容再统一考试。

### 6.4 阶段 4：收尾

#### 学习确认（~10 秒）

让学员建立"我学到了"的自我认知——不是再验一次，而是元认知确认。

示例提问："刚才过的四种风格，你觉得对你当前带团队最有用的是哪个？"

学员的回答写入 Blackboard，教练态接手时知道学员关注点在哪。

#### 实践桥接

| 情况 | 处理 |
|------|------|
| 有即将发生的管理场景（从中期存储/行动项读取） | 直接关联："下周你要和小王做委派谈话，刚学的 S2 风格正好能用" |
| 无具体场景 | 低门槛实践建议："这周找一个机会，试试用提问代替直接指示" |

输出物：

- 脚手架（一页纸 / 议程卡片 / 要点提示）→ 写入实践状态
- 回流时间锚点 → 设定在实践预计发生时间之后（如"下周三谈完之后回来聊聊"）

#### 状态交接

LecturerPersona 往 Blackboard 写入教学结果（Handoff 状态包），CoachPersona 读取后自然衔接——学员感知到的是"Leo 接着跟我聊"。

### 6.5 Design 细化：四阶段输入/输出契约

#### 阶段 1：锚定

| | 内容 |
|---|------|
| 输入 | Handoff triggerContext + emotionSnapshot + cognitiveState |
| 输出 | Blackboard `teachingState.stage=ANCHORING` + knowledgeBlocks（从教学内容提取） |
| 退出 | 学员回应预激活提问 → 进入演绎 |
| 跳过 | `skipAnchoring=true`（断点续教）→ 直接演绎 |

规则：从 Handoff 摘要出发，用学员话题开场；预激活只问一个；答偏了从最基础块开始（不算失败）。

**【Stress-test B1 修正】专家声称"我全都会"的处理策略**：不测试、不质疑——直接进对练/模拟场景。结果自然分层：
- 处理得好 → 确认掌握，不加学习负担 + **触发知识萃取**（InsightCapture 采集专家经验，推给 N+1 和 TD/LD）
- 处理还成 → 学员自然反思（情景导入+冲突导入的综合效果），这本身就是最好的教学
- 处理一塌糊涂 → 学员自己服了，后续教学阻力为零

这是面对有经验学员（Dreyfus L3+）的核心策略：**用场景说话，不用嘴说话**。

#### 阶段 2：内容演绎

| 学员层级 | 演绎策略 |
|---------|---------|
| L1-L2 | 先给框架再举例，多用类比 |
| L3 | 框架+情境，引入"你之前可能遇到过…" |
| L4-L5 | 从经验切入反推原理，挑战性问题引导 |

Cowan 4 chunks 控制：`chunksInFlight >= 4` → 强制进入验证。验证通过后归零。

**【外审修正】** 简化模式按负荷类型具象化（同 §四）。

#### 阶段 3：验证

详见 §七 Design 细化。

#### 阶段 4：收尾

- 元认知提问只问一个（"最有用的是哪个？"），学员敷衍不追问 → 直接进实践桥接
- 有即将发生的管理场景 → 直接关联；无场景 → 低门槛建议
- 脚手架必须具体可用（不是"注意运用"，是"面谈时先问这三个问题"）
- 回流锚点设在实践后 1-2 天（给消化空间）

#### 中断处理

所有阶段的中断（硬停/学员跳出/超纲）都写断点到中期存储。下次恢复：加载断点 → `skipAnchoring=true` → 从断点继续。

## 七、验证策略选择器

### 7.1 八种验证方法

| 验证方式 | 验什么 | Bloom 层级 | 适合谁 | 信号强度 | 时间成本 |
|---------|--------|-----------|--------|---------|---------|
| **快速回忆** | 记住了没 | 记忆 | 新手 | 弱 | 极低 |
| **概念辨析** | 理解了没（能区分相似概念） | 理解 | 新手→熟手 | 中 | 低 |
| **费曼讲解** | 真懂还是假懂（能用自己的话讲清楚） | 理解→应用 | 全阶段 | 强 | 中 |
| **情境判断** | 能不能用（给场景，判断该怎么做） | 应用 | 熟手 | 中 | 中 |
| **案例分析** | 能不能拆（分析案例中的问题和原因） | 分析 | 熟手→专家 | 强 | 高 |
| **角色扮演** | 能不能做（模拟真实场景执行） | 应用→创造 | 全阶段 | 极强 | 高 |
| **迁移测试** | 迁移能力（在新场景中应用已学知识） | 分析→评价 | 专家 | 极强 | 高 |
| **教别人** | 深度内化（能教会另一个"人"） | 评价→创造 | 专家 | 极强 | 极高 |

### 7.2 自动选择策略

**验证策略 = f(知识类型, 学员层级, 认知负荷状态)**

三个维度交叉决定用哪种验证方法：

#### 维度 1：知识类型（从教学内容标签获取）

| 知识类型 | 偏好验证方式 | 理由 |
|---------|------------|------|
| 概念性知识 | 费曼讲解 / 概念辨析 | 概念需要"用自己的话说出来"才算理解 |
| 程序性知识 | 情境判断 / 角色扮演 | 程序需要在场景中执行才算会 |
| 元认知知识 | 案例分析 / 迁移测试 | 元认知需要跨场景迁移才算掌握 |

#### 维度 2：学员 Dreyfus 层级（从长期画像获取）

| Dreyfus 层级 | 偏好验证方式 | 理由 |
|-------------|------------|------|
| 新手（L1-L2） | 快速回忆 + 费曼讲解 | 先确认记住了，再确认理解了 |
| 熟手（L3） | 情境判断 + 角色扮演 | 已有知识基础，需要在场景中检验 |
| 专家（L4-L5） | 迁移测试 + 教别人 | 知识已内化，需要检验迁移和深度 |

#### 维度 3：认知负荷状态（从 Blackboard 实时获取）

| 负荷状态 | 验证策略调整 | 理由 |
|---------|------------|------|
| 负荷高 | 降级 → 轻量验证（快速回忆 / 概念辨析） | 学员已经累了，不要再加重 |
| 负荷适中 | 正常 → 标准验证（费曼 / 情境判断） | 常规状态 |
| 心流区（germane 高 + extraneous 低） | 升级 → 挑战级验证（角色扮演 / 迁移测试） | ZPD 黄金窗口，正是推进的好时机 |

### 7.3 验证结果处理

| 结果 | 动作 | 后续 |
|------|------|------|
| **通过** | 写入 Blackboard（验证通过 + 掌握度评分）→ 沉淀中期 | 进入收尾或下一个知识块 |
| **部分通过** | 针对性补充（不重讲全部）→ 换一种方式再验一次 | 补充后通过 → 正常流转 |
| **未通过** | 判断原因：概念没理解（退回演绎阶段）vs 应用不会（加脚手架重试） | 区分"不懂"和"不会用" |
| **未通过且同策略已试过** | 换验证策略（同一策略最多用 1 次）→ 两种策略都失败 → 记录中期，下次回来 | 不死磕，给学员消化时间 |

### 7.4 教-验嵌套结构

```
锚定
  → [演绎知识块 1 → 验证 1]
    → [演绎知识块 2 → 验证 2]
      → [演绎知识块 3 → 验证 3]
        → ...
          → 收尾
```

每个"教-验"小循环是独立的——前一个通过才进入下一个。

#### 退出条件

| 条件 | 处理 |
|------|------|
| 所有知识块验证通过 | 正常收尾（§6.4） |
| 认知负荷硬停（composite ≥ 0.75） | 保存断点到中期存储，下次从断点继续 |
| 学员主动跳出 | 尊重自主权，记录已完成/未完成进度 |
| 连续两个知识块验证失败 | 暂停教学，记录困难点 → 建议学员先消化，下次继续 |

### 7.5 Design 细化：决策树与行为证据模型

#### 选择决策树

```
Step 1: 排除已用策略（同块同策略限 1 次）
Step 2: 认知负荷门控
  ├─ 硬停区（ext≥0.6 或 int≥0.7）→ 不验证，写断点退出
  ├─ 高负荷 → 强制降级（quick_recall / concept_distinction）→ Step 4
  ├─ 心流区 → 强制升级（role_play / transfer_test / teach_others）→ Step 3
  └─ 常规区 → Step 3
Step 3: 知识类型 × Dreyfus 交叉选择（每格首选+备选）
Step 4: 尊严覆盖层（dreyfus≥4 且被降级 → 重框话术）
Step 5: 输出 { strategy, prompt, isDowngraded }
```

知识类型 × Dreyfus 交叉矩阵：

| | L1-L2 | L3 | L4-L5 |
|---|-------|-----|-------|
| 概念性 | quick_recall → feynman | feynman → concept_dist | teach_others → transfer |
| 程序性 | concept_dist → scenario | scenario → role_play | role_play → transfer |
| 元认知 | feynman → concept_dist | case_analysis → scenario | transfer → teach_others |

#### 验证 = 行为证据采集

验证的核心产出是 **EvidenceEntry**（行为观察记录），不是分数：

| 字段 | 说明 |
|------|------|
| `behaviorObserved.knows` | 能说出概念/原理 |
| `behaviorObserved.canUse` | 在场景中尝试应用 |
| `behaviorObserved.usedCorrectly` | 行为基本正确 |
| `evidenceSummary` | 1-2 句行为描述（观察记录，非评判） |
| `bloomLevel` | 验证触及的 Bloom 层级 |
| `confidenceInEvidence` | 证据可靠度 |
| `targetDashboard` | 能力画像 dashboard（供人观察） |

判定逻辑：
- 知道 + 能用 + 用对 → **pass**
- 知道 + （能用 或 用对）→ **pass**（行为证据足够就放行）
- 知道但不会用 → **partial**（给场景让学员试）
- 结构性不理解 → **fail**

**关键转变**：不是"答对了没"，而是"我们拿到了足够的行为 evidence 让人去观察"。

**【Stress-test S5 → B3 修正】学员挑战教学内容的处理策略**：不是"记录并上报"——是**请教**。Agent 切入 BEI（行为事件访谈）模式，"请教"学员的实战经验。四层收益：
1. **学员被重视**：有"赢了一把"的爽感，信任度飙升
2. **发现好经验**：系统真正捕获到组织内的实战知识，EvidenceEntry 标记 `type: expert_challenge`，推送 N+1 和 TD/LD
3. **补充组织 context**：基于业务场景的 know-how 比教科书更有价值
4. **万一学员错了？** BEI 过程中自然暴露——回到 B1 路径（用场景说话）

这不是"验证失败的异常处理"——这是 Leo 面对经验丰富学员时的**核心交互模式**。

#### mastery 评分（服务于贝叶斯更新）

按验证策略的 Bloom 层级加权：quick_recall=0.3, concept_distinction=0.45, feynman=0.6, scenario_judgment=0.65, case_analysis=0.75, role_play=0.8, transfer_test=0.9, teach_others=0.95。partial 打六折。

#### 尊严覆盖层

**【外审修正】** dreyfus≥4 被降级时的话术重框：
- quick_recall → "帮我梳理几个关键点的逻辑关系"
- concept_distinction → "从你的实战经验看，最关键的区别是什么"
- 其他轻量策略 → "作为过来人，你会怎么跟新人解释这个"

| D-9 | 认知负荷融合权重 | A: 硬0.6:软0.4 / B: 硬0.4:软0.6 / C: 动态（MVP 硬0.3:软0.7） | **C（简化版）** | MVP 硬信号不成熟，软信号主导；手动分阶段步进 |

| D-11 | Handoff 内容形态 | A: 搬运原始对话片段 / B: 压缩摘要 + 索引定位 | **B** | 记忆系统是图书馆，Handoff 是借书卡不是复印件 |
| D-12 | 教学态期间闲聊处理 | A: 外部拦截 / B: 教学态自主判断 | **B** | 教学态有最丰富教学上下文，能判断是顺承/延伸/超纲 |
| D-13 | 人格状态沉淀路径 | A: 跳过中期直写长期 / B: 先中期灰度 3 次再推长期 | **B** | 直写有过拟合和灾难性遗忘风险 |
| D-14 | Dreyfus 降级方式 | A: 纯公式 / B: 纯 LLM / C: 规则初筛+仲裁执行 | **C** | 纯公式易受波动误判，纯 LLM 成本高且不可复现 |
