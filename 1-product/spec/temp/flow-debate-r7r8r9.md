## 一、flow-01-layout-auth（三栏布局框架 + 账户与权限）

### 1.1 问题清单

**P1 [交互设计师] 折叠/展开（A1.1.1）缺少键盘快捷键**
折叠操作仅支持鼠标点击图标，无键盘快捷键。高频用户（尤其HR/运营）长期使用效率低。

**P2 [魔鬼代言人] 宽度拖拽比例重置策略过于激进（A2.2.3）**
用户手动调整比例后切换场景即丢弃。若学员在大厅精心调整了适合自己屏幕的比例，每次从教室返回都要重新调。文档明确写"丢弃手动调整"——对高频切换用户体验差。

**P3 [学员] 移动端提示过于粗暴（A4.2）**
窗口<1024px直接全屏遮罩"请使用电脑访问"并阻断所有操作。如果学员在平板（如iPad 1024x768横屏刚好达标，竖屏不达标）上使用，旋转屏幕就会被遮挡且无法继续。缺少对平板横屏的友好引导。

**P4 [HR] HR体验课程时数据隔离规则（B3.3）有二义性**
"HR账号在学员列表中→数据计入统计"。但HR体验课程是为了了解课程，其学习数据不应污染学员统计。即使HR在学员列表中，也应区分"正式学习"和"体验学习"。

**P5 [课程设计师] 双向联动中Agent自动解读（A3.1.1）缺少解读的上下文范围限制**
学员点击看板数据点后Agent生成解读消息，但未定义解读范围。Agent可能对一个进度数据生成过长的分析，打断当前辅导对话节奏。

**P6 [魔鬼代言人] 断点恢复弹窗（B5.2.2-B5.2.4）对测评教室没有确认弹窗**
授课和对练教室断点恢复都有弹窗让用户选择，但测评教室（B5.2.4）"直接加载测评教室从断点继续"。测评中途退出可能是因为网络异常或时间不够——直接恢复可能让学员感到被迫继续测评，缺少"回到大厅"选项。

**P7 [交互设计师] 对话→看板联动的实体高亮（A3.2.1）持续5秒后淡出——多轮对话中多个高亮如何处理未定义**
如果Agent连续两条消息分别引用不同实体，第一个还在高亮时第二个也触发，视觉冲突规则不明确。

**P8 [学员] 忘记密码（B1.5）只提示"联系运营"**
无自助找回密码机制。如果运营不在线（深夜/周末），学员完全无法登录。

### 1.2 修改建议

| 编号 | 建议 |
|------|------|
| P1 | 补充键盘快捷键（如 Ctrl+B / Cmd+B 折叠目录区），在A1.1.1正常流程末尾增加"支持快捷键触发"。 |
| P2 | 改为"场景内宽度调整保存至session级别"：同一次登录session内，回到已调整过的场景时恢复用户的手动比例；仅跨session时重置为默认。 |
| P3 | 补充异常流程：窗口宽度在960-1024px时显示"建议横屏使用"提示（非阻断）；<960px才全屏阻断。 |
| P4 | 明确规则：HR体验课程的数据一律标记 `is_experience: true`，无论HR是否在学员列表中。HR作为学员正式学习时需通过学员身份登录（非"体验课程"入口），数据才计入统计。 |
| P5 | 在A3.1.1正常流程步骤3补充：Agent解读消息限制在2-3句话以内（约50-80字），不做深度分析；如学员继续追问则展开。 |
| P6 | B5.2.4补充断点恢复弹窗，与授课/对练保持一致：提供"继续测评"和"回到大厅"两个选项。量表测评的"回到大厅"选项附带提示"未完成的测评下次可继续"。 |
| P7 | 补充规则：新高亮触发时，之前未淡出的高亮立即淡出（即同一时间最多1个高亮生效）。或改为：多个高亮可并存，各自独立计时5秒后淡出。选择一种并写入文档。 |
| P8 | 补充"短信/邮箱验证码重置密码"的自助流程作为本期或下期规划。本期若不做，至少提供运营联系方式的多种渠道（电话+企业微信+邮箱）。 |

## 二、flow-02-coaching（辅导模块 — 大厅）

### 2.1 问题清单

**P9 [学员] 首次引导（A1）缺少学员已知信息的交代**
教练自我介绍了"我是谁、能做什么、任务列表"，但没有告诉学员"这个培训项目的总体目标是什么、大概多久完成"。学员首次进入缺少全局预期。

**P10 [魔鬼代言人] 非首次登录的欢迎消息（A2.1.1）每次都发——可能产生审美疲劳**
学员如果一天登录3次，每次都收到"欢迎回来+推荐任务"消息。文档未定义频率控制。高频登录用户会觉得啰嗦。

**P11 [课程设计师] 日常辅导对话的知识范围（A3.1.3）与授课小解答（flow-03 A4.1）的边界在学员视角不清晰**
辅导教练"读取整个项目知识范围"，授课AI"限于当前Activity"。学员可能困惑：为什么在教室里问不到的问题，回大厅就能问到？没有在流程中写明如何向学员解释这个边界差异。

**P12 [交互设计师] 消息嵌入交互元素（A4.2）中"确认按钮"的过期处理未定义**
教练消息中嵌入的"是否进入教室？[是][否]"按钮，如果学员不点击继续对话，这些按钮是否失效？向上滚动到历史消息时按钮是否仍可点击？

**P13 [HR] 辅导模块完全没有HR视角的流程**
flow-02标题是"辅导模块（大厅）"，但全部流程只写了学员端。HR端大厅的Agent交互流程（如HR向Agent提问项目进度、某学员情况）完全缺失。

**P14 [魔鬼代言人] 工作实践反馈存储（A3.3.2）中structured_tags的提取准确性无兜底**
"后台自动提取结构化信息"——如果AI提取的标签不准确（如把"积极反馈"标记为"消极反馈"），后续报告引用时会产生误导。文档只有"存储失败→静默重试"的异常，没有"提取不准确"的质量保障。

**P15 [学员] 左侧目录（BL1.1）中Activity的前置条件锁定逻辑缺失**
BL1.2提到"Activity不可用（前置条件未满足）→锁定状态"，但前置条件是什么未定义。是线性顺序锁（必须按顺序学）还是自由选择？如果锁定，锁定规则来自哪里？

**P16 [交互设计师] 看板组件的布局顺序（BM1-BM7）是否可自定义未说明**
大厅看板有进度、课程推荐、消息中心、笔记资料、工具卡片、待办任务多个区域，上下排列顺序是固定的还是可调的？不同学员可能关注点不同。

**P17 [魔鬼代言人] 工具卡片分享（BM5.4）的权限控制缺失**
分享链接生成后，非学员是否可以打开？如果链接可公开访问，可能涉及课程内容知识产权泄露。

**P18 [学员] 催学消息（BM3.3）的退订/设置机制缺失**
学员无法控制催学消息的频率或关闭催学。虽然内容是价值导向的，但频繁推送仍可能产生抵触。

### 2.2 修改建议

| 编号 | 建议 |
|------|------|
| P9 | A1.1.1开场消息中增加：教练简要介绍项目概况（"这次培训共有X个课程，预计需要Y小时完成"），让学员建立全局预期。 |
| P10 | A2.1.1增加频率控制规则：同一天内第2次及以后登录，简化为"欢迎回来"不附带任务推荐；或仅在距上次登录>4小时时才发送完整欢迎消息。 |
| P11 | 在授课小解答（flow-03 A4.1）的异常流程中，增加AI提示语模板：当学员问到超出Activity范围的问题时，AI除了说"可以回大厅问辅导教练"外，简要解释原因："在这里我的知识范围限于当前课程，辅导教练了解你学过的所有内容，能给你更全面的建议。" |
| P12 | 补充规则：嵌入式交互按钮有效期为当前对话轮次。一旦学员发送新消息（无论是否点击按钮），历史消息中的按钮变为disabled状态（灰色不可点击），保留文字但不可交互。 |
| P13 | 新增B-HR章节，描述HR端大厅Agent的基础交互流程（HR提问学员进度→Agent汇总回答；HR查看报告→Agent解读数据），或在文档开头明确标注"本文档仅覆盖学员端辅导流程，HR端Agent流程在flow-06-management中描述"并确认flow-06确实覆盖。 |
| P14 | A3.3.2增加质量校验步骤：structured_tags提取完成后，对关键字段（如emotional_state、specific_challenge）做置信度评分；低于阈值的标签标记为"待人工确认"，不直接用于报告。 |
| P15 | BL1.2增加说明：前置条件由AOM配置（如"Activity B 要求 Activity A 状态为已完成"）。本期默认线性顺序解锁。在目录区锁定节点上hover显示"请先完成[前置Activity名称]"。 |
| P16 | 补充说明：本期看板组件顺序固定（进度→课程推荐→待办→笔记→工具→消息），后续版本考虑自定义排序。 |
| P17 | BM5.4补充：分享链接需要登录验证才能查看内容。非系统用户打开链接时显示登录页面。或改为"分享链接仅在项目成员间有效"。 |
| P18 | 补充：学员可在设置中调整催学消息频率（每天最多/每周最多）或关闭催学。本期若不做设置页面，至少在消息列表底部提供"减少此类提醒"按钮。 |

## 三、flow-03-teaching（授课模块 — 教室一）

### 3.1 问题清单

**P19 [课程设计师] SCO调度引擎（A0.1）异常2"某个SCO加载失败→跳过"过于冒险**
直接跳过一个SCO可能导致后续SCO的前置知识缺失。尤其FEEDBACK_COLLECT被跳过后，FEEDBACK_REVIEW将无数据可用。跳过策略需要区分SCO类型。

**P20 [学员] 视频播放（A2.1）"自动开始播放"可能在公共场合造成尴尬**
学员可能在办公室使用系统，视频自动播放声音会引起注意。缺少静音自动播放或"点击播放"选项。

**P21 [交互设计师] Quiz问答题（A3.1）"最多5轮"的退出条件不明确**
苏格拉底式追问"最多5轮后AI给出总结"——但如果学员第2轮就回答得很好呢？只写了最大值，没写AI如何判断"够了"的条件。

**P22 [魔鬼代言人] 闭环反馈课前采集（A5.1）的"足够信息"判断标准缺失**
A5.1步骤5"AI判断已采集到足够信息"——什么叫足够？是至少1个场景？至少包含人物+事件+情绪？没有明确的退出条件，AI可能过早结束或过度追问。

**P23 [学员] 插嘴提问（A7.1）在视频播放中的体验未定义**
A7.1.1说"在AI讲解/Quiz/视频播放中学员可打断提问"，但视频播放时右侧对话区的AI不在讲解状态，学员输入问题后AI是否暂停视频？视频声音和AI回复的文字同时进行是否冲突？

**P24 [课程设计师] KGP未设计互动时的fallback（A1.2.4）标记了[待确认]但缺少临时方案**
这个待确认项会直接影响AI的行为。如果没有决策，开发时无法实现。需要给出临时默认行为。

**P25 [魔鬼代言人] 自适应L1（B1）的"语言风格匹配"可能导致AI不专业**
如果学员用非常口语化的表达（甚至不礼貌的语言），AI也匹配其风格，可能损害产品的专业形象。缺少风格匹配的底线约束。

**P26 [交互设计师] PPT讲解（A1.1.1）中看板区只显示一张PPT图片，缺少翻页控制**
NARRATION SCO加载一张PPT，但文档未描述：谁控制翻页？是AI讲完一页自动翻？还是学员可以手动翻？翻页的触发机制完全缺失。

**P27 [HR] Activity结束收尾（H2）缺少学习数据上报/同步给HR的时机**
H2汇总了学习数据，但没有说明数据何时对HR可见。HR端看到的数据是实时的还是有延迟？

**P28 [学员] 重新学习（G1-G2）缺少学员知情确认**
"重新开始"会新建学习记录，但学员可能不清楚原始记录是否保留。缺少确认提示。

**P29 [魔鬼代言人] FEEDBACK_REVIEW课前数据为空（A5.5）标记了[待确认]**
这是一个关键的降级路径。如果学员拒绝分享后REVIEW环节完全没有方案，整个闭环反馈功能断裂。

**P30 [课程设计师] 打标SCO置信度阈值（A0.3异常2、B2.5）硬编码为0.6**
0.6是否合理需要测试验证。文档应标注这是可配置参数，而非固定值。

### 3.2 修改建议

| 编号 | 建议 |
|------|------|
| P19 | A0.1异常2改为分类处理：关键SCO（FEEDBACK_COLLECT、QUIZ）加载失败→重试3次，仍失败→提示学员"该环节暂时无法加载"并提供"跳过"/"重试"/"返回大厅"选项；非关键SCO（NARRATION、VIDEO）可跳过但记录日志。 |
| P20 | A2.1步骤3改为：视频默认静音自动播放（显示"点击开启声音"的浮层提示），或改为不自动播放（显示大播放按钮，学员点击后开始）。建议后者。 |
| P21 | A3.1补充AI判断逻辑：当学员回答已覆盖该知识点的核心要素（由Quiz SCO脚本定义的关键点列表），AI可提前结束追问并给出总结。"最多5轮"是上限保护，非目标轮数。 |
| P22 | A5.1步骤5补充退出条件：AI至少采集到1个具体场景（包含人物+事件+学员的处理方式），或学员明确拒绝后按A5.4处理。采集轮次上限为5轮对话。 |
| P23 | 补充A7.1在视频场景的行为：学员在视频播放中输入问题→系统自动暂停视频→AI回答问题→回答完毕后提示"视频已暂停，点击继续播放"或自动恢复播放。 |
| P24 | A1.2.4临时默认行为：KGP未设计互动时，AI按blueprint纯讲解，不自主插入互动。后续版本若决定允许AI自主互动，再调整。 |
| P25 | B1.1增加约束：AI风格匹配有底线——始终保持专业、友善的基调。可以口语化但不可使用不当用语、不可过度随意。在System Prompt中增加风格底线指令。 |
| P26 | A1.1.1补充翻页机制：NARRATION是单页SCO（一个NARRATION SCO = 一张PPT页），翻页由调度引擎推进下一个SCO实现。在当前SCO讲解完成前，学员不可手动翻页（但可通过左侧缩略图查看已完成的页面）。将此规则写入A0.1的补充说明。 |
| P27 | H2补充：学习数据汇总后异步同步到HR端数据看板，延迟不超过5分钟。HR端数据展示刷新机制由flow-06定义。 |
| P28 | G1-G2步骤1后增加确认弹窗："重新学习将从头开始，你之前的学习记录会完整保留。确定重新开始吗？[确定] [取消]" |
| P29 | A5.5确定默认方案：REVIEW时课前数据为空→使用"方案C：引导学员现场分享一个简单场景"作为默认降级行为。AI说："我们之前没来得及聊你的具体情况。不过没关系，现在想想你最近在[主题]方面有什么具体的经历？哪怕一个简单的场景也行。"采集1轮后进入REVIEW正常流程。 |
| P30 | A0.3和B2.5中将0.6标注为"可配置参数 confidence_threshold，默认0.6"，写入系统配置表，可按项目或按维度调整。 |

## 四、跨模块衔接问题

### 4.1 衔接问题清单

**X1 大厅→教室切换时辅导对话的保存/恢复机制三处定义不完全一致**
- flow-01 A2.1.1步骤2说"辅导对话保存到后台"
- flow-01 A2.1.2步骤3说"恢复辅导教练Agent上下文（含教室中的学习记忆）"
- flow-02 A4.3说"之前的辅导对话消息全部保留可见"
三处都在说同一件事，但描述粒度不同。建议在flow-01 A2.1中统一写清保存内容（完整对话历史+Agent记忆状态），flow-02 A4.3直接引用flow-01定义。

**X2 flow-02辅导模块C节（教室进出与导航）与flow-01 A2节（场景切换）高度重复**
flow-02的C1-C6几乎全部引用flow-01的A2.1.x流程，但又各自写了一段简化版。这种重复容易导致两处定义不一致。建议flow-02的C节只保留辅导视角的补充说明（如教练消息行为），场景切换的技术流程统一指向flow-01。

**X3 flow-03工具卡片同步到大厅（D3）引用了flow-02的BM5，但BM5中没有说明"来源为教室"的工具卡片的特殊标识**
教室中产生的工具卡片（含预制和AI动态生成）到大厅后是否标注来源？学员能否区分哪些是课程中获得的、哪些是AI生成的？

**X4 flow-03的Activity结束收尾（H2）触发"衔接卡片"引用flow-02 C3.1，但衔接卡片的UI是在flow-02中定义的**
逻辑上衔接卡片是教室内行为（属于授课模块范畴），却在辅导模块中定义。模块归属不清晰。建议将衔接卡片UI定义移到flow-03的H2中，flow-02只描述"学员通过衔接卡片回到大厅后教练的行为"。

**X5 三个文档对"Agent切换"的描述散落在多处**
- flow-01 A1.3.1：加载Agent类型
- flow-01 A2.1.1步骤3：加载目标教室的Agent上下文
- flow-02 A1.1.1：辅导教练Context包加载
- flow-03 H1步骤2：重置AI上下文
缺少一个统一的"Agent上下文切换协议"说明：切换时保留什么、丢弃什么、新加载什么。建议在flow-01 A2.1中增加一个Agent切换子流程，其他模块引用。

**X6 flow-02的记忆引用规则（A3.4）与flow-03的引用槽机制（A1.3）是两套独立的引用系统，但共享同一个Database**
辅导教练引用记忆用"System Prompt指令控制"，授课AI引用记忆用"referenceSlots配置"。两套机制读取同一个Database但规则不同。需要确认：同一条记忆在辅导和授课中被引用时，引用格式是否一致？是否有并发读写冲突？

**X7 断点恢复（flow-01 B5）的弹窗出现在"登录后"，但flow-02的非首次进入欢迎消息（A2.1.1）也出现在"进入大厅时"——两者的时序冲突**
如果学员上次在教室中途退出，登录后先弹断点恢复弹窗，学员选择"回到大厅"，此时flow-02的A2.1.1是否触发？还是弹窗覆盖了欢迎消息？需明确时序：断点弹窗→用户选择→若选大厅→触发A2.1.1。

## 五、整体评价

三份文档的结构规范、覆盖面广，正常流程描述详尽，尤其是综合流程的端到端示例对开发和测试非常有价值。主要问题集中在：

1. **异常/边界场景覆盖不足**（P6/P19/P22/P23/P29）——关键降级路径需要明确默认方案
2. **跨模块定义重复与不一致**（X1/X2/X5）——需要建立引用规范避免分歧
3. **待确认项需尽快关闭**（A1.2.4/A5.5）——这些直接影响开发实现
4. **HR视角缺失**（P13/P27）——辅导模块和授课模块几乎只有学员流程

建议优先处理P19、P22、P26、P29、X5、X7这6项，它们直接影响核心流程的可实现性。
