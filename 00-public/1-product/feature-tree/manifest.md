### 字段说明

| 列 | 含义 | 用途 |
|---|------|------|
| **#** | 模块编号 01–10 | 与 `feature-tree/NN-xxx.md` 子文件一一对应，给跨引用（如 `02.B5`）一个稳定前缀 |
| **模块** | 模块业务名 | 一眼识别场域/子系统，与 §2 骨架、§6 依赖矩阵标题保持一致 |
| **Owner** | 责任人（多人并列时首位为主 Owner） | 决策路由与对齐入口：找谁拍板、找谁审阅 |
| **L1** | 一级节点数 | 模块**广度**指标，判断骨架是否过粗/过细 |
| **L2** | 二级节点数 | 模块**深度/体量**指标，用于工作量估算、Sprint 分配、进度百分比 |
| **★变更** | 本版相比上版的结构性新增/修改数 | 变更**热度**指标，★ 多的模块审阅时优先扫（明细见 §3） |
| **待确认** | 尚未拍板的开放问题数 | 模块**阻塞/风险**指标，判断是否 ready 进入开发（明细见 §4） |
| **ADR** | 关联架构决策记录编号 | 决策**依据**指针，改动前必须回看的架构约束（明细见 §5） |

> **合计行**给出全产品总盘面：L1 / L2 / ★变更 / 待确认 的总和 + ADR 总数，一行回答"盘子多大 / 动了多少 / 还有多少未决 / 背后几条架构约束"。

## 8. 横切定义映射表 ★

> 跨多个模块共享的核心定义。**每个概念有单一权威归属（主定义）**，其他模块只引用不重复。修改主定义时必须同步通知所有消费方。

| 横切概念 | 采集侧 | 主定义（权威归属） | 消费侧 |

| **学员画像** | Inquiry F2（BEI/画像/困惑/高光） | **09-Eval B1** 学习特质碎片 | Leo B4 · Neo A1 · Hall E9（个人画像面板） |
| **memory_id** | Lecture C5 FEEDBACK_COLLECT 生成 | **07-Infra N3** | Lecture C6 · Leo B4 · Report H |

### 修改协议

当主定义发生变化（schema 改、口径改、字段加减）：

1. 主定义方发起变更提案
2. 同步通知所有消费方 Owner
3. 消费方评估影响范围
4. 达成一致后同步更新各模块的"→ 见 XXX"引用
5. Manifest §9 表格同步更新

### 消费方引用约定

各模块 L2 在涉及横切概念处，用 `→ 见 [主定义位置]` 标注。例如：
- Hall E1 "累计学习时长" → 见 Eval B7
- Leo B4 "记忆引用" → 见 Infra N3 / Eval B1
