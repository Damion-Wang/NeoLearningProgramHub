## 用途

在产品/技术/业务决策点上，用主持人 + 挑战者 + 主题角色 进行结构化辩论，强制暴露各方立场冲突，让用户在充分信息下拍板。

适用场景：
- 多个方案选其一（如：导出格式 PDF 还是 Markdown）
- 架构决策（如：单端口 vs 多端口）
- 信息架构 / 模块划分（如：催学规则放 config 还是单独模块）
- 主题深化讨论（如：项目生命周期 / GUI spec / Agent BDD）
- 任何"看起来都对但只能选一个"的设计岔路

## 何时触发

**显式触发**（用户明确说）：
- "做个辩论"
- "辩论一下"
- "5 方对线"
- "开 debate"
- "让 PM/Dev/UX/Student/HR 都说说"

**隐式触发**（Claude 主动建议）：
- 用户给出 2 个及以上方案，且问"选哪个"
- 用户在做关键架构决策，单一视角不够
- 用户已收集多方反馈但意见冲突
- 主题跨多议题（≥3 个不可独立的子问题）

主动建议时的措辞：
> "这个问题涉及 [产品/技术/体验/学员/管理] 多方权衡，建议走多角色辩论流程，需要我启动吗？"

## 完整流程（5 Phases）

### Phase 0 · 底层共识对齐 ⭐ v2.0 新增

**目的**：拉齐主持人与 DM 的底层假设、术语、价值观、目标。**避免 plan 在错误前提下做**。

**触发**：
- 主题复杂跨多领域
- DM 已有大量隐性假设
- 主题涉及 PM 与 DM 价值观

**操作**：
1. 主持人列 10-20 个 "底层共识题"
2. 用 AskUserQuestion 工具批量问（每批 4 题）
3. 题型：
   - **价值观对齐题**："你认为产品最关键的体验是什么"
   - **术语澄清题**："X 在你语境里是 Y 还是 Z"
   - **优先级题**："如果只能保 1 个，是 A 还是 B"
   - **边界题**："X 的范围是 Y 到 Z 吗"
   - **风险题**："最坏情况是什么"
   - **颗粒度题**："要具体 behavior 故事还是维度框架"⭐
4. DM 多轮答完后主持人总结"底层共识"
5. 如 DM 反馈与主持人假设差异大 → 重新问 + 拉齐

**产出**：`00-底层共识.md`

**结构模板**：
```markdown
# 底层共识 · v1.0
## DM 给出的关键假设
## 术语澄清
## 价值观对齐
## 优先级 ranking
## 颗粒度边界
## 主持人 takeaway（如何用于 plan）
```

### Phase 1 · 信息预备 ⭐ v2.0 新增

**目的**：把所有"必读"和"必搜"信息预先准备好，debate 中只需引用，不打断节奏。

**操作**：
1. 主持人按 debate 主题列：
   - **项目内 must-read 文件清单**（按议题分组）
     - 每文件标"为什么必读"
     - 推荐：每议题 2-3 个文件
   - **网络 must-search 主题清单**（按议题分组）
     - 每主题标"为什么必搜"
     - 推荐：每议题 1-2 个主题
2. 主持人执行：
   - Read 项目内所有 must-read 文件
   - WebSearch + WebFetch 所有 must-search 主题
   - 整合成"知识背景包"（含原文引用 + 主持人总结）
3. 角色发言时引用知识背景包

**产出**：`01-信息预备包.md`

**结构模板**：
```markdown
# 信息预备包 · v1.0
## 议题 1
### 项目内必读
- 文件 X：[原文摘录 + 关键 takeaway]
### 网络必搜
- 主题 Y：[搜索结果摘录 + 关键 takeaway]
## 议题 2
... 同上
## 主持人整合 takeaway
[给所有角色的"已知信息"基线]
```

**经验值**：
- must-read 总量控制在 20K 字以内（角色读完不疲倦）
- must-search 总量控制在 5 个主题以内
- 多了 → debate 拖慢；少了 → 角色发言空洞

### Phase 2 · debate plan（v2.0 升级）

**前置**：必须先做 Phase 0 + Phase 1。

**操作**：用 AskUserQuestion 工具问 8-12 题确认：
- 议题清单（3-5 个议题）
- 角色配置（主持 + 挑战者 + 主题角色 3-5 个）
- 三轮调度（一轮一议题 / 三议题并行 / 等）
- 共识判定权（主持人裁 / DM 拍 / 多角色同意）
- 网络搜索时机（仅轮间 / 角色发言中 / 主持人请）
- DM 互动密度（轮间题量）
- 文件存放位置
- 完成后处理（todo 入 4-decisions / spec sync / 等）

**产出**：`debate-plan.md`

**结构模板**：
```markdown
# debate-plan · v1.0
## 主题
## 角色配置
## 议题清单
## 流程机制
## 文件结构
## 各轮议题展开
## 节奏建议
## 启动门槛
## 完成后动作
```

### Phase 3 · 每轮 debate ⭐ v2.0 升级 - 4 子环节

#### 子环节 3.1 · 轮流发言（Round-Robin）

- 每个角色按规定顺序发言
- 字数限制：50-150 字
- 每个角色必须给"立场 + 一句关键依据"
- 主持人不打断，不总结
- **目的**：建立各角色基础立场，避免角色相互"附和"

#### 子环节 3.2 · 逐对讨论（One-on-One）

- 主持人识别核心冲突点（≥1 个，≤3 个）
- 每个冲突挑选最强的两方"单挑"（A vs B）
- 单挑限 2-3 回合：
  - 回合 1：A 反驳 B
  - 回合 2：B 回应 A
  - 回合 3：A 终结（可选）
- 其他角色不打扰
- **目的**：深挖核心矛盾，不让冲突被多人发言冲淡

#### 子环节 3.3 · 自由辩论挑战（Free-for-All）

- 挑战者主导：质疑全场（多 vs 1）
- 任何角色可加入支持挑战者或反对
- 限 5-10 轮发言
- **挑战者重点**：
  - 质疑 PM 角色"拍脑袋"
  - 挖隐藏假设
  - 推翻表面共识
- **目的**：暴露 PM 角色（Claude）盲点

#### 子环节 3.4 · 主持人裁定

- 综合 3.1-3.3 → 共识 / 待 DM 拍 / 仍冲突
- 共识 → 写入"阶段性结论"草稿
- 待 DM → 进入轮间 DM 互动
- 仍冲突 → 标记"待下轮再辩"

**产出**：`00-轮内自由讨论记录.md`（一字不拉记录所有发言）

**结构模板**：
```markdown
# 轮 N · 主题 · 轮内自由讨论记录
## § 0 主持人开场
## § 1 议题 1
### 1.1 子环节 3.1 · 轮流发言
[每角色 50-150 字]
### 1.2 子环节 3.2 · 逐对讨论
[冲突识别 + 单挑 2-3 回合]
### 1.3 子环节 3.3 · 自由辩论
[挑战者主导 5-10 轮]
### 1.4 子环节 3.4 · 主持人裁定
## § 2 议题 2
... 同上
## § N 主持人轮 N 总结
```

### Phase 4 · 轮间环节

- **02-阶段调研报告.md**：网络搜索补缺（如 Phase 1 不够 + 辩论中发现）。**按需**——内部素材足够时可跳过
- **03-DM 互动确认.md**：用 AskUserQuestion 工具问 3-5 题
  - DM 反馈可能推翻轮内决议（这是设计的合理环节）
  - 不锁题量，主持人按需
- **04-阶段性结论.md**：基于"自由讨论 + 调研 + DM 反馈"重写共识
- **05-DM 讨论总结.md**：DM 与 Claude 的产品决策对话备份 + PM 角色反思

## 主持人能力清单 ⭐ v2.0 升级

### 基础能力（v1.0 已有）
- 列议题
- 识别冲突
- 触发反驳
- 总结裁定共识
- 控制节奏

### 新增能力（v2.0）

#### 1. 识别"轴错误"
当多方都在 X 轴争论但实际答案在 Y 轴时主持人应识别。

**案例**（2026-04-26 教训）：
- 轮 3 议题"不给答案的渐进比例" → PM/挑战者都在 W 阶段轴争论（W1 / W6 / W12）
- DM 反馈：实际轴是**场景类型**（辅导给 / 教学不给）
- 教训：当争论都聚焦"比例"时，主持人应问 "我们是不是错了维度？"

**操作**：自由辩论 5 轮后如冲突未解，主持人主动问"我们是不是在错的轴上争？真正的轴是什么？"

#### 2. 颗粒度判断
DM 想要"具体故事"还是"维度框架"——主持人 Phase 0 应问清楚。

**案例**（2026-04-26 教训）：
- 轮 3 PM 写 15 个具体 BDD 故事
- DM 反馈："不要具体 behaviour，要 behaviour 设计维度和场景"
- 教训：在 Phase 0 加颗粒度题（"X 是想要具体 N 个例子还是维度框架？"）

#### 3. 共识 vs 假共识
5 角色都说"OK"不一定是真共识，可能是疲倦或回避。

**信号**：
- 角色发言越来越短
- 没人问问题
- 反驳变成"我同意挑战者"
- 共识达成快得反常

**操作**：怀疑假共识时，主持人主动让挑战者再质疑。

#### 4. 及时打断
自由辩论中如果某角色长时间垄断，主持人可截断。

**操作**：
- 单角色连续 3 轮发言 → 主持人打断"还有谁要说"
- 角色偏题（不在议题内）→ 主持人拉回"让我们回到 X"
- 重复观点 → 主持人总结+下一议题

## 角色配置规则 ⭐ v2.0 升级

- **主持人**：1 个，必须有
- **挑战者**：1 个，必须有（独立于其他角色）
  - 立场：反对一切、挖矛盾、问"如果不…会怎样"
  - 价值：不在"每次赢"，在"质疑暴露盲点"
- **主题相关角色**：3-5 个，按主题选
  - 产品决策：PM / UX / 学员代表 / HR 代表 / 业务专家
  - 技术决策：PM / Dev / Architect / QA / SecOps
  - 商业决策：PM / 销售 / 客户成功 / 财务 / 法务
  - 教学/AI：PM / 教学专家 / 学员 / Agent 设计师 / Memory 专家
- **总数控制 5-7 个**（含主持 + 挑战者）

**角色立场规范**：
- 立场必须"硬"（避免"也行也不行"中庸表达）
- 优先用第一人称（"我作为 PM 认为…"）
- 必须有"立场 + 依据"

## 经验教训 ⭐ v2.0 新增

来自 2026-04-26 spec 深化 debate（首次完整 v2.0 流程的部分应用）：

### 教训 1 · 底层共识缺失（→ v2.0 新增 Phase 0）
- DM 在轮 1 临时引入"服务周期 vs 项目周期"二元模型
- 这是个根本概念，**应在 Phase 0 拉齐**
- v2.0 Phase 0 必须问"在你语境里项目周期是什么 / 服务周期是什么"

### 教训 2 · 轴错误（→ v2.0 主持人能力 1）
- 轮 3 PM + 挑战者都困在"渐进比例 vs 突变"
- DM 才点出"轴是场景类型不是 W 阶段"
- v2.0 主持人在自由辩论 5 轮后必须主动问"轴是不是错了"

### 教训 3 · 颗粒度错误（→ v2.0 主持人能力 2）
- 轮 3 PM 写 15 个具体 BDD 故事
- DM 实际想"维度框架"
- v2.0 Phase 0 必须有颗粒度题

### 教训 4 · 信息搜索打断节奏（→ v2.0 新增 Phase 1）
- 轮 1+2 的 WebSearch 在轮间，轮 3 跳过
- 不一致 + 临时打断
- v2.0 Phase 1 预先准备 → debate 中只引用

### 教训 5 · 挑战者不是每次赢
- 议题 4 挑战者被业务专家反驳成功
- 这是**合理 outcome**——挑战者价值在质疑暴露盲点，不在每次赢
- v2.0 角色立场规范明确"挑战者价值不在每次赢"

### 教训 6 · 0424 决策可被推翻
- 本次推翻 2 条 0424 决策（S-12 / C-05）
- 说明 debate 有"挑战既定"功能，不是只为决定新议题
- v2.0 主持人可主动对照过往决策，挑战其合理性

## 注意事项

- **Phase 0 不可省略**（除非主题极简单，DM 已明确背景）
- **Phase 1 不可省略**（即使内部素材丰富，也要先整理 must-read 清单）
- **挑战者不可省略**（独立于其他角色，专门质疑）
- **DM 互动每轮 3-5 题**（不锁题量上限）
- **一字不拉记录**（debate 价值在过程，不只在结论）
- 反驳轮次最多 2-3 回合，避免无限循环
- 用户拍板后，决策原文写入归档文件
- 如本轮决策影响架构，应建议用户在 `00-public/4-decisions/` 新增 ADR
