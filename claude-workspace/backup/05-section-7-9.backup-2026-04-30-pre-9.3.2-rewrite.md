#### 9.3.2.1 业务定义

##### 9.3.2.1.1 一句话定位

> practice（对练）是学员在拟真业务场景中通过角色扮演**动手做**的场域 —— 由 **Actor**（剧本驱动的扮演对象）+ **导演**（后台分析与指挥，学员不可见）+ **Neo**（旁路答疑与辅导）三个 Agent 协作演绎，覆盖 **情景导入 → 角色扮演 → 反思复盘** 三个阶段，目的是把目标技能从"知道"变成"会用"，并产出可追溯的行为数据。

##### 9.3.2.1.2 三 Agent 总览

practice 是学员端 4 个场域中**唯一含三个 Agent**的场域：

| 角色 | 前台呈现 | 主要职责 | 驱动方式 |
|------|---------|---------|---------|
| **Actor**（演员）| 具体业务人物（如"张总"/"客户王经理"/"下属小李"）—— 学员可见、可对话 | 按剧本设定的 **性格 / 目标 / 触发点 / 软化点 / 底线 / 情绪天花板** 进行表演 | 剧本驱动 |
| **导演**（Director）| **不暴露给学员** —— 后台运行，学员无感 | 实时分析学员每轮发言 → 决定 Actor 的场景走向、情绪变化、下一句话 | 剧本驱动 |
| **Neo**（旁路 Coach）| Neo（与大厅 / lecture 同一身份，但本场域以 Coach 态出现）| ① 情景导入阶段：铺垫引导 ② 角色扮演阶段：主动 + 被动答疑辅导 ③ 反思复盘阶段：苏格拉底追问 + 总结 | Neo 自身判断（以剧本 + 导演指令为上下文） |

> **协作关系（一句话）**：Actor ↔ 学员 是主线对话；导演在后台指挥 Actor；Neo 在右栏旁路提供答疑与辅导，**不打断**主线对话。详细协作流程见 § 9.3.2.4.8。

#### 9.3.2.2 界面布局形态

practice 由 **三个连续阶段**（情景导入 / 角色扮演 / 反思复盘，详见 § 9.3.2.3）组成，每个阶段对应**一种独立布局**：

| 阶段 | 布局形态 | 与统一布局关系 |
|------|---------|--------------|
| **① 情景导入** | **两栏**（课件区 + Neo Chat） + 左侧覆盖式抽屉 | 沿用 lecture 风格 |
| **② 角色扮演** | **三栏**（Actor 信息区 + Actor 对话区 + Neo Chat） + 左侧覆盖式抽屉 | ★ 例外布局（practice 独有） |
| **③ 反思复盘** | **两栏**（课件区 + Neo Chat） + 左侧覆盖式抽屉 | 沿用 lecture 风格 |

> **关于"三个阶段"的表述说明**：本节"三个阶段"是产品视角的概念抽象，**对应 AOM 脚本层面的演绎编排单元**。在技术方案确定前，产品 spec 用"阶段"统一描述；**具体是否在研发实现层落地为"SCO"数据结构，由研发根据系统需求与开发进度把控**——产品 spec 不强约束实现路径。
>
> **与 § 4.2 学员端"4 场域统一布局"的关系**：情景导入 / 反思复盘沿用统一布局（左侧覆盖式抽屉 + 课件区 + 右侧 Neo Chat）；角色扮演阶段是 **唯一三栏例外**——为容纳 Actor 主线对话与 Neo 旁路辅导**同时可见**。
>
> **左侧抽屉**（三阶段共用）：沿用 lecture 抽屉设计（详见 § 9.3.1.2.2），含两 Tab —— Tab1 当前 Activity 阶段进度（情景导入 / 角色扮演 / 反思复盘 / 自由探讨）/ Tab2 当前 Course 的 Activity 列表，可切换其他已解锁 Activity（弹窗确认 + 触发断点续播保存）。
>
> **顶部 subbar**（三阶段共用）：呈现 **「开始对练」/「重新对练」按钮**——
> - **情景导入阶段**：按钮文案为「开始对练」，点击进入角色扮演阶段
> - **角色扮演 / 反思复盘 / 自由探讨阶段**：按钮文案变为「重新对练」，点击重新进入角色扮演（详见 § 9.3.2.7.3 重练机制）

**核心要点**：

- **左侧覆盖式抽屉**：与情景导入 / 反思复盘共用同一组件（Tab1 阶段进度 / Tab2 Activity 列表）
- **顶部 Subbar**：右侧呈现 **「重新对练」按钮** —— 学员任意时刻点击可重新进入对练（详见 § 9.3.2.7.3 重练机制）
- **左栏 · Actor 信息区**（拆分为上下两个独立窗口）：
  - **上窗口（Actor 身份 + 实时状态）**：Actor 头像 / Actor 名 + 身份 / 当前情绪标签 / **情绪曲线**（T1-T5 折线 + emoji 节点 + 色带，详见 § 9.3.2.3.3）
  - **下窗口（Actor 背景资料）**：背景 / 性格 / 方法工具 等可折叠展开区域 —— 学员需要时展开查看，不需要时折叠不占视觉
  - 两窗口在视觉上明确分隔（如分隔线 / 间距 / 背景色差异，具体由 GUI 设计阶段确定）
- **中栏 · Actor 对话区**：承载 Actor ↔ 学员 多轮对话主线 —— 顶部状态条（"对练进行中 / 第 N 轮"），主体为对话流，每条消息标注角色（"X · ACTOR"）+ 轮次（"T1" / "T2"）+ 情绪标签（"情绪 · 回避"等），底部输入框
- **右栏 · Neo Chat**：与情景导入 / 反思复盘的右栏同构（Neo 头像 + Chat 流 + 输入框），但承载**答疑辅导**而非铺垫引导；输入框标注 **"问 Neo……不影响左侧对练"** 强化 Neo 不打断 Actor 主线
- **双独立输入框**：中栏对 Actor / 右栏对 Neo —— 学员自主选择何时在哪边发言

> **三栏布局的核心目的**：让 Actor 主线对话与 Neo 旁路辅导**同时可见**，避免学员在两路对话间频繁切换抽屉、丢失上下文。
>
> **左栏拆上下两窗口的目的**：把"Actor 实时状态"（头像 / 情绪 / 曲线）与"Actor 背景资料"（性格 / 方法工具）分开 —— 上窗口是动态信息（每轮变），下窗口是静态资料（可折叠回顾），视觉聚焦更清晰。

##### 9.3.2.2.3 反思复盘布局（两栏）

```
┌─────────────────────────────────────────────────────────────────────┐
│ Topbar（项目名 / Course / Activity / 进度 X/N 轮）                   │
├─────────────────────────────────────────────────────────────────────┤
│ Subbar（路径面包屑 / 当前阶段标识）              [「重新对练」按钮]   │
├──┬──────────────────────────────────────┬───────────────────────────┤
│抽│                                      │                           │
│屉│  课件区（左侧）                       │  Neo Chat（右侧）          │
│  │                                      │                           │
│  │  · 对练小结                           │  · Neo 头像                │
│  │    - 目标达成（✅/⚠️/❌）              │  · 复盘追问口播             │
│  │    - 高光表现                         │   - 苏格拉底式追问           │
│  │    - 待改进点                         │   - 必须给出总结收尾          │
│  │    - 对话回看（按轮次 +              │                           │
│  │      关键节点标注）                    │                           │
│  │                                      │                           │

```

**核心要点**：

- **左侧覆盖式抽屉**：与情景导入 / 角色扮演共用同一组件
- **顶部 Subbar**：右侧呈现 **「重新对练」按钮** —— 学员任意时刻点击可重新进入对练
- **课件区**：全幅展示对练小结（目标达成 / 高光 / 待改进 / 对话回看，详见 § 9.3.2.3.4）
- **Neo Chat**：做复盘追问 —— 苏格拉底式引导反思 + 必须给出明确总结
- **联动**：学员点击左栏对话回看的某一轮 → 右栏 Neo **自动**对该轮做定向分析（无需学员手动输入）

> **后续阶段**：反思复盘完成后，对练 Activity 进入"自由探讨阶段" —— 学员可继续在两栏布局下与 Neo 自由讨论本次对练的任意话题（详见 § 9.3.2.7 完成与衔接，自由探讨阶段保持 Subbar「重新对练」按钮可用）。

##### 9.3.2.2.4 状态指示

| 状态 | 触发 | 视觉提示 |
|------|------|---------|
| **学习计时** | 默认 | Topbar 右侧呈现本 Activity 累计时长（不要求精确，本期不写画像） |
| **进度计数** | 角色扮演阶段 | Topbar 右侧呈现"进度 X/N 轮"（X = 当前轮次，N = 剧本预设上限） |
| **当前情绪标签** | 角色扮演阶段，每轮更新 | 左栏 Actor 头像下方呈现当前情绪标签（如"😐 中性"/ "回避"）|

> **不做的状态**：
> - **当前阶段标识** —— 学员在抽屉 Tab1 已能看到当前所处阶段，不再单独标
> - **暂停** —— Actor 对话与 Neo 对话是**两路并行的独立对话流**，没有"统一暂停"概念；学员任意时刻在哪栏说话由自己控制
> - **重练中** —— 重练只是新一轮对话起始，不需要单独的状态指示；进度计数自然反映轮次

#### 9.3.2.3 对练 Activity 内容模型

##### 9.3.2.3.1 三阶段总览

对练 Activity 由 **三个连续阶段** 组成（KGP 编排时按阶段颗粒度提供内容）：

| 阶段 | 主体 | 学员看到 | KGP 提供（AOM 内容包） |
|------|------|---------|----------------------|
| **① 情景导入** | Neo + 背景 markdown | 场景设定 / Actor 人设介绍 / 任务说明 / 可用工具 | 场景文案 / 人设描述 / 学员任务 / 知识点回顾清单 |
| **② 角色扮演** | Actor + Neo 旁路 | 与 Actor 多轮对话；Neo 在右栏旁观 + 适时介入 | 对练剧本（Actor 对白库 / 触发条件 / 软化点 / 底线 / 情绪天花板 / 成功标准 / 技能阶梯） |
| **③ 反思复盘** | Neo + 对练小结 markdown | 实时生成的对练小结 markdown + Neo 苏格拉底追问 | 复盘追问框架（哪些行为该追问 / 怎么追问 / 引导指向行为承诺） |

> **关键设计**：对练 Activity 拥有**自己独立的 AOM 数据结构**——剧本（角色心理画像 / 行为模式 / 递进式反应链 / 成功标准 / 技能阶梯 等）是 practice 场域专属的内容产物，不与其他场域共用。AOM 数据骨架详见 § 5（注：详细字段属研发实现层）。
>
> **关于"阶段"的实现表述**：本节"三个阶段"是产品视角的概念抽象。在研发实现层是否落地为"三个 SCO"或其他数据结构，由研发根据系统需求与开发进度把控（详见 § 9.3.2.2 顶部说明）。

##### 9.3.2.3.2 情景导入阶段详细

**功能定位**：让学员在与 Actor 对话之前**完整理解**自己处在什么场景、扮演什么角色、要达成什么目标、有什么工具可用。

**左栏：背景 markdown 内容规范**

| 模块 | 内容 |
|------|------|
| **场景** | 学员所处的业务情境（如"你是产品部主管李明，市场部张总找你借 3 个销售支援活动"）|
| **双方角色** | 学员扮演的角色身份 + Actor 扮演的角色身份（含双方背景 / 当前压力）|
| **任务目标** | 学员需要达成什么（主目标 + 分项目标）|
| **可用资源** | 学员扮演这个角色时可调动的资源 / 筹码（如"团队人员"/"预算额度"/"时间窗口"/"上级背书"等）—— 让学员有"手里有什么牌"的认知 |
| **可用工具 / 知识点回顾** | 学员可调用的方法 / 框架 + 与本次对练相关的关键概念（如 PIN 法 / GROW 模型 等）—— 轻量回顾，不展开讲解 |

**右栏：Neo 铺垫口播范围**

- ✅ Neo 可做：解释场景背景 / 角色关系 / 任务目标 / 提示可用工具 / 回答学员关于规则和流程的疑问
- ❌ Neo 不可做：透露剧本中"学员该自己想到的方案" / 预判对练走向（如"对方一开始会很强硬"）/ 直接告诉学员"正确答案"或最优策略

**入口**：顶部 Subbar「开始对练」按钮 —— 学员阅读完背景信息、确认理解后点击进入角色扮演阶段，整体布局切换为三栏形态（详见 § 9.3.2.2.2）。

##### 9.3.2.3.3 角色扮演阶段详细

**功能定位**：对练的核心阶段 —— 学员与 Actor 多轮对话，Neo 在右栏旁路答疑辅导，导演在后台指挥 Actor。

**对练启动**

学员点击「开始对练」后，**直接进入剧本驱动的角色扮演** —— 没有固定开场白。学员可以先说，也可以等 Actor 先说，由剧本设定决定，不约束统一形态。

**Actor 情绪曲线**

- **5 档情绪等级**：从消极到积极一共 5 档（消极 / 偏消极 / 中性 / 偏积极 / 积极）
- **实时更新**：每轮对话后，导演根据对话推进更新当前档位
- **作用**：让学员直观看到自己的策略如何影响 Actor 情绪
- 呈现位置：左栏 Actor 信息区，头像下方（详见 § 9.3.2.2.2 三栏布局；具体视觉形态由 GUI 设计阶段确定）

**Neo 旁路答疑辅导机制**

```
Actor 角色扮演在中栏对话区持续进行
        ↓
Neo 在右栏实时观察学员每轮发言
        ↓
发现知识应用卡点时，Neo 在右栏给出辅导建议：
  "你注意到对方刚才说了'排期压力大'吗？
   这背后可能藏着他的 Interest。试试追问一下？"
        ↓
学员自主选择：
  (A) 暂时不理 Neo，继续在中栏和 Actor 对话
  (B) 在右栏和 Neo 聊一下，消化后再回到中栏继续
        ↓
学员也可随时主动在右栏向 Neo 提问
```

**辅导原则**：

| 原则 | 说明 |
|------|------|
| **不打断 Actor 对话** | Neo 的建议出现在右栏，不暂停中栏的模拟对话（★ 核心约束）|
| **学员自主** | 学员自己决定是否停下来和 Neo 聊，Neo 不强制中断 |
| **卡点驱动（主动）** | 发现知识应用卡点时主动给建议，不做无信号的干预 |
| **随时可问（被动）** | 学员随时可在右栏向 Neo 提问 |
| **简短精准** | 每次建议 1-2 句话，点到即止，不展开讲解 |
| **不替代思考** | 给方向提示，不给具体答案（"试试追问"而非"你应该说..."）|
| **剧本为上下文** | Neo 的判断以剧本和导演给 Actor 的指令为上下文，不脱离教学目标 |

**剧本结束条件**

由剧本定义触发条件，常见三类：

- **轮数上限**：对话达到剧本预设的最大轮数
- **目标达成**：学员的方案满足剧本"成功标准"，导演判定 Actor 接受
- **死循环检测**：学员连续 N 轮在表面僵持 / 重复同样策略，导演判定无法推进

结束后 Actor 退出对话，自动进入反思复盘阶段。

##### 9.3.2.3.4 反思复盘阶段详细

**功能定位**：对练结束后帮助学员**反思关键决策点**，把"刚才发生了什么"转化为"下次该怎么做"，是行为养成的关键收尾。

**左栏：对练小结**（实时生成）

| 模块 | 内容 |
|------|------|
| **目标达成情况** | ✅ 主目标达成 / ⚠️ 部分达成 / ❌ 未达成 |
| **高光表现** | 学员表现亮眼的关键决策点（如"第 3 轮开始探索 Interest，从表面僵持转为深入探询"）|
| **待改进点** | 学员可优化的关键节点（如"前 2 轮停留在表面对抗，未主动运用 PIN 法"）|
| **对话回看** | 按轮次完整展示对练记录 + 关键节点标注（如"第 3 轮 · 关键转折点"）|

**右栏：Neo 复盘追问**

| 行为 | 规范 |
|------|------|
| **苏格拉底式追问** | 引导学员自己发现问题（"你觉得是什么让你在那个时候突然换了一种方式？"），不直接告诉学员"你哪里做得好/不好"|
| **必须给出总结** | 追问之后**必须**给出明确总结和待改进项陈述 —— **不能只追问留开放状态不收尾**（★ 核心约束）|
| **引用具体行为** | 引用具体对话轮次做定向分析（"第 3 轮你这样说..."），不泛泛而谈（避免"你整体还行"这种空话）|
| **对比多次练习** | 如学员有多次重练记录（详见 § 9.3.2.7.3），Neo 对比多次表现差异 |

**对话回看分析**

- 学员在左栏对练小结中点击某一轮对话
- 右栏 Neo **自动**对该轮进行定向分析（无需学员手动输入提问）
- Neo 分析中引用剧本的成功标准和技能阶梯

**完成条件**：学员在反思复盘阶段确认完毕（具体确认动作由 GUI 设计阶段确定）= 对练 Activity 完成，弹出完成 Modal。

##### 9.3.2.3.5 三种对练变体

practice 本期支持 **3 种变体**，通过不同剧本实现，**平台演绎功能不变**（UI / 交互 / AOM 数据骨架统一）。Neo 的答疑与辅导适用于所有变体：

| 变体 | 说明 | Neo 答疑与辅导侧重 |
|------|------|-------------------|
| **基础 Roleplay** | 学员扮演自身的目标角色，Actor 扮演对方。最常用形态 | 辅导学员识别和应用目标技能 |
| **角色互换** | Actor 扮演学员的角色，学员扮演对方。复盘时切回原身份反思 | 辅导学员理解对方状态、提供方向建议 |
| **费曼** | Actor 扮演什么都不知道的学生，学员当老师讲解。倒逼深度理解 | 辅导学员成为老师讲解知识，提示遗漏 |

> **变体由剧本决定**：KGP 在 AOM 编排时通过剧本字段标识变体类型，平台运行时按相同流程演绎。

##### 9.3.2.3.6 多版本切换 + 隐性打标

**多版本机制**：同一对练 Activity 可挂多个 **剧本版本**，按学员标签切换。

实际效果（学员视角）：

- 学员甲（销售标签）→ 进入"销售场景版"剧本（如"客户砍价"）
- 学员乙（建筑标签）→ 进入"建筑场景版"剧本（如"工程对接"）
- **教学目标相同 / 业务场景不同**

**学员标签的获取方式**：

| 阶段 | 打标方式 |
|------|---------|
| **早期（v0.5 本期）** | KGP 在情景导入阶段内置打标点位（如内置一个轻量诊断节点），规则由内容老师写入 |
| **后期（演进方向）** | Neo 自主判定标签 / 置信度 |

**标签作用域**：

- 对练 Activity 内多次打标，后一次打标覆盖前一次（同维度）
- **标签仅本对练 Activity 内有效**——下一对练 Activity 如需个性化版本切换需在自己的情景导入阶段重新打标

> **本节不写实现层细节** —— 切换机制 / 打标具体实现 等属 Prosona 演绎架构，由研发设计。

本期 3 类管理员**共用同一管理端**，权限差异详见 backlog（待 § 10 详细设计阶段）。

### 10.2 看板 Operation

#### 10.2.1 业务定义

管理员日常监控项目运行健康度的场域。三维度监控 + 热力图 + Ora 双交互。

#### 10.2.2 三维度监控

| 维度 | 内容 |
|------|------|
| **项目维度** | 项目整体健康度 / 进度 / 完成率 |
| **人员维度** | 学员个体进度 / 异常学员 / 活跃度分布 |
| **内容维度** | 各 Course 学习情况 / 卡点分布 / Quiz 通关率 |

#### 10.2.3 热力图

学员行为热力可视化——每个学员每日活跃度色块 / 每个 Course 学员通过率色块。

#### 10.2.4 Ora 双交互模式
| 模式 | 触发 | 行为 |
|------|------|------|
| **看板被动呈现** | 默认呈现 | 三维度数据 + 热力图 + Ora 自动生成的项目健康度解读 |
| **Ora 问答** | 管理员主动提问 | "为什么 W4 脱训突增？" / "张三这周参与度怎样？" → Ora 用证据回答 |

> **本期不做**：看板点击 → Ora 实时解释——这是未来形态（详见 § 3.3.4）

#### 10.2.5 异常学员处置

按 § 6.6.2：

- **平台层**：Ora 看板 / 问答 中发现异常学员（脱训 / 进度落后 / Quiz 频繁失败）
- **线下层**：项目运营**线下跟进**该学员（电话 / 微信 / 1on1 沟通）
- **平台不强制干预**——干预靠人 / 渠道完成

#### 10.2.6 颜色条 / 排序 / 详情页

- **颜色条标识健康度**（绿 / 黄 / 红）
- **异常自动排顶**（异常项目 / 异常学员 自动排在顶部）
- **详情页"建议动作"** —— Ora 给出建议动作（详见 § 3.3.5）

### 10.3 团队报告生成 Team Report-Gen

> 这是 § 10 的**核心模块**——管理端的主要交付能力，承担中期反馈 / 半程 review / 结项报告 / 剩余服务期内的新报告 全部生成场景（详见 § 6.7 + § 6.8）。

#### 10.3.1 业务定义

管理员产出对外可举证报告的场域——Ora 拉全程数据 + 多轮对话生成初稿 + 管理员 HITL 编辑 + 导出 PDF / Word。

#### 10.3.2 引擎核心 · 每管理员独立报告库
- **每个管理员**有**独立的报告库**——存放该管理员生成 / 编辑的全部 team report
- **里程碑触发时**给每个管理员一份**独立草稿副本**
- **本期不做协同分享 / 不做实时多人编辑** —— 需协同时由管理员**导出后线下同步**

> 这个设计简化了多人协同的复杂度——研发只需保证每个管理员的报告库独立，不需做实时协同编辑。

#### 10.3.3 prompt 模板由"关联动作 / 评估要点"驱动
- 管理员**不需要选模板，也不手写 prompt**
- 管理员在创建 team report（或里程碑配置）时**填"关联动作 / 评估要点"**（如"评估学员中期参与度 + 概念掌握度"）
- 系统据此**自动生成**对应的 prompt 模板

> 本期 v0.5.0 优先做 **"结项报告 prompt 模板"** —— 用作里程碑结营时的默认 prompt。其他模板（中期 / 半程 / 自定义）后续逐步加（详见 § 6.7.1）。

#### 10.3.4 多轮对话生成 + HITL 编辑

生成流程：

```
里程碑到期触发 / 管理员手动触发
   ↓
Ora 拉项目全程数据
   ↓
Ora 与管理员多轮对话（确认范围 / 学员维度 / 数据筛选 / 文本风格）
   ↓
Ora 生成初稿
   ↓
管理员 HITL 编辑（增删改 / 调整结构 / 改文案）
   ↓
导出 PDF / Word
```

#### 10.3.5 与里程碑的关联

详见 § 6.6.3 + § 6.7.1：

- 里程碑到期当天 → Ora 自动生成 team report 草稿（每管理员一份副本）
- 站内消息提前 3 天每日提醒管理员
- Ora 在草稿生成后即时推送通知
- 过期未处理 → 提醒保留在列表（不堆累 / 不重复推送 / 不升级渠道）

#### 10.3.6 模板含义澄清
按 § 6.7.1：

> 本节"模板"指的是 **prompt 约束的写作规则**（即给 Ora 的"写作 brief / 文章骨架 / 核心要素清单"），**不是固定文章模板或 HTML 模板**。
>
> Ora 基于模板的写作约束 + 项目实际数据 + 关联动作 / 评估要点 → 生成具体文本。

#### 10.3.7 本期不做

- ❌ **学员个人结营报告** —— 学员小结全部在阶段 5 学习期内通过 Course 个人小结 Activity 完成（详见 § 6.7）
- ❌ **多模板自由切换 UI** —— 本期仅"结项报告 prompt 模板"
- ❌ **协同分享 / 实时多人编辑** —— 见 § 10.3.2

### 10.5 消息 Message

#### 10.5.1 业务定义

管理员对学员定向沟通的场域——支持手动发送富文本消息 + 系统自动催学消息的统一查看与跟踪。

#### 10.5.2 GUI 与交互

| 子模块 | 功能 |
|-------|------|
| **手动定向发送** | 项目运营从学员名单中**手动选人** + 编辑富文本 + 发送 |
| **发送记录** | 手动 + 自动催学消息**统一查看**——含对象 / 时间 / 内容 / 投递日志 |

#### 10.5.3 与催学规则的关系

按 § 6.6.5：

- **全员催学** = 通过催学规则自动触发（系统层）
- **异常学员定向催** = 通过本模块**手动选人**发送（运营层）
- 两者的发送记录在本模块**统一呈现**

#### 10.5.4 投递日志

- 完整记录：谁收到 / 是否查看 / 是否点击链接
- 本期**不支持学员回应**（学员无法在催学消息上回复）

#### 10.5.5 本期不做

- ❌ **邮件 / IM / SMS 渠道**（仅站内）
- ❌ **消息已读率 dashboard**（投递日志后台可查，前端不展示）
- ❌ **消息模板库 UI**（文案模板配置在催学规则模块，详见 § 6.6.5）
