# Round 2: 修改建议

---

## 建议 01【高·结构遗漏】HR端补充催学配置入口

**问题：** 需求文档14.1.5明确HR也可配置催学规则，功能树HR端完全缺失。

**修改方案：** 在HR端新增模块或在F1导航中增加"催学"入口，复用运营端I模块的规则配置能力。

```
建议新增 HR端催学入口（复用运营端能力）
├── 催学规则列表（读写）
│   └── 与运营端I1共享规则配置界面
└── 催学效果概览（只读）
    └── 展示当前项目催学效果漏斗
```

**影响范围：** F1.1.1导航项需增加"催学"；需明确HR和运营同时配置催学规则时的冲突处理策略（谁优先？合并？）。

---

## 建议 02【高·逻辑矛盾】C5 分组标签的生成机制需明确

**问题：** "高活跃/稳健/需关注"标签需要自动分类逻辑，但本期原则是"不做智能评价"。

**修改方案（二选一）：**

- **方案A（推荐）：** 标签改为基于规则的阈值分组，非智能评价。运营在项目配置中设定阈值（如完成率<30%=需关注），系统按阈值机械打标。这是规则配置，不是智能评价。
- **方案B：** 本期不做自动分组标签，改为运营手动标注。

无论选哪个方案，C5.1.1需补充标签生成逻辑说明。

---

## 建议 03【高·优先级不清】B2.1.1 学习效果标注为待定

**问题：** 需求文档标注"希望有，具体指标待定"，功能树把它列为正式节点容易误导开发。

**修改方案：** B2.1.1追加标注：

```
B2.1.1 效果指标 — 数据看板效果模块 — 展示测评分数、对练评分、能力维度变化
       ⚠ 优先级：P2（希望有），具体指标待定，不阻塞V1发布
```

---

## 建议 04【高·技术可行性】D2.2 Agent记忆偏好降级处理

**问题：** 跨会话记忆+偏好学习是高复杂度功能，可能超出本期Agent技术栈。

**修改方案：** 拆为两期：

- **本期（V1）：** Agent支持在单次会话内记忆上下文，HR可以手动选择"使用上次报告模板"
- **未来：** Agent自动学习偏好，跨会话记忆

功能树D2.2.1修改为：

```
D2.2.1 模板复用 — 多次生成报告时 — HR可选择"使用上次报告模板"，Agent按模板生成（本期不做跨会话自动记忆）
```

---

## 建议 05【高·权限设计】L1 Agent实例与权限隔离方案

**问题：** "同一实例+不同权限"在功能树中缺少实现说明。

**修改方案：** L1.1.1重写为：

```
L1.1.1 与HR Agent一致 — 本期 — 运营端与HR端共享同一Agent模型和能力集，
       通过用户角色（HR/运营）动态控制数据访问范围：
       HR角色 → 本项目学员数据 + 报告数据
       运营角色 → 全部项目数据 + 配置权限 + 系统数据
```

---

## 建议 06【高·批量流程】K3 审核队列补充批量操作

**问题：** 30份报告逐份审核效率太低，且缺少失败处理。

**修改方案：**

```
K3 审核队列（修改后）
├── K3.1 逐份审核
│   └── K3.1.1 快速审核 — 运营进入审核队列 — 逐份快速审核每份报告
├── K3.2 Agent辅助审核
│   └── K3.2.1 数据解释 — 运营标注"数据不对" — Agent立即给出数据来源和计算逻辑
├── K3.3 批量操作（新增）
│   ├── K3.3.1 全部通过 — 运营确认无需逐份审核 — 一键全部标记为"通过"
│   └── K3.3.2 批量标记 — 运营选择多份报告 — 批量标记"通过"或"需修改"
└── K3.4 失败重试（新增）
    └── K3.4.1 重新生成 — 生成失败的报告 — 支持单份或批量重新生成，展示失败原因
```

---

## 建议 07【中·边界澄清】I1催学规则 vs J2提醒规则 明确区分

**问题：** I1面向学员（催学），J2面向运营自己（提醒），但配置逻辑相似容易混淆。

**修改方案：** 在I模块和J模块的根节点追加区分说明：

```
催学管理（对象：学员，动作：给学员发消息）
数据展示 > 可配置提醒（对象：运营自己，动作：在看板高亮/推送通知给运营）
```

---

## 建议 08【中·数据一致性】HR端A模块与B模块的数据去重

**问题：** A（进度看板）和B（数据看板）都有进度数据，HR分不清区别。

**修改方案：** 明确定位区分：

- **A 进度看板：** 聚焦"人"，按学员维度展示进度，可钻取到个人详情——日常巡检用
- **B 数据看板：** 聚焦"指标"，展示聚合统计指标和趋势——汇报和分析用

在A和B的根节点各加一行定位说明。

---

## 建议 09【中·HR/运营对齐】被动信号数据是否进入HR端

**问题：** 运营端J1.4有"被动信号聚合"，HR端B模块没有。

**修改方案：** 建议本期HR端不展示被动信号。原因：被动信号（停留时长、中途离开率、回翻率）偏运营分析维度，HR更关注结果（进度、效果、完课率）。如需调整，补充到B模块即可。

---

## 建议 10【中·模板体系】统一报告模板管理

**问题：** D2（HR Agent生成报告）和K1.3（报告工厂选模板）各自提到模板，未说明是否统一。

## 四个重点关注项的最终结论

### 1. HR Agent与运营Agent本期一致的设计

**结论：方案可行，但需补充权限隔离说明（建议05）。**

"同一Agent实例"在本期是合理的简化——30-40人项目规模下HR和运营的提问模式差异不大。但功能树必须写清楚：同一模型+角色动态权限，HR只能看本项目数据，运营可看全部数据+配置权限。未来根据差异化需求再拆分。

### 2. 催学规则可配置

**结论：功能树有结构性遗漏（建议01），催学模块设计本身合理。**

运营端I模块的催学规则设计（可配条件+模板+策略+升级机制+效果漏斗）与需求文档高度对齐，可配置性充分。但HR端完全缺失催学入口，需补充。同时需明确HR和运营都配置催学规则时的冲突解决策略。

### 3. 数据展示不做智能评价

**结论：大部分已对齐，C5标签存在一处矛盾（建议02）。**

B模块和J模块都遵循了"呈现数据+可配置提醒"的原则。唯一矛盾点是C5的"高活跃/稳健/需关注"分组标签——自动分组本质上是一种评价判断。建议改为基于阈值的规则分组（运营配置阈值，系统机械执行），这样在"不做智能评价"的约束下仍可实现。

### 4. 报告工厂批量流程

**结论：核心流程完整，缺少效率和容错设计（建议06）。**

K1选参数→K2批量生成→K3审核→K4导出的四步流程与需求文档15.1.4完全对齐。但30份报告的实际操作场景中，"逐份审核"效率太低，需要批量通过/批量标记能力；且并行生成必然有失败情况，需补充重试机制。
