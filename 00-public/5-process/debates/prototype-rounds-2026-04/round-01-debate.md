# 第一轮辩论：从用户旅程出发，确定"要讲几个故事"

> 辩论日期：2026-04-22
> 参与角色：PM（产品经理）、Dev（研发经理）、UX（交互设计师）、Student Rep（学员代表）、HR Rep（企业HR代表）

---

## 第一回合：各角色提出核心旅程主张

### PM（产品经理）

各位，我们今天的目标不是讨论"需要多少张原型图"，而是"需要讲清楚哪些故事，让研发团队看完之后能理解整个产品的运转逻辑"。

我梳理了所有 spec 文件中的用户故事，学员端有五大用户旅程值得作为核心故事线：

1. **新学员首次登录旅程**（`learner/00-overview.md` 场景一）——从登录到首次见到 Leo、首次进入授课场域、首次记笔记，这条线要完整讲一遍，因为它涉及 Leo 自我介绍、需求感知推荐、大厅看板初始状态、SCO 锁定、笔记悬浮球全部的"首次状态"。

2. **授课完整25分钟旅程**（`learner/02-lecture-zone.md` PIN 法用户故事）——这不是选做，是必做。包含 FEEDBACK_COLLECT、PPT 讲解、Video 复盘、Quiz 答题、答疑口播、FEEDBACK_REVIEW 闭环，是整个 AOM 引擎的完整展示。

3. **对练完整旅程**（`learner/03-practice-zone.md` 张磊对练用户故事）——三阶段：情景导入、角色扮演+Neo答疑与辅导双栏、反思复盘+对练小结。三种变体（Roleplay/角色互换/费曼）至少要在快照中体现一种，另外两种以注释说明。

4. **Leo 深度辅导旅程**——hall 文件里有五个用户故事，我认为至少要选两个来展示 Leo 的核心差异化价值：故事二（工作受挫后情绪感知+知行差距）和故事四（复杂场景+STRATEGIST+课程推荐）。

5. **管理端完整生命周期**——从开营配置（`management/03-project-config.md`）到日常巡检（`management/01-operation.md`）到团队报告生成（`management/02-report-gen.md`）到督学消息（`management/04-message.md`）。

管理端的故事必须和学员端有数据穿透关系。比如学员端张磊在对练中的进步，管理端的 Ora 能追溯到原始对话——这种跨端数据流转如果不在原型中体现出来，研发不可能理解记忆工程的实际运作。

---

### Dev（研发经理）

PM 说得有道理，但我有不同的优先级判断。作为要真正开始写代码的人，我关心的不是"故事多完整"，而是"组件和布局模式是否够清楚"。

我看完 spec 之后，发现产品本质上只有**三种布局模式**：
- 学员端双栏（左栏内容+右栏AI对话）——大厅、授课、对练、报告都是这个
- 管理端双栏（左栏看板/编辑器+右栏Ora）——首页、报告生成
- 单栏全幅——管理端消息、配置；调研场域的核心调研阶段

所以我其实需要的核心旅程是能够覆盖**所有不同的布局和交互模式**的最少故事集。具体来说：

1. **大厅完整状态**（初始+学习中+学完一个Course后）——我需要看到看板区各个组件在不同数据量下的呈现。
2. **授课场域的 SCO 类型全覆盖**——不需要讲25分钟的故事，我需要看到每种 SCO 的快照。每种一帧就够了。
3. **对练场域三阶段各一帧**——情景导入、角色扮演（双输入框布局很关键）、反思复盘。
4. **管理端配置的开营流程**——这是最早被使用的模块。
5. **通用栏状态**——topbar、铃铛消息面板、头像下拉菜单、端口切换。

**我和 PM 的分歧在于：PM 想讲"故事"，我想看"状态和组件"。我建议以"旅程为线索、状态快照为节点"来组织——不是拍电影，是画分镜头。**

---

### UX（交互设计师）

Dev 说得很实际。但是作为做交互设计的人，我更关心**状态转换和边界场景**。

spec 里有大量精妙的交互设计，如果原型不把转换过程讲清楚，快照会变成一堆静态图片没人理解。我认为以下**关键转换**必须在旅程中被覆盖：

1. **大厅到场域的进入转换**——`learner/00-overview.md` 说"0.3秒过渡动画"，但更关键的是：学员点击 Leo 推荐的课程卡片 vs 点击看板课程目录的 Activity 卡片，进入场域后 Neo 的首次自我介绍 vs 非首次进入的差异。

2. **授课中的"打断-恢复"流**——`02-lecture-zone.md` 交互#3"学员插嘴提问"描述了授课口播被打断后恢复的流程。

3. **对练双栏的独立性**——`03-practice-zone.md` 里左栏和右栏各有独立输入框。这个"两条独立消息流"的快照如果不画，前端一定会做成一个聊天窗口。

4. **调研场域的栏数变化**——`04-inquiry-zone.md` 描述了三阶段栏数不同（双栏→单栏沉浸→双栏）。

5. **管理端 Ora 跨页记忆**——原型需要展示"管理员在首页问了一个问题，切到报告页后 Ora 还记得"。

6. **笔记悬浮球的叠加态**——原型需要在至少两个不同场域中展示笔记悬浮球+悬浮窗的状态。

---

### Student Rep（学员代表）

各位说的都是技术角度。我作为学员，想说说我的困惑。我看完 spec 之后，最大的问题是：**这个产品我到底怎么用？**

我觉得原型需要回答学员最基本的五个问题：

1. **我登录后看到什么？**——大厅的第一屏。
2. **我怎么上课？**——PPT 在左边，Neo 在右边，我是看左边还是听右边？
3. **对练到底是什么感觉？**——左边是"张总"跟我说话，右边是 Neo 在旁边指导，我怎么同时处理两边？
4. **Leo 和 Neo 到底什么区别？**——原型需要让我看到在大厅和 Leo 聊天是一种感觉，进了课堂和 Neo 聊天是另一种感觉。
5. **报告对我有什么用？**——不只是展示报告长什么样，而是展示互动。

**我特别强调：`01-hall.md` 里的用户故事二"张磊工作受挫后找 Leo 聊"那个旅程，对于学员来说是最核心的差异化体验。** Leo 的价值在于他能接住我的情绪，引用我在课堂上的真实表现来指导我——这个如果不在原型中呈现，学员不会理解为什么要跟一个 AI 聊天。

---

### HR Rep（企业HR代表）

我主要关注管理端。我每天的核心诉求就两个：**随时知道培训进展如何**，和**月底能拿出一份像样的报告给老板**。

原型必须讲清楚以下管理端旅程：

1. **日常巡检旅程**（`management/01-operation.md` 用户故事一）——登录→看总览→问 Ora→看学员详情→看内容维度。
2. **被领导追问时的应急查询**（用户故事二）——"张磊现在学到哪里了？"→ Ora 秒回。
3. **团队报告生成完整流程**（`management/02-report-gen.md`）——从新建到导出。
4. **开营配置旅程**（`management/03-project-config.md`）——名单导入后的预览是什么样？
5. **数据追溯到原始对话**（用户故事七）——Ora 帮我找到学员说过的原话。

**我和 PM 的共识是：管理端和学员端必须有数据穿透的展示。我补充一点：消息模块虽然简单，但快照不能省。**

## 第二回合：交叉辩论与分歧聚焦

### Dev（研发经理）

我听了大家的发言，现在有三个核心分歧需要解决：

**分歧一：故事粒度**。PM 想要完整的25分钟授课旅程，我只需要每种 SCO 类型一帧。学员代表想要"感觉"，我需要"组件规格"。我建议折中：**以用户旅程串联，但在关键节点标注所有需要出现的状态快照**。比如授课旅程，故事线走一遍，但每个 SCO 类型的快照作为附页展示。

**分歧二：管理端的优先级**。HR 希望管理端四个模块都有完整旅程，但我认为配置和消息是纯 GUI 表单，快照难度很低。核心旅程应该聚焦在有 Ora 参与的首页和报告生成。

**分歧三：调研场域（P3）**。UX 说至少要有占位说明。我的态度是：如果六月不做，原型就不画。如果是 P3 争取项，就单独放一页标注"P3 如果做，长这样"。

---

### PM（产品经理）反驳

Dev 的折中建议我部分接受。但有一点我必须坚持：**Leo 辅导旅程不能压缩为一帧**。

`01-hall.md` 里有五个 Leo 用户故事，每一个展示了不同的核心能力。这五个故事如果都压缩，Leo 就变成了"一个聊天框"。Leo 是这个产品的**灵魂**——`requirements-v0.4.0.md` 明确说"Leo（Coach）是学员端大厅的核心"（ADR-003）。

我建议至少保留**两个完整的 Leo 对话旅程**（故事二和故事四），其余三个以注释说明"Leo 还支持以下场景"。

但我同意 Dev 说的：调研场域 P3 不进核心旅程，单独附页标注。

---

### UX（交互设计师）回应

我要为状态转换争一个位置。PM 和 Dev 都在说"内容旅程"，但我关心的是**交互模式的清晰传达**。

精简后我认为不可省略的转换点：
1. **授课中断-恢复**——可以用两帧快照+箭头标注，不需要独立旅程。
2. **对练双栏独立输入**——需要一帧明确标注"左栏输入区发消息给 Actor，右栏输入区发消息给 Neo"。
3. **场域结束→下一任务衔接**——Activity 完成时的最后一屏不能省。
4. **报告渐进式呈现**——如果不标注，研发会一次性加载全部内容。

我不再坚持笔记悬浮球要在每个场域都展示了——同意 Dev 的建议，在两个场域中展示即可。

---

### Student Rep（学员代表）补充

我同意 PM 的判断——Leo 是灵魂。但我想补充一个旅程，大家都没有提到：**自由复习旅程**。

`learner/00-overview.md` 和 `02-lecture-zone.md` 都提到"自由复习"是本期做的功能。如果不把这个旅程在原型中呈现，学员不知道"学完了还能不能回去看"，而且研发也不知道"已完成 Activity 的场域状态"和"正在学习的场域状态"有什么区别。

---

### HR Rep（企业HR代表）反驳 Dev

Dev 说配置和消息可以作为附录。我部分同意消息可以简化，但**配置模块的开营流程不能降级**。

原因是：`management/03-project-config.md` 的用户故事涉及**系统状态的根本切换**（`requirements-v0.4.0.md` 3.2节：开营前学员端不可登录，开营后四模块全部解锁）。如果开营配置的旅程不清晰，研发不知道"确认开营"这个按钮按下去之后系统发生了什么变化。

另外我同意 PM 说的管理端和学员端需要数据穿透展示。我提议在报告生成旅程中，专门展示 **Ora 追溯到学员端原始对话记录**的交互。

## 第三回合：收敛共识与旅程定稿

### PM（产品经理）收敛

经过两轮讨论，我来做一次收敛：

**共识**：
1. 以旅程为线索、以关键状态快照为节点来组织原型
2. Leo 辅导旅程必须深度展示（至少两个完整对话场景）
3. 管理端需要有数据穿透到学员端的展示
4. 调研场域（P3）不进核心旅程，单独附页
5. 关键交互转换点需要在快照旁用箭头和注释标注

**已解决的分歧**：

| 分歧 | 各方立场 | 解决方式 |
|------|---------|---------|
| 故事 vs 组件 | PM/学员主张完整旅程；Dev 主张组件状态 | **折中**：旅程为主线+每条旅程附组件状态总览 |
| Leo 旅程深度 | Dev 认为一帧够；PM/学员坚持至少两个场景 | **PM/学员胜**：保留两个完整 Leo 对话旅程 |
| 配置模块定位 | Dev 认为可降级；HR 坚持涉及系统状态切换 | **HR 胜**：开营配置保留为核心旅程 |
| 调研P3 | UX 建议占位；Dev 不想画不做的东西 | **折中**：P3 附页单独展示 |
| 消息模块 | HR 要求完整；Dev 认为太简单 | **折中**：核心旅程但精简 |
| 自由复习 | 学员代表认为必须独立展示 | **学员代表胜**：独立旅程 |

### Dev（研发经理）追加

我接受这个结论。但追加一个要求：**每个核心旅程，除了故事线快照外，需要一张"组件状态总览"附页**。这样设计师出的原型我能直接用于前端组件拆分。

### UX（交互设计师）补充

完全同意 Dev 的附页建议。另外确认：**场域间直跳**不需要独立旅程，在授课结束时标注即可。

### Student Rep（学员代表）确认

报告场域应该作为独立旅程——它的"已生成内容+AI 讨论"模式和授课完全不同。

### PM（产品经理）最终定稿

好，报告场域确认为独立旅程。管理端首页三个维度各出一帧在日常巡检旅程中串联。

## 最终产出：核心用户旅程清单（13条 + 1个P3附页）

### 学员端（J1-J7）

**J1. 新学员首次登录** — 登录→大厅空状态→Leo自我介绍→需求感知→进入授课→Neo首次自我介绍
- 参考 spec：`learner/00-overview.md` 场景一

**J2. 授课场域完整教学** — FEEDBACK_COLLECT→PPT讲解（含插嘴答疑中断-恢复）→Video→Quiz→FEEDBACK_REVIEW→Activity完成→下一任务
- 参考 spec：`learner/02-lecture-zone.md` PIN法用户故事

**J3. 对练场域完整训练** — 情景导入→角色扮演双栏（Actor+Neo独立输入）→Neo主动推送+学员提问→对练小结→复盘追问→重练
- 参考 spec：`learner/03-practice-zone.md` 张磊用户故事

**J4. Leo深度辅导（两个场景）** — A:情绪感知+知行差距+行动计划 B:复杂场景+STRATEGIST+框架引导+课程推荐
- 参考 spec：`learner/01-hall.md` 用户故事二和四

**J5. 报告场域查阅与解读** — 祝贺→三部分渐进呈现→证据查看→图表→Neo讨论→导出
- 参考 spec：`learner/05-report-zone.md` 张磊用户故事

**J6. 自由复习** — 大厅→已完成Activity（SCO全解锁）→Neo答疑讨论→翻看之前页面
- 参考 spec：`learner/02-lecture-zone.md` 自由复习部分

**J7. 大厅看板日常使用（学习中期）** — 热力图+工具库+笔记库+待办+消息面板+Leo推荐
- 参考 spec：`learner/01-hall.md` 用户故事一

### 管理端（J8-J12）

**J8. 开营配置** — 开营前状态（导航锁定）→项目信息→名单导入→角色分配→催学规则→个性化→内容预览→确认开营
- 参考 spec：`management/03-project-config.md`

**J9. 日常巡检** — 项目维度（热力图+Ora查询）→人员维度（学员详情→个人报告）→内容维度（柱状图+Ora解读）
- 参考 spec：`management/01-operation.md` 用户故事一至四

**J10. 团队报告生成** — 报告库→新建表单→Ora生成→HITL编辑→Ora查询+修改→导出→追溯证据
- 参考 spec：`management/02-report-gen.md` 王经理用户故事

**J11. 督学消息** — 新建消息（选人+编辑）→发送→发送记录（手动+自动+状态）
- 参考 spec：`management/04-message.md`

**J12. Ora跨页记忆与深度追溯** — 首页问Ora→切报告页Ora记忆连续→追溯到学员端原始对话
- 参考 spec：`management/00-overview.md` Ora共享记忆

### 通用（J13）

**J13. 通用栏与跨端体验** — topbar全景→铃铛消息面板→头像下拉→个人设置→端口切换→帮助
- 参考 spec：`common/00-overview.md`、`common/01-general.md`、`common/02-notification.md`

### P3附页

**P3. 调研场域** — 三阶段栏数变化+对话式+量表两种形态
- 参考 spec：`learner/04-inquiry-zone.md`

## 待产品负责人确认的7个开放问题

1. Leo后台逻辑在原型中标注到什么深度？
2. Ora追溯原始对话的展示格式（文本引用/对话气泡/弹窗）？
3. 笔记悬浮球在对练双栏中的位置？
4. 学员端和管理端热力图视觉规格是否统一？
5. 断点续播的UI呈现（弹窗提醒 vs 静默恢复）？
6. 对练三种变体中角色互换和费曼是否需要单独快照？
7. 开营前vs开营后的导航状态是否需要两帧对比？
