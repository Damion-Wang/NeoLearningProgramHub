## 零、前置议题：交互范式统一

**PM：** 在进入逐快照 brief 之前，有一个重要的设计原则需要先对齐——**减少交互范式种类**。目前 spec 中出现了多种切换机制：授课场域左侧 SCO 列表点击切换、对练/调研 Phase Tab 切换、管理端维度 Tab 切换、配置页锚点导航……如果每个模块都发明自己的切换范式，用户学习成本会很高。

**UX：** 我梳理一下当前涉及的切换场景和已确认/待讨论的方案：

| 切换场景 | 当前方案 | 性质 |
|---------|---------|------|
| 授课 SCO 切换 | 左侧导航列表点击（R2已确认） | 内容单元顺序切换 |
| 授课 Activity 列表 | 左侧导航 Tab2（R2已确认） | 课程目录查看 |
| 对练三阶段切换 | 左导航内嵌步骤条 stepper（R2已确认） | 流程阶段切换 |
| 调研三阶段切换 | 待定 | 流程阶段切换 |
| 管理端首页三维度 | 左栏顶部 Tab（R2辩论结论） | 视图维度切换 |
| 管理端消息两视图 | Tab 切换（R2辩论结论） | 功能视图切换 |
| 管理端配置子功能区 | 左侧子导航锚点（R2已确认） | 长表单分区导航 |
| 管理端报告库→编辑器 | 点击列表项→进入（R2已确认） | 列表→详情 drill-down |

**Dev：** 从开发角度，我建议归纳为 **4 种基础范式**，所有切换场景从中选用：

1. **左导航列表**：点击列表项 → 右侧内容区替换（用于：SCO切换、Activity列表）
2. **左导航步骤条**：纵向 stepper，阶段顺序明确，支持回看已完成阶段（用于：对练阶段、调研阶段）
3. **内容区顶部 Tab**：水平标签页，Tab 之间平等无顺序（用于：管理端维度、消息视图）
4. **列表→详情 drill-down**：点击列表项进入详情视图，左上角返回链接（用于：报告库→编辑器）

辅助范式：
- **左侧子导航锚点**：长页面定位，仅用于配置页（性质是同一表单的分区导航，不算视图切换）
- **弹窗/下拉**：叠加层交互（消息面板、头像菜单、设置弹窗等）

**Student：** 从学员视角，对练和调研都是"分阶段推进的流程"，用统一的左导航步骤条很自然。但步骤条和 SCO 列表的视觉要区分开——步骤条是 3 个固定阶段，SCO 列表可能有 10+ 个条目。

**HR：** 管理端的维度 Tab 和消息 Tab 都是水平标签页，这很一致。配置页用锚点导航也合理——它本质是一个长表单，不是多个独立视图。

**PM：** 好。那调研场域（S18）的三阶段切换也用左导航步骤条，和对练完全一致。这样"学员端场域内的阶段推进"只有一种范式。

**UX：** 补充一点视觉区分规则：
- **左导航列表**（授课）：每项有序号+标题+状态图标（✅▶🔒），类似目录
- **左导航步骤条**（对练/调研）：3 项固定，用连接线串联，当前阶段高亮+进行中动画，已完成打勾，未到达灰色
- 两者在左导航同一区域，但视觉语言不同——列表是"可随机访问的目录"，步骤条是"必须顺序推进的流程"

**Dev：** 从实现角度，步骤条和列表可以共用同一个左导航容器组件，只是内部渲染模式不同。这个复用很好做。

**全体：** 通过。交互范式统一为 4+2 种，各场景归属明确。

## 一、S01 — 登录页

### 辩论

**PM：** 登录页是全产品入口，布局 L-H（全屏无导航）。内容简单：Logo + 登录表单 + 登录按钮 + 产品标语。

**UX：** 登录页需要传达产品调性。建议左右分屏——左半是品牌视觉区（产品名+标语+装饰图），右半是登录表单区。这是 SaaS 产品的标准登录页布局。

**Dev：** 只有账号密码登录，不做第三方 SSO 和验证码？

**PM：** 对，本期仅账号密码。后期对接企业 GPB（通用平台总线）做 SSO。登录页只是占位，核心是让开发者知道入口在这里。

**Student：** 登录页上有"忘记密码"入口吗？

**PM：** 不需要。本期账号由管理员批量导入创建，密码重置走管理员渠道。

## 二、S02 — 大厅空状态（首次登录）

### 辩论

**PM：** 首次登录大厅，所有看板数据为空/零，课程全部锁定，工具库为空。核心看点是右侧 Leo 的自我介绍和需求感知对话。

**UX：** 空状态的看板区不能只放空白——每个组件需要有明确的空态设计。比如学习进度显示 0%，课程目录所有卡片灰色锁定，热力图无数据但保留格子框架，工具库显示"暂无工具，完成课程后逐步解锁"。

**Student：** 首次进入最重要的是 Leo 的自我介绍。我进来看到一堆空白看板会很茫然，Leo 的对话必须在第一时间抓住注意力。

**Dev：** 右侧 Chat 区可以设置自动滚动到底部，这样 Leo 的首条消息进入视口。或者首次进入时左侧看板区略暗，右侧 Chat 区高亮？

**UX：** 不需要做亮度差异——Leo 的消息气泡足够醒目。关键是 Leo 的首条消息要包含一张自我介绍卡片（头像+名字+身份说明），视觉上比普通文字消息更突出。

**PM：** Leo 的首次对话需要覆盖：自我介绍 → 能力说明 → 需求感知提问 → 学员回答 → 课程推荐卡片。在 mock 数据里我们写 2-3 轮。

## 三、S03 — 授课场域 · PPT SCO

### 辩论

**PM：** S03 是授课场域的默认视图，展示 PPT 类型 SCO。同时承载笔记悬浮球完整交互链和自由复习模式差异注释。

**UX：** PPT SCO 的中间看板区渲染一张 PPT 图片。这里有个关键问题——用户提到会提供"实际 PPT 页面图片"用于原型展示。所以我们在 brief 里标注 mock 图片的内容主题，实际图片由用户后续提供。

**Dev：** 左侧导航需要同时展示两个 Tab——Tab1 是 SCO 列表（当前 Activity 的所有 SCO），Tab2 是 Activity 列表（整个 Course 的 Activity）。R2 确认了用 SCO 列表点击切换内容，不是顶部 Tab。两个 Tab 切换器放在左导航顶部。

**Student：** Neo 的首次自我介绍要在 Chat 区顶部展示——"你好，我是 Neo，你的课堂导师…"。这和 Leo 的自我介绍卡片对应。

**PM：** 还有语音输入按钮——spec 明确说支持"点击语音按钮→说话→自动转文字→按发送"。输入框旁边要有一个麦克风图标。

**UX：** 笔记悬浮球的完整交互链在这个快照展示。需要有多个状态层：
1. 默认：右下角小圆球
2. 点击后：下拉菜单（新建笔记 / 查看笔记）
3. 新建笔记：弹出可拖拽编辑窗
4. 查看笔记：弹出笔记列表窗

这些是叠加态，在 HTML 里用 JS toggle 切换。

## 八、S08 — 对练场域 · 情景导入

### 辩论

**PM：** 进入对练场域，R2 确认了三阶段用左导航内嵌步骤条。但用户在最后确认时提出"合理利用范式，尽量减少种类"——比如对练的阶段切换可以类比授课的 SCO 目录。我们需要讨论：步骤条 vs 目录列表？

**UX：** 这两者的核心区别是：
- **目录列表**（授课）：条目数量不固定（5-15个 SCO），支持随机访问已完成项，是"内容目录"
- **步骤条**（对练/调研）：固定 3 个阶段，必须顺序推进，是"流程阶段"

如果把对练三阶段做成目录列表样式，会产生误导——学员可能以为可以跳过情景导入直接进角色扮演。

**Dev：** 但从范式统一角度，可以把步骤条做成和目录列表视觉上相近的形态——同样在左导航区域，只是：
1. 条目固定为 3 个
2. 用连接线串联（而不是独立条目）
3. 未到达阶段灰色不可点击
4. 当前阶段有"进行中"高亮

这样视觉上和 SCO 列表是"同一个容器的两种渲染模式"，用户不会觉得是完全不同的范式。

**Student：** 反思复盘阶段可以回看角色扮演的对话——这时候步骤条上"角色扮演"应该是可点击的吗？

**PM：** 不。对话回看在反思复盘的左栏内展示（"对话回看"部分），不需要切回角色扮演阶段。步骤条只表示"你当前在哪个阶段"，不是回退机制。重练用"重新练习"按钮触发。

**UX：** 那我确定方案：左导航步骤条渲染为纵向三项，视觉上和 SCO 列表在同一容器，但用连接线+阶段状态图标区分。已完成阶段可以点击回看（只是看，不是重新进入）。

**PM：** 不，已完成阶段不可点击回看。步骤条纯粹是状态指示器，不是导航。所有回看功能在反思复盘阶段的左栏内完成。

**全体：** 通过。步骤条 = 状态指示器，不可点击切换。

## 十三、S13 — 管理端 · 项目配置

### 辩论（含 R2 遗留议题：确认开营交互）

**PM：** R2 留了"确认开营的交互形式"到 R3 讨论。有三个选项：二次确认弹窗、Toggle 两帧切换、仅展示开营前。

**UX：** 我建议 **Toggle 两帧切换**——配置页默认展示"开营前"状态（可编辑），顶部有一个 Toggle（"开营前 / 开营后"），切换到"开营后"时所有表单字段置灰不可编辑，底部显示"已开营 · 2026-04-20 确认"。这样开发者能直观看到两种状态的对比。

**HR：** 我喜欢这个方案。实际使用时管理员只会看到其中一种状态——开营前可编辑，开营后只读。但在原型里用 Toggle 切换能让开发者理解两种状态的差异。

**Dev：** 点击"确认开营"按钮时需不需要二次确认弹窗？

**PM：** 需要。这是不可逆操作——一旦确认开营，学员端就开放了，配置不可修改。必须有二次确认弹窗。

**UX：** 那方案是：
1. 默认展示"开营前"状态（全部可编辑）
2. 底部[确认开营]按钮 → 点击弹出二次确认弹窗
3. 确认后 → 页面切换到"开营后"状态（全部置灰，按钮变为"已开营"）

**全体：** 通过。
