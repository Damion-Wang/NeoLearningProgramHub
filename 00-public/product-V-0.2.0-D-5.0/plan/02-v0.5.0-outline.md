# requirements v0.5.0 · 大纲（v0.4）

> **作用**：研发第一次读就理解总体设计思路，并跳转 spec/design/ 对应文件做细节
> **状态**：v0.4 草稿（基于 v0.3 + 9 项 OPT 修订，目标 ≤1000 字，顶层规划不卷细节）

---

## 1 · 愿景

**睿学（NeoLearning）** 是面向**企业培训**的 AI-Native 学习陪伴平台。本期交付**睿学培训项目（NeoLearningProgram）**。

- **对学员**：1v1 老师 Neo 在 6 个月项目周期内持续陪伴，跨大厅 + 4 场域真正掌握课程内容并迁移到真实工作行为（不止"学了"，而是"会了"）
- **对企业 / HR**：基于全过程行为数据产出**可举证的行为改变报告**，把培训从"成本中心"升级为"价值证据中心"
- **对供应商运营**：系统自动执行催学规则，Ora 解读数据，单项目运营人力 ≤ 传统项目 50%

→ 详见 spec/design/requirements §1

---

## 2 · Neo 教学方法论（第一性原理）

**Neo = 1v1 线下授课老师的 AI 孪生**——通过 **讲授 / 解惑 / 讨论 / 考核 / 练习 / 追问 / 反思** 7 个动作，**带领学生掌握内容**。Neo **有自己的教学方法论和教学立场**，不是聊天伙伴、不是百科全书、不是应声虫。

→ 详见 spec/design/common/03-ai-brand

---

## 3 · 产品底层原则（PM 决策总尺度）

PM 在做任何产品决策时拿这三把尺子量。

### 3.1 第一性 · 让 Neo 更像 1v1 老师吗？
（详见 §2）

### 3.2 懂你 · Neo 越用越懂学员
跨场域 / 跨课程 / 跨会话 都记得，不重置。学员第 6 个月的 Neo 比第 1 周更精准理解他。

### 3.3 像人 · Agent 有立场不当应声虫
每个 Agent（Neo / Ora）有**自己的画像 + 立场 + voice**——**绝不无理由表扬**，**学员错时直接指出**。Neo 按场域差异化（大厅陪伴 / 授课推进 / 对练旁观 / 调研接纳 / 报告反思），但身份一致。

→ 详见 spec/design/requirements §11 + common/03-ai-brand

## 4 · 角色与端口

### 4.1 学员端 · Neo（唯一 AI 角色）
跨大厅 + 4 场域统一身份。对练 Actor / 导演为子 Agent，**不是顶层角色**。

### 4.2 管理端 · Ora
数据解读 + 报告辅助 + 证据追溯。与 Neo **互补不隔离**：Neo 对单个学员负责，Ora 对全部学员负责，共享统一 Database。

### 4.3 端口
学员端 / 管理端两个端口。HR 与运营**统一为"管理员"**。

→ 详见 spec/design/requirements §4

## 5 · 数据模型 AOM

### 5.1 五层结构
`Project → Course → Activity（15-25 分钟）→ SCO（最小内容单元）→ Segment（自适应版本）`

### 5.2 SCO 类型（4 种）
- **SLICE** PPT 讲解
- **VIDEO** 案例视频
- **QUIZ** 互动测验（问答 / 选择 / 填空 / 判断 / 连线 / 排序）
- **FEEDBACK** 含 COLLECT + REVIEW 双子类型，memory_id 关联形成闭环

**ASSESSMENT_TAG 不是 SCO**——是**打标机制**，决定后续 SCO 选哪个 Segment（L2 自适应）。

### 5.3 Course 清单（暂用，最终待定）
demo 按"基层管理者"暂列 8 Course。**正式投产前由内容侧重新评估**。

→ 详见 spec/design/requirements §3.4 + §6 + §8.3

## 6 · 项目全生命周期（8 阶段）

### 6.1 服务 / 项目周期二元
- 服务周期 ≤ 12 月（账号活的总时长）
- 项目周期 ≤ 11 月（督促学员的时间锚）
- 二者必有 ≥ 1 月差

### 6.2 阶段 1 · 商机识别
销售在天使客户库 / 母公司客户库识别 lead。

### 6.3 阶段 2 · 试运行体验（独立小项目）
**签单前**客户用真实小项目体验睿学（5-15 名 × 2-4 周 × 2-3 Course），客户基于试运行决定是否签单。

### 6.4 阶段 3 · 客户意愿确认 + 签单
基于试运行 → 合同（项目周期 + 服务周期 + 名额 + 内容范围）→ 商品平台下单 → 创建项目实例。本期不做订阅。

### 6.5 阶段 4 · 开营组织（重点）
admin 配置项目 / 名单导入 / 角色分配 / 催学规则 / 平台个性化 / 启动会 / 确认开营。

### 6.6 阶段 5 · 学习期（重点 · 项目周期主体）
**学员**：Neo 持续陪伴 + 按 Course 学 / 自由复习 / 重练 / 笔记
**管理员**：Ora 看板 + 异常学员跟进 + 中期反馈 + 变更管理（调名单 / 调课 / 调规则）
**关键节点**：W2 首次脱训预警 → W4 中期反馈 → W12 半程 review → W20 进入结营

### 6.7 阶段 6 · 结营（重点）
结项报告（Ora 拉全程数据 + HITL 编辑 + 多模板）→ 成果汇报会（HR 主导）→ 学员收尾。

### 6.8 阶段 7 · 结营到服务期结束（重点 · 简化）
账号仍可登录看历史 + 复习。**本期不实现复杂续约 / 退课**。

### 6.9 阶段 8 · 归档（手工）
服务到期 → **手工归档** → 账号 disabled。

→ 详见 spec/design/requirements §3 + management/03-project-config

## 7 · 学员端

### 7.1 旅程
**项目期**：Neo 持续陪伴 → **项目完成**：Neo 祝贺 + 邀请回来 → **服务到期**：账号关闭。

### 7.2 大厅 Hall
- **看板**：项目概览 + 课程目录 + **热力图** + 待办 + 社区占位
- **4 库 GUI**（4 tab + 数量显示）：成果库 / 发现库 / 工具库 / 笔记库
- **Neo 大厅 Coach**：陪伴 / 跨课程辅导 / 应用咨询（**可给答案**）/ 督学

### 7.3 授课 Lecture
Neo 按 AOM 脚本结构化教学（讲授 + Quiz + 答疑），SCO 锁定 + 防挂机 + 闭环反馈。**不给答案**。

### 7.4 对练 Practice
Actor + Neo 双栏 GUI，后台导演驱动剧本。**逃生通道**：卡死时 1 次 Actor 情境提示 → 2 次 Neo 切讲授。

### 7.5 调研 Inquiry（P3）
Neo 对话访谈或 GUI 量表，三阶段：说明 → 核心调研 → 成果澄清。**不剧透**。

### 7.6 报告 Report
个人学习报告（内容回顾 + 表现复盘 + 应用计划）+ Neo 解读对话。**解读 + 反问**。

### 7.7 笔记
**悬浮球**贯穿大厅和场域，随时记录（打字 / 粘贴 / 截图）。归档于大厅笔记库。

→ 详见 spec/design/learner/

## 8 · 管理端

### 8.1 首页 Operation（看板）
**三维度**（项目 + 人员 + 内容）+ **热力图** + Ora 解读。颜色条标识健康度 + 异常自动排顶。本期不做主动推送。

### 8.2 报告生成 Report-Gen
团队报告：表单 → Ora 生成 → HITL 编辑 → 导出。多模板（中期 / 结项 / 自定义）。**持续价值追踪报告 V2 启动**。

### 8.3 配置 Config
项目信息 / 名单 / 角色 / **催学规则**（多渠道 + 4 阶段漏斗）/ 个性化 / Token 统计 P3 / 内容预览 / 确认开营。**含服务 / 项目周期二元 + 变更管理 SOP**。

### 8.4 消息 Message
管理员手动定向发送 + 发送记录（手动 + 自动统一查看）。

→ 详见 spec/design/management/

## 10 · 跨模块系统

### 10.A 架构决策（ADR 级）

#### 10.A.1 内容驱动 [ADR-004]
平台是引擎，内容是燃料。教学行为由 AOM / 对练剧本 / 调研剧本控制，平台不硬编码。

#### 10.A.2 记忆工程 [ADR-005]
三级管道：课堂内采集 → 大厅辅导引用 → 管理端汇聚。Neo / Ora 共享统一 Database。

### 10.B 实现机制

#### 10.B.1 自适应体系
**L1** 实时调整 / **L2** ASSESSMENT_TAG + Segment 切换 / **L3 / L4** 未来。

#### 10.B.2 AI 质量控制
Agent 输出前 self-check（基于 Soul）+ 反幻觉（无证据时实诚拒答）。

#### 10.B.3 举证机制
报告"高光 / 待改进"由 AI 回顾全部对话标注，可追溯到 SCO 级证据。

→ 详见 spec/design/requirements §8
