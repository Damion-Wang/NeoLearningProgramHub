## 9 · 管理端

> 本章描述管理端的整体形态 + 4 个核心模块。
> AI 角色 = Ora（详见 § 3.3）。
> 用户 = HR + 项目运营 + admin（详见 § 4.3）。
> 详细字段 / UI 细节 详见 [04-v0.5.0-backlog.md](04-v0.5.0-backlog.md)。

### 9.1 管理端总览

#### 9.1.1 管理员旅程

```
登录管理端
   ↓
进入首页（看板 Operation）
   ↓
日常巡视 → Ora 看板（被动呈现）+ Ora 问答（主动对话）
   ↓
异常学员发现 → 线下跟进
   ↓
里程碑到期 → team report 草稿生成 → HITL 编辑 → 导出
   ↓
配置变更 → program config（学员名单 / 项目周期 / 催学规则 / 平台个性化）
   ↓
（结营后）→ 状态切换为剩余服务期 → 持续可生成新报告
```

#### 9.1.2 管理端模块清单

| 模块 | 类型 | 一句话 |
|------|------|--------|
| **首页 Operation** | 监控 | Ora 看板 + Ora 问答 + 异常学员发现 |
| **团队报告生成 Team Report-Gen** | 交付 | Ora 拉数据 + 多轮对话 + HITL 编辑 + 导出 |
| **配置 Program Config** | 配置 | 项目信息 / 学员名单 / 角色分配 / 催学规则 / 平台个性化 |
| **消息 Message** | 沟通 | 手动定向发送富文本 + 发送记录 |

#### 9.1.3 入口

管理员账号登录 → 默认进入 Operation 看板。

#### 9.1.4 HR vs 项目运营 vs admin 协同

| 角色 | 主要行为模式 |
|------|------------|
| **admin** | 项目根账号——开营前配置全权 + 创建管理员账号 + 锁定服务周期 |
| **HR** | 不定期统观——看板巡视 + 阶段性 team report 生成 + 对外汇报 |
| **项目运营** | 日常执行——异常学员发现 / 学员名单变更 / 催学规则配置 / 报告生成执行 |

本期 3 类管理员**共用同一管理端**，权限差异详见 backlog（待 § 8 详细设计阶段）。

### 9.2 看板 Operation

#### 9.2.1 业务定义

管理员日常监控项目运行健康度的场域。三维度监控 + 热力图 + Ora 双交互。

#### 9.2.2 三维度监控

| 维度 | 内容 |
|------|------|
| **项目维度** | 项目整体健康度 / 进度 / 完成率 |
| **人员维度** | 学员个体进度 / 异常学员 / 活跃度分布 |
| **内容维度** | 各 Course 学习情况 / 卡点分布 / Quiz 通关率 |

#### 9.2.3 热力图

学员行为热力可视化——每个学员每日活跃度色块 / 每个 Course 学员通过率色块。

#### 9.2.4 Ora 双交互模式
| 模式 | 触发 | 行为 |
|------|------|------|
| **看板被动呈现** | 默认呈现 | 三维度数据 + 热力图 + Ora 自动生成的项目健康度解读 |
| **Ora 问答** | 管理员主动提问 | "为什么 W4 脱训突增？" / "张三这周参与度怎样？" → Ora 用证据回答 |

> **本期不做**：看板点击 → Ora 实时解释——这是未来形态（详见 § 3.3.4）

#### 9.2.5 异常学员处置

按 § 6.6.2：

- **平台层**：Ora 看板 / 问答 中发现异常学员（脱训 / 进度落后 / Quiz 频繁失败）
- **线下层**：项目运营**线下跟进**该学员（电话 / 微信 / 1on1 沟通）
- **平台不强制干预**——干预靠人 / 渠道完成

#### 9.2.6 颜色条 / 排序 / 详情页

- **颜色条标识健康度**（绿 / 黄 / 红）
- **异常自动排顶**（异常项目 / 异常学员 自动排在顶部）
- **详情页"建议动作"** —— Ora 给出建议动作（详见 § 3.3.5）

### 9.3 团队报告生成 Team Report-Gen

> 这是 § 8 的**核心模块**——管理端的主要交付能力，承担中期反馈 / 半程 review / 结项报告 / 剩余服务期内的新报告 全部生成场景（详见 § 6.7 + § 6.8）。

#### 9.3.1 业务定义

管理员产出对外可举证报告的场域——Ora 拉全程数据 + 多轮对话生成初稿 + 管理员 HITL 编辑 + 导出 PDF / Word。

#### 9.3.2 引擎核心 · 每管理员独立报告库
- **每个管理员**有**独立的报告库**——存放该管理员生成 / 编辑的全部 team report
- **里程碑触发时**给每个管理员一份**独立草稿副本**
- **本期不做协同分享 / 不做实时多人编辑** —— 需协同时由管理员**导出后线下同步**

> 这个设计简化了多人协同的复杂度——研发只需保证每个管理员的报告库独立，不需做实时协同编辑。

#### 9.3.3 prompt 模板由"关联动作 / 评估要点"驱动
- 管理员**不需要选模板，也不手写 prompt**
- 管理员在创建 team report（或里程碑配置）时**填"关联动作 / 评估要点"**（如"评估学员中期参与度 + 概念掌握度"）
- 系统据此**自动生成**对应的 prompt 模板

> 本期 v0.5.0 优先做 **"结项报告 prompt 模板"** —— 用作里程碑结营时的默认 prompt。其他模板（中期 / 半程 / 自定义）后续逐步加（详见 § 6.7.1）。

#### 9.3.4 多轮对话生成 + HITL 编辑

生成流程：

```
里程碑到期触发 / 管理员手动触发
   ↓
Ora 拉项目全程数据
   ↓
Ora 与管理员多轮对话（确认范围 / 学员维度 / 数据筛选 / 文本风格）
   ↓
Ora 生成初稿
   ↓
管理员 HITL 编辑（增删改 / 调整结构 / 改文案）
   ↓
导出 PDF / Word
```

#### 9.3.5 与里程碑的关联

详见 § 6.6.3 + § 6.7.1：

- 里程碑到期当天 → Ora 自动生成 team report 草稿（每管理员一份副本）
- 站内消息提前 3 天每日提醒管理员
- Ora 在草稿生成后即时推送通知
- 过期未处理 → 提醒保留在列表（不堆累 / 不重复推送 / 不升级渠道）

#### 9.3.6 模板含义澄清
按 § 6.7.1：

> 本节"模板"指的是 **prompt 约束的写作规则**（即给 Ora 的"写作 brief / 文章骨架 / 核心要素清单"），**不是固定文章模板或 HTML 模板**。
>
> Ora 基于模板的写作约束 + 项目实际数据 + 关联动作 / 评估要点 → 生成具体文本。

#### 9.3.7 本期不做

- ❌ **学员个人结营报告** —— 学员小结全部在阶段 5 学习期内通过 Course 个人小结 Activity 完成（详见 § 6.7）
- ❌ **多模板自由切换 UI** —— 本期仅"结项报告 prompt 模板"
- ❌ **协同分享 / 实时多人编辑** —— 见 § 9.3.2

### 9.4 配置 Program Config

> 本节给出 program config 的入口与功能模块清单。详细字段 / 必填选填 / 默认值 详见 § 4.4 / § 6.4 / § 6.5 / § 6.6 + [04-v0.5.0-backlog.md § 1](04-v0.5.0-backlog.md)。

#### 9.4.1 入口

管理端导航栏 → "配置 Program Config" → 多 tab 形态布局：

- 项目信息 tab
- 人员名单与角色 tab
- 催学规则 tab
- 平台个性化 tab
- 内容预览 tab
- 里程碑 tab（详见 § 6.6.3）
- 变更日志 tab —— **本期不做**（详见 § 6.6.4）

> tab 划分细节属于 D 板块议题（详见 backlog § 三）—— 本期暂列上述清单，最终 tab 结构待 § 9.4 详细设计阶段确认。

#### 9.4.2 配置阶段切换

按 § 6.5 关键设计：

- **开营前**：admin 可全量编辑（含特殊安全措施修改服务周期）
- **"确认开营"按钮触发**：连锁动作（账号开放 + 系统消息 + 状态切换）
- **开营后**：进入只读 + 变更管理状态——大部分字段锁定，仅允许部分变更（学员名单增减 / 项目周期 / 催学规则 / 平台个性化 / 等，详见 § 6.6.4）

#### 9.4.3 关键功能引用

| 功能 | 详细位置 |
|------|---------|
| 项目信息 字段 必填 / 选填 / 默认值 | § 6.5 |
| 人员名单批量导入 + 角色分配（多选）| § 6.5 |
| 催学规则配置（仅站内 + 模板变量 + 投递日志）| § 6.6.5 + backlog § 1.2 |
| 平台个性化（Logo / 项目名 / Neo Ora 名字 等可改）| § 6.6.6 + backlog § 1.3 |

| 全阶段功能矩阵 | backlog § 1.4 |

#### 9.4.4 本期不做

- ❌ **预览模式**（以学员身份模拟登录）—— 本期通过"内容预览"接口直接进入体验即可（详见 § 6.5）
- ❌ **变更操作日志** —— 详见 § 6.6.4
- ❌ **学员分组 / 学员标签** —— 详见 § 6.5

### 9.5 消息 Message

#### 9.5.1 业务定义

管理员对学员定向沟通的场域——支持手动发送富文本消息 + 系统自动催学消息的统一查看与跟踪。

#### 9.5.2 GUI 与交互

| 子模块 | 功能 |
|-------|------|
| **手动定向发送** | 项目运营从学员名单中**手动选人** + 编辑富文本 + 发送 |
| **发送记录** | 手动 + 自动催学消息**统一查看**——含对象 / 时间 / 内容 / 投递日志 |

#### 9.5.3 与催学规则的关系

按 § 6.6.5：

- **全员催学** = 通过催学规则自动触发（系统层）
- **异常学员定向催** = 通过本模块**手动选人**发送（运营层）
- 两者的发送记录在本模块**统一呈现**

#### 9.5.4 投递日志

- 完整记录：谁收到 / 是否查看 / 是否点击链接
- 本期**不支持学员回应**（学员无法在催学消息上回复）

#### 9.5.5 本期不做

- ❌ **邮件 / IM / SMS 渠道**（仅站内）
- ❌ **消息已读率 dashboard**（投递日志后台可查，前端不展示）
- ❌ **消息模板库 UI**（文案模板配置在催学规则模块，详见 § 6.6.5）
