## E1: 天使客户深度访谈

### 测试假设

**"2000+人大型企业的HR/培训经理确实因'无法证明培训ROI'而面临政治生存压力，并且愿意为能解决这个问题的AI产品单独付费。"** [Assumption]

### 为什么这个实验最优先

- 所有收入假设建立在客户愿意付费的前提之上 [Data Gap]
- 60家天使客户是最大的未利用资产，真实需求完全未知 [Data — confidence-dashboard]
- 成本为零（内部资源），信息密度最高

### 执行步骤

1. **筛选（Day 1-2）**：从60家天使客户中筛选15-20家，覆盖以下维度：
   - 行业分布：金融 5家、制造 4家、零售 3家、其他 3-5家
   - 企业规模：2,000-10,000人 10家、10,000-50,000人 5-10家
   - 现有绚星产品使用深度：深度用户 8家、浅度用户 7家
2. **约访（Day 2-3）**：通过绚星销售团队/CSM引荐，约30分钟1v1视频访谈
3. **执行访谈（Day 4-12）**：每天2-3场，使用标准化访谈脚本（详见experiment-design.md）
4. **数据编码（Day 13-14）**：将访谈记录按痛点维度、付费意愿、预算信号等编码分析

### 衡量指标

| 指标 | 定义 | 数据来源 |
|------|------|----------|
| 痛点共鸣率 | 主动提及或强烈认同"无法证明培训效果"的受访者比例 | 访谈编码 |
| 付费意愿率 | 明确表示"愿意单独付费购买AI培训产品"的比例 | 直接提问 |
| 预算信号强度 | 能说出具体预算范围或明确预算来源的比例 | 访谈编码 |
| 决策链复杂度 | 从培训经理到签约的平均决策层级数 | 访谈编码 |

### 验证 vs 否定标准

| 结果 | 判定 | 后续动作 |
|------|------|----------|
| **≥70%痛点共鸣 + ≥50%付费意愿** | 验证通过 | 加速推进MVP |
| **50-70%痛点共鸣 或 30-50%付费意愿** | 条件验证 | 调整价值主张定位，重新测试 |
| **<50%痛点共鸣 或 <30%付费意愿** | 假设否定 | 暂停MVP开发，重新诊断客户痛点 |

### 时间与成本

- **时间：** 2周（Day 1-14）
- **成本：** 0元（内部人力）
- **人力：** 产品负责人 + 1名售前专家，各投入约40%

## E2: Van Westendorp 定价敏感度测试

### 测试假设

**"睿学NeoLearning基管项目的账号定价在1,600-1,800元/人区间是市场可接受的。"** [Assumption — 当前定价完全未经客户验证，是最大盲区]

### 为什么排第二

- 定价是商业模式的核心假设 [Data — confidence-dashboard置信度评级"低"]
- 可与E1访谈同步执行，无额外成本
- 定价错误将直接导致收入目标1,300-1,600万不可达

### 执行步骤

1. **嵌入E1访谈**：在每场天使客户访谈的后半段加入Van Westendorp四问题
2. **问题设计**（详见experiment-design.md中的中文版本）：
   - Q1: 太便宜（质量存疑）的价格
   - Q2: 便宜（划算）的价格
   - Q3: 贵（但可接受）的价格
   - Q4: 太贵（不会考虑）的价格
3. **补充问题**：
   - "如果按项目制收费（~5万/项目，30人），vs 按账号收费（1,600元/人），您更倾向哪种？"
   - "您的部门今年培训预算中，有多少比例可用于尝试AI培训产品？"
4. **数据分析（Day 14-15）**：绘制Van Westendorp价格敏感度曲线，确定：
   - PMC（价格无差异点）
   - OPP（最优价格点）
   - 可接受价格范围

### 衡量指标

| 指标 | 定义 |
|------|------|
| 最优价格点（OPP） | "太贵"与"太便宜"曲线交叉点 |
| 可接受价格范围 | "便宜"与"贵"曲线之间的区间 |
| 定价模式偏好 | 账号制 vs 项目制 vs 混合制的偏好占比 |

### 验证 vs 否定标准

| 结果 | 判定 | 后续动作 |
|------|------|----------|
| **OPP落在1,200-2,000元区间** | 验证通过 | 在OPP附近定价，启动销售 |
| **OPP落在800-1,200元区间** | 部分验证 | 下调定价或增强高端价值包装 |
| **OPP < 800元 或 >60%受访者认为1,600元"太贵"** | 假设否定 | 重新设计定价结构（考虑按效果付费、订阅制等） |

### 时间与成本

- **时间：** 与E1同步，+1天数据分析
- **成本：** 0元
- **人力：** 复用E1团队

## E3: AI测评原型测试

### 测试假设

**"AI测评模块能产出可区分于通用完课数据的Kirkpatrick L3行为层证据，且HR认为这些证据具有汇报可信度。"** [Assumption — AI测评是核心差异化但技术验证为零]

### 为什么排第三

- AI测评是睿学核心差异化的基石 [Data — mvp-definition Red Flag]
- 如果AI无法产出可信的L3行为证据，整个产品退化为"更好的视频课"，无法支撑溢价 [Opinion]
- 需要少量开发投入，但信息价值极高

### 执行步骤

1. **设计测评框架（Week 1）**：
   - 与领域专家共同定义横向协同的3-5个核心胜任力维度
   - 设计2-3个情境模拟场景（跨部门资源争夺、目标冲突协调等）
   - 定义每个维度的评分量表（对标得助智能14维度标准 [Data]）
2. **构建测评原型（Week 2）**：
   - 基于现有AI对练模块改造，增加结构化评分输出
   - 输出样本：能力雷达图 + 行为举证数据 + 前后对比报告
3. **内部测试（Week 2末）**：
   - 10名内部人员完成测评，产出样本报告
   - 检验：L3行为数据是否可区分于简单的完课/考试数据
4. **HR可信度测试（Week 3）**：
   - 将样本报告展示给5位天使客户HR（从E1中筛选）
   - 收集反馈："这份报告能帮你向管理层汇报吗？缺什么？"

### 衡量指标

| 指标 | 定义 |
|------|------|
| L3数据可区分性 | 盲测中，专家能否区分AI测评报告 vs 通用完课报告 |
| HR可信度评分 | 5位HR对报告"汇报可用性"的评分（1-5分） |
| 数据维度丰富度 | AI评估维度数量 vs 传统考试维度数量 |

### 验证 vs 否定标准

| 结果 | 判定 | 后续动作 |
|------|------|----------|
| **盲测区分率100% + HR可信度评分≥3.5/5** | 验证通过 | AI测评列为MVP最高优先级功能 |
| **盲测可区分但HR评分2.5-3.5** | 条件验证 | 与HR共创改进报告格式，再测一轮 |
| **盲测无法可靠区分 或 HR评分<2.5** | 假设否定 | 触发Kill Criteria——重新评估产品核心定位 |

### 时间与成本

- **时间：** 3周（可与E1/E2并行启动）
- **成本：** ~2万元（领域专家咨询费 + 原型开发人力机会成本）
- **人力：** 1名AI工程师（50%）+ 1名领域专家（30%）+ 产品负责人（20%）

## E4: Concierge MVP（横向协同内容）

### 测试假设

**"使用横向协同内容的完整AI培训项目，能让学员产生真实的能力提升，并让HR获得可用于汇报的效果报告。"** [Assumption]

### 为什么排第四

- 这是最接近真实产品体验的验证，但成本最高
- 需要E1-E3的信号支持才值得投入
- Concierge模式：AI完成教学交互，人工补足未完成的自动化环节

### 执行步骤

1. **客户筛选（Week 1）**：从E1访谈中筛选3-5家高意向天使客户
2. **项目启动（Week 1-2）**：
   - 签署免费试用协议（明确：试用免费，正式版定价1,600元/人）
   - 每家企业选10-15名学员参与（非30人全量，控制风险）
3. **项目执行（Week 2-5）**：30天横向协同培训项目
   - AI授课 + AI对练：使用已有Demo/模块
   - AI辅导：如未开发完成，由领域专家人工扮演"AI辅导"角色（Concierge）
   - AI测评：使用E3原型
   - AI报告：如未自动化，由团队手工编制（基于AI数据）
   - 运营催学：人工执行（模拟AI运营模块）
4. **数据收集（全程）**：学员互动数据、完课率、能力前后测对比
5. **报告交付（Week 5）**：向每家企业HR交付完整培训效果报告

### 衡量指标

| 指标 | 目标 | 标签 |
|------|------|------|
| 学员完课率 | ≥75%（传统视频课~50%） | [Assumption] |
| 学员满意度 | ≥4.2/5.0 | [Assumption] |
| 能力前后测提升幅度 | 可量化的正向变化 | [Assumption] |
| HR报告可用性评分 | ≥4.0/5.0（"可直接用于向管理层汇报"） | [Assumption] |
| 付费转化意愿 | ≥3/5家明确表达付费意愿 | [Assumption] |

### 验证 vs 否定标准

| 结果 | 判定 | 后续动作 |
|------|------|----------|
| **完课率≥75% + HR报告评分≥4.0 + ≥3/5付费意愿** | 验证通过 | 全力推进MVP产品化 + 扩大天使客户规模 |
| **完课率60-75% 或 HR评分3.0-4.0** | 条件验证 | 定向改进弱项，再做一轮3家客户测试 |
| **完课率<60% 或 HR评分<3.0 或 0/5付费意愿** | 假设否定 | 触发Kill Criteria——评估是否需要根本性产品调整 |

### 时间与成本

- **时间：** 4周（紧凑型项目周期）
- **成本：** ~5万元（领域专家人工成本 + Concierge运营人力 + LLM推理费）
- **人力：** 产品+运营+领域专家，约3人全职投入

## E5: 首项目NPS调查

### 测试假设

**"完成首个AI培训项目的客户达到产品-市场匹配的早期信号（NPS≥40）。"** [Assumption]

### 为什么排第五

- NPS只在客户真实使用产品后才有意义
- 是对E4结果的量化确认
- 成本为零，但信息价值依赖E4的执行质量

### 执行步骤

1. **问卷设计（E4执行期间准备）**：
   - 核心NPS问题："您有多大可能向同行推荐睿学？"（0-10分）
   - 追问："推荐/不推荐的最主要原因是什么？"
   - 补充维度：
     - "培训效果可见性"单项NPS
     - "AI互动体验"单项NPS
     - "报告质量"单项NPS
     - Sean Ellis测试问题："如果明天无法继续使用睿学，您会有多失望？"
2. **发放时机**：E4项目结项后3天内
3. **双视角收集**：
   - HR/培训经理视角（决策者）
   - 学员视角（使用者）
4. **数据分析**：
   - 计算整体NPS和各维度NPS
   - 分析推荐者/贬损者的关键驱动因素
   - Sean Ellis测试："非常失望"比例

### 衡量指标

| 指标 | 目标 |
|------|------|
| 整体NPS（HR视角） | ≥40 |
| 整体NPS（学员视角） | ≥30 |
| "培训效果可见性"NPS | ≥50（核心价值主张） |
| Sean Ellis"非常失望"比例 | ≥40%（PMF信号） |

### 验证 vs 否定标准

| 结果 | 判定 | 后续动作 |
|------|------|----------|
| **HR NPS≥40 + Sean Ellis≥40%** | PMF早期信号确认 | 启动规模化销售 |
| **HR NPS 20-40 或 Sean Ellis 25-40%** | 接近PMF | 针对贬损者反馈改进，再测一轮 |
| **HR NPS<20 或 Sean Ellis<25%** | PMF不成立 | 回到E1重新诊断——可能是产品问题，也可能是客户选择问题 |

### 时间与成本

- **时间：** E4结项后1周
- **成本：** 0元
- **人力：** 产品负责人，约20%

## Flags

**Red Flags:**
- 5个实验中有3个（E1、E2、E5）成本为零——不是因为省钱，而是因为**这些信息本应在产品开发前就获取**。目前所有关键商业假设都未经客户验证即进入开发阶段。 [Data — confidence-dashboard]
- E3（AI测评）是核心差异化验证，但排在第三不是因为不重要，而是因为需要一定开发投入。如果时间紧迫，E3应与E1同步启动。 [Opinion]

**Yellow Flags:**
- E4 Concierge MVP的"人工补足"成本可能高于估计——如果自动化程度低，每个项目需要大量人工介入 [Assumption]
- 天使客户访谈可能存在社交期望偏差（"老客户碍于情面说好话"）——建议引入非绚星现有客户作为对照组 [Opinion]
