## 一、架构图解读

架构图呈现了比我们之前白板确认更清晰的模块关系：

```
┌──────────┐ ┌─────────────────────────────────────────┐  ┌──────────────┐
│ 学员看板  │ │  可交互式授课                             │  │ 行为记录     │
│ ·课程目录 │ │  ·讲述        ┌──────────┐              │  │ 默认报告     │
│ ·学习统计 │ │  ·Quiz/callback│Activity │              │  │ dashboard   │
│ ·消息中心 │ │               │ 答疑     │              │  │              │
│ ·笔记资料 │ ├──────────────────────────┤   database  │  │              │
│          │ │  对练  → Roleplay → 反思  │   ←──────→  │  │              │
│          │ ├──────────────────────────┤              │  │              │
│          │ │  测评  → 对话    → 分析   │              │  │ 解读&素材生成 │
├──────────┤ ├──────────────────────────┤              │  │              │
│ Project  │ │  个人报告       → 解析    │              │  │              │
│ 辅导     │ │                          │              │  │              │
│          │ │  ···（更多模块类型）       │              │  │              │
└──────────┘ └─────────────────────────────────────────┘  └──────────────┘

             ┌──────────────────────────────────────────────────────────┐
             │ 运营  跟 逼 催  dashboard  临时任务下发    │ 运营Agent │
             └──────────────────────────────────────────────────────────┘
```

### 架构图中的关键发现

1. **"个人报告"是第四个学习模块类型**——和授课/对练/测评平级，不是附属功能
2. **"Activity 答疑"出现在授课模块旁边**——授课中的答疑是一个独立功能块
3. **对练的三段结构更清晰**：对练 → Roleplay → 反思（三个阶段）
4. **测评的两段结构**：对话 → 分析（测+评分开）
5. **个人报告也有两段**：报告 → 解析
6. **"···"表示更多模块类型**——预留了扩展位
7. **右侧数据层**：行为记录 + 默认报告 + dashboard + 解读&素材生成（四个产出）
8. **学员看板**明确列出4项：课程目录、学习统计、消息中心、笔记资料

## 二、与我们已有文档的差异对比

### 差异1：个人报告是独立模块（新发现）

| 我们文档 | 豆包记录+架构图 |
|---------|---------------|
| 报告作为"报告智能体"的功能，归属管理端 | **个人学习报告是学员端的一个独立Activity类型**，和授课/对练/测评平级 |
| 学员只能"查看个人学习报告" | 学员可以和AI讨论报告："你为什么给我这么低分？"——有**报告解析**环节 |

**创始人原话**："个人学习报告不是固定生产在某一个位置，可以在每个Course学完后有一个个人学习报告，由Agent帮忙进行解读。"

**影响**：需要在学员端行为规格中新增"个人报告模块"，作为第四种教学模块类型。

---

### 差异2：记忆工程（全新内容，需求文档中完全缺失）

豆包记录中有大段关于记忆工程的讨论，我们的文档里只有"统一Database"这个概念，没有设计记忆如何流转。

**创始人描述的记忆工程三级管道：**

1. **课堂内记忆**：授课/对练/测评中采集的数据（学员输入的真实场景、行为表现、测评结果）
2. **大厅记忆**：课堂数据流转给辅导智能体，辅导可以引用"你在对练中说了XX"来给建议
3. **管理端记忆**：课堂+大厅数据流转给报告智能体，生成HR可用的报告

**关键痛点（内测发现）**：学员在课前认真分享了真实工作问题，但后续课程完全没有回应这些问题——"情感错付"。

**解决方案思路**：新增"闭环反馈SCO"——课前采集 → 存入记忆 → 课后用学到的知识复盘学员的真实问题。

**影响**：需要在产品需求文档中新增"记忆工程"章节，定义数据流转规则。

---

### 差异3：辅导模块的知识库不用RAG

| 我们文档 | 豆包记录 |
|---------|---------|
| 未讨论辅导模块的知识检索方式 | **明确不用RAG（向量检索）**，用"渐进式披露"的文件层级检索——每层有README，像读文件系统一样逐层匹配 |

**创始人原话**："整个这个里面是不要用到传统RAG...应该是用类似渐进式披露这样的知识库方式，每一个层级的文件夹都有一个readme。"

**影响**：这是一个技术架构决策，应记录为ADR。

---

### 差异4：学员看板细节更丰富

| 我们文档 | 架构图 |
|---------|--------|
| 学习进度、课程结构、个人笔记、可带走内容、待办任务/作业 | **课程目录、学习统计、消息中心、笔记资料** |

**差异**："消息中心"我们没有，它是接收运营下发消息和催学提醒的地方。"学习统计"比"学习进度"更丰富。

---

### 差异5：对练的三段结构在架构图中更清晰

| 我们文档 | 架构图 |
|---------|--------|
| Roleplay对话 + 苏格拉底反思 + 可完整重练 | **对练 → Roleplay → 反思**（三段明确分开，每段可能是独立SCO） |

结合豆包记录的补充："对练之前有情景导入，对练结束后有反思，你可以理解为这三个东西是不同的SCO。"

所以对练Activity下的SCO结构是：
1. SCO: 情景导入（场景说明+角色分配+任务目标）
2. SCO: Roleplay 对话
3. SCO: 反思辅导（苏格拉底追问）

---

### 差异6：测评的两段结构

| 我们文档 | 豆包记录 |
|---------|---------|
| 测评作为整体 | **"测"和"评"分开**——前面的测（对话收集/量表）+ 后面的评（对话解读，可选项） |

---

### 差异7：授课中的"小解答"细节

豆包记录补充了我们文档中的"小解答SCO"的具体行为："基于学员之前的真实问题和课程中发生过的记忆"进行解答。这要求授课模块能访问记忆工程中学员之前的输入。

---

### 差异8：闭环反馈SCO（全新概念）

**这是内测后产生的新需求**——课前采集学员真实场景，课后用学到的知识复盘。

创始人原话："一种SCO叫双端的SCO，课前获取，课后复盘，记忆工程比较直接，不至于越早聊的东西就忘掉。"

**和我们讨论过的"小测评SCO"区别**：小测评是即时的知识检查，闭环反馈是跨时间的场景复用。

---

### 差异9：项目规模更具体

| 我们文档 | 豆包记录 |
|---------|---------|
| 2000+人企业，30人/项目 | **30-40人学员 + 1-2个HR账号 + 1-2个运营账号，项目周期约6个月** |

---

### 差异10：运营端的"临时任务"更具体

豆包记录描述："比如大家一块说一下今天学习到哪些东西，用一句话金句总结"——这种非必须的临时任务由运营端下发，学员在看板消息中心接收。

创始人还提到学员提交后，运营端"可以看到所有人提交的内容"，像"单向群聊窗口"——运营能看所有人，学员只看运营的下发。

## 三、架构图与我们文档一致的部分（确认正确）

1. ✅ 辅导模块在左侧，作为主入口/大厅
2. ✅ 三种主要教学模块类型（授课/对练/测评）
3. ✅ 统一Database收集所有数据
4. ✅ 右侧管理端有行为记录、报告、Dashboard、解读&素材生成
5. ✅ 运营端功能：跟逼催 + Dashboard + 临时任务下发 + 运营Agent
6. ✅ 三端合一（学员/HR/运营共用，权限区分）
7. ✅ 半结构化内容驱动架构（KGP生产数据包）

## 五、需要和创始人确认的问题

1. 个人报告模块：是作为每个Course结束后的固定Activity，还是可在任意位置插入？
2. 闭环反馈SCO：这个是本期（6月）就要做的，还是迭代方向？
3. 记忆工程的本期范围：三级管道全做还是先做课堂→Database→报告这条单向链路？
4. 架构图中的"Activity答疑"和辅导模块的答疑是什么关系？是授课中的插嘴提问功能，还是独立的答疑Activity？
