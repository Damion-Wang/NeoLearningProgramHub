# 问题1：分段式自适应——打标SCO独立化的完整设计

## 主持人引入

创始人在R1回应中做出了一个重要的架构决策：**互动SCO和打标SCO分离**。打标SCO基于上下文判定，影响范围只到下一个打标SCO为止。一个Activity内可有多个打标点，形成"分段式自适应"。

网络调研发现：行业已出现"课程内多判定点"的设计模式。Learning Pool、Mindsmith等平台采用贝叶斯知识追踪（BKT）+ IRT逻辑，将诊断评估嵌入场景本身。Indiana州教育系统引入了全年3个检查点的机制。调研结论：分段判定点设计可行且有行业先例，建议将评估嵌入场景，采用概率模型而非简单二值判断。

我们需要讨论：打标SCO的AOM数据模型、打标结果的存储格式、多打标点的编排模式、打标如何影响对练版本选择。

---

### 课程设计师提出方案A：显式打标SCO + 规则引擎模型

我的方案是将打标SCO设计为一种独立的SCO类型，有自己的AOM数据结构，与Quiz SCO明确区分。

**打标SCO的AOM数据模型**：

```json
{
  "scoType": "ASSESSMENT_TAG",
  "scoId": "tag-001",
  "tagConfig": {
    "tagDimension": "conflict_style",
    "possibleTags": ["defensive_delay", "interest_bargain", "emotional_resist"],
    "dataSource": "preceding_scos",
    "dataSourceRange": ["sco-intro-01", "sco-quiz-01", "sco-quiz-02"],
    "tagLogic": {
      "type": "rule_based",
      "rules": [
        {
          "condition": "student mentions avoidance behavior >= 2 times",
          "tag": "defensive_delay",
          "confidence_threshold": 0.7
        }
      ],
      "fallback_tag": "interest_bargain"
    },
    "effectScope": {
      "start": "next_sco",
      "end": "next_assessment_tag_sco OR activity_end"
    }
  }
}
```

与Quiz SCO的核心区别：
- Quiz SCO有`passConditions`和`solution`，目的是评估知识掌握；打标SCO没有对错之分，目的是分类
- Quiz SCO结果产生"认知转化碎片"用于报告；打标SCO结果产生"画像标签"用于路径切换
- Quiz SCO对学员可见（答题过程）；打标SCO对学员不可见（后台判定）

**打标结果存储格式**：

```json
{
  "student_id": "xxx",
  "tag_sco_id": "tag-001",
  "activity_id": "act-01",
  "timestamp": "2026-06-15T14:35:00",
  "tag_dimension": "conflict_style",
  "assigned_tag": "defensive_delay",
  "confidence": 0.82,
  "evidence_summary": "学员在Quiz-01中选择'先拖延等领导协调'；在互动中两次提到'这种事不急'",
  "data_source_scos": ["sco-intro-01", "sco-quiz-01"],
  "effective_until": "tag-002 OR activity_end"
}
```

**典型编排模式**（一个授课Activity内）：

```
SCO-1: 闭环反馈（课前采集） → 互动，采集学员场景
SCO-2: 打标SCO-A            → 基于SCO-1判定冲突类型标签
SCO-3: PPT讲解（知识导入）   → 按标签选择Segment版本
SCO-4: 视频案例              → 按标签选择Segment版本
SCO-5: Quiz                  → 检验理解
SCO-6: 打标SCO-B             → 基于SCO-3~5表现重新判定
SCO-7: PPT讲解（进阶知识）   → 按新标签选择Segment版本
SCO-8: 小解答                → 不受标签影响
```

**对练版本选择**：对练Activity也有Segment版本管理。对练Activity在编排中读取最近一个打标SCO的标签结果，匹配对练剧本版本。

```json
{
  "activityType": "roleplay",
  "segmentSelection": {
    "readTagFrom": "tag-002",
    "versionMapping": {
      "defensive_delay": "roleplay-segment-gentle",
      "interest_bargain": "roleplay-segment-negotiation",
      "emotional_resist": "roleplay-segment-emotional"
    }
  }
}
```

### 交互设计师提出方案B：隐式嵌入式打标 + 概率累积模型

我的方案不同于显式的独立打标SCO。我认为应该将打标逻辑嵌入到已有的互动SCO中，用概率累积的方式实现更自然的自适应。

**核心理念**：调研显示最佳实践是"将评估嵌入场景本身"——学员的每一个决策都同时充当诊断数据点。我认为不需要一个独立的、停顿式的"打标SCO"，而是让每个互动SCO都携带打标权重，后台持续累积概率。

**数据模型**：

每个互动SCO增加一个可选的`tagContribution`字段：

```json
{
  "scoType": "QUIZ",
  "scoId": "quiz-01",
  "tagContribution": {
    "dimension": "conflict_style",
    "signals": [
      {
        "behavior_pattern": "选择回避类选项",
        "tag": "defensive_delay",
        "weight": 0.3
      },
      {
        "behavior_pattern": "选择直面类选项",
        "tag": "interest_bargain",
        "weight": 0.3
      }
    ]
  }
}
```

**概率累积与决策点**：

系统后台维护一个实时概率向量。每经过一个带`tagContribution`的SCO，概率向量就更新一次（类似BKT）。当概率向量中某个标签的概率超过阈值（如0.7），即确定标签。

需要版本切换的SCO设置一个`tagDecisionPoint: true`，此时系统基于当前概率向量选择Segment版本。如果概率都不够高，使用默认版本。

**优势**：
- 学员体验连续，没有"停下来被评估"的感觉
- 数据累积更丰富，每个互动都贡献信息
- 支持"越来越准"的渐进式画像
- 完全符合调研中BKT模型的最佳实践

**编排模式**：

```
SCO-1: 闭环反馈（课前采集） → tagContribution: weight 0.4
SCO-2: PPT讲解（知识导入）   → tagDecisionPoint: true → 选版本
SCO-3: Quiz                  → tagContribution: weight 0.3
SCO-4: 视频案例              → tagDecisionPoint: true → 选版本（更准了）
...
```

### 魔鬼代言人攻击

**攻击方案A**：

1. **编排复杂度爆炸**：一个Activity内如果有2个打标点，每个打标点3种标签，后续SCO就需要准备 3x3=9 个Segment版本？如果8个Course都这么做，KGP的内容制作量翻了多少倍？课程设计师说"2-3个打标点"，实际执行时KGP团队能撑住吗？

2. **打标范围的"恰好到下一个打标点"很脆弱**：如果一个Activity内只安排了1个打标点，那打标影响的范围就是"到Activity结束"——这跟课前一次测评有什么区别？分段式自适应的优势就体现不出来了。

3. **跨Activity的打标继承问题**：创始人说"影响范围只到下一个打标SCO为止"，但对练是独立的Activity，对练需要读取授课Activity的打标结果。如果打标范围只在Activity内，跨Activity怎么传递？

**攻击方案B**：

1. **概率模型的可解释性差**：当运营在后台看到"张三的defensive_delay概率是0.68"，这个数字怎么解释？0.68到底够不够确定？不同内容设计师设置的阈值不一致怎么办？

2. **冷启动问题**：第一个需要版本切换的SCO出现时，可能只有一个互动SCO贡献了信号，概率模型的数据量根本不够做出可靠判断。这时候要么用默认版本（那为什么做自适应？），要么硬猜（风险更大）。

3. **给KGP增加了认知负担**：KGP内容设计师现在不仅要写教学脚本，还要给每个互动SCO配weight值？设计师懂BKT概率模型吗？这不是把技术复杂度转嫁给了内容团队？

### 主持人综合建议

综合五个角色的观点，我建议采用**混合方案**：

**结论：方案A的显式打标SCO + 方案B的概率累积逻辑**

1. **保留显式打标SCO作为AOM层的编排单元**——KGP内容设计师在Activity编排时明确放置打标SCO，标注"此处做一次判定"。这符合"内容驱动行为"的设计哲学，让设计师控制"何时判定"。

2. **打标SCO内部采用概率累积逻辑**——打标SCO不是自己做判定，而是汇总前面所有互动SCO累积的信号，用概率模型输出标签。这意味着互动SCO的数据都会被利用，但"决策时机"由打标SCO决定。

3. **打标结果存储为确定标签 + 置信度**——输出一个明确标签（满足HR报告需求），同时存储置信度（供系统和调试使用）。

4. **低置信度降级为默认版本**——当置信度低于阈值（如0.6），使用默认Segment版本，不硬猜。打标结果标记为"default_fallback"。

5. **跨Activity的标签传递**——打标结果写入统一Database的学员标签表，对练Activity可以读取。"影响范围只到下一个打标SCO"是Activity内的规则；跨Activity时，对练读取的是"最新有效标签"。

6. **KGP不需要配概率权重**——概率模型的参数由技术团队根据互动类型设定默认值，KGP只需要在打标SCO中定义"标签维度"和"可能的标签值"。

**需要创始人决策的分歧点**：
- 打标SCO对学员是否完全透明？（即：打标SCO是否有任何前台表现？还是纯后台行为？）
- 跨Activity的标签传递规则——"影响范围只到下一个打标SCO为止"是否只适用于Activity内？

# 问题2：三栏布局的联动设计

## 主持人引入

创始人确认：三栏不是主辅关系，而是**默认占比差异+联动协作**。教室也是三栏（不是全屏）。R1发现的核心矛盾是：对话密集场景（Quiz/Roleplay/BEI）中间看板利用率低。创始人回应"不是问题——对练有角色形象，测评有背景信息"。

但交互设计师仍有疑虑：不同场景的默认比例未定义、看板区和对话区的具体联动机制未明确。调研发现Sana的"上下文感知侧边栏"、ShadCN的可折叠导航+多区域布局是值得参考的模式。

### 课程设计师提出方案A：场景预设比例 + AOM控制看板内容

我的方案是在AOM数据模型中增加看板内容控制字段，让内容设计师精确控制每个场景下中间看板展示什么。

**不同场景的默认三栏比例**：

| 场景 | 左栏(目录) | 中栏(看板) | 右栏(对话) | 看板内容 |
|------|-----------|-----------|-----------|---------|
| 大厅（首次进入） | 15% | 50% | 35% | 学习看板（进度/课程/待办） |
| 大厅（日常） | 15% | 45% | 40% | 学习看板 |
| 授课-PPT讲解 | 10% | 55% | 35% | PPT课件 |
| 授课-视频 | 10% | 60% | 30% | 视频播放器 |
| 授课-Quiz对话 | 10% | 40% | 50% | **Quiz关联知识卡片 + 当前进度指示** |
| 对练-情景导入 | 10% | 55% | 35% | 场景描述+角色信息卡片 |
| 对练-角色扮演 | 10% | 35% | 55% | **角色形象+情绪指示+技能提示卡** |
| 对练-反思复盘 | 10% | 50% | 40% | **对话记录回看面板** |
| 测评-对话式 | 10% | 40% | 50% | 背景信息+已采集信息摘要 |
| 测评-量表式 | 10% | 60% | 30% | 量表GUI |
| 报告 | 10% | 55% | 35% | 报告内容+BI图表 |
| HR端 | 15% | 50% | 35% | 管理看板 |

**AOM中增加看板控制字段**：

```json
{
  "scoType": "QUIZ",
  "kanbanConfig": {
    "defaultContent": "knowledge_card",
    "knowledgeCardRef": "card-pin-method",
    "progressIndicator": true,
    "interactiveElements": [
      {
        "type": "tip_card",
        "trigger": "student_stuck_3_rounds",
        "content": "提示：回忆一下PIN法的三个步骤"
      }
    ]
  }
}
```

这样Quiz对话时中间看板不是空白，而是显示与当前Quiz相关的知识卡片、技能提示等辅助信息。

**联动设计**：

看板区 → 对话区（点击触发）：
- 学员点击看板中的"对练第3轮对话"→ 右侧Agent自动生成该轮的分析："这一轮你使用了直接表达策略，效果是..."
- HR点击看板中"张三的能力雷达图" → Agent自动解读："张三在冲突处理维度表现最好，因为..."

对话区 → 看板区（结果联动）：
- Agent在对话中说"你在第5轮的沟通技巧很好" → 看板中第5轮对话高亮标记
- HR问Agent"谁的进度最落后？" → Agent回答后，看板进度表自动高亮该学员

### 交互设计师提出方案B：响应式弹性布局 + 上下文感知面板

我的方案强调弹性和自适应。不是在AOM中硬编码看板内容，而是建立一套通用的响应式规则。

**核心理念**：参考Sana的"上下文感知侧边栏"——看板面板能感知当前对话的上下文，自动展示最相关的信息。

**弹性布局规则**：

```
规则1：当对话区活跃度 > 阈值（如连续3轮对话），自动将比例调整为 10-30-60
规则2：当看板区有新内容加载（如PPT翻页），自动将比例调整为 10-55-35
规则3：用户手动拖拽调整后，锁定用户设置，不再自动调整（直到场景切换）
规则4：场景切换时（如从PPT讲解进入Quiz），重置为该场景的默认比例
```

**上下文感知看板**：

不需要AOM逐个定义看板内容。建立一套通用的看板组件库，系统根据当前场景自动组装：

| 看板组件 | 触发条件 | 内容 |
|---------|---------|------|
| 知识卡片面板 | 当前SCO关联了知识点 | 展示相关知识点摘要 |
| 进度指示器 | 任何Activity中 | 当前SCO位置/总数 |
| 角色信息面板 | 对练场景 | 角色形象+当前情绪状态 |
| 对话回看面板 | 反思复盘场景 | 可选中某轮对话触发Agent分析 |
| 空状态提示 | 无可展示内容时 | "专注对话中" + 简要提示文案 |
| 参考技巧面板 | Quiz/对练中学员连续回答不佳 | 动态展示相关技巧提示 |

**联动机制**（与方案A类似，但更通用）：

```typescript
// 看板 → 对话 联动
onKanbanElementClick(element) {
  if (element.type === 'dialogue_round') {
    agent.triggerAnalysis(`分析这一轮对话的沟通策略`);
  }
  if (element.type === 'chart_data_point') {
    agent.triggerInterpretation(`解读这个数据点`);
  }
}

// 对话 → 看板 联动
onAgentMention(reference) {
  kanban.highlight(reference.targetElement);
  kanban.scrollTo(reference.targetElement);
}
```

**反思复盘时的"对话记录回看"组件**：

这是一个通用看板组件，在反思复盘阶段自动挂载到中间看板区：
- 左侧显示对话轮次列表（缩略）
- 点击某轮展开完整对话内容
- 点击"请AI分析这一轮"按钮 → 右侧Agent触发定向分析
- 支持按"学员发言"/"AI发言"过滤
- 关键轮次（Agent认为有教学意义的）自动标记

**攻击方案A**：

### 交互设计师提出方案B：轻量记忆标签 + AI自主关联

我的方案减少AOM层面的复杂配置，更多依赖AI的自主理解能力。

**课前采集数据的存储**：不做过度结构化，用"标签+原文"的轻量模式：

```json
{
  "memory_id": "mem-001",
  "student_id": "xxx",
  "source": "pre_course_feedback",
  "course_id": "横向协作",
  "tags": ["跨部门冲突", "排期争夺", "资源分配"],
  "summary": "与市场部争夺排期，尝试直接拒绝和找领导协调但效果不佳",
  "raw_text": "[原始对话]",
  "timestamp": "2026-06-15T10:00:00"
}
```

**课中引用：不用引用槽，用System Prompt指令**

不在AOM中预设引用槽，而是在授课AI的System Prompt中加入通用指令：

```
当你讲解知识点时，如果学员的记忆库中有与当前知识点相关的真实场景，
请自然地将该场景融入讲解作为案例。不要生硬引用，要让学员感觉
"这个知识点就是在说我的问题"。
如果没有匹配的场景，使用脚本提供的通用案例。
```

AI通过渐进式文件检索（ADR-006）自主在记忆中找到相关场景，自行决定何时引用。

**课后闭环：由辅导教练在大厅触发**

课后闭环不一定在教室内完成。学员完成授课Activity后回到大厅，辅导教练主动发起复盘对话：

"你刚学完了PIN法。还记得你之前提到和市场部争排期的事吗？用今天学的方法，你觉得可以怎么处理？"

这样做的好处：
- 复盘在辅导对话中进行，更自然（不是教室内的"考试"感）
- 辅导教练有跨模块的完整上下文，复盘质量更高
- 不需要在AOM中定义复杂的"双端SCO"

**辅导教练引用规则：基于AI理解，不硬编码触发条件**

辅导教练的System Prompt包含通用引用策略：

```
你可以引用学员在教室中的学习记忆。引用规则：
1. 学员回到大厅时：主动提及最近完成的Activity中的亮点或待改进点
2. 学员提到某话题时：如果记忆中有相关内容，自然引用
3. 不要每次都引用——如果学员想聊新话题，跟着学员走
4. 引用时要具体，不要泛泛说"你之前学过的"
```

### 魔鬼代言人攻击

**攻击方案A**：

1. **引用槽的覆盖率问题**：KGP设计每页PPT的脚本时，都要考虑"这里能不能插一个引用槽"？8个Course、上百个PPT SCO，有多少适合引用学员场景的？实际上可能只有10-15%的SCO适合引用。其余85%的引用槽配置就是浪费。

2. **双端SCO的技术复杂度**：一个SCO分成pre和post两个阶段，中间隔着若干其他SCO。这在执行引擎层面意味着什么？SCO的状态管理变得复杂——pre阶段完成后SCO状态是"进行中"还是"待完成"？断点续播时如何恢复？

3. **触发规则的过度工程化**：方案A定义了三种触发条件、三种模板、还有概率和匹配逻辑。辅导教练的记忆引用真的需要这么精确的规则吗？LLM本身就擅长根据上下文决定是否引用——给它记忆数据和一条通用指令就够了。过度规则化反而限制了AI的自然性。

**攻击方案B**：

1. **AI自主关联的不可控性**：调研中发现"AI自主发挥讲解质量无法监控"是P0问题（虽然本期不处理）。现在又让AI自主决定引用什么记忆？如果AI在讲"冲突处理"时引用了学员在"目标拆解"课前分享的一个无关场景，谁来发现和纠正？

2. **课后复盘在大厅的定位问题**：如果课后复盘由辅导教练在大厅发起，那闭环反馈SCO的"课后端"就不是AOM控制的了——它变成了辅导模块的行为。这违反了"内容驱动行为"的设计哲学。辅导教练的复盘追问策略、问什么问题、到什么深度，全靠AI自行判断。R1已经指出"反思复盘缺少AOM脚本控制"是P2问题。

3. **如果AI不引用怎么办？** 方案B的System Prompt是"可以引用"，不是"必须引用"。如果AI某天状态不好（prompt太长、token截断、注意力分散），直接跳过了引用，学员又一次"情感错付"。没有兜底机制。

### 主持人综合建议

**建议方案**：保证闭环的结构化链路 + 自然的引用体验

1. **课前采集用方案A的结构化标签格式**——这是HR报告的数据基础。采集后立即进行结构化提取，存储为`structured_tags` + `raw_dialogue`双层格式。

2. **课中引用用"必选引用槽+可选引用槽"的混合模式**：
   - KGP在关键知识点处设置少量（每Activity 1-2个）**必选引用槽**（`required: true`）——AI必须在这些位置引用学员场景
   - 其他位置AI可自主判断是否引用——给System Prompt通用指令即可
   - 这解决了"引用一定发生"（必选）和"引用要自然"（自主）的矛盾

3. **课后复盘保留在AOM中（方案A路线）**——闭环反馈SCO的课后端在AOM中有明确位置和脚本控制。这保证了复盘质量的可控性，符合"内容驱动行为"的设计哲学。但可以简化为"同一Activity的最后一个SCO"，不需要"双端SCO"的复杂设计。

   即：闭环反馈拆成两个独立SCO：
   - `FEEDBACK_COLLECT` SCO：放在Activity开头，负责课前采集
   - `FEEDBACK_REVIEW` SCO：放在Activity末尾，负责课后复盘
   两者通过记忆ID关联，但各自是独立的SCO。

4. **辅导教练的记忆引用用简单规则**：
   - 学员回到大厅时：**必然触发**一次最近学习的引用（不用概率）
   - 学员提到相关话题时：AI自主判断是否引用（System Prompt指令）
   - 不做复杂的条件触发树

5. **对练场景匹配用方案A的版本映射**——打标结果 + 记忆标签共同决定对练Segment版本选择。

**需要创始人决策的分歧点**：
- 课后复盘是在AOM中作为独立SCO（保留在教室内），还是由辅导教练在大厅发起？这影响"内容驱动"的纯粹性。
- 课中引用的"必选引用槽"数量——每Activity 1-2个是否足够？还是每个核心知识点都需要？

# 问题4：报告中的项目内排名设计

## 主持人引入

创始人确认：项目内排名（30-40人）。R1发现的问题是：第一批客户没有常模，排名基础缺失。

调研发现：30-40人的传统排名在统计上存在问题——个体间差异可能不具统计显著性，排名波动大。美国多州采用的SGP（学生成长百分位数）模型是成熟的替代方案——不与所有人比绝对分数，而是与"学术同伴"比成长幅度。AMBOSS的同伴对比功能允许学员与特定同伴群体进行表现对比。

### 课程设计师提出方案A：多维成长雷达 + 分组定位

我的方案是不用传统的单一排名，而是用多维度的成长型展示。

**排名维度设计**：

不做单一总分排名，做3个层次的定位：

**层次1：能力维度雷达图（个人报告核心）**

每个Course有3-5个能力维度（如横向协作课的维度：利益识别、情绪管理、方案协商、关系维护），每个维度用1-5分的标准化量尺。

展示方式：
- 个人雷达图：自己各维度的得分
- 叠加项目均值线：自己 vs 全体均值
- 不展示具体排名名次

**层次2：成长幅度定位（SGP变体）**

参考SGP模型，但简化：不计算复杂的"学术同伴组"，而是在30-40人内按**成长幅度**分三组：

| 分组 | 定义 | 展示文案 |
|------|------|---------|
| 高成长组 | 成长幅度 Top 30% | "你的进步速度在项目中名列前茅" |
| 稳健组 | 成长幅度 Middle 40% | "你保持了稳定的学习节奏" |
| 潜力组 | 成长幅度 Bottom 30% | "你还有很大的提升空间" |

文案积极化——"潜力组"而非"落后组"。

成长幅度 = 课后能力评分 - 课前基线评分（如果有BEI基线）或 = 后半段对练评分 - 前半段对练评分。

**层次3：HR端项目聚合视图**

HR能看到但学员看不到的：
- 全员能力维度分布热力图
- 成长幅度排名完整名单
- 各维度的Top 3和需关注的Bottom 3

**在AOM中的定义**：

```json
    "rankingDimensions": [
      {
        "dimensionName": "利益识别",

        "baselineMeasure": "pre_course_bei OR first_roleplay",
        "growthCalc": "latest_score - baseline_score"

      ]
    }
  }
}
```

### 交互设计师提出方案B：个人成长时间线 + 匿名分位标尺

我的方案从用户体验出发，强调**个人纵向对比**优先于**横向同伴对比**。

**核心组件：个人成长时间线**

在报告中间看板，用时间线可视化展示学员从Course 1到Course 8的成长轨迹：

```
能力评分
  5 |                                    ●
  4 |                        ●   ●
  3 |            ●   ●   ●
  2 |    ●   ●
  1 |●
    +---+---+---+---+---+---+---+---→ Course
      1   2   3   4   5   6   7   8
```

每个数据点可点击展开详情（联动看板→对话区：AI解读"这个阶段你提升最快是因为..."）。

**匿名分位标尺**（附加在时间线上）：

在每个数据点旁边用一个灰色的分位标尺展示"你在项目内的位置"：

```
  ▓▓▓▓▓▓▓▓▓░░░  你的位置: 前35%
  |----|----|----|
  前25%  中50%  后25%
```

不展示具体名次（不说"你排第11"），只展示分位区间。标尺用柔和的灰色调，不抢主信息（成长曲线）的视觉焦点。

**BI图表设计**：

| 图表 | 用途 | 交互 |
|------|------|------|
| 折线图（主） | 个人各维度能力评分变化曲线 | 点击数据点→Agent解读 |
| 分位标尺（附） | 项目内定位（匿名） | 悬浮显示分位数值 |
| 雷达图 | 最新一次的多维度综合展示 | 叠加均值线 |
| 小提琴图（HR端） | 全员某维度的分数分布 | 识别双峰/偏态 |

**避免负面效应的设计细节**：
- 学员报告中永远不显示具体排名名次
- 用分位区间替代排名：前25% / 中间50% / 后25%
- 文案用"你的成长速度"而非"你的排名"
- HR端可以看到完整排名，但学员端不可见

### 魔鬼代言人攻击

**攻击方案A**：

1. **成长幅度的基线问题**：成长幅度 = 课后 - 课前。但课前基线怎么测？如果用BEI，R1说了BEI"是否V1做由内容决定"。如果某些Course没有课前BEI，成长幅度怎么算？用"第一次对练评分"当基线？第一次对练学员还不熟悉系统，分数天然偏低，导致"成长幅度"被人为放大。

2. **三组分法的"天花板效应"**：能力评分用1-5分，本身就有天花板。初始评分高的学员（如4分），提升空间只有1分；初始评分低的（如2分），提升空间有3分。成长幅度天然偏向初始差的学员，对初始好的学员不公平。

3. **"潜力组"的透明度问题**：张三说的对——30人的小圈子，分组信息藏不住。如果有学员知道自己在"潜力组"并且不满，投诉HR"你们的AI系统歧视我"，怎么办？

**攻击方案B**：

1. **匿名分位的统计意义**：30人的前25%就是前7-8个人。学员看到"你在前25%"的反应是"那我大概第7名左右"。想匿名根本匿不了。

2. **个人成长时间线需要跨Course数据**：时间线展示Course 1到Course 8的趋势。但能力维度是按Course定义的（横向协作的维度 ≠ 辅导反馈的维度）。不同Course的评分怎么放在同一条曲线上？需要一个跨Course的统一能力维度体系——当前AOM没有定义。

3. **两个方案都回避了一个问题**：HR真正想要的是**绝对排名**——"这30个人谁最好谁最差"。所有"成长幅度""分组""分位"的美化设计，到了CEO面前可能就被一句"直接告诉我排名"打回来。

---

### 主持人综合建议

**建议方案**：学员端成长优先 + HR端排名透明 + 统一维度体系

1. **学员端报告**：
   - **主信息**：个人成长时间线（方案B）——纵向展示自己每个维度的进步
   - **辅助信息**：匿名分位区间（前25%/中间50%/后25%）——轻量的横向定位
   - **不展示**：具体排名名次、分组标签
   - **文案基调**：全部用成长型语言（"你在XX维度的进步幅度超过了项目中大多数同伴"）

2. **HR端报告**：
   - 完整的成长幅度排名（可排序表格）
   - 各能力维度的全员均分、分布图（柱状图或热力图，不用小提琴图）
   - 聚合统计："平均成长幅度"、"最大提升维度"、"需重点关注学员（成长幅度最低的3人）"
   - HR可以看到每个学员的分组（高成长/稳健/潜力），但这是HR端的分析工具，不是学员端的展示

3. **成长幅度计算**：
   - 优先使用"课前BEI基线 vs 课后评分"
   - 如果没有BEI基线，使用"前1/3学习阶段的平均表现 vs 后1/3学习阶段的平均表现"
   - 引入"成长幅度标准化"——消除初始分数对成长幅度的偏置（高起点学员的小幅提升和低起点学员的大幅提升具有可比性）

4. **跨Course的统一维度体系**：
   - 在Project级定义3-5个**通用能力维度**（如"沟通影响力""冲突管理""目标执行"）
   - 每个Course的细分维度映射到通用维度（横向协作的"利益识别"映射到"沟通影响力"）
   - 时间线展示通用维度的跨Course趋势

5. **BI图表选型**：
   - 学员端：折线图（成长趋势）+ 雷达图（多维度）+ 分位标尺
   - HR端：柱状图（各维度排名）+ 热力图（全员行为分布）+ 进度矩阵表

**需要创始人决策的分歧点**：
- HR端是否直接展示绝对排名？还是只展示成长幅度排名？
- 学员端的分位信息是否展示？（张三提出30人圈子里分位也藏不住）
- 是否需要定义Project级的通用能力维度体系？

### 课程设计师提出方案A：分层触发 + 结构化碎片生产规则

我的方案是为记忆工程的每个环节定义明确的触发规则和数据格式。

**辅导教练引用课堂记忆的触发规则**：

三层触发机制：

```
层级1：必然触发（每次）
- 触发条件：学员完成一个Activity后首次回到大厅
- 动作：辅导教练主动提及刚完成的Activity中最突出的1-2个表现
- 模板："你刚完成了[Activity名称]。[正面评价/观察点]。有什么想聊的吗？"
- 实现：从Database读取该Activity最新的能力评估碎片

层级2：条件触发（话题相关时）
- 触发条件：学员在辅导对话中提到的话题，与某条课堂记忆的capability_tags匹配
- 匹配逻辑：语义相似度 > 阈值 OR 标签精确匹配
- 动作：辅导教练引用该记忆
- 示例：学员说"我今天和同事吵架了" → 匹配到"冲突处理"标签 → 引用对练中的冲突处理表现

层级3：主动回顾（定期）
- 触发条件：学员连续3次回到大厅都没有提及学习内容
- 动作：辅导教练主动进行一次学习回顾
- 模板："我注意到你最近在[课程]的学习中有些有趣的发现，要不要聊聊？"
```

**纯讲述SCO（NARRATION）的数据采集**：

纯讲述SCO确实不产生互动数据。但需求文档说"授课是互动式的"——所以即使是PPT讲解，也应该有互动环节。

解决方案：**不存在"零互动"的SCO**

- PPT讲解SCO：即使学员不主动提问，AI也会在讲解后追加一句"关于这个知识点，你有什么想法？"——这产生了一条互动数据
- 如果学员选择跳过（比如直接点"下一页"），记录"学员跳过互动"——这本身就是学习行为数据（参与度低信号）
- 纯视频SCO：播放完毕后AI自动发起一句简短提问或反应征询

这样每个SCO都至少产生一条行为记录，报告的"学习表现"模块不会缺数据。

### 魔鬼代言人攻击

**攻击方案A**：

1. **碎片生产的AI成本**：每个SCO完成后都要生成碎片——30个学员 x 每人每天完成5-10个SCO = 150-300次碎片生成。每次碎片生成是一次AI推理调用（分析对话、提取标签、生成摘要）。这个调用量是否在GPU预算内？

2. **"不存在零互动SCO"的强制设计违反用户预期**：学员选择安静听课是一种合理的学习行为。强制在每个SCO后追加互动环节，可能把5分钟的PPT讲解延长到7-8分钟。8个Course积累下来，总学时增加了20-30%。

3. **三层触发机制的测试复杂度**：层级1必然触发、层级2条件触发、层级3主动回顾——三层之间会不会冲突？比如学员回到大厅（触发层级1），然后立刻提了一个问题（触发层级2），教练连续发两条引用消息？还是层级1被层级2覆盖？

**攻击方案B**：

1. **被动信号的信息量太低**：停留时长和点击行为只能判断"人在不在"，不能判断"学没学会"。一个学员挂机3分钟和认真听讲3分钟的被动信号完全一样。报告"学习表现"模块如果主要依赖被动信号，数据质量会很差。

2. **事件驱动模型的"遗漏"风险**：方案B的事件模型是穷举式的——列了4个事件就只处理这4种情况。如果出现第5种情况（比如学员在大厅对话中突然分享了一个新的工作困境），没有对应事件处理，这条记忆就丢了？

3. **不打通跨Course记忆是"偷懒"**：创始人在需求文档中多次强调"教练了解你的一切"（§5.2核心体验原则第3条）。如果教练只记得当前Course的内容，这个承诺就是打折的。"先验证单Course"只是延迟问题，不是解决问题。

### 主持人综合建议

**建议方案**：结构化碎片 + 被动信号辅助 + 跨Course记忆打通

1. **辅导教练引用触发规则**：
   - **必然触发**：学员完成Activity首次回到大厅时，教练主动提及。限制为1-2句话，不啰嗦。
   - **条件触发**：学员对话中提到相关话题时，AI自主判断是否引用（System Prompt指令，不硬编码匹配规则）。
   - **去掉"主动回顾"层级**——学员在大厅发呆不应该被打扰。如果需要回顾，由学员主动发起。
   - **防冲突规则**：必然触发和条件触发不会在同一消息中重叠——如果学员回到大厅后立刻说话（触发条件层），跳过必然触发，优先响应学员的话题。

2. **纯讲述SCO的数据采集**：
   - **不强制互动**——纯讲述SCO允许学员安静听完。
   - **采集被动信号**（方案B）：停留时长、是否中途离开、是否回翻、是否点击工具卡片。
   - **在关键知识点处用Quiz SCO保证互动数据**——这是KGP的内容设计职责，不是平台的强制行为。
   - **报告"学习表现"模块的数据来源优先级**：Quiz互动数据 > 对练行为数据 > 测评数据 > 辅导对话数据 > 被动信号数据。纯讲述SCO的被动信号作为补充。

3. **碎片生产规则用方案A的结构化模板**——但优化触发时机：
   - "学习特质碎片"：不是每个SCO生成一个，而是每个**Activity**完成后生成一个汇总碎片（减少AI调用次数）。
   - "认知转化碎片"：每个Quiz闭环后生成（保持不变）。
   - "能力评估碎片"：每次对练完成后生成（保持不变）。

4. **半结构化记录格式**：采用方案A的详细格式（含dialogue_turns、intent标注），但增加方案B的passive_signals字段。两者合并为最终格式。

5. **跨Course记忆打通**——本期打通（方案A路线）。理由：
   - "教练了解你的一切"是核心体验承诺
   - 技术上统一Database天然支持
   - 引用优先级：当前Course > 相关Course > 远期Course
   - 不做复杂的权重衰减——简单的时间排序即可（优先引用最近的记忆）

**需要创始人决策的分歧点**：
- 纯讲述SCO是否在讲解结束后追加一句互动引导？（方案A认为应该追加，张三认为不应该）
- 碎片生产粒度：每个SCO一个碎片 vs 每个Activity一个碎片？（影响AI调用成本和报告数据细度）
- 跨Course记忆打通是否本期实现？

# 全局总结

## 五个问题的建议方案一览

| # | 问题 | 建议方案 | 方案简述 |
|---|------|---------|---------|
| 1 | 分段式自适应打标设计 | 混合方案（A结构+B逻辑） | 显式打标SCO + 概率累积判定 + 低置信度降级 |
| 2 | 三栏布局联动设计 | 预设比例+组件库 | 场景预设比例 + 通用看板组件库 + 双向联动 |
| 3 | 闭环反馈数据链路 | 结构化+自然引用 | 结构化采集 + 必选/可选引用槽 + 独立SCO（非双端） |
| 4 | 报告排名设计 | 学员成长+HR排名 | 学员端成长时间线+分位 / HR端完整排名+聚合 |
| 5 | 记忆触发时机 | 分层触发+被动信号 | 必然+条件触发 / 被动信号辅助 / 跨Course打通 |

## 需要创始人决策的7个分歧点

1. **打标SCO对学员是否完全透明**——是纯后台行为，还是有前台表现？
2. **跨Activity的标签传递规则**——"影响范围只到下一个打标SCO为止"仅限Activity内？
3. **Quiz对话时中间看板展示什么**——关联知识卡片？还是其他？
4. **课后复盘在教室内（AOM控制）还是大厅（辅导教练发起）**？
5. **HR端是否展示绝对排名**——还是只展示成长幅度排名？
6. **纯讲述SCO是否追加互动引导**？
7. **跨Course记忆是否本期打通**？
