# 第一部分：这是什么

---

## 1. 产品定义

### 1.1 一句话定义

AI 老师是一个面向企业培训场景的 AI 原生学习陪伴系统——通过"辅导大厅 + 多间教室"的空间模型，让 30-40 名基层管理者在 6 个月周期内，获得 1v1 AI 教练的持续陪伴、结构化课程的自适应学习、沉浸式对练的行为训练，最终产出可举证的行为改变报告。

### 1.2 核心价值主张

- **对学员**：不再是被动"刷课"，而是有一个懂你的教练随时在身边，遇到真实问题随时问、推荐你去学最需要的课、对练中陪你反复练
- **对 HR**：不再拿"课程完成率"交差，而是拿到基于全过程数据的行为改变证据报告，可以直接向 CEO 汇报
- **对运营**：不再靠人工一个一个催学、手工拼报告，而是系统自动执行可配置的运营规则、数据呈现、一键生产报告

### 1.3 产品比喻

> 平台是钢琴，内容是乐谱，AI 是演奏者。
>
> 同一架钢琴可以演奏不同曲目。KGP（内容制作工具）生产的 AOM 数据包就是乐谱，Prosona 执行引擎按照乐谱演奏，学员听到的是完整的音乐体验。更换乐谱（内容包），就能交付完全不同主题的培训项目。

### 1.4 与传统培训的关系

AI 老师覆盖 OMO 混合培养模式中的线上环节，替代原来"人工硬混"的多系统拼接。不覆盖线下集训、售前 POC、签单等线下环节。

### 1.5 核心设计哲学

以下设计哲学贯穿全产品，指导所有功能设计：

1. **AI 不做"判官"**——所有学员输入都有价值，不判断真假。例如 BEI 中学员编造的案例，也是学习状况的体现
2. **内容驱动行为**——平台不硬编码教学逻辑，所有教学行为由内容脚本控制
3. **看板区与对话区联动**——三端统一三栏布局，看板区默认占比更大，但看板区和 Agent 对话区双向联动（如点击看板数据可触发 Agent 解读，Agent 建议可联动看板高亮）
4. **LUI 优先于 GUI**——在新手引导等场景，优先用对话式交互（LUI）而非传统 GUI 元素

### 1.6 模块描述

产品定义模块确立了整个系统的定位和价值。AI 老师不是一个课程播放器，而是一个可编排的 AI 原生培训项目交付引擎。它通过自适应的 AI 动作序列（授课/辅导/测评/对练/萃取），将企业培训从"不可验证的课程完成"升级为"可举证的行为改变"。

### 1.7 用户故事

- **作为学员**，我希望 AI 教练了解我的真实工作困境，能基于我的具体情境给出建议，以便我获得个性化的辅导
  - AC: Given 学员描述工作困境，When AI 回复，Then 引用学员描述的具体细节（人名/场景/情绪），不使用泛化回答
- **作为学员**，我希望 AI 教练能根据我的学习进度和能力画像推荐最适合我的下一个学习任务，以便我的学习路径是个性化的
  - AC: Given 学员回到大厅，When 教练推荐任务，Then 推荐基于学员当前进度和未完成任务
- **作为学员**，我希望在学习结束后能感受到自己在工作中的行为有了改变，以便培训不仅是知识传递
  - AC: Given 学员完成一个 Course，When 查看报告，Then 至少展示3个具体行为改变证据
- **作为 HR**，我希望随时了解培训项目的整体进度和每个学员的个人进度，以便我对项目状态心中有数
  - AC: Given HR 登录，When 打开进度看板，Then 显示整体完成率 + 每个学员进度百分比 + 最后登录时间
- **作为 HR**，我希望通过 Agent 一句话就能生成向 CEO 汇报的 PDF 报告，以便我不需要手工制作汇报材料
  - AC: Given HR 说"帮我生成本月报告"，When Agent 完成，Then 产出包含进度/效果/需关注学员的 PDF
- **作为运营**，我希望系统自动执行日常运营工作（催学、数据展示、报告生成），我只需要关注异常情况，以便同时管理多个培训项目
- **作为 KGP 内容设计师**，我希望我制作的脚本和剧本能精确控制 AI 的教学行为，以便保证每个学员获得高质量的学习体验
- **作为产品负责人**，我希望产品架构支持内容与平台解耦，更换内容包就能交付不同主题的培训，以便一次平台建设支撑多条业务线

### 1.8 已确认决策

- 第一版定位为"项目版"而非"企业版"——每个项目有明确的起止周期（约6个月）
- 产品覆盖线上环节，不覆盖线下集训、售前等

### 1.9 本期不做的功能

- 企业版功能（多项目并行管理、企业级权限体系）——本期聚焦单项目交付
- AI 讲解质量监控——AI 自主发挥讲解时每个学员听到不同版本，本期不建立质量审核/抽检机制。原因：授课模块使用半预制 SCO 偏离空间小，先依赖 System Prompt 中的知识点断言清单做基础预防（Round-02 R1 确认）
- 行为改变 ROI 数据链路——系统内表现（对练评分、Quiz 成绩）到真实工作场景行为改变之间的数据链路本期不打通。原因：辅导模块对工作行为的探索已经比传统竞品多很多，完整 ROI 链路留待后续迭代（Round-02 R1 确认）

## 2. 角色

### 2.1 四个角色

| 角色 | 归属 | 核心诉求 | 与系统的关系 |
|------|------|---------|------------|
| **学员** | 客户企业 | 解决真实工作问题，完成培训任务 | 学习者，系统的核心用户 |
| **HR** | 客户企业 | 向 CEO 汇报培训成果，验证投入产出 | 项目发起人和验收者 |
| **运营** | 供应商（绚星团队派出） | 高效管理项目全程，保证交付质量 | 项目交付执行者 |
| **KGP（内容制作）** | 供应商 | 生产高质量的课程内容包 | 不是软件端口，是内容供应链 |

### 2.2 角色关系

```
KGP 生产内容包 ──→ 运营配置项目 ──→ 学员学习体验
                                    ↑
                           HR 发起项目需求
                           HR 全程查看进度和报告
                           运营全程监控和干预
```

### 2.3 核心功能详述

**学员**：通过辅导大厅和多间教室完成学习旅程，获得 1v1 AI 教练陪伴。学员是系统最核心的用户——所有功能设计都以学员学习体验为中心。

**HR**：30-40人规模暂无学员组/部门分组的划分。HR 关注的是整体项目进度和个人维度的学习效果。HR 可以切换为学员身份体验课程。

**运营**：本期运营端的 Agent 与 HR 端的 Agent 保持一致，不做独立的运营 Agent。运营的核心工作是项目配置、进度监控、催学管理和报告生产。

**KGP**：不是软件端口，而是内容供应链。KGP 通过专门的生产工具制作 AOM 数据包，包含所有教学内容、剧本、脚本。

### 2.4 交互细节

- 学员首次登录后进入辅导大厅，AI 教练直接通过对话自我介绍（不使用 overlay 引导层）
- HR 登录后进入三栏布局的管理界面，左侧目录导航，中间管理看板，右侧 Agent 对话区
- 运营登录后进入同样的三栏布局，看到运营相关的功能入口
- 所有角色通过同一 Web 应用的权限控制区分入口

### 2.5 用户故事

- **作为学员**，我希望有一个 AI 教练了解我的真实工作困境，能随时回答我的问题，以便我在实际工作中做出改变
- **作为 HR**，我希望随时了解每个学员的学习进度和效果，以便我向管理层汇报时有据可依
- **作为运营**，我希望在一个界面上完成项目配置、进度查看、催学和报告生成，以便我高效管理项目
- **作为 KGP 内容设计师**，我希望通过编写脚本就能精确控制 AI 的教学行为，以便我不需要修改代码就能调整教学体验
- **作为 HR**，我希望能切换为学员身份亲自体验课程，以便我理解学员的实际学习感受

### 2.6 已确认决策

- 30-40人规模暂无学员组/部门分组的划分——HR 端不需要"销售部门""技术部门"等分组概念
- 本期运营端 Agent 和 HR 端 Agent 保持一致，不做独立运营 Agent
- 隐私不是核心问题——企业培训场景下学员对数据可见性有预期

### 2.7 本期不做的功能

- 学员分组/部门划分——30-40人规模暂时不需要
- 独立运营 Agent——本期与 HR Agent 保持一致，未来根据需求独立化

## 3. 项目生命周期

### 3.1 全流程时间线

```
线下售前 → POC → 签单 → [产品覆盖范围开始] 
  → 开营选人 → 任务下发 → 线上学习（6个月持续）
  → 实时线上辅导 → HR 随时查报告 → 线下集训（人工）
  → 运营写成果 → 结项报告 → [产品覆盖范围结束]
  → 线下颁奖/汇报
```

### 3.2 四阶段泳道

| 阶段 | KGP | 运营 | HR | 学员 |
|------|-----|------|-----|------|
| **开营前** | 制作全部课程内容（SCO + 多版本 + 对练剧本 + 测评剧本 + 辅导知识库） | — | — | — |
| **开营** | — | 导入名单、配置项目、任务下发、通知学员 | 提供参训名单（线下） | 收到通知、首次登录 |

| **结项** | — | 报告智能体批量生产报告、整理成果、输出文件包 | 看报告、Agent 解答、生产汇报报告 | 查看个人学习报告、与 AI 讨论报告 |

**泳道变更说明**：

### 3.3 项目规模

- 单项目学员数：30-40 人
- HR 账号：1-2 个
- 运营账号：1-2 个
- 项目周期：约 6 个月
- 第一个内容主题：基层管理者（8 个 Course）

## 4. 三端总览

### 4.1 三端合一 Web 应用 [ADR-001]

学员端、HR 端、运营端是同一个 Web 应用，通过权限控制不同角色看到的入口和功能。HR 可以切换为学员身份体验课程。

### 4.2 三端统一布局 [ADR-007]

三端（学员/HR/运营）的界面布局统一为三栏结构：

```
┌────────────────────────────────────────────────────────────────┐
│  [目录区(可折叠)]  |  [看板区(GUI, 默认宽)]  |  [对话区(Agent, 宽度可调)]  │
│                    |                        |                      │
│  · 项目导航        |  · 学习进度看板         |  · AI 教练对话        │
│  · 课程目录        |  · 课程内容看板         |  · Agent 辅助         │
│  · 功能入口        |  · 管理功能看板         |  · 宽度可调           │
│                    |                        |                      │
│  ← 可隐藏折叠      |  ← 主要交互区域         |  ← 宽度可拖拽调整     │
└────────────────────────────────────────────────────────────────┘
```

**布局核心原则**：
- **看板区与对话区联动**——中间看板区默认占比更大，右侧对话区宽度可调。两区双向联动：看板中的数据可点击触发 Agent 解读，Agent 的分析结果可联动看板高亮/跳转对应数据
- 类似 Claude Code 的布局——左侧导航 + 中间主内容 + 右侧对话
- 对话区宽度可调——用户可以根据需要拖拽调整对话区宽度
- 左侧目录区可隐藏折叠——需要更大看板空间时折叠左侧导航

**三栏默认比例定义（左:中:右）**：

为每种场景设定固定的默认三栏比例（不做自动弹性调整），用户可手动拖拽微调。**手动调整仅在当前场景内有效，切换场景时恢复该场景的默认预设值**（Round-02 R3 创始人决策）：

| 场景 | 左:中:右 | 设计意图 |
|------|---------|---------|
| 大厅 | 15:50:35 | 看板主导，辅导对话常驻 |
| 授课-PPT | 10:55:35 | 课件展示为主，对话辅助 |
| 授课-视频 | 10:65:25 | 视频全宽展示优先 |
| 授课-Quiz | 10:40:50 | 对话互动加重，看板展示题干+辅助文件 |
| 对练-导入 | 10:55:35 | 情景信息卡片为主 |
| 对练-演绎 | 10:35:55 | 角色对话加重，看板展示角色形象+场景 |
| 对练-复盘 | 10:50:40 | 对话回看面板为主，对话辅助分析 |
| 测评-对话 | 10:40:50 | 对话式测评以右侧为主 |
| 测评-量表 | 10:60:30 | 量表 GUI 为主 |
| 报告 | 10:55:35 | 报告内容展示为主 |
| HR/运营端 | 15:50:35 | 管理看板为主 |

**看板↔对话双向联动机制**：
- **看板→对话**：看板元素（数据点、图表、卡片等）可点击，点击后触发右侧 Agent 自动解读该元素（学员不需要手动问"这个数据什么意思"）
- **对话→看板**：Agent 对话中提到的实体（知识点、数据指标、学员姓名等）在看板中自动高亮或跳转到对应位置

**三端的看板区内容不同**：
- 学员端中间：学习看板（进度/课程/笔记/工具/待办）
- HR 端中间：管理看板（项目进度/报告/数据）
- 运营端中间：运营看板（项目配置/数据展示/催学/报告工厂）

**三端的对话区角色不同**：
- 学员端右侧：辅导教练（1v1 陪伴辅导）
- HR 端右侧：HR Agent（项目查询/报告生产）
- 运营端右侧：Agent（本期与 HR Agent 一致）

- **作为学员**，我希望在大厅左侧目录区通过多层折叠导航快速找到要学的课程，以便我高效定位学习内容

- **作为学员**，我希望在大厅中间看板看到进度、待办、笔记和工具卡片，以便我对学习状态一目了然
  - AC: Given 学员在大厅，When 查看看板，Then 至少显示进度（百分比/环形图）+ 推荐任务卡片 + 待办列表
- **作为学员**，我希望在大厅右侧对话区随时向辅导教练提问，以便我随时获得帮助

- **作为 HR**，我希望拖动对话区的宽度来调整界面比例，在需要仔细看数据时缩小对话区、在需要深入问 Agent 时放大对话区，以便灵活使用界面
- **作为学员**，我希望进入教室时界面切换为教学模式（左侧显示课程目录、中间显示课件、右侧显示AI对话），以便我专注于当前学习任务
- **作为学员**，我希望看板上的数据点可以点击后让右侧 Agent 自动解读，以便我不需要手动问"这个数据什么意思"
- **作为学员**，我希望 AI 对话中提到数据指标时看板自动高亮或跳转到对应位置，以便我直观看到 AI 在说什么
  - AC: Given AI 对话中提到指标，When 看板中存在该指标，Then 看板自动高亮对应元素，高亮持续至少5秒
- **作为学员**，我希望切换教室场景时三栏比例自动恢复为该场景的默认值，以便每次进入场景都有最优的视觉布局

- **作为学员**，我希望左侧目录折叠时中间和右侧自动扩展填充空间，以便我获得更大的内容展示区
  - AC: Given 学员折叠左侧目录，When 折叠完成，Then 中右两栏按比例重新分配空间

### 4.6 已确认决策

- [ADR-007] 三端布局统一为三栏结构：左侧目录/导航（可隐藏折叠）| 中间看板区（GUI，默认占比更大）| 右侧对话区（Agent，宽度可调）
- 看板区和对话区双向联动——不是主辅关系，而是默认展示占比的差异+联动协作
- 学员端教室也是三栏布局，但三栏内容随教室类型变化（授课/对练/测评/报告各不同）
- 三栏默认比例为 11 种场景预设固定值（Round-02 R2 确认），用户可手动拖拽微调，不做自动弹性调整
- 看板↔对话双向联动为本期必做（Round-02 R2 确认）：看板元素点击→触发 Agent 解读；Agent 提到的实体→看板高亮/跳转
- **三栏比例手动调整仅在当前场景内有效，切换场景时恢复该场景的默认预设值**（Round-02 R3 创始人决策）——不做 Session 内保持，每次切换都回默认
- 左侧目录折叠时，中间和右侧按比例重新分配空间（Round-02 R3 新增）

- 移动端适配——本期仅做桌面 Web 端
- 三栏布局的个性化配置保存——未来可记住用户偏好的列宽

## 5. 学员端空间模型：大厅 + 教室

### 5.1 空间隐喻

学员端采用"游戏大厅 + 教室"的空间模型——大厅是学员的"家"，教室是进入完成任务的地方。大厅和教室都是三栏布局，但三栏的内容不同。

```
┌─────────────────┐     进入教室        ┌────────────────────────┐
│                 │ ──────────────→    │                        │
│   大厅(三栏)    │                     │    教室（三栏，内容切换）  │
│  目录|看板|对话  │  ←──────────────    │  授课室 | 对练室        │
│                 │     回到大厅         │  考场   | 报告室        │
│                 │     或直达下一间     │  ...更多教室（未来）    │
└─────────────────┘                     └────────────────────────┘
```

### 5.2 核心体验原则

1. **大厅永远可以回来**——无论在哪间教室，随时可以返回大厅找教练
2. **教室之间可以直达**——完成一间教室后，可以直接进入下一间，不必经过大厅
3. **教练了解你的一切**——教室里发生的事，教练都知道，回到大厅时教练能引用你在教室里的表现
4. **辅导是主入口但不是强制路径** [ADR-003]——学员可以通过辅导对话进入教室，也可以通过看板课程目录直接进入
5. **所有 Activity 支持断点续播和重新学习**——中途退出后可以从断点继续，也可以选择重新学习/重新练习

### 5.3 大厅布局（三栏模式）

**左侧：目录区（可折叠）**

| 目录项 | 内容 |
|--------|------|
| 课程目录 | Project 完整结构导航（多层折叠：Course → Activity → SCO） |
| 学习进度 | 简要进度指示 |
| 消息中心 | 未读消息数量标识 |

**中间：学习看板（主交互区）**

| 看板组件 | 内容 | 交互 |
|---------|------|------|
| 学习进度 | 个人完成情况（百分比/环形图） | 查看详情 |
| 课程内容 | 当前推荐的学习任务和课程卡片 | 点击进入教室 |
| 消息中心 | 运营下发消息、催学提醒、系统通知 | 查看/标记已读 |
| 笔记资料 | 个人学习笔记（AI 自动生成 + 手动记录） | 查看/编辑/导出 |
| 工具卡片 | 可带走的实用工具（学习卡片、表格模板、工具文本），清晰展示，支持查看和下载 PDF | 查看/下载/分享 |
| 待办任务 | 当前待完成的任务列表 | 点击直接进入 |

**大厅看板相关用户故事（Round-02 R3 新增）**：
- **作为学员**，我希望在消息中心查看催学提醒和运营通知，以便我不会错过重要信息
- **作为学员**，我希望查看 AI 自动生成的学习笔记，并能手动添加自己的笔记，以便我积累学习成果
- **作为学员**，我希望下载工具卡片（PDF 格式），以便我在工作中随时参考使用
- **作为学员**，我希望催学提醒频率不过高且内容有价值（告诉我来了有什么好处），以便提醒是有帮助的而不是骚扰

**右侧：辅导对话（Chat UI，宽度可调）**

1v1 AI 教练对话界面，长期陪伴关系。

### 5.4 进入教室的三种方式

1. **AI 推荐**：辅导教练在对话中推荐下一个学习任务，学员点击任务卡片进入
2. **自主选择**：学员在左侧目录区或中间看板的课程目录中浏览，点击任意 Activity 进入
3. **自动衔接**：上一间教室完成后，系统自动展示"下一个任务"卡片，学员点击直接进入下一间教室

### 5.5 教室的通用行为

- **内容切换**：进入教室时 0.3 秒过渡动画，三栏布局保持不变，但三栏内容从大厅模式切换为教室模式
- **顶部悬浮导航**：显示"返回辅导"按钮 + 当前任务名称
- **隐藏课程目录**：教室内有可展开的隐藏课程目录，多层折叠显示（Course → Activity → SCO），学员可以在教室内直接导航到任意位置，不必回到大厅
- **断点续播**：所有 Activity 支持断点续播——中途退出后再次进入时从上次位置继续。**断点续播遇到后台 SCO（如 `ASSESSMENT_TAG`）时自动跳过，从上次有交互的 SCO 之后继续**（Round-02 R3 创始人决策）
- **重新学习**：学员可以选择重新开始当前 Activity（对练重新练习 = 仅重练 Roleplay 阶段，不含前导的情景导入）。**重新学习时保留原始学习数据存档，新旧记录独立，不覆盖也不追加**（Round-02 R3 创始人决策）
- **完成后衔接**：教室最后一屏自动显示"下一个任务：[名称]，预计 [X分钟]"+ "开始"按钮（主要动作）+ "回到辅导教练"链接（次要动作），衔接信息包含下一个任务的简要描述和预估时长

### 5.6 交互细节

- 进入教室：用户点击任务卡片 → 0.3秒过渡动画 → 三栏内容切换为教室模式
- 返回大厅：点击顶部"返回辅导"按钮 → 0.3秒过渡动画 → 恢复三栏布局
- 断点续播：系统自动记录用户离开教室时的位置 → 再次进入时弹出"从上次位置继续"或"重新开始"的选项
- 教室内导航：点击顶部隐藏目录图标 → 展开课程目录侧边栏 → 点击目标位置 → 跳转并关闭目录

### 5.7 用户故事

- **作为学员**，我希望大厅像一个"家"，左侧有课程导航，中间有我的学习看板，右侧教练随时可以对话，以便我在学习过程中不感到孤单
- **作为学员**，我希望在教室里学习时可以打开课程目录直接跳转到其他课程，以便我按需学习而不是必须按顺序
- **作为学员**，我希望完成一个任务后能直接开始下一个，不需要每次都回到大厅，以便我保持学习的连贯性
- **作为学员**，我希望中途退出教室后再次进入时能从上次位置继续，以便我不用重新学习已完成的部分
- **作为学员**，我希望对练结束后如果想重新练习，只需要重练角色扮演部分而不用重看情景导入，以便我高效地反复练习核心环节
- **作为学员**，我希望断点续播时看到进度提示（上次位置和时间），以便我快速回忆上次学到哪里
  - AC: Given 学员断点续播，When 进入教室，Then 提示包含上次位置（SCO名/页码/轮次）和离开时间
- **作为学员**，我希望重新学习不覆盖之前的记录，以便我可以对比前后两次的学习表现
  - AC: Given 学员选择重新学习，When 完成后查看记录，Then 历史记录完整保留，新记录独立存储
- **作为学员**，我希望断点续播后 AI 记住之前的上下文（包括课前采集的内容），以便对话衔接自然
  - AC: Given 学员断点续播，When 继续与 AI 对话，Then AI 可引用断点前的 FEEDBACK_COLLECT 等内容
- **作为学员**，我希望教室进出有平滑过渡动画，以便空间切换感受自然
  - AC: Given 学员进入/退出教室，When 界面切换，Then 有0.3秒过渡动画，不出现白屏或闪烁

### 5.8 已确认决策

- 所有 Activity 支持断点续播 + 重新学习/重新练习
- 对练的"重新练习"= 仅重练 Roleplay 阶段，不含情景导入前导
- 断点恢复策略统一：从断点继续（不区分中断时长）
- 断点续播遇到后台 SCO（如 `ASSESSMENT_TAG`）时自动跳过，从上次有交互的 SCO 之后继续（Round-02 R3 创始人决策）
- 重新学习时保留原始学习数据存档，新旧记录独立——不覆盖、不追加（Round-02 R3 创始人决策）

## 6. HR 管理端结构

### 6.1 核心能力

| 能力 | 描述 |
|------|------|
| 学习进度看板 | 查看所有学员的学习进度、完成率、活跃度（个人维度） |
| 项目数据看板 | 呈现学习进度数据，供 HR 自行判断（本期不做智能评价） |
| 报告查看 | 查看个人学习报告、项目整体报告 |
| HR Agent | 通过对话式 Agent 随时询问项目情况、生产汇报素材 |
| 汇报报告生产 | Agent 辅助生成汇报报告（本期仅支持报告格式，不支持 PPT） |
| 体验课程 | 切换为学员身份，亲自体验课程内容 |

### 6.2 HR Agent 的交互模式

HR Agent 是一个对话式智能体，贯穿学习期和结项期：

1. HR 提出需求："帮我准备给 CEO 的月度汇报报告"
2. Agent 确认模板、重点维度、特殊要求（记忆 HR 历史偏好，越来越少问）
3. 生成初稿 → 在线预览 → HR 标注修改 → Agent 修改 → 确认 → 下载 PDF
4. 数据可追溯——CEO 追问某个数字时，可以点进去看原始记录

HR Agent 不使用"销售部门""技术部门"等分组概念（30-40人规模暂无学员组划分）。Agent 的查询维度以个人和整体两个视角为主：

## 9. 大厅：辅导模块

### 9.1 模块描述

辅导模块是学员端的主界面和主入口 [ADR-003]。它模拟一个长期陪伴的 1v1 AI 教练，了解学员的所有学习历史，能回答工作中的真实问题，推荐合适的学习内容。辅导模块位于三栏布局的右侧对话区。

经市场验证：学员对正式培训的抵触越来越多，而陪伴式辅导是学员真实需要的——有问题就能问，问的是真实发生的问题。辅导模块产生的对话和洞察，对学员感知和 HR 端结果留存都更有价值。

**辅导模块 vs 授课答疑的区分**：
- **辅导**：大厅里的陪伴式辅导教练，范围覆盖整个项目的所有知识内容，长期陪伴关系
- **答疑**：授课 Activity 内的简单答疑（小解答 SCO），范围限于当前 Activity 内的知识内容和聊天记录

### 9.2 辅导教练的角色定位

辅导智能体的角色优先级：
1. **首先是辅导**——陪伴式回答学员的真实工作问题
2. **其次才是督学**——推荐任务、提醒进度
3. **还要教会用户**——引导学员学会使用平台工具、知道能问什么问题

### 9.3 辅导教练人格

本期辅导教练人格固定（名称、语气风格、人格特质由团队统一设定），但架构上按 Context 包隔离设计：
- 人格设定以独立的 Context 包形式存在，与核心逻辑分离
- 本期使用固定的 Context 包，假装是 KGP 生产的
- 未来 KGP 按此标准生产不同人格的 Context 包，实现人格可替换
- 这样做的目的是为未来不同项目使用不同辅导教练人格打下架构基础

### 9.4 辅导教练的核心行为

#### 首次进入（LUI 思维）

不使用 overlay 引导层——这是 GUI 的新手引导思路。LUI 该怎么做？直接辅导老师自我介绍。

- 用户进入系统后，右侧对话区直接显示辅导教练的自我介绍消息
- AI 第一条消息（自然亲切）：自我介绍 + 简要说明能做什么 + 当前有哪些学习任务
- 对话区域中自然嵌入可点击的任务卡片（标题 + 预计时长 + 进度状态）
- 左侧看板和中间看板同时展示，学员可自由探索
- 学员可选择：(A) 直接点任务卡片进入教室；(B) 在对话中打字互动；(C) 在看板自行浏览
- **核心原则：AI 主动但不阻塞——辅导对话 always available，不是 mandatory path**

#### 再次进入

- AI 消息："欢迎回来。你上次完成了[X]，今天的任务是[Y]。" + 任务卡片
- 不再重复自我介绍

#### 所有任务完成后

- AI 消息："你目前的任务都完成了。有什么想聊的吗？或者可以看看你的学习笔记。"
- 中间看板高亮笔记和可带走内容

#### 日常辅导对话

- 学员提出工作问题时，教练基于学员上下文给出针对性建议——不推回"通用理论"
- 学员说"我们公司刚裁了30%"→ 教练不说"我的建议基于通用管理理论"→ 教练说"裁员后团队焦虑的情况下，正向反馈的关键不是表扬业绩，而是认可留下来的人的情感付出。比如你可以这样说..."
- 如果学员问到 AI 确实无法回答的企业内部信息，坦诚告知并提供管理视角参考
- 教练引用学员在教室中的表现："你在对练中三次成功使用了共赢策略，说明你已经掌握了PIN法的核心"

#### 辅导教练引用规则（Round-02 R2+R3 确认）

辅导教练引用学员学习记忆的触发规则明确定义如下：

- **必然触发**：学员每次回到大厅时，教练都主动提一句刚学的内容（1-2 句话，简短不啰嗦）（Round-02 R3 创始人决策：每次都提，不只是完成 Activity 后）。例如："你刚学完的 PIN 法三步走，在你之前提到的跨部门协调场景中特别实用。"
- **条件触发**：日常辅导对话中，当话题与学员之前学习的内容相关时，AI 自主判断引用（通过 System Prompt 指令控制）
- **防冲突规则**：学员回大厅后如果立刻说话，优先响应学员的话题，然后再找合适时机提及刚学内容（不跳过，延迟触发）（Round-02 R3 矛盾修正：将"跳过"改为"延迟"）
- **"不打扰"的含义**：教练回大厅时的必然引用是开场白性质，发出后如果学员不回应则不再追问——"不打扰"指的是不反复追问沉默的学员，而非不主动发起引用（Round-02 R3 矛盾2修正）

#### 跨 Course 记忆引用（Round-02 R2+R3 确认）

辅导教练的记忆范围本期打通跨 Course 边界，教练可以引用学员在所有 Course 中分享的内容，实现"教练了解你的一切"的承诺。

**引用优先级**：
1. **当前 Course**：最优先引用学员在当前正在学习的 Course 中的记录
2. **相关 Course**：其次引用与当前话题主题相关的其他 Course 中的记录（主题相关性优先于时间近度）
3. **远期 Course**：最后引用时间较远或主题关联度较低的 Course 中的记录

采用简单时间排序，本期不做复杂权重衰减算法。

**跨 Course 引用的自然性要求（Round-02 R3 新增）**：
- 教练跨课程引用时必须自然融入对话，使用学员的具体场景描述而非课程名称。例如："你之前处理市场部合作时..."，不说"根据横向协作课程记录..."
- 跨课程引用优先选择与当前话题最相关的内容，而非时间最近的无关记录

#### 知识检索方式 [ADR-006]

辅导教练的知识库不使用传统 RAG（向量检索），而是采用渐进式披露的文件层级检索方式：
- 知识库按主题分层级目录
- 每个层级有 README 文件，描述该层包含的内容索引
- AI 像读文件系统一样逐层匹配，从顶层 README 开始，逐步深入找到相关内容
- 检索过程可追溯、可调试，不是黑盒

### 9.5 教练的感知能力

辅导教练可以感知和引用：

| 数据来源 | 感知内容 | 引用方式示例 |
|---------|---------|------------|
| 授课模块 | Quiz 回答情况、学习时长、互动内容、所有对话记录 | "你在横向协作的PIN法测试中答对了3道题" |
| 对练模块 | 对练表现、角色扮演中的行为、反思结论 | "你在对练中提到的方案，其实可以用在现在这个场景" |
| 测评模块 | 测评结果、学员画像、能力评估 | "根据你的测评，你在冲突处理方面有很好的基础" |
| 辅导历史 | 之前聊过的问题、分享过的困境 | "你上周提到的跨部门协作问题，解决了吗？" |

### 9.6 交互细节

- 辅导对话在三栏布局的右侧，宽度可调
- 对话区底部有输入框，支持文字输入和语音输入（未来）
- AI 回复中可以嵌入任务卡片、知识卡片等可交互元素
- 学员点击任务卡片后三栏内容切换进入教室模式
- 对话历史保留，回到大厅后可以继续之前的对话

### 9.7 用户故事

- **作为学员**，我希望首次进入系统时辅导教练直接自我介绍并告诉我该做什么，而不是弹出一个引导弹窗让我点"下一步"，以便我自然地开始学习
- **作为学员**，我希望辅导教练了解我在课堂中的表现，能主动引用我说过的话来给建议，以便我感受到"AI 记得我"
- **作为学员**，我希望在辅导对话中问到的工作问题，教练能给出基于我的具体情境的建议，以便我获得的不是通用答案而是实用指导
- **作为学员**，我希望调整对话区的宽度，在需要专注看看板时缩小对话区、在需要深入对话时放大对话区，以便灵活使用界面
- **作为学员**，我希望辅导教练能区分我在课堂中的问题（简单答疑）和大厅里的问题（深入辅导），给我不同深度的回应，以便我在不同场景获得合适的帮助
- **作为学员**，我希望辅导教练在我每次回到大厅时都简短提及刚学的内容（1-2句话），而不是长篇总结，以便我快速获得反馈而不觉得啰嗦
- **作为学员**，我希望辅导教练记得我在所有课程中分享的内容（不只是当前课程），以便我不需要重复讲述之前分享过的工作问题
- **作为学员**，我希望教练跨课程引用时自然融入对话（说"你之前处理市场部合作时..."而不是"根据横向协作课程记录..."），以便我感受到教练真正了解我
- **作为学员**，我希望教练跨课程引用时优先选择与当前话题最相关的内容，而非时间最近但无关的记录，以便引用有价值
- **作为学员**，我希望教练发出必然引用时如果我先说话了，教练优先响应我的话题，之后再找合适时机提及刚学的内容，以便对话节奏自然

## 10. 教室一：授课模块

### 10.1 模块描述

授课模块是 AI 老师执行结构化教学的教室。AI 按照 KGP 预制的授课脚本（AOM 数据包中的 scoFlow），依次演绎一系列教学内容：PPT 讲解、案例视频播放、Quiz 互动、小解答、知识回顾。

授课模块的核心理念是"结构化演绎"——教学节奏由内容设计师预设，AI 按照脚本的 blueprint（教学设计意图）和 rule（交互规则）执行，但在每一步互动中保持自适应的灵活性。

**授课是互动式的**——不是单向的视频播放。授课中的所有对话都要记录（半结构化），包括学员的提问、AI 的回答、Quiz 的答题过程等。

### 10.2 SCO 类型

每个 Activity（学习活动）由多个 SCO（最小学习内容单元）组成，SCO 类型包括：

| SCO 类型 | AOM scoType | 交互方式 | 描述 |
|---------|------------|---------|------|
| **PPT 讲解** | `NARRATION` | AI 根据提示和学员状态自主发挥 | AI 根据 PPT 上的知识要点和教学提示，结合学员当前状态自主讲解——不是照着念 PPT。纯讲述 SCO 需要加入与内容教学相关的互动（不能过分泛化），具体互动设计由 KGP 在内容层面解决 |
| **案例视频** | `VIDEO` | 自动播放 | 播放预制的场景案例视频（当前为伪视频格式，未来可能是 MP4） |
| **Quiz 互动** | `QUIZ` | AI 出题 + 学员回答 | 多种类型：引发思考、带入场景、检验理解、策略应用 |
| **小解答** | `QA` | 对话式 | AI 基于学员在当前 Activity 内的真实问题和聊天记录进行解答 |
| **闭环反馈-课前采集** | `FEEDBACK_COLLECT` | 对话式 | 通过对话获取学员在某主题下的真实工作场景和困境，存入记忆工程（Round-02 R2 新增独立 SCO 类型） |
| **闭环反馈-课后复盘** | `FEEDBACK_REVIEW` | 对话式 | 课后引导学员用所学知识重新拆解和解决自己的真实问题，在 Activity 末尾完成（Round-02 R2 新增独立 SCO 类型） |
| **打标判定** | `ASSESSMENT_TAG` | 纯后台（学员无感知） | 独立的打标 SCO，汇总前序互动 SCO 的累积信号进行标签判定，影响后续 Segment 版本选择（Round-02 R1/R2 新增） |

**PPT 讲解的自主发挥机制**：
- AI 拿到的是一页 PPT 图片 + 教学提示（blueprint）+ 交互规则（rule）
- AI 不是照着念 PPT 上的文字，而是根据提示理解教学意图，结合学员当前状态（之前的回答、画像标签、学习进度等）自主组织讲解语言
- 这意味着同一页 PPT，面对不同学员，AI 的讲解方式、举例、深度可能不同
- 教学提示控制的是"方向"（该讲什么、不该讲什么、重点在哪），AI 决定的是"怎么讲"

**纯讲述 SCO 的互动要求（Round-02 R2 确认）**：
- 纯讲述 SCO 需要加入互动，但要找到平衡点——之前加了太碎，去掉太死板
- 核心要求：互动必须与当前内容教学相关，不能过分泛化（如"你觉得呢？"这种万能追问不行）
- 这是内容设计层面的解法，而非平台层面——KGP 在 SCO 脚本中设计与知识点直接相关的互动问题
- 学员不被强制互动——如果学员选择安静听完，其参与度通过被动信号（停留时长、是否回翻、是否点击工具卡片等）被采集

**授课脚本引用槽（Round-02 R2 确认）**：
- AOM 授课脚本新增 `referenceSlots` 字段，用于在关键知识点处设置"引用槽"
- 引用槽分为必选（`required: true`）和可选（`required: false`）两种：
  - **必选引用槽**：AI 必须在此处引用学员的真实场景（来自闭环反馈课前采集的记忆），确保闭环反馈不被遗漏
  - **可选引用槽**：AI 根据上下文自主判断是否在此处引用学员场景
- KGP 在关键知识点设 1-2 个必选引用槽，其余位置 AI 自主判断

### 10.3 Quiz 交互类型（本期）

- **问答题**：开放式回答，AI 苏格拉底式启发引导（最多 5 轮）
- **选择题**：单选/多选
- **判断题**：对错判断
- **连线题**：概念配对

**Quiz 看板展示（Round-02 R2 确认）**：
- Quiz 对话时，中间看板展示**题干 + 辅助文件**（图片、表格等）
- 辅助图片支持点击放大查看（模态框展示，支持缩放和关闭）（Round-02 R3 新增）
- 小视频作为辅助文件本期不做，留待后续迭代
- 看板默认组件：关联知识卡片 + 进度指示

**Quiz 答题反馈（Round-02 R3 新增）**：
- 学员 Quiz 答完后，看到 AI 的反馈和解析——不只是对错判断，而是解释为什么对/为什么错
- 学员不知道答案时可以说"不知道"或"给我个提示"，AI 给出引导性提示而非直接告诉答案

### 10.4 闭环反馈 SCO

本期做完整版。这是内测后产生的核心需求——解决"情感错付"问题。

**问题背景**：内测中，学员在课前认真分享了真实工作困境，但后续课程完全没有回应这些问题——"那么认真地把我的事情都告诉你了，结果一点都没给我回应"。

**解决方案（Round-02 R2+R3 更新）**：闭环反馈拆分为两个独立 SCO 类型（不做"双端 SCO"复杂设计），通过 `memory_id` 关联：

1. **`FEEDBACK_COLLECT` SCO（课前采集）**：放在 Activity 开头，通过自然对话（非问卷格式）获取学员在某主题下的真实工作场景和困境，存入记忆工程。AI 使用开放式提问引导分享，不使用封闭式问卷格式（Round-02 R3 新增）
2. **`FEEDBACK_REVIEW` SCO（课后复盘）**：放在 Activity 末尾，调取学员之前输入的场景（通过 `memory_id` 关联，确保精确匹配），引导学员用所学知识重新拆解和解决自己的真实问题。课后复盘在 Activity 内完成，不在大厅进行（Round-02 R2 确认）

**学员拒绝分享的处理（Round-02 R3 创始人决策）**：
- 学员在课前采集中拒绝分享时的处理方式**由脚本控制**——不同课程可以有不同的处理策略
- KGP 在 `FEEDBACK_COLLECT` SCO 的脚本中定义拒绝分享时的行为（如最多追问2次后跳过、必须获取至少1个场景、直接跳过等）
- 平台不硬编码统一的拒绝处理规则

**课前采集记忆存储模板（Round-02 R2 确认）**：

采集的学员场景数据采用双层存储格式：
- **`structured_tags` 层**：结构化标签，包含 `topic_dimension`（话题维度）、`scenario_type`（场景类型）、`scenario_keywords`（场景关键词）、`emotional_state`（情绪状态）、`specific_challenge`（具体挑战）等字段
- **`raw_dialogue` 层**：原始对话记录，保留完整的对话上下文

双层存储确保既有结构化字段便于检索和匹配，又保留原始信息的完整性。

闭环反馈 SCO 与普通 Quiz 的核心区别：
- Quiz 是独立的测评工具，数据不主动关联后续学习
- 闭环反馈 SCO 是闭环设计——课前收集的场景数据会存入记忆，由 `FEEDBACK_REVIEW` SCO 触发课后复盘
- 闭环反馈的复盘阶段会结合学员的真实场景进行个性化拆解
- 两个 SCO 之间通过 `memory_id` 关联，数据链路完整可追溯

### 10.5 小解答 SCO

本期覆盖——小解答的记忆范围是当前 Activity 内的知识内容和聊天记录。

小解答不同于大厅的辅导答疑：
- 小解答：范围限于当前 Activity 内，回答授课过程中产生的简单问题
- 辅导答疑：范围覆盖整个项目，提供深度的陪伴式辅导

### 10.6 授课中的自适应 [ADR-004]

**L1 自适应（每次交互）**：LLM 在每次回复中根据学员的回答调整语言、深度、举例方式。

**L2 自适应（SCO 版本切换）**：同一教学位置有多个版本的 SCO（通过 Segment 版本管理），系统根据学员画像选择最匹配的版本。

- 例如："视频案例与解析"SCO 有三个版本——防御拖延型、利益博弈型、情绪对抗型
- 学员在课前互动中被打上标签后，系统自动选择对应版本的案例播放
- 切换对学员透明，AI 自然过渡："让我换个角度来看这个问题"

自适应规则由内容设计者定义，不固定权重——因为自适应不只是难度问题，可能是角度问题，只有内容设计者才知道怎么切换。

#### 打标 SCO 独立化（Round-02 R1/R2 确认）

互动 SCO 和打标 SCO 明确分离——互动负责采集学员信息，打标负责基于累积信号进行标签判定。

**`ASSESSMENT_TAG` SCO 设计**：
- 打标 SCO 作为 AOM 中独立的 SCO 类型（`scoType: ASSESSMENT_TAG`），KGP 在 Activity 编排时显式放置
- 打标 SCO 不自己做独立判定，而是汇总前序互动 SCO 的累积信号，用概率模型输出标签
- 一个 Activity 内可以有多个打标点，形成"分段式自适应"

**打标结果存储字段**：

| 字段 | 说明 | 示例 |
|------|------|------|
| `tag_dimension` | 标签维度 | "冲突处理风格" |
| `assigned_tag` | 判定标签 | "利益博弈型" |
| `confidence` | 置信度（0-1） | 0.78 |
| `evidence_summary` | 判定依据摘要 | "学员3次提及资源交换，2次回避情感诉求" |
| `effective_until` | 有效截止位置 | 下一个打标 SCO 的 ID，或 Activity 结束 |

**打标影响范围规则（Round-02 R1+R2+R3 确认）**：
- 打标结果的影响范围只到下一个打标 SCO 为止——每段影响范围独立。一个 Activity 内多个打标 SCO 时，后一个打标 SCO 执行后，前一个的标签不再影响后续 Segment 选择（Round-02 R3 新增明确）
- **标签严格限于 Activity 内，跨 Activity 不传递**——对练 Activity 无法自动获取授课 Activity 的打标结果
- 对练如果需要个性化版本切换，需要在自己的导学阶段内置打标 SCO，导学阶段的打标完成后自动匹配对应版本的对练剧本（Round-02 R3 新增明确）
- 这简化了数据流设计，避免跨 Activity 的复杂标签依赖

**低置信度降级策略**：
- 置信度低于 0.6 时，降级为默认 Segment 版本，不硬猜
- 宁可给学员默认版本，也不给一个猜错的版本（Round-02 R2 确认）

**打标对学员的透明度（Round-02 R2 确认）**：
- 打标判定过程对学员是纯后台行为，学员无感知
- 但应让学员知道有几个版本存在——通过前一页 PPT 的内容表述，或通过 UI 功能展示（具体方式待定）
- KGP 不需要配置概率权重——概率模型参数由技术团队按互动类型设定默认值

**打标 SCO 的 AOM 配置**：
```
{
  "scoType": "ASSESSMENT_TAG",
  "tagConfig": {
    "tagDimension": "冲突处理风格",
    "possibleTags": ["防御拖延型", "利益博弈型", "情绪对抗型"],
    "dataSourceRange": "前序所有互动SCO",
    "fallbackStrategy": "默认Segment版本"
  }
}
```

### 10.7 授课中的互动

- **插嘴提问**：学员可以在任何时候打断 AI 讲解提出问题，AI 回答后继续讲解
- **主动回顾**：学员可以要求 AI 重新讲解之前某一页的内容，讲完后拉回当前位置
- **获得工具卡片**：AI 在讲解中适时生成可带走的工具卡片/表格模板，出现在看板的工具区
- **断点续播**：中途退出后从上次位置继续
- **重新学习**：可以选择从头开始当前 Activity

### 10.8 教室内导航

教学页面有隐藏的课程目录，多层折叠展示（Course → Activity → SCO），学员可以在教室内直接导航到任意位置。

### 10.9 交互细节

- PPT 讲解：左侧显示 PPT 图片（HTML 或图片渲染），右侧显示 AI 讲解对话。AI 不是照念，而是基于教学提示自主发挥。纯讲述 SCO 中的互动问题直接关联当前知识点，AI 回应将学员回答与知识点关联
- 案例视频：全屏播放视频内容，播放完毕后 AI 进行简要总结并引导学员思考（Round-02 R3 新增：视频播放后 AI 不只是切换到下一步，而是简要总结关键点并引导学员结合自身经验思考）
- Quiz：AI 在对话中出题，学员在对话中回答（问答题）或通过 GUI 组件回答（选择题/判断题/连线题）。答完后 AI 给出反馈和解析；学员不知道答案时 AI 给引导性提示
- 小解答：学员在授课过程中提出问题，AI 基于当前 Activity 的知识内容和之前的聊天记录进行解答
- 闭环反馈（课前）：AI 通过自然对话（开放式提问，非问卷格式）引导学员描述真实工作场景 → 记录到记忆。学员拒绝分享时的处理由脚本控制
- 闭环反馈（课后）：AI 调取课前记录的场景（通过 memory_id 精确关联） → 引导学员用所学知识重新分析
- 打标判定前：通过前序 PPT 内容自然提示学员"接下来有多个版本"，让学员知道内容会个性化匹配

## 11. 教室二：对练模块

### 11.1 模块描述

对练模块是基于场景的角色扮演训练教室。AI 模拟一个真实角色（如难缠的跨部门同事），与学员进行沉浸式对话练习。对练的核心目的是能力评估和行为训练，而非知识传递。

### 11.2 对练三阶段

对练 Activity 下包含三个阶段（每个阶段可视为独立 SCO）：

**阶段一：情景导入（导学）**

系统依次展示三组信息卡片：
1. 背景介绍：你扮演的角色信息 + AI 扮演的角色信息 + 场景描述
2. 目标介绍：本次对练的主目标和分项目标
3. 技能介绍：需要运用的核心技能和知识点

学员了解场景后点击"开始对练"。

**阶段二：角色扮演（演绎）**

这是对练的核心阶段。没有旁白——直接进入角色对话。

1. AI 角色直接说出开场白（不使用旁白描述场景氛围）
2. 学员和 AI 角色进行多轮对话（约10-15-20轮）
3. AI 角色的行为完全由对练剧本控制——角色的性格、目标、触发点、软化点、底线、行为模式等全部预设
4. AI 角色的情绪会被学员行为影响——有效沟通则情绪缓和，忽视/粗暴则情绪升级
5. 对练结束时生成目标达成评估
6. **对练结束机制由剧本控制**（Round-02 R1 确认）——剧本定义对练的结束条件（轮数上限、目标达成、死循环检测等），平台不硬编码结束规则

**关于旁白**：演绎阶段不设置旁白。之前 v3.0 中"旁白以沉浸式文字描述场景氛围"的设计已取消——直接进入角色对话更自然。

**阶段三：反思复盘**

1. AI 退出角色，切换为辅导教练身份
2. 调出演绎阶段的完整对话记录供学员回看——中间看板展示**"对话回看"面板**（新增看板组件，Round-02 R2 确认）
3. 学员可以点击对话回看面板中的某一轮对话，触发右侧 Agent 对该轮进行定向分析
4. 苏格拉底式追问引导学员反思关键决策点
5. 对照剧本中的成功标准，帮助学员发现改进空间

### 11.3 导演-演员分离架构（产品视角）

对练中有两个"幕后角色"协同工作：

- **导演**：理解剧本全局，分析学员每一轮发言，决定下一步场景走向和情绪变化
- **演员**：按照导演的指令和角色人设进行表演，不知道剧本的全局走向

学员只看到"演员"（AI角色），不知道"导演"的存在。这种分离确保了 AI 角色的表演自然真实。

### 11.4 对练多版本与打标匹配（Round-02 R1 确认）

对练 Activity 也支持 Segment 版本管理——基于打标结果，可以匹配不同版本的对练剧本，解决"对练场景与学员真实场景无关联"的问题。

**工作方式**：
- 对练 Activity 下的 Segment 代表不同版本的对练剧本（如"销售场景版"/"产品场景版"/"管理场景版"）
- 如果对练需要个性化版本切换，需要在导学阶段内置 `ASSESSMENT_TAG` SCO 进行打标（因为标签严格限 Activity 内，无法读取授课 Activity 的打标结果）
- 打标结果决定后续演绎阶段使用哪个版本的剧本

### 11.5 对练剧本如何控制行为 [ADR-004]

对练的情绪边界、行为规则等全部由对练剧本控制，平台不做主动控制——因为不同场景的情绪尺度完全不同（裁员谈话 vs 日常反馈）。

**情绪天花板由剧本定义，不是平台硬编码**——剧本设计者根据场景需要决定情绪可以升级到什么程度。裁员谈话的情绪天花板可能高于日常反馈场景，这是内容设计的决策，不是平台层面的约束。

剧本定义的关键控制元素：

| 控制元素 | 说明 | 示例 |
|---------|------|------|
| 角色心理画像 | 显性目标、隐性目标、触发点、软化点、底线 | "显性目标：不接受加急；隐性目标：被尊重" |
| 行为模式 | 递进式反应链 | "一上来冰山态度→施压则翻旧账→尊重则展现配合" |
| 防剧透指令 | AI 绝对不能主动提出某些方案 | "绝不主动提出延期方案，必须等学员想到" |
| 成功标准 | 分阶段的通关条件 | "阶段1通过：对方情绪降温到可对话程度" |
| 技能阶梯 | 每个能力维度的 L1-L5 量化标准 | "L1：直接表达诉求；L5：以对方利益为切入点重构议题" |
| 情绪天花板 | 本剧本中情绪最高可升级到什么程度 | "最高：中度不满（不会升级为愤怒离场）" |

### 11.6 断点恢复与重新练习

- **断点续播**：中途退出后，统一从断点继续（不区分中断时长）
- **重新练习**：重练 = 仅重练 Roleplay 阶段（阶段二：角色扮演），不重做情景导入（阶段一）
- 重练时保留上次的情景导入信息，直接从 AI 角色开场白开始

### 11.7 对练玩法

**本版本**：
- 真实场景角色扮演（1v1 Roleplay）

**未来迭代**：
- 角色互换（换位思考——通过不同的剧本设计实现）
- 费曼学习法（AI 扮学生，学员教 AI）
- 辩论（观点对抗式练习）
- 多人模拟（多个 AI 角色同时参与）

### 11.8 交互细节

- 情景导入阶段：系统依次展示三组信息卡片 → 学员阅读后点击"开始对练" → 进入角色扮演
- 角色扮演阶段：AI 角色直接说出开场白 → 学员输入回应 → 多轮对话交替进行 → 达到结束条件后系统提示"对练结束"
- 反思复盘阶段：AI 切换身份 → 展示对话回看 → 苏格拉底式追问 → 生成对练评估报告
- 断点续播：再次进入时系统提示"从上次位置继续"或"重新练习" → 选择继续则恢复到中断位置 → 选择重练则从 Roleplay 开场白重新开始

### 11.9 用户故事

- **作为学员**，我希望对练中的 AI 角色像真人一样有情绪反应，我的沟通方式能真正影响对方的态度，以便我体验到沟通技巧的实际效果
- **作为学员**，我希望对练结束后有一个复盘环节，教练帮我回看对话找出改进点，以便我知道自己哪里做得好、哪里需要提升
- **作为学员**，我希望对练场景和我的真实工作场景相关，以便我练习的技能能直接迁移到工作中
- **作为学员**，我希望对练匹配到的剧本与我的工作相关，且我不需要知道还有其他版本——所有内容都是为我匹配的，以便沉浸式体验
  - AC: Given 学员被匹配到特定版本剧本，When 进入对练，Then 所有内容均为匹配版本，无"你被分配到X版本"的提示
- **作为学员**，我希望对练中途被打断后能从上次位置继续，不用重头开始，以便我的时间不被浪费
- **作为学员**，我希望重新练习对练时只需要重练对话部分，不用再看一遍场景介绍，以便我高效地反复练习核心环节
- **作为学员**，我希望在对练复盘时，中间看板按轮次清晰展示对话记录，每条消息标注角色名，我可以点击某一轮让 AI 自动分析，以便我能具体回顾自己的表现
  - AC: Given 学员在复盘阶段，When 点击某轮对话，Then Agent 自动生成该轮分析，无需手动输入
- **作为学员**，我希望对练中跑题时 AI 能温和引导我回到主题，以便我的练习时间不被浪费
  - AC: Given 学员对话偏离对练主题，When AI 检测到跑题，Then 以角色身份温和引导回主题，不打破沉浸感
- **作为 KGP 内容设计师**，我希望为对练创建多版本剧本（如销售版/产品版/管理版），让打标为"销售"后自动选择销售版剧本，以便实现个性化对练
- **作为 KGP 内容设计师**，我希望在剧本中定义对练的结束条件（轮数上限/目标达成/死循环检测），以便不同对练有不同的结束规则
  - AC: Given 剧本定义了结束条件，When 条件触发，Then 系统按剧本规则结束对练

### 11.10 已确认决策

### 11.11 本期不做的功能

## 12. 教室三：测评模块

### 12.1 模块描述

测评模块是 AI 通过对话或 GUI 量表采集学员真实信息并进行评估的教室。测评不是"考试"，而是"深度访谈"或"标准化量表"——AI 遵照后台指令，通过自然对话或量表交互获取学员的真实行为数据。

### 12.2 两种测评类型

测评模块明确支持两种类型，根据访谈剧本的要求选择：

**类型一：对话式测评**
- AI 通过自然对话引导学员讲述真实经历
- 对话路径由访谈剧本控制
- 适用于：BEI 行为事件访谈、学员画像采集、困惑收集、高光萃取

**类型二：GUI 量表测评**
- 类似 MBTI 的标准化量表，学员通过 GUI 界面进行选择
- 量表题目和计分逻辑由访谈剧本定义
- 适用于：结构化能力评估、态度量表、自我认知评估

### 12.3 "测"与"评"分开

测评模块分为两个阶段：

**阶段一：测（信息采集）**
- 对话式采集：AI 通过自然对话引导学员讲述真实经历
- 或量表式采集：GUI 界面上的标准化量表

**阶段二：评（分析评估，可选）**
- AI 基于采集的数据进行分析和评估
- 以对话形式向学员解读评估结果
- 学员可以与 AI 讨论评估结论

### 12.4 对话式测评的具体方法

BEI 只是测评模块下的一种方法，还有其他类型：

| 测评类型 | 目的 | 核心方法 |
|---------|------|---------|
| **BEI（行为事件访谈）** | 采集学员在真实场景中的行为证据 | STAR 结构追问（Situation/Task/Action/Result） |
| **学员画像** | 了解学员的工作背景和管理风格 | 开放式对话 + 结构化提取 |
| **困惑收集** | 收集学员在某主题下的真实困惑 | 引导式提问 |
| **高光萃取** | 萃取学员已有的成功经验 | 深度追问最佳实践 |

具体行为由背后的访谈剧本控制 [ADR-004]。

### 12.5 BEI 追问策略

1. 开放式提问引出案例
2. STAR 完整性检查（后台自动判断 Situation/Task/Action/Result 是否齐全）
3. 针对性追问缺失元素（自然语气，不像审讯）
   - 特别关注：学员说"我们"时追问"你个人的作用"；Action 模糊时追问具体动作
4. 回避行为识别：连续 2 次模糊/转移话题 → 不再纠缠，切换到新案例
5. 同一案例最多追问 3 轮；不完整的案例记录完整度评分，案例不完整本身就是评估信息

**BEI 编造识别**：不考虑真假——编出来的假案例也是学员学习状况的一种体现。AI 不做"判官"，所有学员输入都有价值。

### 12.6 BEI 时机

BEI 是否在 V1 做由内容决定——平台支持，内容设计者选择是否使用。BEI 时机由内容设计决定，不固定——可以在课程开头做基线测评，也可以在课程结束做效果测评，取决于教学设计。

### 12.7 GUI 量表测评

GUI 量表的交互设计：
- 标准化的量表界面，学员通过点选/滑块等 GUI 组件完成
- 量表题目、选项、计分逻辑全部由访谈剧本定义
- 完成后自动计分，可选"评"阶段解读结果
- 量表结果写入统一 Database，供报告和辅导引用

### 12.8 交互细节

- 对话式测评：AI 以自然对话开场 → 按剧本引导学员讲述经历 → 后台自动检查信息完整性 → 追问缺失元素 → 采集完成后进入评阶段
- GUI 量表测评：系统展示量表界面 → 学员逐题作答 → 提交后自动计分 → 可选进入 AI 解读对话
- 评阶段：AI 展示评估结果 → 学员可以提问和讨论 → AI 引用具体行为证据解释结论

### 12.9 用户故事

- **作为学员**，我希望测评过程像一次自然的对话，而不是被审讯的感觉，以便我愿意分享真实经历
- **作为学员**，我希望测评结束后能看到 AI 对我的评估解读，并能和 AI 讨论我不认同的地方，以便评估结果对我有实际价值
- **作为学员**，我希望量表测评的界面简洁直观，点选操作流畅，以便我能快速完成标准化评估
- **作为学员**，我希望 AI 在测评中不会纠缠我不想回答的问题，适时换个话题继续，以便我不会感到被逼迫
- **作为 KGP 内容设计师**，我希望能通过剧本灵活选择对话式或量表式测评，以便不同课程可以用最合适的测评方式

### 12.10 已确认决策

- [ADR-004] 测评行为由访谈剧本控制
- 明确两种测评类型：对话式 + GUI 量表，根据访谈剧本要求选择
- BEI 只是一种测评方法，不是唯一的
- BEI 是否 V1 做由内容决定——平台支持，内容设计者选择
- BEI 编造识别：不考虑真假——AI 不做"判官"
- 测评中 AI 的追问路径围绕 KGP 定义的能力维度展开

### 12.11 本期不做的功能

- BEI 编造识别——设计哲学决定不做，不是技术限制
- 测评结果自动影响学习路径——本期测评结果供报告和辅导引用，不自动调整路径（属于 L3 自适应，未来迭代）

## 13. 教室四：个人报告模块

### 13.1 模块描述

个人报告是学员端的第四个教学模块类型，和授课/对练/测评平级——不是附属功能，而是独立的学习体验。学员不仅能看到报告，还能和 AI 讨论报告。

**报告位置**：每个 Course 结束后固定生成一份个人学习报告。未来可从辅导模块随时进入查看。

### 13.2 报告两阶段

**阶段一：生成**
- 学员完成一个 Course 后进入报告教室
- 系统播放祝贺动画和预告文案
- 报告在后台并行生成（多模块数据汇聚）
- 分段渐进式呈现给学员（高光时刻 → 能力成长 → 学习建议）

**阶段二：解析**
- 学员可以与 AI 讨论报告内容
- "你为什么给我这个维度这么低的分？"
- AI 引用具体的行为证据解释评估结论
- 学员可以对报告中不认同的点提出反馈

### 13.3 报告内容模块

报告不包含"基础信息"模块（学员姓名、学习时长等），直接聚焦有价值的分析维度：

| 报告模块 | 内容 |
|---------|------|
| 高光时刻 | 精选学员在学习过程中的最佳表现（top 3） |
| 能力成长 | 按能力维度的评估和变化趋势 |
| 认知转化 | 学员在 Quiz 中展现的认知变化 |
| 学习表现 | 各模块参与度、互动质量 |
| 技能提升 | 基于对练的技能评分变化 |
| 成长表现 | 学习特质、认知转变、个人优势 |
| 综合评估 | 汇总所有维度的综合结论 |
| 行动建议 | 针对性的下一步提升方向 |

### 13.4 学员端报告展示规范（Round-02 R2+R3 确认）

学员端报告以个人成长为核心，不展示排名名次：

- **主信息**：个人成长时间线——折线图展示各维度跨 Course 的进步趋势
- **辅助信息**：匿名分位区间（前 25% / 中间 50% / 后 25%），用柔和灰色标尺展示，帮助学员了解自己在项目中的相对位置
- **不展示**：具体排名名次和分组标签（如"高成长组"等标签学员不可见）
- **文案风格**：使用成长型语言，强调进步而非排位

排名计算基于项目内（30-40 人内部排名），不跨项目比较（Round-02 R1 确认）。

**学员端与 HR 端排名数据隔离（Round-02 R3 矛盾3修正）**：
- 学员端"不展示排名"与 HR 端"排名可排序"是有意设计，但需 **API 级隔离**确保数据不泄漏
- 学员端报告 API 不返回排名字段，仅返回分位区间标识（"前25%"/"中间50%"/"后25%"）
- HR 端报告 API 返回完整排名数据（进度排名、互动排名、可排序、可筛选）
- 两端使用不同的 API 端点或数据视图，前端无法通过学员端接口获取排名详情

### 13.5 数据采集机制

报告不是在最后一刻才生成的，而是在学习过程中持续积累数据碎片。

**碎片生产规则（Round-02 R2 优化）**：
- **学习特质碎片**：从"每 SCO 一个"优化为**每 Activity 一个汇总碎片**——减少 AI 调用成本，同时保证数据粒度满足报告需求
- **认知转化碎片**：每个 Quiz 答题闭环后生成（不变）
- **能力评估碎片**：每次对练完成后生成（不变）

**碎片来源优先级**（用于报告数据权重）：
1. Quiz 互动数据（最高权重）
2. 对练行为数据
3. 测评数据
4. 辅导对话数据
5. 被动信号数据（停留时长、是否回翻等，最低权重）

报告生成时拉取全部碎片汇总分析。

### 13.6 BI 可视化图表

报告中需要嵌入数据可视化图表，增强报告的说服力和可读性。需要建设一个**通用图表组件库**，本期必须有但优先级属于优化项。

**必需图表类型**：

| 图表类型 | 适用场景 |
|---------|---------|
| 雷达图 | 能力维度综合展示（如沟通/授权/激励等多维度对比） |
| 热力图 | 学习行为分布（如各模块参与度、各时间段活跃度） |
| 饼图 | 占比类数据（如各能力等级分布、学习时间分配） |
| 折线图 | 趋势变化（如能力评分随课程推进的变化曲线） |
| 柱状图 | 对比类数据（如各 Course 的完成度、对练评分对比） |

**设计原则**：
- 组件库通用化——不仅用于个人报告，也用于 HR 端和运营端的看板
- 图表可交互——点击图表元素可联动 Agent 对话区进行深入解读
- 数据可追溯——点击图表中的数据点可查看原始行为记录

### 13.7 交互细节

- 学员完成一个 Course 的所有 Activity 后，系统提示"生成个人报告"→ 进入报告教室
- 报告教室三栏展示：左侧 Project 目录 | 中间渐进式展示报告各模块（含 BI 图表） | 右侧 AI 对话入口（讨论报告）
- 学员点击任意报告模块可以展开详情
- 学员点击图表中的数据点，右侧 Agent 自动解读该数据背后的行为证据
- 学员可以在对话区与 AI 讨论报告：提问 → AI 引用行为证据回答
- 报告可导出为 PDF

### 13.8 用户故事

- **作为学员**，我希望每个课程模块学完后能看到一份有洞察力的个人报告，以便我清楚自己的成长和不足
- **作为学员**，我希望能和 AI 讨论报告中的评估结论，质疑不合理的评分，以便报告不是单向的"宣判"而是双向的"对话"
- **作为学员**，我希望报告直接呈现有价值的分析（高光时刻、能力变化），而不是先看一堆基础信息（姓名、时长），以便我第一时间看到最重要的内容
- **作为学员**，我希望报告中引用我在学习中的具体行为（"你在第三次对练中首次主动使用了共赢策略"），以便评估结论有据可依
- **作为学员**，我希望报告中有个人成长时间线（折线图），展示各维度跨 Course 的进步趋势，以便我直观看到自己的成长轨迹
  - AC: Given 学员查看报告，When 报告加载完成，Then 展示折线图，X轴为时间/Course，Y轴为各能力维度得分
- **作为学员**，我希望报告中的相对位置只展示匿名分位区间（前25%/中间50%/后25%），不展示具体排名，以便我了解自己的位置而不产生不必要的比较焦虑
- **作为学员**，我希望报告以分段渐进式呈现（高光时刻→能力成长→学习建议），以便阅读体验有节奏感
- **作为学员**，我希望未来能从辅导教练那里随时进入查看之前的报告，以便我随时回顾自己的成长

### 13.9 已确认决策

- 个人报告去掉"基础信息"模块——直接聚焦有价值的分析维度
- 报告位置：每个 Course 结束后固定生成，未来可从辅导随时进入
- 报告是独立的教学模块，学员可以与 AI 讨论报告内容
- 学员端报告展示个人成长时间线+匿名分位区间，不展示具体排名名次（Round-02 R2 确认）
- 排名基于项目内（30-40 人），不跨项目比较（Round-02 R1 确认）
- 碎片生产优化：学习特质碎片从每 SCO 一个改为每 Activity 一个汇总碎片（Round-02 R2 确认）

### 13.10 本期不做的功能

- 从辅导模块随时进入查看报告——本期报告只在 Course 结束后固定生成，未来迭代支持随时进入

## 14. HR 管理端

### 14.1 核心功能

#### 14.1.1 学习进度看板

- 查看所有学员的整体完成进度（百分比、环形图）
- 按 Course/Activity 维度查看细分进度（Activity 级粒度，本期不做 SCO 级，Round-02 R1 确认）
- 按学员维度查看个人进度详情
- 学习进度从个人角度出发——呈现每个人的完成情况，不做部门分组

#### 14.1.2 项目数据看板

本期先呈现数据，不做智能评价。呈现的维度：

| 数据维度 | 描述 | 本期实现 |
|---------|------|---------|
| 学习进度 | 个人和整体的完成率、进度分布 | 必须有 |
| 学习效果 | 测评分数、对练评分、能力维度变化 | 希望有，具体指标待定 |
| 学员活跃度 | 平均登录频率、平均学习时长 | 有 |
| 完课率 | 各模块的完成率统计 | 有 |

AI 质量维度不放在 HR 端——学员不关心，HR 也不需要关心。AI 质量数据放在后台，供运营和技术团队查看。

#### 14.1.3 报告查看与定制

- 查看个人学习报告（与学员看到的同一份报告）
- 查看项目整体报告（所有学员的汇总分析）
- 通过 HR Agent 定制特殊报告（如按特定维度切片等）

#### 14.1.3a HR 端报告排名规范（Round-02 R2 确认）

HR 端报告排名做了重大简化——**排名不看能力**，先实现以下两种排名维度：

| 排名维度 | 说明 | 可操作性 |
|---------|------|---------|
| **进度排名** | 谁完成得多——基于 Activity 完成数/百分比 | 可排序、可筛选 |
| **互动排名** | 谁互动得多——基于互动 token 量 | 可排序、可筛选 |

**不做能力维度排名**——因为能力评分的可信度和可比性在本期无法保证（样本量小、评分标准化未完成）。

**HR 端报告内容**：
- 完整的进度排名表和互动排名表（可排序、可筛选）
- 各 Course 完成率统计和分布
- 聚合统计：平均完成进度、平均互动量、需关注学员 Top 3（基于进度落后指标）
- HR 可见分组标签（如"高活跃/稳健/需关注"），学员不可见

**成长幅度计算规则**（为未来能力排名预留）：
- 优先方法：课前 BEI 基线 vs 课后评分
- 备选方法：前 1/3 阶段均值 vs 后 1/3 阶段均值
- 引入标准化，消除初始分数偏置

**跨 Course 统一能力维度**（为未来迭代预留）：
- Project 级定义 3-5 个通用能力维度
- 各 Course 细分维度映射到通用维度
- AOM Report 模块新增 `rankingDimensions` 和 `groupingStrategy` 配置

#### 14.1.4 HR Agent

HR Agent 的交互模式：
- 自然语言提问："本周谁的进度最快？"、"所有学员的平均完成率是多少？"
- 生成汇报报告：PDF 格式，记忆 HR 的模板偏好（本期不生成 PPT）
- 解读数据：CEO 追问时，可以逐层钻取到原始数据
- 贯穿学习期和结项期——不是只有结项时才用

#### 14.1.5 催学规则配置

HR 端也可以配置催学规则，与运营端共享规则配置能力。

### 14.2 交互细节

- HR 进入三栏布局后，左侧目录导航到不同功能
- 点击"学习进度"→ 中间看板展示全部学员的进度表格和图表
- 点击某个学员 → 展开个人详情面板，显示个人的 Course/Activity 完成情况
- 右侧 Agent 随时可对话："帮我看看张三的学习表现"→ Agent 返回张三的数据摘要
- 报告生成流程：HR 向 Agent 描述需求 → Agent 确认 → 生成报告 → HR 预览 → 修改 → 下载 PDF

### 14.3 用户故事

- **作为 HR**，我希望一打开管理端就能看到项目整体进度数据，一眼判断项目是否按计划推进，以便我高效管理培训项目
- **作为 HR**，我希望通过 Agent 说"帮我生成一份本月的项目进展报告"就能得到高质量的 PDF 报告，以便我不需要手工制作汇报材料
- **作为 HR**，我希望报告中的行为改变结论有学员的真实行为证据支撑（"在第三次对练中，张三首次主动使用了倾听-回应-探询的三步沟通法"），以便 CEO 信服培训效果
- **作为 HR**，我希望自定义催学规则（比如"连续3天未登录的学员发提醒"），以便催学策略符合我们企业的管理风格
- **作为 HR**，我希望 Agent 不用"销售部门的平均完成率"这样的分组概念来展示数据，而是直接告诉我每个人的情况，以便我不需要先建分组就能使用
- **作为 HR**，我希望报告中每个学员的进度排名和互动排名可以排序和筛选，以便我快速找到进步最大和需要关注的学员
- **作为 HR**，我希望项目报告中展示各 Course 的全员完成率分布，以便我向 CEO 汇报"整体学习进展如何"
- **作为 HR**，我希望进度排名表中看到具体完成数据（已完成/总Activity数、百分比、最后学习时间），以便我了解每个学员的详细进度
  - AC: Given HR 查看进度排名，When 排名表加载，Then 每行包含已完成Activity数/总数、百分比、最后学习时间
- **作为 HR**，我希望理解"互动量排名"的含义和计算方式，以便我正确解读排名数据
  - AC: Given HR 查看互动排名，When 查看说明，Then 界面展示互动排名计算说明（基于互动 token 量）
- **作为 HR**，我希望在看板显著位置看到"需关注学员 Top 3"（进度最低的3人），以便我快速定位需要重点关注的学员
  - AC: Given HR 打开看板，When 看板加载，Then 显著位置展示进度最低3人（含姓名、进度百分比、最后登录时间）
- **作为 HR**，我希望切换学员身份体验课程后产生的数据不污染学员统计，以便报告数据准确
- **作为 HR**，我希望 Agent 回答时标注数据更新时间，以便我知道看到的数据是否是最新的
  - AC: Given HR 提问 Agent，When Agent 返回数据，Then 回答中包含"数据截至 YYYY-MM-DD HH:MM"
- **作为 HR**，我希望 Agent 无法回答某个问题时坦诚告知并建议我联系运营，以便我不会收到不准确的信息
  - AC: Given HR 提问超出 Agent 能力范围，When Agent 无法回答，Then 回复"这个问题我暂时无法回答，建议联系运营团队"

### 14.4 已确认决策

- 去掉"销售部门"等分组概念——30-40人暂无学员组划分
- 项目数据看板：学习进度必须有（个人角度），学习效果待定，AI质量不放学员端/HR端
- 管理端数据粒度为 Activity 级，本期不做 SCO 级进度展示（Round-02 R1 确认）
- HR 端排名不看能力——先实现进度排名+互动排名（token），不做能力排名（Round-02 R2 确认）
- 本期先呈现数据，不做智能评价
- 催学规则可配置，HR 端和运营端都可配置
- 本期仅生成报告，不生成 PPT

### 14.5 本期不做的功能

- CEO 汇报 PPT 生成——先做好报告格式
- 智能健康度评价——先呈现数据
- 学员分组分析——暂无分组需求
- AI 质量看板——学员和 HR 不关心，放后台
- 能力维度排名——能力评分的可信度和可比性在本期无法保证（Round-02 R2 确认）
- SCO 级进度展示——Activity 级粒度已够用（Round-02 R1 确认）

## 15. 运营管理端

### 15.1 核心功能

#### 15.1.1 项目配置

- 导入学员名单（Excel/CSV）

#### 15.1.2 数据展示看板

本期不做智能预警评价——先把数据呈现出来，运营自己判断。

## 16. 内容驱动 [ADR-004]

### 16.1 核心理念

产品的灵魂不在平台，在内容。平台提供执行引擎，内容提供具体的教学行为。同一个引擎 + 不同的脚本 = 不同的培训产品。

> AOM 是乐谱，Prosona 是钢琴。给什么乐谱，就演奏什么音乐。

### 16.2 AOM 五层数据模型

```
Project（项目/课程包）
  └── Course（课程/模块）
       └── Activity（活动：授课/对练/测评/报告）
            └── SCO（最小学习内容单元）
                 └── Segment（SCO 的版本）
```

| 层级 | 说明 | 示例 |
|------|------|------|
| Project | 完整的培训项目 | "基层管理者培养项目" |
| Course | 主题模块（原 Module 层） | "横向协作" |
| Activity | 一次学习活动 | "PIN法三步走"（授课）、"跨部门协商对练"（对练） |
| SCO | 一个教学单元 | 一页 PPT 讲解、一道 Quiz、一段视频 |
| Segment | 同一 SCO 的不同版本 | "字数多版" / "字数少版"；"销售场景版" / "产品场景版" |

### 16.3 SCO 与 Segment 的关系（重要修正）

**SCO 和 Segment 一对一对应**——这是对 v3.0 理解的重大修正。

- 一页 PPT = 一个 SCO
- 一个视频 = 一个 SCO
- 一个 Quiz = 一个 SCO
- 每个 SCO 对应一个 Segment

**Segment 不是"内容片段"（PPT图片URL、视频帧列表），而是同一 SCO 的不同版本。**

- 一个 SCO 下有多个 Segment = 该 SCO 的多个版本（如字数多版/字数少版、销售场景版/产品场景版）
- 不再是"一个 SCO 下多页连续讲解"的旧模型

**自适应时切换的是 Segment（版本），不是 SCO**：
- L2 自适应的工作方式：根据学员画像标签，选择该 SCO 下最匹配的 Segment（版本）
- 同一个教学位置，不同学员可能看到不同版本的内容
- 版本由 KGP 半预制，自适应切换

### 16.4 AOM 数据模型扩展（Round-02 R2 确认）

基于第二次辩论确认的结论，AOM 五层数据模型新增以下 SCO 类型和字段：

**新增 SCO 类型**：

| SCO 类型 | scoType | 说明 |
|---------|---------|------|
| 打标判定 | `ASSESSMENT_TAG` | 独立打标 SCO，含 `tagConfig`（标签维度、可能标签值、数据来源范围、降级策略） |
| 闭环反馈-课前采集 | `FEEDBACK_COLLECT` | 课前端采集学员真实场景，通过 `memory_id` 与课后复盘关联 |
| 闭环反馈-课后复盘 | `FEEDBACK_REVIEW` | 课后端引导学员复盘，通过 `memory_id` 读取课前采集数据 |

**新增字段**：

| 层级 | 字段 | 说明 |
|------|------|------|
| SCO 层 | `referenceSlots` | 引用槽数组，每个槽含 `slotId`、`required`（是否必选）、`description` |
| Project 层 | `universalCapabilityDimensions` | 通用能力维度映射体系，3-5 个通用维度 |
| Report 层 | `rankingDimensions` | 排名维度配置（本期：进度排名+互动排名） |
| Report 层 | `groupingStrategy` | 分组策略配置（HR 可见分组，学员不可见） |

### 16.5 内容驱动行为的证据

以下教学行为全部由内容脚本控制，而非平台硬编码：

| 行为 | 控制元素 | 存在于 |
|------|---------|--------|
| AI 何时出题、出什么题 | passConditions + rule | AOM SCO 层 |
| AI 讲解的教学策略 | blueprint | AOM SCO 层 |
| AI 评判答案的标准 | solution + passConditions | AOM SCO 层 |
| AI 在授课中引用学员场景 | referenceSlots（必选/可选） | AOM SCO 层 |
| AI 在对练中的情绪表现 | behavioral_patterns + triggers/softeners | 对练剧本 |
| AI 在对练中的情绪天花板 | 剧本定义的情绪上限 | 对练剧本 |
| AI 在测评中的追问策略 | 访谈剧本 | 测评剧本 |
| 自适应版本切换 | Segment 版本标签 | AOM Segment 层 |
| 打标判定 | tagConfig（维度/标签值/降级策略） | AOM ASSESSMENT_TAG SCO |
| 对练结束条件 | 剧本定义的结束规则 | 对练剧本 |
| AI 的收尾话术 | answerAnalysis | AOM SCO 层 |
| 催学规则 | 可配置规则（非硬编码） | 运营端配置 |

### 16.6 预制菜模式 [ADR-002]

第一版所有内容由 KGP 预制交付：
- Project/Course/Activity/SCO 全部由供应商制作
- 管理员不配置课程结构
- 未来迭代为 Agent 平台模式，让企业可自助制作

### 16.7 交互细节

- KGP 通过生产工具制作 AOM 数据包，导出为半结构化数据格式
- 运营端导入内容包，系统自动解析 Project/Course/Activity/SCO/Segment 层级结构
- AI 执行引擎在运行时读取 AOM 数据，按脚本演绎教学行为
- 自适应切换时，系统根据学员标签在同一 SCO 的多个 Segment 中选择最匹配的版本

### 16.8 用户故事

- **作为 KGP 内容设计师**，我希望通过编写脚本就能精确控制 AI 的教学行为（何时提问、如何评判、何时切换版本），以便我不需要修改代码就能调整教学体验
- **作为产品负责人**，我希望更换内容包就能交付完全不同主题的培训项目（从基层管理者到销售培训），以便一次平台建设支撑多条业务线
- **作为 KGP 内容设计师**，我希望为同一个 SCO 创建多个 Segment（版本），让不同学员看到最适合自己的内容版本，以便实现个性化学习体验
- **作为 KGP 内容设计师**，我希望在对练剧本中定义情绪天花板，控制 AI 角色的情绪上限，以便不同场景有不同的情绪尺度
- **作为运营**，我希望导入内容包后系统自动解析课程结构，我不需要手动配置每一个 SCO，以便快速启动项目
- **作为 KGP 内容设计师**，我希望在一个 Activity 内放置多个打标 SCO，每个打标 SCO 的影响范围独立到下一个打标 SCO，以便实现分段式自适应
- **作为 KGP 内容设计师**，我希望在对练导学阶段内置打标 SCO 实现个性化版本匹配，导学打标后演绎阶段自动匹配对应版本的剧本

### 16.9 已确认决策

- SCO 和 Segment 一对一对应，Segment 用作版本管理
- 自适应切换的是 Segment（版本），不是 SCO
- 情绪天花板由剧本定义，不是平台硬编码
- 催学规则可配置，不硬编码
- AOM 数据模型新增三种 SCO 类型：`ASSESSMENT_TAG`、`FEEDBACK_COLLECT`、`FEEDBACK_REVIEW`（Round-02 R2 确认）
- AOM SCO 层新增 `referenceSlots` 字段（Round-02 R2 确认）
- AOM Project 层新增 `universalCapabilityDimensions` 通用能力维度映射（Round-02 R2 确认）
- AOM Report 层新增 `rankingDimensions` 和 `groupingStrategy` 配置（Round-02 R2 确认）

## 17. 记忆工程 [ADR-005]

### 17.1 核心问题

内测痛点："情感错付"——学员课前认真分享了真实工作问题，课后完全没有回应。如果记忆链断了，整个产品就变成了另一个"刷课"系统。

### 17.2 三级记忆管道

本期实现完整的三级记忆管道 [ADR-005]。所有数据写入统一 Database，各端按权限读取。

**第一级：课堂内记忆采集**

| 数据来源 | 采集内容 | 格式 |
|---------|---------|------|
| 授课模块 | 闭环反馈 SCO 收集的学员真实场景、Quiz 回答、所有互动对话 | 半结构化记录 |
| 对练模块 | 对练对话记录、行为表现、能力评估碎片 | 结构化 + 原始对话 |
| 测评模块 | BEI 案例、学员画像、困惑记录、高光经验、量表得分 | 结构化摘要 |

**关键变更**：授课也是互动式的，都要原始对话记录，尽量用半结构化记录（不只是 Quiz 回答和闭环反馈）。

所有数据实时汇总到统一 Database。

**第二级：大厅辅导引用课堂记忆**

辅导智能体读取课堂数据，在辅导对话中引用：
- "你在对练中三次成功使用了共赢策略"
- "上次你提到的跨部门协作问题，在刚学的PIN法课程里正好有解决方案"
- "你在测评中提到的团队管理困惑，要不要深入聊聊？"

**第三级：管理端记忆汇聚生成报告**

报告智能体读取全部数据（课堂 + 辅导），生成 HR 可用的行为改变证据报告。数据来源于统一 Database，各端按权限和视图差异读取。

### 17.3 闭环反馈机制

```
课前：学员分享真实工作困境 
  → 记忆存储（统一 Database）
课中：AI 用这些困境举例、用知识点关联解答
  → 记忆引用
课后：复盘"你之前说的那个问题，现在你会怎么处理？"
  → 记忆应用
辅导：随时引用"你在对练中说过..."
  → 跨模块记忆
报告：基于全过程数据生成行为改变证据
  → 记忆汇聚
```

### 17.4 半结构化记录的设计

所有模块的对话记录采用半结构化格式：

```
{
  "timestamp": "2026-06-15T14:30:00",
  "student_id": "xxx",
  "activity_id": "xxx",
  "sco_id": "xxx",
  "interaction_type": "quiz_answer | free_dialogue | roleplay | assessment | feedback_collect | feedback_review",
  "raw_dialogue": [原始对话记录],
  "ai_summary": "AI 生成的摘要",
  "capability_tags": ["反馈沟通", "冲突处理"],
  "behavior_observations": ["使用了共赢策略", "首次主动倾听"],
  "passive_signals": {
    "dwell_time_seconds": 180,
    "left_midway": false,
    "scrolled_back": true,
    "clicked_tool_cards": ["PIN法工具卡片"]
  }
}
```

**`passive_signals` 字段（Round-02 R2 新增）**：
- `dwell_time_seconds`：停留时长（秒）
- `left_midway`：是否中途离开
- `scrolled_back`：是否回翻浏览
- `clicked_tool_cards`：是否点击工具卡片（记录具体卡片名称）

被动信号主要用于纯讲述 SCO 等学员不主动互动的场景，作为学习参与度的辅助指标。被动信号在报告数据中的权重最低（低于 Quiz 互动、对练行为、测评、辅导对话）。

这样既保留了原始对话的完整性，又有结构化字段便于检索、统计和报告生成。

### 17.4a 课前采集记忆存储模板（Round-02 R2 确认）

闭环反馈 `FEEDBACK_COLLECT` SCO 采集的学员场景数据，使用以下双层存储模板：

```
{
  "memory_id": "mem_xxx",
  "student_id": "xxx",
  "activity_id": "xxx",
  "sco_id": "xxx",
  "collect_timestamp": "2026-06-15T09:00:00",
  "structured_tags": {
    "topic_dimension": "横向协作",
    "scenario_type": "跨部门资源协调",
    "scenario_keywords": ["资源争夺", "优先级冲突", "KPI不一致"],
    "emotional_state": "焦虑",
    "specific_challenge": "市场部要求紧急支援但产品路线图已满"
  },
  "raw_dialogue": [
    {"role": "ai", "content": "你在横向协作中遇到过什么困难？"},
    {"role": "student", "content": "上周市场部的张总找我..."},
    ...
  ]
}
```

**`structured_tags` 层**用于：
- 后续 `FEEDBACK_REVIEW` SCO 精确调取学员场景
- 授课中 `referenceSlots`（引用槽）的场景匹配
- 辅导教练的跨 Course 记忆引用

**`raw_dialogue` 层**用于：
- 报告中的行为证据追溯
- AI 在复盘时引用学员的原话

### 17.4b 跨 Course 记忆工程（Round-02 R2 确认）

辅导教练的记忆范围本期打通跨 Course 边界：

**引用优先级**：
1. 当前 Course 记忆（最优先）
2. 相关 Course 记忆（主题相关）
3. 远期 Course 记忆（时间较远或主题关联度低）

**实现方式**：
- 简单时间排序，本期不做复杂权重衰减
- 所有 Course 的记忆统一存储在同一 Database，通过 `course_id` 字段区分来源
- 辅导教练在检索记忆时，按优先级规则从全部 Course 中筛选相关记录

### 17.5 交互细节

- 记忆采集对学员透明——学员不需要感知"记忆正在被采集"
- 辅导教练引用记忆时自然融入对话，不说"根据你的对练记录..."，而是说"你之前在和张楠的沟通中提到过..."
- 报告中引用记忆时标注来源，支持追溯到原始对话

### 17.6 用户故事

- **作为学员**，我希望课前分享的工作困境在课后被 AI 重新提起，引导我用所学知识解决，以便我的每一次分享都不白费
- **作为学员**，我希望辅导教练能引用我在课堂中的表现给建议（"你在对练中发现的那个方法，可以用在现在这个场景"），以便学习内容之间是连贯的
- **作为学员**，我希望系统自动记录被动信号（停留时长、回翻、点击）时我完全无感知，以便我的学习体验不被打扰
  - AC: Given 系统采集被动信号，When 学员正常学习，Then 学员界面无"正在记录"等任何提示
- **作为 HR**，我希望报告中的数据来自学员的真实学习过程（而非事后问卷），以便评估结论真实可信
- **作为学员**，我希望授课中我回答的每一个问题、提的每一个疑问都被记录下来，未来教练和报告能引用，以便我的每次互动都有价值
- **作为运营**，我希望能追溯到学员的原始对话记录，以便在审核报告时验证数据来源

### 17.7 已确认决策

- 授课也是互动式的，都要原始对话记录，尽量用半结构化记录
- 统一 Database，三端按权限读取，不做方向性数据流转
- 记忆管道本期三级全做
- 半结构化记录新增 `passive_signals` 字段（停留时长、是否离开、是否回翻、是否点击），用于纯讲述 SCO 等无主动互动场景（Round-02 R2 确认）
- 课前采集记忆使用 `structured_tags` + `raw_dialogue` 双层存储模板（Round-02 R2 确认）
- 碎片生产规则优化：学习特质碎片每 Activity 一个汇总 / Quiz 级认知转化 / 对练级能力评估（Round-02 R2 确认）
- 辅导教练回大厅必然引用一次最近学习+话题条件触发，不做主动回顾打扰（Round-02 R2 确认）
- 跨 Course 记忆本期打通，引用优先级：当前 Course > 相关 Course > 远期 Course（Round-02 R2 确认）

## 18. 自适应

### 18.1 自适应层级

| 层级 | 范围 | 机制 | 本期实现 |
|------|------|------|---------|
| **L1** | 每次对话交互 | LLM 实时根据学员回答调整语言、深度、举例 | 是 |
| **L2** | SCO 版本切换 | 根据学员画像标签匹配 Segment（版本）选择 | 是 |
| **L3** | Activity 顺序调整 | 根据学员能力跳过/重排学习活动 | 未来迭代 |
| **L4** | 动态内容生成 | 实时生成新 Quiz/新模块/临时文件 | 未来迭代 |

### 18.2 L2 版本切换的工作方式（Round-02 R1/R2 更新）

L2 自适应通过打标 SCO（`ASSESSMENT_TAG`）独立判定学员标签，再基于标签匹配 Segment 版本：

1. **互动 SCO** 采集学员特征（课前互动、Quiz 回答、对话内容等）
2. **打标 SCO**（`ASSESSMENT_TAG`）汇总前序互动 SCO 的累积信号，用概率模型输出标签（如"防御拖延型"/"利益博弈型"/"情绪对抗型"）
3. 后续 SCO 中，系统根据打标结果匹配 Segment（版本）选择最适合的版本
4. 同一 SCO 下的多个 Segment = 多个版本，选择与学员标签匹配的那个

**版本切换的本质**：不是切换 SCO（教学位置不变），而是切换 Segment（同一位置的不同内容版本）。

**打标影响范围（Round-02 R1+R2 确认）**：
- 打标结果影响范围只到下一个打标 SCO 为止，每段影响范围独立
- 标签严格限于 Activity 内，跨 Activity 不传递
- 低置信度（< 0.6）时降级为默认 Segment 版本
- 打标对学员为纯后台行为，但通过 PPT 内容或 UI 让学员知道有多个版本（具体方式待定）

### 18.3 自适应原则

- 不定固定权重——规则由内容设计者根据教学目标定义
- 自适应不只是难度问题，可能是角度问题
- 切换对学员透明——AI 自然过渡，不说"你被切换到简单版本"
- 路径个性化，能力评估维度标准化——两者不矛盾

### 18.4 交互细节

- 打标判定过程对学员是纯后台行为，学员无感知
- 但应让学员知道有几个版本存在——通过前一页 PPT 的内容表述或 UI 功能展示（具体方式待定，Round-02 R2 确认）
- AI 在切换版本时自然过渡："让我换个角度来看这个问题"或直接用新版本内容讲解
- 管理端可以查看每个学员被分配的版本和打标详情（运营/KGP调试用）

### 18.5 用户故事

- **作为学员**，我希望学习内容自动匹配我的工作场景（销售岗位看销售案例，产品岗位看产品案例），以便内容对我有实际参考价值
- **作为学员**，我希望 AI 在切换内容版本时自然过渡，我感受到的是"AI 懂我"而不是"我被降级了"
- **作为 KGP 内容设计师**，我希望通过 Segment 版本管理为不同类型的学员准备不同的内容版本，以便实现精细化的自适应教学
- **作为 KGP 内容设计师**，我希望自定义版本切换的触发条件和规则，以便自适应策略完全由教学设计驱动
- **作为运营**，我希望能查看每个学员被分配的内容版本，以便了解自适应系统的实际运行效果

### 18.6 已确认决策

- 自适应切换的是 Segment（版本），不是 SCO
- 自适应规则不定固定权重，由内容设计决定
- L1 + L2 本期实现，L3/L4 未来迭代
- 互动 SCO 和打标 SCO 分离——互动负责采集，打标负责判定（Round-02 R1 确认）
- 打标结果严格限 Activity 内，跨 Activity 不传递（Round-02 R2 确认）
- 低置信度（< 0.6）降级为默认版本，不硬猜（Round-02 R2 确认）
- 打标对学员纯后台判定，但让学员知道有几个版本（Round-02 R2 确认）

## 19. 闭环反馈

### 19.1 问题背景

内测核心痛点：学员在课前分享了真实工作困境，但后续课程完全没有回应——"情感错付"。

### 19.2 闭环设计

本期做完整版。闭环反馈不是单一功能，而是贯穿整个学习旅程的设计原则。

闭环反馈拆分为两个独立 SCO 类型（Round-02 R2 确认），通过 `memory_id` 关联：

| 阶段 | SCO 类型 | 动作 | 数据流向 |
|------|---------|------|---------|
| **课前采集** | `FEEDBACK_COLLECT` SCO（Activity 开头） | 通过对话获取学员在某主题下的真实场景，以 `structured_tags` + `raw_dialogue` 双层格式存储 | → 记忆存储（统一 Database） |
| **课中关联** | 授课 SCO（含 `referenceSlots`） | AI 在必选引用槽处必须引用学员场景，可选位置 AI 自主判断 | 记忆 → 授课引用 |
| **课后复盘** | `FEEDBACK_REVIEW` SCO（Activity 末尾） | 通过 `memory_id` 调取课前采集数据，引导学员用所学知识重新拆解自己的真实问题 | 记忆 → 复盘 SCO |
| **辅导引用** | 辅导教练（大厅） | 回大厅必然触发 1 次引用（1-2 句话），话题相关时条件触发 | 记忆 → 辅导引用 |
| **报告呈现** | 报告模块 | 在报告中展示从"初始问题"到"知识应用"的完整证据链 | 记忆 → 报告 |

### 19.3 交互细节

- 课前采集：AI 自然对话引导 → "你在横向协作中遇到过什么困难？" → 学员描述真实场景 → 系统存入记忆
- 课中关联：AI 在讲解知识点时引用 → "就像你之前提到的那个跨部门协调的情况..." → 知识点与场景关联
- 课后复盘：AI 调取记忆 → "还记得你之前提到的那个情况吗？学完PIN法后，你现在会怎么处理？" → 学员用新知识分析
- 辅导引用：回到大厅后 → "你在课上重新分析了那个跨部门协作的场景，思路很棒。在实际工作中遇到类似情况可以..."

### 19.4 用户故事

- **作为学员**，我希望我分享的每一个真实工作问题最终都能得到回应——要么在课中被解答，要么在课后被引导解决，以便我愿意继续真诚分享
- **作为学员**，我希望课后复盘时 AI 能准确地引用我课前分享的具体场景，而不是泛泛地说"你之前的问题"，以便我感受到 AI 真的记住了我的分享
- **作为学员**，我希望辅导教练能把课上的复盘结论延伸到日常辅导中，给我更多在工作中应用的建议，以便学习和工作真正打通
- **作为学员**，我希望课前采集中不想分享时 AI 尊重我的选择（具体处理方式因课程而异），以便我不感到被强迫
- **作为 HR**，我希望报告中能展示学员从"提出问题"到"学会解决"的完整证据链，以便证明培训确实产生了效果
- **作为 KGP 内容设计师**，我希望通过闭环反馈 SCO 的设计控制课前采集和课后复盘的内容，包括学员拒绝分享时的处理策略，以便每门课程都有针对性的闭环反馈

### 19.5 已确认决策

- 闭环反馈本期做完整版
- 闭环反馈拆分为 `FEEDBACK_COLLECT`（课前采集）和 `FEEDBACK_REVIEW`（课后复盘）两个独立 SCO 类型，通过 `memory_id` 关联（Round-02 R2 确认）
- 课后复盘在 Activity 内完成（`FEEDBACK_REVIEW` SCO 放在 Activity 末尾），不在大厅进行（Round-02 R2 确认）
- 课前采集数据使用 `structured_tags` + `raw_dialogue` 双层存储（Round-02 R2 确认）
- 授课脚本通过 `referenceSlots`（含 `required` 标志）控制课中引用（Round-02 R2 确认）
- 辅导教练回大厅必然引用+话题条件触发（Round-02 R2 确认）
- 学员拒绝分享时的处理由脚本控制，不同课程可有不同策略（Round-02 R3 创始人决策）

## 20. AI 质量控制

### 20.1 定位

AI 质量控制作为后台质量控制机制，优先级低于前台功能。本期做基础版本——能后台查找溯源即可，不做复杂的四层架构。当前安排了后门（技术调试入口）。

### 20.2 基础版本功能

**核心能力：后台查找溯源**

运营和技术团队能够：
1. **查找**：按学员、时间、模块等维度检索 AI 输出的历史记录
2. **溯源**：查看每一条 AI 回复的上下文（学员输入、System Prompt、知识库引用、输出结果）
3. **标注**：对有问题的 AI 输出进行标注和分类
4. **后门**：技术团队通过后门入口进行调试和快速修正

**基础预防措施**：
- 授课模块使用半预制 SCO，偏离空间小
- AI 的 System Prompt 写入核心知识点断言清单（KGP 制作时提供）
- 对练模块 AI 角色的 System Prompt 定义行为边界

**学员端不呈现 AI 质量数据**——学员不关心也不需要知道 AI 质量控制的存在。

### 20.3 交互细节

- 运营/技术人员通过后台界面（不是三栏布局的运营端）进入 AI 质量控制面板
- 搜索栏支持按学员ID、时间范围、模块类型、关键词等条件检索
- 检索结果展示 AI 输出列表，点击每条可以查看完整上下文
- 标注功能：对有问题的输出打标签（"知识点错误"/"脱离角色"/"不当回复"等）
- 后门入口：技术团队可以查看和修改 System Prompt、知识点清单等

### 20.4 用户故事

- **作为运营**，我希望能在后台按学员和时间维度检索 AI 的历史输出，以便发现问题时快速定位
- **作为技术人员**，我希望能查看每条 AI 回复的完整上下文（输入、Prompt、知识库引用），以便准确诊断问题原因
- **作为运营**，我希望对有问题的 AI 输出进行标注分类，以便统计问题类型和频率
- **作为技术人员**，我希望通过后门快速修正 AI 的知识点清单或 Prompt，以便在发现问题后及时修复
- **作为产品负责人**，我希望 AI 质量控制不影响前台功能的开发进度，以便团队优先保证学员和管理端的核心体验

### 20.5 已确认决策

- 做基础版本，不做复杂的四层架构（预防-检测-响应-修复）
- 核心是后台查找溯源能力
- 当前安排了后门用于技术调试
- 优先级低于前台功能
- AI 质量数据不放在学员端或 HR 端

### 20.6 本期不做的功能

- 复杂的四层质量控制架构（预防→检测→响应→修复）——本期仅做基础预防+后台溯源
- 实时异常检测（5类异常信号自动检测）——未来迭代
- 分级响应机制（轻度/中度/重度/违规四级）——未来迭代
- 事后主动通知受影响学员——未来迭代
- HR 端 AI 质量看板——学员和 HR 不关心

## 21. 举证机制

### 21.1 核心承诺

产品的核心价值主张是"可举证的行为改变"——不是"课程完成率"，而是用行为证据证明培训产生了实际效果。

### 21.2 能力维度标准化

KGP 在内容制作阶段为每个项目定义 3-5 个能力维度（如"反馈沟通""冲突处理"等），作为贯穿全模块的统一评估坐标。

路径个性化，评估维度标准化——每个学员走的学习路径可能不同（通过 Segment 版本切换），但最终都在同一组能力维度上被评估。

### 21.3 证据采集方式

| 模块 | 采集方式 | 产出 |
|------|---------|------|
| 授课 | Quiz 结果 + 所有互动对话记录（半结构化）映射到能力维度 | 知识掌握度证据 |
| 对练 | AI 实时标注行为观察点（每个对练预绑定 1-2 个能力维度） | 行为表现证据 |
| 辅导 | AI 后台识别学员提问涉及的能力维度 | 求助行为和问题摘要 |
| 测评 | BEI 追问路径 + GUI 量表得分围绕能力维度展开 | STAR 案例 + 量表评分 |

### 21.4 AI 双轨运行

AI 在所有模块中执行"双轨运行"：
- **前台**：正常教学/辅导，不被后台采集干扰
- **后台**：异步标注能力维度和表现信号

关键约束：后台标注不得干扰前台对话的自然性——学员的操作性问题就回答操作性问题，不为采集数据而改变对话方向。

### 21.5 行为改变证据链

报告按能力维度汇聚全模块证据，生成"学习旅程证据链"：

```
初始表现（课前测评/学员画像）
  → 学习过程（授课互动/对练行为/辅导求助）
  → 最终表现（课后测评/对练改善/辅导对话质量提升）
```

数据可追溯到原始记录（半结构化对话记录）。

### 21.6 报告定位

"基于模拟情境的行为改变指标，作为传统评估方法的有力补充"——"补充"而非"替代"。

### 21.7 交互细节

- 证据采集对学员透明——学员在正常学习过程中自然产生证据，不需要额外操作
- 报告中的每个结论都有"查看证据"链接 → 点击展开原始对话记录
- HR 在查看报告时可以逐层钻取：结论 → 证据摘要 → 原始对话

### 21.8 用户故事

- **作为 HR**，我希望报告呈现的不是"完成了多少课时"，而是"在哪些能力维度上发生了什么行为改变"，以便 CEO 理解培训的实际价值
- **作为 HR**，我希望报告中的每个结论都可以追溯到学员的具体行为记录，以便在被追问时有据可查
- **作为学员**，我希望学习过程中不会因为"数据采集"而被干扰正常学习体验，以便我专注于学习本身
- **作为 KGP 内容设计师**，我希望为每个项目定义统一的能力维度，让不同模块的数据都映射到同一组维度上，以便生成连贯的证据链
- **作为运营**，我希望报告中的证据可以追溯到原始对话记录，以便在审核时验证数据的真实性

### 21.9 已确认决策

- 路径个性化，评估维度标准化——两者不矛盾
- AI 双轨运行：前台教学不受后台采集干扰
- 授课也记录原始对话（半结构化），作为证据来源

## 23. 迭代方向

### 23.1 六大迭代维度

| 维度 | 含义 | 示例 |
|------|------|------|
| **自适应（高级）** | 从版本切换到路径调整再到动态生成 | 学员某知识点已掌握 → 跳过；学员说"再给我出一道题" → 动态生成 |
| **模块质量** | 每个模块内部的智能性和交互丰富度 | 对练中教练实时介入；更丰富的 Quiz 交互 |
| **内容覆盖** | 同一软件系统支持不同内容主题 | 销售培训、项目管理、AI 协同等 |
| **增加模块类型** | 授课/对练/测评之外的新教室 | 沙盘推演、研讨、播客、案例拆解工坊 |
| **感知** | 感知学员学习状态、情绪、心理变化 | 交互埋点、情绪分析深化 |
| **定制** | 满足企业的定制需求 | 企业画像接入（非企业知识库）、品牌定制 |

### 23.2 近期优先（V2 方向）

1. **PPT 生成**——HR 汇报材料从报告升级到 PPT
2. **独立运营 Agent**——根据运营特殊需求独立化
3. **临时任务下发**——"单向群聊"模式
4. **智能预警**——在数据呈现基础上加入智能判断
5. **AI 质量控制深化**——从基础溯源升级到实时检测和分级响应
6. **企业画像接入**——获取方式待定
7. **角色互换对练**——通过不同剧本设计实现换位思考
8. **数据双向回流**——管理端分析结果反向优化学员端体验
9. **从辅导随时进入报告**——辅导中可随时查看历史报告

### 23.3 中远期方向

1. **多人对练**——多个 AI 角色 + 多个学员的团队模拟
2. **案例拆解工坊**——独立的新教学模块类型
3. **沙盘推演**——类似下棋的管理决策推演
4. **研讨社区**——AI 组织的学员讨论群
5. **跨项目学习**——学员在一个平台上参与多个培训项目，记忆和画像跨项目累积
6. **动态内容生成**——AI 根据学员实时需求生成新的教学内容
7. **辅导人格可替换**——通过 KGP 生产不同的 Context 包实现
8. **辅导风格按课程调整**——不同课程匹配不同辅导风格
