# 消息提醒模块

## 模块定位

消息提醒是平台向学员/HR/运营推送通知的统一通道。入口在 topbar 的铃铛🔔按钮，点击后下拉消息面板。本期消息只读，不支持回复。

---

## 入口与交互

- **入口**：topbar 右侧铃铛图标🔔
- **未读标识**：有未读消息时铃铛旁显示红点
- **点击行为**：下拉消息面板
- **面板布局**：
  - 顶部：标题"消息" + 右上角"全部已读"按钮
  - 中间：消息列表（按时间倒序）
  - 每条消息：固定高度展示，超出高度折叠，点击展开查看全量内容
  - 消息内容支持：文本 + 图片 + 链接
  - 点击消息本身：展开/折叠，不跳转。消息内的链接可点击跳转
  - 展开消息后自动标记为已读

---

## 三类消息

| 类型 | 来源 | 典型内容 | 触发方式 |
|------|------|---------|---------|
| **平台消息** | 系统 | 功能更新通知、内容更新提醒、版本发布说明 | 后台发布 |
| **系统消息** | 自动规则 | 催学提醒（"你已3天未登录"）、学习进度提醒 | 管理端配置的催学规则自动触发 |
| **运营消息** | 运营人员 | 群发通知、定点发送给特定学员的消息、学习鼓励 | 运营在管理端手动发送 |

## 用户故事：学员收到催学消息

### 场景背景
学员王芳已经 3 天没有登录学习了，管理端配置的自动催学规则触发了一条系统消息。

### 完整旅程

**步骤 1 — 消息触达**

王芳登录系统后，看到 topbar 铃铛图标旁有红点。

**步骤 2 — 查看消息**

她点击铃铛，下拉面板弹出，最顶部是一条未读消息（固定高度，显示摘要）：
> **系统消息** · 2分钟前
> "王芳你好，'横向协作'课程有新的学习内容等你来探索..."

**步骤 3 — 展开消息**

王芳点击这条消息，消息展开显示全量内容：
> "王芳你好，'横向协作'课程有新的学习内容等你来探索。上次你学到了 PIN 法概念讲解，继续完成可以解锁 PIN 法实战工具包。[点击继续学习](链接)"

消息展开后自动标记为已读。王芳点击消息中的"点击继续学习"链接，跳转到授课场域。

**步骤 4 — 批量已读**

王芳回到大厅，再次点击铃铛，看到还有 2 条旧消息（一条平台更新通知、一条运营鼓励消息）。她点击右上角"全部已读"，红点消失。

## 消息规则配置（管理端侧）

消息规则的配置和管理在管理端的「项目配置」中完成（详见 [management/03-project-config.md](../management/03-project-config.md)），不在消息提醒模块本身。

本模块只负责**接收和展示**消息，不负责配置和发送。

---

## 消息内容原则

> 该原则主要针对**系统消息**（自动催学类）。平台消息和运营消息由人工编写，不受此约束。

- **价值导向，而非催促导向**——催学消息应告诉学员"来了有什么好处"，而不是"你多久没来了"
  - 正确示例："'横向协作'课程有新内容等你来探索，继续完成可以解锁 PIN 法实战工具包"
  - 避免示例："你已经3天没有登录了，请尽快完成学习任务"
- **实现路径**：
  - **本期优先**：固定回复模板（运营可自定义模板，模板默认遵循价值导向原则）
  - **争取实现**：根据学员实际学习情况，通过大模型生成个性化催学消息（如引用学员上次学到的具体内容）

## 催学效果追踪

本期仅做站内消息，追踪漏斗为：**登录 → 查看消息 → 完成活动**（站内消息必须登录后才能看到，因此"登录"在"查看"之前）。

- **本期**：记录消息发送状态 + 已读状态
- **未来迭代**：根据 GPB 能力增加站外提醒渠道（微信/邮箱/短信等），站外渠道的漏斗为"发送 → 打开 → 登录 → 完成活动"，追踪能力更完整

追踪数据在管理端消息模块中展示（详见 [management/04-message.md](../management/[待审阅]04-message.md)）。

---

## 已确认决策

- 入口在 topbar 铃铛按钮，两端（学员端/管理端）统一
- 本期消息只读，不支持回复
- "全部已读"按钮在面板右上角
- 三类消息：平台/系统/运营
- 消息内容遵循"价值导向"原则
- 催学规则配置在管理端项目配置中，不在此模块
- 消息渠道：本期仅站内消息；未来根据 GPB 能力增加站外渠道（微信/邮箱/短信等）
- 纯 GUI，不涉及 Agent

## 反向补充：催学渠道优先级 + 4 阶段漏斗

> 来源：v0.3.3 §6.x 第 620-624 行 + §15.1.3 第 1650-1654 行（迁移于 2026-04-27）

本节是对上面「催学效果追踪」章节的精确化补充——明确 V1 必做/可选渠道，以及漏斗的 4 个阶段。

### 渠道优先级

| 优先级 | 渠道 | 说明 |
|--------|------|------|
| **V1 必做** | 产品内消息（辅导对话顶部 + 站内铃铛） | 学员登录后立即可见，对应漏斗的"查看"阶段 |
| **V1 必做** | Email | 学员账号关联邮箱，系统自动发送 |
| **V1 可选** | 企业微信 | 需企业 IT 预先完成 OAuth 授权 |
| **V1 可选** | 钉钉 | 需企业 IT 预先完成 OAuth 授权 |

> 注：本文上文「催学效果追踪」中描述的"本期仅做站内消息"对应 P0 最小必做范围；Email 是 V1 必做的第二条渠道（与 P0 同期），企业微信/钉钉为 V1 可选项（视客户 OAuth 就绪情况）。

### 完整 4 阶段漏斗

每条催学消息记录完整漏斗，用于评估规则效果：

```
发送 → 查看 → 登录 → 完成活动
```

| 阶段 | 站内消息含义 | 站外渠道（Email/企微/钉钉）含义 |
|------|--------------|-------------------------------|
| **发送** | 系统下发消息记录 | 系统调用渠道 API 完成投递 |
| **查看** | 学员打开站内铃铛或进入大厅 | 邮件被打开 / 消息送达回执 |
| **登录** | （站内消息此阶段已等同于查看） | 学员从催学链接进入平台 |
| **完成活动** | 学员完成催学指向的 Activity | 同左 |

> 站内消息天然要求"先登录后查看"，因此在站内场景下追踪漏斗为「登录 → 查看 → 完成活动」3 阶段；站外渠道追踪完整 4 阶段。

### 配置入口

催学规则及触发条件在**管理端配置页**设置（详见 [management/03-project-config.md](../management/03-project-config.md)），消息发送记录与漏斗数据在**管理端消息模块**查看（详见 [management/04-message.md](../management/04-message.md)）。
