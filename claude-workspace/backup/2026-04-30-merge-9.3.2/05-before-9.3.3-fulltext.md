#### 9.3.3 大纲（v4）

> **与 lecture / practice 8 子节结构差异**：
> - 多出 § 9.3.3.4 一整节专写"报告"——报告是 recap 的核心交付物，不是普通子节
> - **取消 SCO 编排**改为"单一对话场景"——recap 是融合的对话场景，不是结构化的内容序列
> - **Neo 行为按 3 种态切换**（首聊 / 回访 / 复习）—— 多次进入 + 报告会变是 recap 独有
> - **SCO 锁定较弱**——recap 整个 Activity 入口可随时进，与 lecture / practice 严格顺序锁定不同
> - 完成判定有 milestone 概念 + "数据更新不退回未完成"

##### 9.3.3.1 业务定义

| 子节 | 内容要点 |
|------|---------|
| **9.3.3.1.1 一句话定位** | recap = Course 内的"反思镜场域" —— 学员可随时进入查看 Neo 对自己学习的当前判断，并与 Neo 对话挖掘洞察。报告随前序 Activity 完成增量更新 |
| **9.3.3.1.2 三场域差异 + 与 practice 复盘的边界** | lecture(吸收) / practice(行为养成) / recap(反思迁移)；与 practice Activity 内复盘的关键边界：practice 复盘 = Activity 内单次行为复盘 / recap = Course 级跨多 Activity 的反思整合 |
| **9.3.3.1.3 "随时打开 + 增量更新"机制（★）** | 区别于 lecture / practice 严格 SCO 锁定，recap 可随时进入。报告以"前序 Activity 完成"为更新单位，后台静默更新，学员进入时直接看最新版本 |
| **9.3.3.1.4 学员进入时的 4 种状态** | 空白态（0 个前序完成）/ 部分态（部分完成）/ 完整态（全部前序 ≥ 1 次完成）/ 重练态（某 practice 多次重练）—— 报告随状态自适应 |
| **9.3.3.1.5 成功标准** | 学员首次完整走完 Neo Walk-through + 对话有实质性反思 = milestone 完成（详见 § 9.3.3.8.1）|

##### 9.3.3.3 对话场景编排（取代传统 SCO 编排）

> **设计理由**：recap 是融合的对话场景（看报告 + 与 Neo 聊 + Neo 给行动建议三件事在同一屏幕同时进行），不是分阶段的内容序列。强行套 SCO 会扭曲体验。recap 内部行为是产品级统一规则，KGP 不干预。

| 子节 | 内容要点 |
|------|---------|
| **9.3.3.3.1 场景结构** | 单一连续对话流：进入 → Neo 按当前态开场 → 学员 ↔ Neo 自由对话 → 首聊态走完后 Neo 自动产出应用计划入大厅待办 → 完成或退出 |
| **9.3.3.3.2 KGP 在 recap 上的能力边界** | KGP 只控制两件事：① 把 recap Activity 摆放在 AOM 中的位置 / ② 决定 recap 关联哪些前序 Activity 作为数据源（**关联范围由 KGP 自由控制，不强制与 Course 绑定**——本期典型场景是关联同 Course 内前序 Activity，但功能上不限制 KGP 跨 Course 关联或仅关联部分）。recap 内部行为（Walk-through 节奏、反思引导、应用计划产出、完成判定）= 产品级统一规则，KGP 不干预 |
| **9.3.3.3.3 Neo 多次进入差异化（★ 三态切换）** | 首聊态（首次进入或上次未走完）= 完整 Walk-through 5 块 + 走完后产出应用计划 / 回访态（再次进入 + 有实质性变化）= 不重讲全程，主动指出变化点 / 复习态（再次进入 + 无实质变化）= **Neo 自主判断是否开场**（可问可不问），主体是被动答疑 |
| **9.3.3.3.4 三态切换的数据信号** | 综合多个信号 Neo 自主判断（spec 不规定固定阈值）。**关联 Activity 内信号**：是否有新关联 Activity 完成 / 关联 Activity 内是否有新高光。**全场景信号**：6 维画像是否有明显变化（来源不限）/ 学员在大厅或其他场景是否有新对话 / 学员上次在 recap 内跟 Neo 聊到哪 / 上次 Walk-through 走到哪一块 |
| **9.3.3.3.5 学员对"重新 Walk-through"的主动权** | 学员通过对话表达诉求（"我们再过一遍"）— Neo 自然响应；不提供独立按钮（贴合"老师"体感，详见 § 9.3.3.6.6）|
| **9.3.3.3.6 应用计划产出（★ 重要）** | **应用计划由 Neo 产出**（不是学员手写承诺），是 Neo 基于报告全数据 + 跨场域记忆给学员的"建议下一步行动"（1-3 条具体小任务粒度）。**仅在首聊态走完 Walk-through 后生成一次**，后续不更新（即使报告增量更新、学员重进 recap，应用计划不再变）。生成后**自动加入大厅"待办清单"**（沿用 § 9.2.2.1 (1) E + § 9.2.1 C.3.4 待办清单机制），后续追踪 / 勾选完成 / Neo 反馈追问 全部在大厅待办流程内 |
| **9.3.3.3.7 recap 在 AOM 中的位置** | 由 KGP 决定（典型末尾、可半程、可项目末，详见 § 6.7）|

##### 9.3.3.4 Course 个人报告 · 数据与生成（★ 核心节）

> **作用**：定义报告的生成机制 + 数据来源 + 内容骨架 + 状态自适应规则。报告是 Neo 引导对话的"道具"，写到 70%，留 30% 给 Neo Chat 展开。

| 子节 | 内容要点 |
|------|---------|
| **9.3.3.4.1 生成机制** | 触发：每次"前序 Activity 完成"事件触发后台增量重新生成；学员进入 recap 时直接看最新缓存。**不保留历史快照**（每次只显示当前最新；学员看上次的版本本期不做）|
| **9.3.3.4.2 报告骨架 · 5 块结构** | ① Course 锚点（学员目标 + 时间）/ ② 走过的轨迹（部分态主显，完整态压缩；含互动量/停留时长的"投入度"一句话）/ ③ 6 维画像 · 当前快照（★ 灵魂）/ ④ 关键证据（高光 + 卡点 + 「我以为 vs 实际」偏差）/ ⑤ Neo 留言（基于全数据 + 跨场域记忆 + 学员开营目标的"班主任谈话"）|
| **9.3.3.4.3 数据来源全表** | 5 大类标注可用性等级：A. AOM 内容侧 / B. 学员前序行为（B.1 lecture / B.2 practice / B.3 跨 Activity 互动量）/ C. Neo 沉淀（含 6 维画像 / 4 维实时感知 / 高光库 / 行为模式标签）/ D. 学员主动产出（笔记 / 跨场域 Neo 记忆）/ E. 开营项目级（学员目标 / 项目周期 / 标签）。每数据项标"直接可用 ✅ / 需 KGP 配合 ⚠️ / 本期争取 ⚠️" |
| **9.3.3.4.4 报告呈现规范** | markdown 主体 + 必要图表嵌入（最多 2-3 张）；图表 = 结构化数据可视化（如 Quiz 命中柱图、6 维画像变化图），markdown = 非结构化洞察。报告**不可修改**（学员侧）+ **场域内不可导出**（沿用 § 9.3.5.6）。报告全长 500-800 字 / 阅读时长 2-3 分钟 |
| **9.3.3.4.5 状态自适应矩阵（★）** | 5 块 × 4 状态（空白 / 部分 / 完整 / 重练）的呈现规则表——每块在数据不足时如何降级（不显示 / 占位 / 标注） |
| **9.3.3.4.6 6 维画像在报告中的呈现** | **直接呈现"当前快照"**（不澄清跨场域数据来源——6 维画像本就是综合产物）。具体形态（数值表 / 进度条 / 雷达图 / 文字描述）—— PM 决策点 D1 |
| **9.3.3.4.7 关键证据块的内容选择规则（含作用域）** | 关键证据是"显性证据"——**严格限定 recap 关联 Activity 内**，关联外的不进。高光 1-2 条（数据源：关联 Activity 内高光库）/ 卡点 1-2 条（数据源：关联 practice 复盘 + 关联 lecture Quiz 错题模式）/ 「我以为 vs 实际」偏差 0-2 条（依赖 KGP 在 lecture FEEDBACK_COLLECT 设结构化字段，PM 决策点 D5）。关联内数据不足时**宁可空着也不补关联外**（保护"老师真实"感）|
| **9.3.3.4.8 Neo 留言的生成原则** | "隐性背景"——基于全数据 + 跨场域记忆 + 学员开营目标的"班主任谈话"，**不限定数据来源**（含大厅 / 跨 Course）；表达观察 / 担心 / 期待；师长口吻；为 Neo Chat 提供锚点。一次性生成，不可改 |
| **9.3.3.4.9 报告 diff 标识机制** | 报告生成时记录 vs 上版的变化点（哪些数据是新的 / 哪些块有变化）—— 给 Neo 判断回访态用 + Subbar 小红点提示用。学员侧不需看到详细 diff |
| **9.3.3.4.10 数据降级原则** | 数据不足时 spec 默认行为：不硬塞 / 不编造 / 写占位说明（保护"老师真实"感）。每块都有降级规则（详见 § 9.3.3.4.5） |
| **9.3.3.4.11 历史快照保留规则** | 本期不保留历史快照——报告是当下视图，每次更新覆盖旧版本 |

##### 9.3.3.5 Neo 行为（反思态）

| 子节 | 内容要点 |
|------|---------|
| **9.3.3.5.1 反思态定位** | 同 Neo 身份（与 § 3.2 + § 9.3.0 一致）/ 在 recap 中以"反思态"出现 —— 区别于 lecture Tutor 态 / practice Coach 态 |
| **9.3.3.5.2 7 动作分工** | 反思 ★ 主场 / 追问 ★ / 讨论 ★ / 解惑 ✓ / 考核 ✓ / 讲授（轻量补漏，不主动）/ 练习 —— 不出现 |
| **9.3.3.5.3 三态行为规范** | 首聊态 = 完整 Walk-through 5 块 + 走完后产出应用计划入大厅待办；回访态 = 主动指出变化点 + 引用上次对话 + 询问学员近况；复习态 = **Neo 自主判断是否开场**（可问可不问），主体被动答疑（详见 § 9.3.3.3.3） |
| **9.3.3.5.4 Walk-through 节奏** | Neo 主动逐块引导学员看报告（首聊态时）；不让学员自己默读完事 |
| **9.3.3.5.5 留白式追问规范** | "你看到这条洞察吗？想到了什么？" / "为什么你觉得这维度低？" —— 不替学员说总结，引发学员自陈（沿用 § 9.3.3.7.1 ★ 核心约束）|
| **9.3.3.5.6 跨场域记忆调用（★）** | Neo 主动引用学员开营时说过的目标 / lecture FEEDBACK 里的自陈 / practice 演练里的具体行为；让"老师记得你"成为体感 |
| **9.3.3.5.7 必收尾约束** | Neo 追问之后必须给收尾（沿用 practice 复盘 § 9.3.2.6.5）—— 不能只追问留开放状态 |

##### 9.3.3.6 学员行为与核心交互

| 子节 | 内容要点 |
|------|---------|
| **9.3.3.6.1 输入方式** | 右栏 Neo Chat 单输入框（打字 / 语音）—— recap 内学员只跟 Neo 对话，不写应用计划（应用计划 Neo 产出） |
| **9.3.3.6.2 核心交互一览** | 多数沿用 lecture（防挂机 / 笔记悬浮球 / Activity 间跳转 / 断点续播）|
| **9.3.3.6.3 "随时进入"特别机制** | 学员可在 Course 任意时刻通过抽屉 Tab2 跳进 recap；4 状态进入处理详见 § 9.3.3.1.4 + § 9.3.3.4.5 |
| **9.3.3.6.4 部分态进入时的引导** | Neo 第一句明确："这只是基于你已完成的部分，回头多做几个再来看，画像会更准" |
| **9.3.3.6.5 应用计划呈现** | Neo 产出（详见 § 9.3.3.3.6 + § 9.3.3.4），学员侧不编辑、不确认；首聊态走完 Walk-through 后报告内显示"应用计划已加入大厅待办清单 ✓ {N 项}"+ 跳转大厅链接 |
| **9.3.3.6.6 学员主动请求"重新 Walk-through"** | 通过对话表达（"我们再过一遍"）—— Neo 自然响应；不提供独立按钮 |
| **9.3.3.6.7 重看 / 重进** | 完成后再进 = 三态切换决定行为（见 § 9.3.3.3.3）；报告随前序新完成自动更新；学员看到"📊 报告有更新"小红点时进入会触发 Neo 回访态 |

##### 9.3.3.7 关键约束

| 子节 | 内容要点 |
|------|---------|
| **9.3.3.7.1 不替学员总结洞察（★ 核心）** | Neo 引导学员说，不替学员说；防止报告 / 对话变成 Neo 的"评分书" |
| **9.3.3.7.2 不预设标准答案 / 不给"应该"** | 沿用 § 2.6.2 + § 9.3.1.6.1 |
| **9.3.3.7.3 报告不可修改** | 整体一次性生成，学员侧不可改；下次更新覆盖 |
| **9.3.3.7.4 报告导出规则** | 场域内不导出（沿用 § 9.3.5.6）；大厅报告库可导出 |
| **9.3.3.7.5 与管理端 team report 边界** | Course 个人报告 = 学员侧反思场景 / team report = 管理端阶段性对外汇报；数据可能共享（如 6 维画像）但呈现完全分开 |
| **9.3.3.7.6 数据降级原则** | 不硬塞 / 不编造，呼应 § 9.3.3.4.10 |
| **9.3.3.7.7 6 维画像可见性** | 学员侧报告露出（含趋势）/ 管理端 D-5.0 不露出（与 § 9.3.2.3.3 矩阵决策一致）|
| **9.3.3.7.8 取消 SCO 锁定** | recap 整个 Activity 可随时进；不存在 SCO 序列，与 lecture / practice 严格顺序锁定不同 |
| **9.3.3.7.9 Walk-through 主动权** | 学员通过对话表达"重新走" —— 不提供独立按钮 |
| **9.3.3.7.10 显性证据限定关联 Activity** | 报告内可被学员看到具体来源的"证据类"内容（高光 / 卡点 / Quiz 错题模式 / 对练具体行为）严格限定 KGP 配的关联 Activity 内；关联外的不进 |

### 10.1 管理端总览

```
登录管理端
```

#### 10.1.4 HR vs 项目运营 vs admin 协同

| 角色 | 主要行为模式 |

### 10.2 看板 Operation

#### 10.2.1 业务定义

管理员日常监控项目运行健康度的场域。三维度监控 + 热力图 + Ora 双交互。

#### 10.2.2 三维度监控

| 维度 | 内容 |
|------|------|
| **项目维度** | 项目整体健康度 / 进度 / 完成率 |
| **人员维度** | 学员个体进度 / 异常学员 / 活跃度分布 |
| **内容维度** | 各 Course 学习情况 / 卡点分布 / Quiz 通关率 |

#### 10.2.3 热力图

学员行为热力可视化——每个学员每日活跃度色块 / 每个 Course 学员通过率色块。

#### 10.2.4 Ora 双交互模式
| 模式 | 触发 | 行为 |
|------|------|------|
| **看板被动呈现** | 默认呈现 | 三维度数据 + 热力图 + Ora 自动生成的项目健康度解读 |
| **Ora 问答** | 管理员主动提问 | "为什么 W4 脱训突增？" / "张三这周参与度怎样？" → Ora 用证据回答 |

> **本期不做**：看板点击 → Ora 实时解释——这是未来形态（详见 § 3.3.4）

#### 10.2.5 异常学员处置

按 § 6.6.2：

- **平台层**：Ora 看板 / 问答 中发现异常学员（脱训 / 进度落后 / Quiz 频繁失败）
- **线下层**：项目运营**线下跟进**该学员（电话 / 微信 / 1on1 沟通）
- **平台不强制干预**——干预靠人 / 渠道完成

#### 10.2.6 颜色条 / 排序 / 详情页

- **颜色条标识健康度**（绿 / 黄 / 红）
- **异常自动排顶**（异常项目 / 异常学员 自动排在顶部）
- **详情页"建议动作"** —— Ora 给出建议动作（详见 § 3.3.5）

### 10.3 团队报告生成 Team Report-Gen

> 这是 § 10 的**核心模块**——管理端的主要交付能力，承担中期反馈 / 半程 review / 结项报告 / 剩余服务期内的新报告 全部生成场景（详见 § 6.7 + § 6.8）。

#### 10.3.1 业务定义

管理员产出对外可举证报告的场域——Ora 拉全程数据 + 多轮对话生成初稿 + 管理员 HITL 编辑 + 导出 PDF / Word。

#### 10.3.2 引擎核心 · 每管理员独立报告库
- **每个管理员**有**独立的报告库**——存放该管理员生成 / 编辑的全部 team report
- **里程碑触发时**给每个管理员一份**独立草稿副本**
- **本期不做协同分享 / 不做实时多人编辑** —— 需协同时由管理员**导出后线下同步**

> 这个设计简化了多人协同的复杂度——研发只需保证每个管理员的报告库独立，不需做实时协同编辑。

#### 10.3.3 prompt 模板由"关联动作 / 评估要点"驱动
- 管理员**不需要选模板，也不手写 prompt**
- 管理员在创建 team report（或里程碑配置）时**填"关联动作 / 评估要点"**（如"评估学员中期参与度 + 概念掌握度"）
- 系统据此**自动生成**对应的 prompt 模板

> 本期 v0.5.0 优先做 **"结项报告 prompt 模板"** —— 用作里程碑结营时的默认 prompt。其他模板（中期 / 半程 / 自定义）后续逐步加（详见 § 6.7.1）。

#### 10.3.4 多轮对话生成 + HITL 编辑

生成流程：

```
里程碑到期触发 / 管理员手动触发
   ↓
Ora 拉项目全程数据
   ↓
Ora 与管理员多轮对话（确认范围 / 学员维度 / 数据筛选 / 文本风格）
   ↓
Ora 生成初稿
   ↓
管理员 HITL 编辑（增删改 / 调整结构 / 改文案）
   ↓
导出 PDF / Word
```

#### 10.3.5 与里程碑的关联

详见 § 6.6.3 + § 6.7.1：

- 里程碑到期当天 → Ora 自动生成 team report 草稿（每管理员一份副本）
- 站内消息提前 3 天每日提醒管理员
- Ora 在草稿生成后即时推送通知
- 过期未处理 → 提醒保留在列表（不堆累 / 不重复推送 / 不升级渠道）

#### 10.3.6 模板含义澄清
按 § 6.7.1：

> 本节"模板"指的是 **prompt 约束的写作规则**（即给 Ora 的"写作 brief / 文章骨架 / 核心要素清单"），**不是固定文章模板或 HTML 模板**。
>
> Ora 基于模板的写作约束 + 项目实际数据 + 关联动作 / 评估要点 → 生成具体文本。
