## 用途

在产品/技术决策点上，用 5 个固定角色（PM、Dev、UX、Student Rep、HR Rep）进行结构化辩论，强制暴露各方立场冲突，让用户在充分信息下拍板。

适用场景：
- 多个产品方案选其一（如：导出格式选 PDF 还是 Markdown？）
- 架构决策（如：单端口 vs. 多端口）
- 信息架构 / 模块划分（如：催学规则放 config 还是单独模块）
- 任何"看起来都对但只能选一个"的设计岔路

## 何时触发

**显式触发**（用户明确说）：
- "5 角色辩论一下"
- "做个辩论"
- "辩论这个问题"
- "5 方对线"
- "让 PM/Dev/UX/Student/HR 都说说"

**隐式触发**（Claude 主动建议）：
- 用户给出 2 个及以上方案，且问"选哪个"
- 用户在做关键架构决策，单一视角不够
- 用户已收集多方反馈但意见冲突

主动建议时的措辞：
> "这个问题涉及 [产品/技术/体验/学员/管理] 多方权衡，建议走 5 角色辩论流程，需要我启动吗？"

## 5 个角色 + 立场

| 角色 | 简称 | 优先关注 | 典型立场 |
|------|------|----------|----------|
| 产品经理 | PM | 用户体验闭环、商业可行性 | "这个流程必须能让用户跑完闭环，否则就是半成品" |
| 研发经理 | Dev | 技术实现复杂度、性能 | "这个方案改动太大，2 周内做不完" |
| 交互设计师 | UX | 视觉一致性、交互流畅 | "组件库里没这个模式，新增会破坏一致性" |
| 学员代表 | Student Rep | 新手友好、学习负担 | "我学习时不想看到 5 个按钮，选项太多就放弃了" |
| 企业 HR 代表 | HR Rep | 管理效率、数据可见性 | "我要能一眼看到 100 个员工的进度，不是逐个点开" |

**角色发言规范**：
- 每次发言 50-150 字
- 必须包含「立场 + 依据」（依据可来自项目 spec、行业经验、用户反馈）
- 立场要鲜明，避免"也行也不行"的中庸表达
- 优先用第一人称（"我作为 PM 认为……"）

## 标准流程

### Step 1：列出待辩论议题

主持人（Claude）先开场：
```
本轮辩论主题：[主题]
待辩论议题（1-3 个）：
1. [议题 1]
2. [议题 2]
3. [议题 3]
```

如议题超过 3 个，建议用户拆分成多轮，避免单轮过长。

### Step 2：每个议题 5 角色依次发言

按 PM → Dev → UX → Student → HR 顺序，每个角色一段：
```
**PM**：[立场 + 依据，50-150 字]
**Dev**：[立场 + 依据]
**UX**：[立场 + 依据]
**Student**：[立场 + 依据]
**HR**：[立场 + 依据]
```

### Step 3：识别冲突，触发反驳

主持人扫描 5 段发言，标出关键冲突：
```
冲突点：
- PM vs Dev：PM 要 X 功能，Dev 说 2 周做不完
- UX vs Student：UX 要保持一致性，Student 觉得现在太复杂
```

冲突方点名互相回应一轮（每方 100 字内）：
```
**PM 反驳 Dev**：……
**Dev 回应 PM**：……
```

### Step 4：主持人总结

```
已对齐：
- [双方/多方达成一致的点]

待用户决策：
1. [选项 A：xxx（支持方：PM、Student）]
2. [选项 B：xxx（支持方：Dev、HR）]
3. [选项 C：折中：xxx]
```

### Step 5：用户拍板

用户回复后，主持人记录决策：
```
## 用户决策（[日期]）
选定：选项 [X]
理由：[用户给出的或主持人推断的理由]
```

### Step 6：归档

输出文件：`process/debates/round-N-confirmed.md`
- N 自增（查 process/debates/ 目录已有最大 N）
- 含完整发言、反驳、决策、归档时间戳

## 输入 / 输出

**输入**（用户提供或 Claude 提取）：
- 主题（一句话）
- 议题列表（1-3 个）
- 背景 / 当前已知信息（可选，越多越好）

**输出**：
- 实时对话：5 角色发言 + 反驳 + 决策选项
- 归档文件：`process/debates/round-N-[主题简写]-confirmed.md`

## 范例参考

本项目原型规划阶段已完成 3 轮辩论（导出格式、催学规则归属、HITL 模式），归档于 `process/debates/`。可作为本 skill 的标准范例。

## 注意事项

- 角色立场要"硬"：不要让所有角色都附和 PM，制造真实冲突才有价值
- 反驳轮次最多 1-2 轮，避免无限循环
- 用户拍板后，必须把决策原文（含选项编号）写入归档文件，方便后续追溯
- 如本轮决策影响架构，应同步建议用户在 `00-public/4-decisions/` 新增 ADR
