# 01-overview.md 审阅记录

## 总体评估

§ 10 章首 + § 10.1 总览定位准确——把"双状态 / 4 模块清单 / 入口 / 角色协同 / Ora 双实例 + 4 类卡片 Truth Source"集中在一个 126 行的入口文件里，**作为 06-management/ 板块的索引页是合格的**。子节顺序清晰（业务定义 → 行为表 → 模块清单 → 入口 → 角色 → Ora Truth Source），与 D1 维度要求的 6 子节模式完全一致。但 Ora Truth Source（§ 10.1.6）有 **2 处具体子节落点指向不准** + 模块清单与"4 模块 vs 5 文件"的呈现存在轻微割裂。

## P1 · 建议改（结构不清晰 / 内部重复 / MEC 重叠）

- **§ 10.1.3 模块清单 4 模块 vs 文件目录 5 文件 张力未在文件内消解**：CLAUDE.md 提示 "管理端 4 模块 vs 5 模块"——01-overview.md 自己是第 5 个文件但被叫"总览"不是"模块"，理念上 OK，但章首"本章描述管理端的整体形态 + 4 个核心模块"与文件结构并存时**没有任何一句话说"01-overview 不是模块、是入口/索引"**。建议章首加一句："本章拆 5 文件——本文件 = 总览 / 后 4 文件 = 4 核心模块"，与 04-toc-tree-2026-05-05.md 的"4 模块"叙事对齐。

- **§ 10.1.6 的章节标题与位置略不平衡**：章节叫"Ora 在管理端的实现：双实例 + 4 类卡片"，承担 Truth Source 职能；但前面 § 10.1.1-1.5 都是"管理端总览级"的（业务定义 / 行为 / 模块清单 / 入口 / 角色协同），到了 § 10.1.6 突然进入"Ora 实现"这种 spec 级具体定义。**建议改标题为"§ 10.1.6 Ora 在管理端的实现（Truth Source）"** 并在小节首句明确说"本节是 Ora 双实例 + 4 类卡片定义的唯一权威源 / § 10.2 / § 10.3 仅各自场景下的使用细化"——让"Truth Source"地位显式化。

- **§ 10.1.5 角色协同表与 § 10.1.2 行为表轻微重叠**：§ 10.1.2 的"开营前 admin 行为 3 类 + 开营后管理员 7 类"已映射对应模块 / § 10.1.5 表又说 "admin = 配置全权 + 创建账号 + 锁定" / "管理员 = 看板巡视 + 异常处置 + ..."——两表信息重叠 60%。**建议**：§ 10.1.5 改为只讲"角色合并理由（HR + 项目运营）"+ 删掉与 10.1.2 重复的"主要行为"列；或反过来 10.1.2 仅列"行为类型 + 时机 + 模块"，删去 admin/管理员 角色判断重复。

- **§ 10.1.4 入口 / § 10.1.6.1 Ora 不参与模块** 在内容上是对偶但分两节给——管理员从导航第 X 项进 / Config + Message 是纯 GUI 不参与 Ora。**建议合并思路**：§ 10.1.4 提到 4 模块入口时已经把 Cfg/Msg 列为纯 GUI 的提示嵌进去，避免两处呼应散乱。
