# 豆包 0414 会议记录分析

**日期：** 2026-04-14
**内容：** 创始人与豆包的学员端用户故事讨论
**目的：** 对比需求文档 v0.3.3，找出新信息、差异、以及对拆解深化的启示

---

## 一、会议中讨论了什么

按顺序覆盖了学员端全部模块的具体用户故事：

1. **授课模块**（最深入，约60%篇幅）
2. **对练模块**（三种变体+重练）
3. **测评模块**（三阶段结构）
4. **个人报告模块**（三部分内容）
5. **大厅/辅导模块**（看板功能）
6. **运营端/HR端**（共性框架）
7. **后台设置模块**（新增内容）

---

## 二、需求文档 v0.3.3 需要调整的内容

### 调整1：授课模块的"具体一节课是怎么上的"——v0.3.3缺这个粒度

v0.3.3 描述了 SCO 类型、自适应、插嘴提问等功能，但**没有一个完整的"一节课从头到尾怎么上"的端到端走查**。

会议中用 PIN 法那节课做了详细走查：
```
Quiz（课前场景采集）→ 负面案例视频 → P概念PPT → I概念PPT → N概念PPT 
→ 概念验证Quiz（连线/选择）→ 正面案例视频 → 案例分析Quiz 
→ 四象限方法PPT（逐象限讲解）→ 综合升华 → 基于学员初始场景的反馈与辅导
```

**启示：** 拆解深化不应该从"功能"维度拆，应该从**"一节具体课是怎么上的"**维度拆。功能树和功能流程太抽象了，缺乏真实课程的场景感。

### 调整2：SCO之间的互动——"既不要每个都加，也不要全去掉"

v0.3.3 说"纯讲述SCO需要加互动，但要找平衡点"，但没给出具体方案。

会议明确了：
- 之前做过两个极端版本（每SCO都插互动 / 全部取消只用Quiz），效果都不好
- **正确方向：AI根据内容逻辑和学员状态自主决定是否插入互动**
- 不要和时间绑定（"每5分钟一次"太机械）
- 要和内容逻辑绑定（"讲完一个完整知识点模块后，如果偏抽象理论，就插一个结合工作的互动"）

### 调整3：SCO锁定机制——v0.3.3没明确说

会议明确：**未播放到的SCO是锁定状态**，学员只能回顾已学完的，不能跳到后面。这在v0.3.3中没有写。

### 调整4：笔记功能的形态——悬浮窗/悬浮球

v0.3.3 说笔记在左侧看板。会议明确：
- **笔记和授课是独立的两条线**
- 用**浮窗或悬浮球**形式实现
- 可以随时打开记录
- 支持打字、编辑、粘贴、插入截图、保存
- **本期不和授课内容打通**（未来再说）

### 调整5：防挂机机制

v0.3.3 没提到。会议明确：**每5分钟进行一次提醒或暂停**。

### 调整6：AI自主出题——放在答疑而非新增SCO

会议讨论了"AI根据内容自主出一道小题考学员"的场景：
- 不应该新增SCO（会影响AOM数据结构）
- **放在答疑模块**——AI在答疑对话中出小题考一下
- 未来是否由AI自主新增SCO，后续再讨论

### 调整7：对练三种变体的确认+重练细节

v0.3.3 列了玩法但不够具体。会议明确了三种变体的完整定义：
1. **基础Roleplay**：学员扮演目标角色，AI扮演被解决的人
2. **角色互换**：AI扮演你的角色，你扮演客户/被解决的人→**复盘时切回原身份**
3. **费曼**：AI扮演什么都不知道的学生，学员当老师讲解

**重练机制**：
- 只做Roleplay阶段的重练（不含前导和复盘）
- **老的练习不被覆盖**，新旧同时记录
- 复盘时可以对比多次练习的表现差异

### 调整8：测评三阶段结构的明确

v0.3.3 说"两段结构（测+评）"，会议调整为**三阶段**：
1. **测评说明**（场景铺垫）：告诉学员测什么、怎么测
2. **核心测评**（执行）：量表或对话式访谈
3. **信息确认复盘**：AI总结收集的信息，逐一向学员确认是否准确

**关键区别**：测评复盘 ≠ 对练复盘。对练复盘是苏格拉底深度思考，测评复盘是**信息确认**（"我总结的对不对？"）

### 调整9：个人报告内容——三部分

v0.3.3 列了8个报告模块，会议简化为三部分：
1. **课程核心内容回顾**——学了什么
2. **个人课堂表现复盘**——高光表现+待改进点（用具体案例支撑）
3. **下一步应用计划**——可落地的行动步骤

### 调整10：工具库——逐步解锁

v0.3.3 说"左侧看板有工具卡片"，会议明确：
- 工具库是左侧看板的子功能
- **随课程进度逐步解锁**（类似游戏背包）
- 解锁后可下载带走

### 调整11：进度看板——热力图+悬停卡片

v0.3.3 只说"学习进度"。会议给出了具体的可视化设计：
- **热力图**：以周/月为单位，颜色深浅代表投入度
- **悬停卡片**：鼠标悬停某天→弹出柱状图，展示各模块（授课/对练/测评/辅导）的具体投入
- 指标：时间+token消耗

### 调整12：AI语速调整——后台设置而非播放界面

v0.3.3 没提到语速。会议明确：
- 1倍速~1.5倍速
- **在后台设置界面调整**，不在播放界面
- 所有对话场景统一应用
- **目的：不让学员以为这是视频课**

### 调整13：运营端/HR端——当前完全一致

v0.3.3 分了HR端和运营端。会议明确：
- **本期两个端口界面完全一致**
- 两个独立端口，但功能和权限相同
- 未来再区分

### 调整14：后台设置模块——新增内容

v0.3.3 中缺失。会议定义了：
- **自动提醒机制配置**（2天/5天等触发周期，可配置）
- **平台个性化配置**（LOGO、老师形象、老师音色）
- **角色管理入口**（当前只有HR+运营，未来可新增组长等）
- **数据统计**：
  - 学习项目进度统计（已进行天数/剩余天数/完成比例）
  - Token消耗统计（总量+按日/周趋势）
- **项目周期配置不开放**——由技术团队后端管控

## 三、对拆解深化的启示

### 核心问题：之前的拆解逻辑不对

之前的拆解路径是：**功能树→功能流程→UX**。这是自顶向下的"系统视角"拆解。

但会议暴露的问题是：**真正的产品细节是在"一节具体课怎么上"这个粒度才能讨论清楚的**。

### 建议的新拆解方式

**不再按"功能模块"拆，而是按"用户故事场景"拆：**

```
场景1：学员上一节PIN法课（25分钟完整旅程）
  → 每个SCO的交互细节
  → AI在什么情况下插入互动
  → 学员插嘴时AI怎么响应
  → 回顾跳转怎么工作
  → 闭环反馈的完整链路

场景2：学员做一次对练（基础Roleplay）
  → 前导铺垫的交互
  → Roleplay的10-20轮对话
  → AI表情切换
  → 反思复盘的苏格拉底追问
  → 重练流程

场景3：学员做一次测评（对话式BEI）
  → 测评说明
  → 访谈过程
  → 信息确认复盘

场景4：学员看个人报告
  → 三部分内容展示
  → AI解读讨论

场景5：学员在大厅的一天
  → 辅导教练对话
  → 看板浏览
  → 工具库解锁
  → 热力图查看
  → 消息提醒

场景6：运营/HR的一天
  → 团队看板
  → Agent查询
  → 报告生成定制
  → 后台设置
```

每个场景做到**"坐在学员/运营旁边看他操作一遍"**的粒度，而不是"功能列表+流程图"的粒度。
