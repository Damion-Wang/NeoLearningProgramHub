# 第二轮产出：快照拆分方案

> 日期：2026-04-20
> 主题：13条旅程如何拆分为静态 HTML 快照，每个快照展示什么

---

## 辩论纪要

### 一、布局骨架分类

**Dev 先发言：** 在开始逐旅程讨论之前，我先把 spec 里识别出来的布局骨架归类，这样后面讨论可以直接引用编号，省得每次重复描述。

| 骨架代号 | 描述 | 适用场景 |
|---------|------|---------|
| **L-A** | Topbar + 可折叠左侧导航 + 中间看板区（长页面滚动）+ 右侧Chat | 学员端大厅 |
| **L-B** | Topbar + 可折叠左侧导航(SCO列表/Activity列表Tab) + 中间课件区 + 右侧Neo Chat | 授课场域 |
| **L-C** | Topbar + 可折叠左侧导航 + 左栏内容 + 右栏Neo Chat（双栏） | 对练(情景导入/反思复盘)、调研(说明/确认)、报告 |
| **L-D** | Topbar + 可折叠左侧导航 + 左栏(头像+背景+模拟对话) + 右栏Neo Chat | 对练(角色扮演阶段)，左栏三区域 |
| **L-E** | Topbar + 可折叠左侧导航 + 全屏单栏沉浸对话 | 调研(核心调研阶段) |
| **L-F** | Topbar + 常显左侧导航 + 左栏看板(Tab切换) + 右栏Ora Chat | 管理端首页、管理端报告编辑 |
| **L-G** | Topbar + 常显左侧导航 + 单栏全幅 | 管理端消息、管理端配置 |
| **L-H** | 全屏登录页（无导航） | 登录 |

**UX 补充：** 学员端的左侧导航是可折叠的（spec 的 ref HTML 里有 collapsed 状态），管理端的常显导航是固定不折叠的。两套导航在视觉上差异明显——学员端更多是课程目录，管理端是功能模块切换（首页/报告/消息/配置）。这个区分在快照里必须体现出来。

**PM 确认：** 对，而且 topbar 在两端完全一致（spec common/00-overview.md 明确说了），差异仅在端口标识文字（"学员端"/"管理端"）和消息内容。快照中 topbar 可以复用同一组件。

### 二、逐旅程拆解辩论

#### J1 — 新学员首次登录

**PM：** J1 覆盖的流程是：登录 → 大厅空状态 → Leo 自我介绍 → 需求感知 → 进入授课 → Neo 首次自我介绍。这里至少需要 3 个快照：登录页、大厅空状态（含 Leo 首次消息）、授课场域（Neo 首次自我介绍）。

**UX：** 登录页是独立布局（L-H），大厅空状态和大厅日常看板（J7）共用布局骨架 L-A 但数据完全不同——看板全部是零/空，课程目录全部锁定，工具库为空。这个差异足够大，必须单独做一个快照。

**Student Rep：** 作为学员，首次进入系统最关键的体验是 Leo 的自我介绍和需求感知对话。这段对话一定要在快照里体现出来，因为它决定了学员的第一印象。

**Dev：** 登录页是纯 GUI，没有复杂交互，做一个快照就够。大厅空状态和 Leo 首次对话可以合并到一个快照里——右侧 Chat 区展示 Leo 的欢迎消息和 2-3 轮需求感知对话。至于"进入授课后 Neo 首次自我介绍"，这可以在 J2 的授课快照里用一个注释标注"首次进入时 Neo 会先自我介绍"，不需要单独做快照。

**PM：** 同意。那 J1 = 2 个快照：S01 登录页 + S02 大厅空状态（含 Leo 首次对话）。

**全体：** 通过。

---

#### J2 — 授课场域完整教学

**PM：** J2 是最复杂的旅程——要覆盖全部 SCO 类型（PPT/Video/Quiz/FEEDBACK_COLLECT/FEEDBACK_REVIEW）+ 插嘴答疑 + 闭环反馈 + Activity 完成页。spec 里 SCO 类型有 5 种，每种看板区内容不同。

**UX：** 这里的关键分歧是：每种 SCO 类型是否需要独立快照？我认为需要，因为看板区内容完全不同——PPT 是图片渲染，Video 是播放器，Quiz 是题干+答题区。开发者必须知道每种 SCO 的看板区长什么样。

**Dev：** 我同意 PPT、Video、Quiz 三种核心 SCO 需要独立快照。但 FEEDBACK_COLLECT 和 FEEDBACK_REVIEW 本质上都是 Quiz 类型（问答题），看板区都是"题干展示+对话区回答"。可以合并为一个快照，用注释区分。

**Student Rep：** 插嘴答疑的体验也很重要——学员在任何 SCO 播放中都可以打字提问，Neo 会暂停讲解回答。这个交互需要在某个快照里体现。

**PM：** 还有 Activity 完成页——"下一个任务"卡片 + "回到辅导"链接。这是场域间直跳的关键节点。

**UX：** 我建议 J2 拆为 5 个快照：
1. PPT SCO（授课口播 + 左侧 SCO 列表含锁定/解锁/当前状态）
2. Video SCO（播放器 + 无授课口播 + 答疑口播对话）
3. Quiz SCO — 选择题/判断题（看板区答题 + Neo 反馈）
4. Quiz SCO — 问答题/FEEDBACK（看板区题干 + 对话区回答 + 闭环反馈）
5. Activity 完成页（下一个任务卡片 + 回到辅导链接）

**Dev：** 问一下，左侧导航的两个 Tab（SCO列表 / Activity列表）在哪个快照里展示 Tab2？

**UX：** 在 PPT SCO 那个快照里做 Tab 切换——默认显示 SCO 列表（Tab1），页面内交互可以切到 Activity 列表（Tab2）。

**Dev：** 好的。那插嘴答疑在哪个快照里体现？

**UX：** 在 Video SCO 那个快照里——视频播放期间 Neo 不做授课口播，但学员可以答疑口播。Chat 区展示一条学员提问 + Neo 回答。

**PM：** Activity 完成页和 J1 的"Neo 首次自我介绍"是否冲突？

**Dev：** 不冲突。Activity 完成页是授课场域最后一屏，和授课进行中的快照共用布局 L-B，只是看板区内容变成了完成卡片。Neo 首次自我介绍发生在第一个 SCO 之前，我们在 PPT SCO 快照的 Chat 区顶部加一句 Neo 自我介绍文本即可。

**全体：** J2 = 5 个快照（S03-S07）。通过。

---

#### J3 — 对练场域完整训练

**PM：** J3 覆盖三个阶段：情景导入 → 角色扮演(含 Neo 答疑与辅导) → 反思复盘(含对练小结)。再加上重练。

**UX：** 三个阶段的布局不同：
- 情景导入：L-C（左栏铺垫markdown + 右栏Neo引导）
- 角色扮演：L-D（左栏三区域 + 右栏Neo辅导）——这是对练特有的布局
- 反思复盘：L-C（左栏对练小结 + 右栏Neo复盘对话）

三个阶段必须各一个快照。ref HTML（03-对练教室.html）里已经有 phase-tab 切换机制，我们可以在一个 HTML 里做 3 个 phase 的 Tab 切换。但作为快照拆分，逻辑上算 3 个独立画面。

**Dev：** 从开发角度，我建议对练用 1 个 HTML 文件 + 3 个 phase Tab 切换，而不是 3 个独立 HTML。ref 已经这样做了。但在快照清单里我们还是标 3 个 snapshot ID。

**Student Rep：** 角色扮演阶段的左栏三区域布局需要特别清晰——上方是 Actor 头像，中部是背景缩略，下方是模拟对话。这个和右栏 Neo 的辅导对话形成"两个独立对话窗口"的体验，是产品最独特的交互之一。

**HR Rep：** 反思复盘的左栏"对练小结"里有"目标达成/高光表现/待改进/对话回看"——这和管理端的报告数据互通。开发者需要在快照里看到小结的数据结构。

**PM：** 重练呢？需要单独快照吗？

**UX：** 不需要。重练是跳过情景导入直接从角色扮演开始，交互和布局一样，只是 Chat 区累积了上一轮记录。我们在角色扮演快照的注释里标注即可。

**全体：** J3 = 3 个快照（S08-S10），物理上可以是 1 个 HTML + 3 Phase Tab。通过。

---

#### J4 — Leo 深度辅导（两场景）

**PM：** J4 有两个场景：A 情绪感知 + 知行差距，B 复杂场景 + 课程推荐。两个场景都发生在大厅，布局都是 L-A。

**UX：** 这两个场景的看板区数据和 J7（日常大厅）是一样的——中期数据状态。差异只在右侧 Leo 对话内容。所以不需要单独的快照，而是在大厅日常状态（J7）的快照中，通过 Chat 区的对话内容来覆盖 J4。

**Student Rep：** 但 J4 的对话内容是产品的核心差异化——情绪感知、知行差距分析、课程推荐卡片。如果只用一个快照，Chat 区要展示这么多对话会不会太长？

**PM：** 这是个好问题。但静态快照中 Chat 区本来就是可滚动的。我们可以在 J7 的大厅快照中，让 Chat 区包含足够丰富的对话——包括登录问候、情绪感知、知行差距追问、课程推荐卡片。这样一个快照就能覆盖 J4 和 J7 的大部分内容。

**Dev：** 我同意 J4 不需要独立快照。但是 spec 里提到 Leo 的对话中会嵌入"任务卡片"（可点击进入授课/对练）和"待办推送"——这些组件在 Chat 区的渲染方式需要在快照里体现。

**UX：** 那我们把 J7 大厅的 Chat 区设计得足够丰富：包含普通对话、情绪响应、任务卡片（可点击）、待办推送卡片。J4 就完全被 J7 吸收。

**全体：** J4 不单独出快照，被 J7 吸收。通过。

---

#### J5 — 报告场域查阅与解读

**PM：** J5 覆盖：报告三部分渐进呈现 + 证据查看 + Neo 讨论 + 导出 PDF。

**UX：** 报告场域始终是 L-C 布局（左栏报告 + 右栏 Neo Chat）。但报告有三部分渐进式呈现，加上证据弹窗。

**Dev：** 报告三部分在同一页面中渐进出现，不是三个独立页面。所以 1 个快照就够——展示三部分全部展开的完整状态，Chat 区有 Neo 解读对话。

**UX：** 还需要一个证据查看的状态——spec 说"点击高光时刻卡片上的'查看证据'链接，弹出对练原始对话记录片段"。这个弹窗是页内交互。

**PM：** 还有导出 PDF 按钮在 topbar。ref HTML（05-报告教室.html）里已经有这个按钮了。

**Dev：** 那 J5 = 1 个快照，报告完整呈现 + Chat 解读 + "查看证据"弹窗作为页内交互。

**Student Rep：** 报告中有 BI 图表（折线图/分位区间标尺）——这些在快照里必须有视觉表达，不能只放占位框。

**全体：** J5 = 1 个快照（S11），含页内弹窗交互。通过。

---

#### J6 — 自由复习

**PM：** J6 是已完成的 Activity 重新进入——SCO 全解锁 + Neo 答疑。

**UX：** 自由复习进入的仍然是授课场域（L-B 布局），唯一区别是左侧 SCO 列表全部为已完成状态（没有锁定的），学员可以自由点击任何 SCO。

**Dev：** 这和 J2 的授课快照布局完全一样，只是 SCO 列表状态不同。需要单独快照吗？

**PM：** 需要。因为开发者必须知道"自由复习模式"和"首次学习模式"的 SCO 列表状态差异。而且 spec 明确说自由复习时 Neo 会"简要讲解该页核心内容（不完整重播）"。

**UX：** 我建议不单独做一个新快照，而是在 J2 的 PPT SCO 快照的注释中标注"自由复习模式差异"——SCO 列表全部解锁、Neo 对话不同。或者在同一个 HTML 里做一个 Tab/Toggle 切换。

**Dev：** 从开发角度，一个注释就够了。不需要独立快照。

**PM：** J6 确实可以被 J2 吸收。但我们需要一个明确的注释。

**全体：** J6 不单独出快照，在 S03（PPT SCO）注释中标注自由复习差异。通过。

---

#### J7 — 大厅看板日常使用

**PM：** J7 是最信息密集的——中期数据状态 + 所有看板组件 + 消息面板。spec 里大厅看板包含：学习进度、课程目录（S型曲线）、热力图（悬停柱状图）、工具库（解锁/锁定）、笔记库、待办任务、个人资产下载、社区（占位）。

**UX：** 这个页面信息密度极高。中间看板区是长页面滚动的，不可能在一屏里放下所有组件。但作为静态快照，我们应该展示完整的长页面——开发者上下滚动就能看到所有组件。

**Student Rep：** 热力图是学员端和管理端统一视觉规格的（第一轮确认），这个在大厅快照里必须清晰展示——包括颜色深浅和悬停柱状图弹窗。

**Dev：** J7 大厅日常状态需要 1 个快照。但消息面板（topbar 铃铛下拉）是页内交互——点击铃铛展开。需要在同一快照里做 dropdown 切换。

**PM：** 加上 J4 被吸收进来的 Leo 对话内容，J7 这个快照承载的信息量巨大。

**UX：** 还有头像下拉菜单（J13 的一部分）——端口切换、账号管理、个人设置等。这些也是页内交互。

**PM：** 那 J7 的快照需要支持的页内交互有：
1. 消息面板下拉（铃铛点击）
2. 头像下拉菜单
3. 热力图悬停柱状图弹窗
4. 工具库点击弹窗预览
5. 看板区滚动

**Dev：** 这么多页内交互建议用 JS 控制 display toggle，ref HTML 已经用了类似的方式。

**全体：** J7 = 1 个快照（S12），页内交互丰富。同时吸收 J4。通过。

---

#### J8 — 开营配置

**PM：** J8 是管理端配置页——开营前锁定状态 → 完整配置 → 确认开营 → 解锁。第一轮确认了"需要两帧对比"（开营前 vs 开营后导航状态）。

**UX：** 配置页是 L-G 布局（常显导航 + 单栏全幅，纯 GUI 无 Ora）。配置页内有多个子功能区：项目信息 / 学员导入 / 角色分配 / 催学规则 / 平台个性化 / 内容预览。

**Dev：** 配置页是否需要展示所有子功能区？还是只展示一个代表性的？

**PM：** 必须展示完整的。HR 需要从快照里理解整个开营流程。但是可以用 Tab/Accordion 在同一页面里切换。

**HR Rep：** 我作为管理员，最关心的是配置页能不能让我一目了然地知道还差什么步骤没完成。开营前锁定状态也很重要——学员端不可访问的提示。

**UX：** 我建议 J8 = 2 个快照：
1. 配置页完整（所有配置项展示，含学员名单导入预览、角色分配列表等），底部有"确认开营"按钮
2. 开营前 vs 开营后的导航状态对比——可以用一个 HTML 里的 Toggle 切换

**Dev：** 开营前 vs 开营后的差异主要在哪里？

**PM：** 主要在学员端——开营前学员无法登录/进入系统。管理端的导航应该都是可用的。所以"两帧对比"更多是描述文字说明，不一定是两个完整页面的对比。

**UX：** 那简化一下：配置页快照里，"确认开营"按钮附近加一个注释区域，展示开营前后的状态说明（学员端锁定/解锁、导航变化等）。不需要单独出第二个快照。

**全体：** J8 = 1 个快照（S13），配置页完整展示 + 开营前后状态对比说明。通过。

---

#### J9 — 日常巡检

**PM：** J9 覆盖管理端首页三维度看板 + 热力图 + Ora 查询 + 学员详情 + 个人报告。spec 里首页有三个 Tab：项目维度（默认）、人员维度、内容维度。

**UX：** 三个 Tab 的内容差异很大：
- 项目维度：整体进度 + 日活热力图 + 学习投入度热力图
- 人员维度：学员列表 + 点击展开详情 + 查看个人报告
- 内容维度：各 Course 完成率柱状图 + Activity 完成率明细

**Dev：** 三个 Tab 共用 L-F 布局（常显导航 + 左栏Tab切换 + 右栏Ora Chat），用 Tab 切换就好，算 1 个快照 + 3 Tab 页内交互。

**HR Rep：** 人员维度里的"学员详情展开"和"个人报告查看"是日常巡检的核心操作——我每天都要点开看几个重点学员。这个交互必须在快照里体现。

**UX：** 学员详情可以做成一个可展开的 panel（在人员维度 Tab 内），个人报告跳转到学员端的报告场域（J5 已覆盖）。

**PM：** Ora 的对话内容在 J9 快照中要展示什么？

**UX：** 展示日常巡检典型对话——管理员问"上周进度最快的3个人是谁"，Ora 给出数据回答。

**Dev：** 还有"图表点击联动 Ora"的交互——管理员点击柱状图，Ora 自动解读。这个是 P3 但 spec 里描述了完整流程。

**PM：** P3 的图表联动在快照里可以不做交互实现，但需要标注。

**全体：** J9 = 1 个快照（S14），3 Tab 页内切换 + 学员详情展开 + Ora 对话。通过。

---

#### J10 — 团队报告生成

**PM：** J10 覆盖：报告库 → 新建 → Ora 生成 → HITL 编辑 → 追溯证据。

**UX：** 报告页有两个视图：报告库首页（列表）和报告编辑器。两个视图布局都是 L-F。

**Dev：** 报告库首页可以和报告编辑器做在一个 HTML 里切换吗？

**UX：** 可以——新建表单弹窗 + 编辑器视图切换。但报告库首页和编辑器在信息层级上差异很大，我建议分 2 个快照。

**PM：** 报告库首页比较简单（报告列表 + 新建按钮），可以和编辑器合并——列表作为初始状态，点击某份报告后切换到编辑器视图。

**Dev：** Ora 的"追溯证据"在对话中用自己语言引述原始对话（第一轮确认），不做弹窗。这个在 Chat 区里展示就好。
