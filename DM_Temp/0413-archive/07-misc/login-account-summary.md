# AI 老师 — 登录与账户体系需求总结

**来源：** 4份参考文档 + 电话会议记录
**日期：** 2026-04-14
**目的：** 整合现有 GPB 账户基础设施能力和 AI 老师的登录需求，明确对接方案

---

## 一、电话会议核心需求（创始人与技术团队）

### 确认的需求

| # | 需求 | 详情 |
|---|------|------|
| 1 | **越轻越好** | 不想在2.0上面做，想自己做掉登录，2.0给个接口跳过来就行 |
| 2 | **接 GPB** | 想接 GPB 的登录体系，但不确定 GPB 能否支持特定需求 |
| 3 | **项目制** | 每次开项目导入一批人（40-50人），项目结束（约6个月）后就结束 |
| 4 | **账号管理** | 几个管理员账号 + 几个运营账号 + 剩下全是学员账号 |
| 5 | **角色权限** | 一个角色对应一种权限，不做复杂的功能权限，角色权限就够了 |
| 6 | **不接组织架构** | 第一个版本不想接企业的组织架构 |
| 7 | **项目=租户** | 每个项目相当于配一个租户，项目隔离 |
| 8 | **新项目新账号** | 下一个项目进来的学员用新账号（暂定），不复用之前的 |
| 9 | **最终交付** | 以一个文件包的形式交付给客户最后的结果 |

### 待定的问题

- 登录是自己开发还是接 GPB？→ **倾向接 GPB**
- GPB 能否支持"每个项目一个租户"的模式？
- 学员如何登录？链接+验证码？还是账号密码？
- 是否需要管理后台？→ **需要，有管理动作（分配管理员等）**

## 二、GPB 账户基础设施能力

### 2.1 GPB 账户体系架构

```
自然人(User) — 全局唯一标识
    │
    ├── 账户(Account) — 身份凭证（手机号/邮箱/三方）
    │
    ├── 组织/租户(Tenant) — 数据隔离、计费基本单元
    │   ├── 成员(Member) — 用户加入组织后的身份
    │   ├── 角色(Role) — RBAC角色（如学员/管理员/运营）
    │   └── 权限(Permission) — 具体操作许可
    │
    └── 权益(Entitlement) — 订阅产品获得的功能和配额
        ├── 功能(Feature)
        └── 配额(Limit)
```

### 2.2 GPB 已支持的登录方式

| 方式 | 国内版 | 说明 |
|------|--------|------|
| 手机号注册/登录 | ✅ | 支持国际号段，验证码登录 |
| 邮箱登录 | ✅ | 验证码方式 |
| 企业 SSO | ✅ | 输入企业域名跳转企业登录 |
| Google/Apple/Microsoft | 仅海外 | 国内不支持 |

### 2.3 GPB 关键能力

- **多租户**：一个用户可关联多个租户，登录后选择进入哪个
- **自动创建租户**：注册后自动为用户创建一个租户
- **成员邀请**：支持邀请加入组织
- **角色权限(RBAC)**：支持角色-权限映射
- **产品隔离**：同一租户下不同产品可有不同权限
- **统一登录域名**：login.radnova.com/产品名
- **个人管理中心**：统一管理账号信息

### 2.4 GPB 注册流程（国内版）

```
输入手机号 → 验证码 → 检查是否有关联组织
  ├── 无组织 → 自动创建一个组织 → 进入
  ├── 1个组织 → 直接进入
  └── 多个组织 → 选择要进入的组织 → 进入
```

## 三、AI 老师对接 GPB 的方案分析

### 3.1 映射关系

| AI 老师概念 | GPB 概念 | 说明 |
|------------|---------|------|
| 培训项目(Project) | 租户(Tenant) | 每个项目 = 一个租户，数据隔离 |
| 学员 | 成员(Member) + 学员角色 | 项目内的普通成员 |
| HR | 成员(Member) + HR角色 | 有管理看板权限 |
| 运营 | 成员(Member) + 运营角色 | 有运营工具权限 |
| 权限控制 | RBAC | 一个角色=一种权限集合 |

### 3.2 推荐对接方案

**方案：接 GPB 登录 + 项目=租户 + 角色权限**

```
项目创建流程：
运营创建项目 → GPB 创建一个新租户 → 导入学员名单 
→ GPB 为每个学员创建账号+发送邀请 → 学员通过手机号验证码登录

登录流程：
学员收到链接 → 打开 login.radnova.com/ai-teacher 
→ 输入手机号+验证码 → GPB 验证 
→ 如果只有一个项目(租户)直接进入
→ 如果有多个项目(租户)选择进入哪个
→ 进入 AI 老师学员端

角色分配：
项目创建时由运营指定：
  - 学员角色 → 看到学员端（大厅+教室）
  - HR角色 → 看到HR管理端
  - 运营角色 → 看到运营管理端
```

### 3.3 需要 GPB 支持的特殊需求

| # | 需求 | GPB 现状 | 差距 |
|---|------|---------|------|
| 1 | 项目结束后租户归档 | 租户无"归档"状态 | 需要 GPB 新增租户归档/冻结能力 |
| 2 | 批量创建账号 | 支持邀请加入 | 需确认是否支持批量导入 Excel 创建 |
| 3 | 新项目新账号 | 同一用户可关联多租户 | 如果同一人参加多个项目，GPB 自动关联，不需要新账号——**与创始人暂定方案矛盾**，需讨论 |
| 4 | 不接组织架构 | GPB 有完整组织架构 | AI 老师不使用组织架构功能，只用角色权限 |
| 5 | 项目内角色简单 | GPB 支持复杂 RBAC | AI 老师只需要3个角色（学员/HR/运营），不需要细粒度权限 |

### 3.4 AI 老师 vs GPB 的关注点差异

| 维度 | GPB 设计目标 | AI 老师需求 |
|------|------------|-----------|
| 租户 | 企业长期使用 | 项目短期使用（6个月） |
| 账号 | 用户长期保留 | 项目结束后可归档 |
| 组织架构 | 多层级部门 | 不需要，扁平角色即可 |
| 权益 | 订阅制/按功能付费 | 项目制（买一个项目=全部功能） |
| 多产品 | 同一租户下多产品 | 只有AI老师一个产品 |

## 四、对 AI 老师产品设计的影响

### 4.1 需要更新功能树的地方

当前 `feature-tree/01-layout-auth.md` 中的账户权限模块是通用设计，需要基于 GPB 对接方案具体化：

1. **B1 登录**：改为 GPB 统一登录（手机号验证码为主），不自建登录
2. **B2 角色权限**：对齐 GPB RBAC，3个角色（学员/HR/运营）
3. **B3 角色切换**：HR→学员切换需要通过 GPB 的权限层实现
4. **B4 项目上下文**：项目=租户，多项目=多租户选择
5. **新增**：项目创建时的 GPB 租户创建+批量账号创建流程

### 4.2 需要和创始人确认的决策

| # | 问题 | 选项 |
|---|------|------|
| 1 | 同一人参加多个项目时，用同一个账号（GPB自动关联多租户）还是每次新建账号？ | GPB 架构下"同一手机号=同一用户"，建议复用账号+多租户选择 |
| 2 | 项目结束后账号怎么处理？ | 冻结租户但保留数据（用于报告交付） |
| 3 | 登录方式：手机号验证码 only？还是也支持邮箱？ | 建议手机号为主，邮箱为备选 |
| 4 | 是否需要企业 SSO？ | 大企业客户可能需要，建议作为可选项 |
| 5 | GPB 的统一登录域名（login.radnova.com/ai-teacher）是否可接受？还是需要独立域名？ | 待确认 |
