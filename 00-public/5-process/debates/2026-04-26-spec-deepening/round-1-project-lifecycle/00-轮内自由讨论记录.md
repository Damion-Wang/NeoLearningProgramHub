## § 1 · 议题 1 · 项目周期 vs 账号有效期 是否分离

### 1.1 各角色第一轮发言

**PM**：

我主张**架构层分离，商业层本期不上线**。

理由：
- 商业上，"年包"和"项目"是两种不同的销售单元。年包 = 12 个月学员账号有效期；项目 = 一次完整培训交付（典型 6 个月）。两者解耦后，能支撑：
  - 年包模式（撑续约低门槛）
  - 项目套餐（典型场景）
  - "1 年账号 + 多个 3 月小项目"复合模式（V2）
- 本期 60 家天使客户多数不需要年包，但**架构留口子**。如果本期把"项目周期 = 账号有效期"写死在 schema 里，V2 改造代价巨大。
- spec 层加一句"本期商业层简化为项目周期 = 账号有效期，架构上保留分离"，研发就明白了。

**UX**：

我担心**学员 UI 复杂化**。

如果学员看到的"我的项目"和"我的账号"是两个不同状态实体，会出现：
- "项目结束了但账号还能用？我现在能干啥？"
- "为什么登录后还能看到 X 但不能点 Y？"
- 状态机复杂 = bug 多 = UX 糟糕

PM 说"架构分离 UI 不感知"，但实际开发中 UI 经常被迫感知——尤其在状态切换的临界点（项目刚结 / 账号过期前）。

我的建议：本期 UI 层强约束"项目周期 = 账号生命周期"。架构能不能分离让研发自决，UX 不接受 UI 暴露分离状态。

**学员代表**：

我作为学员只关心"什么时候能学，什么时候不能学"。

具体场景：
- W0-W24（项目期）：能学、能问、能复习、能下载
- W24+（项目结束）：状态变成什么？

如果"账号还活 6 月但什么都不能做"——这是个**很奇怪的状态**。我会想"既然让我登录，为什么不能继续？"。

如果"账号还活 6 月还能复习 + 看报告"——OK，符合"持续价值"承诺。

如果"账号立即关闭，登录不了"——也 OK，明确干净。

**最不能接受**：能登录但啥都干不成。这是欺骗。

**HR 代表**：

HR 视角**必须分离**。原因是续约谈判。

我作为客户公司 HR，跟睿学销售谈续约时：
- 销售说"明年再买一期项目" → 我会算"投入产出比"，门槛高
- 销售说"账号年包，每年续费" → 门槛低，类似订阅

后者销售更容易促成。**没有"账号年包"概念，续约率就是项目级别的低**。

但我同意 PM 说的"本期不上线"——本期客户都是大客户买项目，年包模式没必要急着推。**架构留口子，V2 推年包商品**。

**业务专家**：

从企业培训实践看，**90% 是项目套餐，10% 才是真正的年包**。

而且企业 L&D 部门买培训的预算决策周期：
- 项目套餐：年初批一次，年中执行，年末验收。**单次决策**。
- 年包：要月度对账、季度续费、年度总结。**多次决策点 = 多个被砍预算的机会**。

所以年包模式商业上**未必更好**。本期专心做项目套餐，把 ROI 做出来，比急着上年包更重要。

我同意 PM 的"架构留口子但本期不展示"。

### 1.2 主持人识别冲突

**主持人**：

冲突点 2 个：
- **PM/HR vs UX**：架构是否分离？UX 担心 UI 失控
- **PM/HR vs 业务专家**：年包模式商业价值是真还是伪？

学员代表的核心诉求是"不能登录但啥都干不成"——这不是分离 vs 不分离，是"结项后状态怎么设计"问题（议题 3 才会展开）。

挑战者请反驳。

### 1.3 挑战者反驳

**挑战者**：

我反驳 PM。

你说"架构留口子"听起来好听，但你能给我一个具体场景，证明本期就要做这个口子？60 家天使客户有几家明确要年包？没有具体客户需求，"口子"就是过度设计——研发会留一堆 dead code，未来重构成本一样高。

我反驳 HR。

你说"年包销售门槛低"——前提是产品已经成型、价值已经被市场验证。**本期连第一个项目都没跑完**，就在想年包续约？这是在 PM 教科书"产品市场契合"还没达成时就规划"复购模型"。本末倒置。

我反驳业务专家。

你说"90% 项目套餐"——但你的数据来源是哪里？基于过去的"传统培训"经验？我们是 AI Native，可能会**改变购买模式**。Khanmigo 就是订阅模式（年包），不是项目套餐。

我支持 UX。

UX 的担心是真的——"项目结束账号还活"这个状态，学员体验一定会出问题。本期就是"项目周期 = 账号生命周期"最干净。

### 1.4 自由讨论

**PM 反驳挑战者**：

具体场景：60 家天使客户中**有 3 家是大型央企**（不能确切提名），它们的采购流程是"先小项目验证 + 后年度框架协议"。如果本期不留口子，明年这 3 家想签框架协议时我们改不动。这不是 dead code，是**已知存在的需求**，只是延迟体现。

**业务专家回应挑战者**：

你说"Khanmigo 是订阅"——但 Khanmigo 是 K-12 / 高等教育 toC 产品。**toB 企业培训和 toC 教育购买模式完全不同**。toB 有采购预算审批 / 年度计划，年包反而**不灵活**（预算用完就废了）。我维持 90% 项目套餐的判断。

**UX 调整立场**：

我不反对架构分离，我反对**UI 暴露分离**。

如果 PM 说"架构分离 + UI 不暴露"，那本期 UI 层就是"项目周期 = 账号生命周期"，跟 UX 立场一致。学员看不到任何"账号还活"的状态。

**HR 加深**：

UX 说的对。**本期 UI = "项目周期 = 账号生命周期"**。架构是研发问题，UX 接受。我妥协。

**学员代表回到话题**：

只要 UI 层是干净的（项目结 = 登不上 OR 项目结 = 还能复习且明确告知），都可以。状态机别让我猜就行。

**业务专家加注**：

补一句：本期"项目周期 = 账号生命周期"还有个**业务好处**——结项汇报会时学员还在系统里，HR 可以即时给学员发"通知/调研问卷"作为结项动作。如果账号已死，这些动作就要改成邮件，不优雅。

**挑战者最后追问**：

PM 说的"3 家央企"是真实数据还是凭印象？

**PM 实诚回应**：

我作为辩论中的 PM 角色，**没有具体数据**。这是我从行业 / 销售流程推断的。如果 DM（产品负责人）能确认 60 家中有几家真要年包，更好。

### 1.5 主持人裁定共识

**主持人**：

议题 1 共识达成。

**结论**：

- V2 启动年包商品形态时，UI 层加暴露 + 状态机扩展

## § 2 · 议题 2 · 试用期定位

### 2.1 各角色第一轮发言

**PM**：

50 题决策定的"研发后天使客户试用调优"是个新阶段，要在 7 阶段里**显式**插入。

我的方案：
- 在"开营组织"前加**试用期**阶段
- 试用期 = 1-2 周 + 缩小学员数（例如 5-10 人 vs 正式 30-40 人）+ 简化课程包（例如 2 Course vs 8 Course）
- 试用期 KPI：DM + 研发 + HR 三方共同评估"产品达标度"，达标后才进入"正式开营"
- 不达标 → 调优 → 再试用 OR 退课

**UX**：

试用期对学员是**糟糕体验**。试用期学员可能踩坑、看到 bug、被改回访问。

我建议：**试用期学员是"内部志愿者"**（DM 同事 / HR 自己 / 公司高管），不是真实学员。真实 30-40 人在正式开营开始。

否则试用期学员会跟普通学员混在一起，体验不一致 = 投诉。

**学员代表**：

UX 说的对。我作为学员**不愿意当小白鼠**，特别是"用我们公司学习预算被试错"这种事。

如果试用期是企业自己的内部志愿者（HR 团队、培训部成员），那是企业内部消化，问题不大。

但如果试用期就是从 30-40 人里选 5-10 人先试用，**剩下 25 人会觉得自己是二等公民**。

**HR 代表**：

HR 想要"试用期"——为了控制风险。买睿学这种 AI 产品，HR 不敢一上来就给 30-40 人用，万一翻车我职业风险大。

但 UX 说的"内部志愿者"也对——我可以让 HR 自己 + 培训部 5-10 人先试 1-2 周，跑通了再上线。

**业务专家**：

从企业培训实践看，**Pilot（试点）阶段**是工业标准。
- 大型客户买培训前都要 Pilot（5-15 人 / 1-2 个月）
- Pilot 阶段重点验证：内容是否合适 / 效果是否可量化 / 学员是否买账
- Pilot 通过后才大规模铺开

所以试用期应该叫 **Pilot 阶段**，企业培训行业的通用术语。

### 2.2 主持人识别冲突

**主持人**：

冲突点：
- **PM 的"试用 vs 正式" 是产品交付侧概念**（研发后调优）
- **UX/学员/HR/业务专家的"Pilot" 是客户购买侧概念**（企业风险控制）

这两者是不同的事，但有重叠！

让挑战者反驳。

### 2.3 挑战者反驳

**挑战者**：

我反驳"试用期是阶段"这个说法。

你们都把"研发调优试用"和"客户 Pilot"混在一起。**这是两件事**：
- "研发调优试用" = 内部测试 + 早期 alpha 客户。是产品 ready 之前的事。**不是项目生命周期的一部分**
- "客户 Pilot" = 卖给客户后的小规模先行。**是项目生命周期里"开营组织"的一个子阶段**

把两者混在一起，会让生命周期变得混乱：
- 客户 A 看的"试用"是 Pilot
- 客户 B 看的"试用"是产品 alpha
- 销售 / HR / 学员的预期完全不同

我的反驳：从生命周期里**砍掉"研发后试用"这个阶段**，因为那是产品 ready 之前的事。生命周期只保留"客户 Pilot 子阶段"。

### 2.4 自由讨论

**PM 接挑战者**：

挑战者点中要害。我重新表态：
- 50 题决策的"研发后天使客户试用调优"是**产品上线前阶段**，不是项目生命周期的一部分
- 项目生命周期里只有"客户 Pilot"子阶段（属于开营组织阶段）

**业务专家**：

同意。Pilot 是开营组织的一个**可选步骤**：
- 大型客户：开营组织 → Pilot 子阶段（5-10 人 / 1-2 周） → 正式开营（30-40 人 / 6 月）
- 小型客户：跳过 Pilot，直接正式开营

合同条款里明确"是否启用 Pilot"。

**HR 同意**：

我同意业务专家。Pilot 写进合同条款，企业 HR 自己决定要不要。

**UX**：

UI 层 Pilot 子阶段不需要新设计——本质是"小规模开营 + 短期"，复用现有开营 UI。

**学员代表**：

如果 Pilot 学员是企业内部志愿者，我作为正式学员就没风险。OK。

### 2.5 主持人裁定共识

**主持人**：

## § 3 · 议题 3 · 结项后学员状态

### 3.1 各角色第一轮发言

**PM**：

三选一：
- A · 立即关闭：账号失效，登录失败
- B · 只读延续：默认 30-90 天延续期，可看历史 / 报告 / 笔记，但不能与 Neo 互动
- C · 部分互动：延续期内可向 Neo 提"应用追问"，但不能进入新 Activity / 新对练

我倾向 **B**。原因：
- 学员体验上"再看一眼"很合理
- 商业上：延续期是"持续价值"承诺的具体落地（0424 决策 N-03）
- HR 续约谈判时可说"学员结项后还在用"

C 选项不推荐——"应用追问"功能边界模糊，会破坏学员对"项目结束"的认知。

**UX**：

UI 层 B 选项最干净：
- 学员登录后看到"项目已结项"横幅
- 大厅看板转为"成果回顾模式"——历史报告 / 笔记 / 工具卡 仍可见
- Neo 对话区显示"你的项目已结束，无法继续对话。如有需要可以下载历史记录或复习已学课程"
- 不需要新 UI 模块，复用现有

C 选项要新 UI（"应用追问"模式跟正常对话视觉混淆，UX 难做）。

**学员代表**：

我从学员角度希望能 C 选项。

具体场景：项目结束 1 个月，我在工作中遇到"PIN 法不会用"，想问 Neo "我用的对吗"。这是真实需求。

但如果 C 做不到，B 也接受——至少能看历史复习。

A（立即关闭）我接受度低——如果学习了 6 月，账号瞬间消失，**情感上很伤**。像分手一样。

**HR 代表**：

HR 倾向 B。
- 商业上"延续期"是续约钩子
- 数据合规上"立即关闭"也方便（合同到期 = 数据冻结）
- C 太复杂 + 边界模糊

但要求**延续期可配置**。某些大型客户合规严格，要求项目结束就关闭。配置项写进合同。

**业务专家**：

工业实践：
- LinkedIn Learning：课程结束后无限期可复习
- Coursera：完成证书后无限期可看课
- 企业培训：通常项目结束后 30-90 天延续访问

**B 选项是工业标准**。延续期 30-90 天可配置。

C 选项业内罕见。原因：会让学员"延期项目"——继续问 Neo = 项目没真正结束。

### 3.2 主持人识别冲突

**主持人**：

5 角色立场都倾向 B（只读延续）。
- C 仅学员代表想要，但学员自己也接受 B 作为兜底
- A 仅作为 HR 的特殊客户配置选项

冲突很小。但**延续期默认值**和**是否完全砍掉 C**还有讨论。

挑战者反驳。

### 3.3 挑战者反驳

**挑战者**：

我反驳"B 是工业标准所以选 B"。

工业实践给的是 toC 教育（LinkedIn / Coursera）。**企业培训 toB 不一定**。
- 企业 toB 培训通常是"封闭式" = 项目结束 = 课程关闭。延续访问的少。
- 你们引用的是消费教育，不是企业培训。

我反驳"延续期是续约钩子"。

延续期免费给学员看 = 客户白嫖。HR 续约时说"学员还在用，不续约也行" = 反作用。
