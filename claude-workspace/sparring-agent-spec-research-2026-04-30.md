## 1. 架构总览

### 1.1 整体设计思路（一句话）

> 对练系统 = **1 个剧本 + 1 个调度中枢 + 4 个工具调用 + 1 个独立判官 + 1 个复盘 skill + 3 路报告产出**，串成 **生成 → 参与 → 结束** 的完整闭环。

剧本是配置源，导演是调度中枢，演员/教练/旁白是导演的三个工具，通过判断和评分独立运行。整体是**KGP 内容生产侧（01）+ 运行期演绎侧（02-08）+ 报告产出侧（09-12）** 的三段式拓扑。

### 1.2 与本仓库 drill 项目的关系

| 维度 | jiyh spec | 本仓库 drill |
|------|----------|----------|
| 视角 | 工程实现侧（prompt 即 spec）| 产品 spec 侧（设计意图）|
| 阶段 | 12 个 prompt 已经成文 | § 9.3.2 写到 9.3.2.3 / 9.3.2.4-7 待深化 |
| 角色清单 | 编剧 + 导演 + 演员 + 教练 + 通过判官 + 复盘 + 评分 + 高光 + 对话分析 + 报告助手（**9 个 Agent / 12 个 prompt**）| Actor + 导演 + Neo（**3 个 Agent**）|
| 复盘 | 独立 prompt（08）+ 独立 Skill（_builtin_conversation / _builtin_write_scratchpad）| Neo 兼任，沿用 Neo 跨场域记忆 |

## 2. 三模式扩展（重点研究）

### 2.1 「三模式」是什么

「三模式」 = 对练的 **3 种变体**：**基础 Roleplay / 角色互换 / 费曼**，**运行时三选一**——根据知识 / 学习内容匹配，每次对练只走一种模式。

### 2.2 三种模式分别是什么

| 模式 | 定义 | 学员扮演 | AI 演员扮演 | 一句话 |
|------|------|---------|------------|--------|
| **基础 Roleplay**（baseline）| 经典角色扮演 | 自身职位（如基层管理者）| 对手 / 对方（如客户 / 下属）| 当前主流形态 |
| **角色互换**（role swap）| 主体对调 | 对手 / 对方 | 学员本职位 | "学员演对手 / AI 演本职" |
| **费曼**（Feynman）| "教别人才是真懂" | 老师（讲解者）| 无知学生（提问者）| 学员讲解，AI 发问 |

### 2.3 三模式对系统的影响（核心结论）

来源：`需求文档/三模式扩展影响清单.md` 全量 157 项字段分析。

| | 基础 Roleplay（基线） | 角色互换 | 费曼 |
|---|:---:|:---:|:---:|
| ✅ 适用 | 100%（基线）| 156 / 157（**99.4%**）| 116 / 157（73.9%）|
| ⚠️ 部分适用 | — | 0 | 23（14.6%）|
| ❌ 不适用 | — | 1 | 18（11.5%）|

#### 2.3.1 角色互换 ≈ 99.4% 复用

**唯一冲突点**：08 复盘的 **L3 Perspective Shift**（"假设你是对方……"）—— 用户已经实地扮演过对方一遍，再被问"假设你是对方"会重复诡异。

**改造方案**：仅需调整 08 prompt 中 L3 的提问角度，把"假设你是对方"改为"切回原身份反思你刚才的扮演"。**字段层全部沿用基础 Roleplay 剧本结构**。

#### 2.3.2 费曼有 14 个高严重风险点

费曼模式中，**14 项 prompt 硬编码冲突**——填字段也解决不了，**必须改 prompt 内核**。其中 **10 项位于 03 导演 / 05 演员 prompt 的硬编码区**：

| # | 硬编码冲突点 | 性质 |
|---|---|---|
| 17 / 106 | 演员显性目标 | 输入空 → 演技空洞 |
| 18 / 107 | 演员隐性目标 | 心理纵深丧失 |
| 21 / 110 | 触发点 | "对方差→你更焦虑"硬编码 vs 费曼无激怒点 |
| 22 / 111 | 软化点 | 同上 |
| 23 / 108 | 底线 | 输入空 → AI 变无原则 |
| 44 | L4 博弈性应对 | 09 评分按"博弈证据"判 71-90，费曼无博弈→错判低级别 |
| 68 / 69 / 70 | 张力维度 | 03 导演硬编码"释放关键信息""激活新冲突""不兜底"，与费曼空转 |
| 73 | 导演 opening | 硬编码"带困境/委屈"指挥 AI |
| 89 | 05 演员核心理念 | "有血有肉真人" vs 费曼"无知学生"定位冲突 |
| 91 | 05 演员反应轴 | 情绪轴硬编码"对方差→更焦虑"，费曼应是"听懂/没听懂"轴 |
| 96 | 05 演员"不说不知道的信息" | 费曼里学生本应不知道，硬编码会让 AI 学生回避提问 |

**结论**：费曼**不是"换一套剧本字段"能解决的**，需要 03 / 05 prompt 大改。所以 jiyh 把费曼归为 🟡 非本期。

#### 2.3.3 对我们 v0.5 的关键启示

> v0.5 § 9.3.2.3.5 三种对练变体写："**变体由剧本决定**——KGP 在 AOM 编排时通过剧本字段标识变体类型，**平台运行时按相同流程演绎**"。

✅ 这与角色互换的实证一致（99.4% 字段复用）。
❌ 但与费曼的实证**冲突**：费曼有 14 项硬编码冲突——平台运行时**不能按相同流程演绎**，必须 mode-aware 的 prompt 分支。

**建议 v0.5 的写法收敛**：
- 角色互换 = 剧本字段层差异，平台运行时同流程
- 费曼 = **prompt 模板层差异**，需要单独的 Actor / 导演 prompt 模板（或本期不做，与 jiyh 同步）

## 3. 剧本要求（01 编剧专家）

### 3.1 编剧四阶段流程（依次执行，上一阶段输出做下一阶段输入）

详见 `Prompt/01-制作-对练剧本.md` § 编剧流程：

1. **构建练习主题**（细分受众 → 识别困境 → 解构能力 → 解构行为）
2. **定义角色**（用户角色 + 演员角色，立体心理动态）
3. **编写场景剧本**（沉浸式描述 + 陷阱清单 + 能力触发地图）
4. **设定技能阶梯**（每个技能 L1-L5）

### 3.2 必备字段全清单

#### 3.2.1 主题字段（阶段一）

`theme.name` / `business_context` / `task_goal`（一条主线）/ `behavior_goals`（2-5 个可观察行为）/ `core_skills` / `knowledge_points` / `选择理由` / `核心挑战` / `常见失败模式` / `预期错误类型 4 类` / `难度分级 4 档`

#### 3.2.2 用户角色字段（阶段二）

`user_role`（**含权限边界 / 禁止行为 / 可用资源 / 决策权限**——不只是岗位职责）

#### 3.2.3 AI 演员字段（阶段二，最重）

| 大类 | 字段 | 说明 |
|------|------|------|
| **基本信息** | 全名 / 性别 / 年龄段 / 身份 / 学历 / 工作经验 / 角色背景 | 注入 05 演员 6 字段 |
| **背景信息** | 完整背景故事（**区分"演员已知"/"未知"客观事实**）/ 与用户关系 / 此前互动 | — |
| **心理动态** | **显性目标**（说出来的）/ **隐性目标**（真正想要的）/ 表面 vs 真实需求 / 顾虑 / 激怒点 / 软化点 / **底线**（绝对不让步）/ **可谈判项**（2-5 条）| 心理纵深的核心 |
| **行为特征** | 性格特点 / 说话风格 / **触发点**（2-5）/ **软化点**（2-5）/ **行为模式**（2-5，如"压力大说话变快"）| 注入 05 演员 |
| **隐藏信息** | 用户未知的关键事实 / 对用户初始态度 | — |

#### 3.2.4 场景字段（阶段三）

`场景标题` / `沉浸式场景描述`（3-5 句电影感，含**肢体 / 面部描述，不含语言**）/ `演员开场白`（1-2 句不像 NPC）/ `演员心理动态`（场景内）/ `陷阱清单`（1-4）/ `挑战清单`（1-4）/ `成功标准` / `失败信号` / **`能力触发地图`（3-6 阶段，每个阶段含决策点 / 正确应对 / 常见错误清单）**

#### 3.2.5 技能阶梯字段（阶段四）

每个技能 L1-L5：

| 等级 | 核心特征 |
|------|---------|
| L1 基础应对 | 能反应，但被动且模板化 |
| L2 结构化应对 | 有逻辑框架，按步骤处理 |
| L3 策略性应对 | 根据对方反应调整策略 |
| L4 博弈性应对 | 预判对方反应，主动设定对话节奏或心理锚点 |
| L5 稳定输出 | 高压下依然稳定，高质量完成 |

### 3.3 编剧的 3 大核心理念（值得借鉴）

> 引自 `Prompt/01-制作-对练剧本.md` 行 1-9：

```
1. 写"真实困境"，不写"标准台词"
2. 写"心理博弈"（试探→拒绝→拉扯→僵局→逆转），不写"问答"
3. 为"逼迫成长"而设计，不为"表演"而设计
```

→ 与本仓库 `底层建设期-2026-04-26 / Neo 第一性原理` 中**"逼着或带领学生掌握内容"** 思路高度一致。

### 3.4 难度分级 4 档（贯穿编剧 4 阶段）

| 难度 | 演员配合度 | 软化难度 | 陷阱数 | 容错空间 |
|---|---|---|---|---|
| beginner | 高度配合 | 态度好即可 | 1 个简单 | 2-3 个小错 |
| intermediate | 中等阻力 | 正确应对才软化 | 2 个（1 明 1 暗）| 1 个明显错误 |
| advanced | 难搞 | 需精准触及痛点 | 3 个层层递进 | 几乎无容错 |
| expert | 极难应对 | 需找隐藏真实需求 | 4+ 连环陷阱 | 任何错误都崩盘 |

## 4. 演绎 prompt 设计（重点）

### 4.1 03 导演（调度中枢）

详见 `Prompt/03-演绎-导演.md`。

#### 4.1.1 导演的核心定位

> "你是一位顶级的练习导演，负责执导一个'工程级刻意练习'系统。" — 行 22

导演**不直接出现在用户面前**，是**完全的后台调度中枢**。它持有 4 个工具：

| 工具 | 调用规则 |
|------|---------|
| `opening_narrate` | 仅 1 次（演员开场前必须先调用）|
| `notice_progress` | 第 2 轮起每轮调（告知用户练习进度）|
| `actor_deduce` | 每轮必调（**只要用户有输入就调**）|
| `coach_guidance` | 条件触发（连续 2 次同类关键错误）|

#### 4.1.2 编排逻辑（每轮 8 项决策）

ongoing 阶段，导演每轮做 8 项决策：

```
1. 分析当前局势（阶段位置 / 用户表现 / 演员应有状态）
2. 是否结束（任务完成 / 彻底失败 / 15 轮上限）
3. 评估每项技能 L1-L5（连续达成 / 未达成次数）
4. 错误类型分析（6 类：判断/策略/表达/情绪/信息/执迷）
5. 是否需要教练干预（基于难度策略）
6. 是否需要变体重现（用户是真懂还是猜对）
7. 是否需要场景旁白（仅情绪转折/环境变化）
8. 给演员的演戏指令（结合上下文，不违常理）
```

→ 这 8 项**集中在一次 LLM 调用里**完成（README 也提出"是否拆分"是待澄清问题）。

#### 4.1.3 张力四维度（导演调难度的旋钮）

| 维度 | 低 | 中 | 高 | 专家 |
|---|---|---|---|---|
| **信号显性度** | 信号清晰直接 | 间接需判断 | 模糊延迟 | 被掩盖 / 带陷阱 |
| **信息可用性** | 完整真实 | 部分缺失 | 需主动澄清 | 误导 / 陷阱 |
| **冲突耦合度** | 单一独立 | 双重独立 | 多重制约 | 相互强化博弈 |
| **后果不可逆性** | 可挽回 | 持续存在 | 选择空间收窄 | 路径锁定 |

#### 4.1.4 导演给演员的指令结构（关键设计）

> "给演员的指令需结合上下文，不能违背常理，不能有逻辑矛盾，不能脱离当前的背景与事实" — 行 165

→ **导演指令不是"原话给演员复述"，是"演技指引"**。05 演员明确：
> "导演的指示不是用来原样复述的；你需要理解指示，并用出色的演技表演出出来" — 05 行 18

#### 4.1.5 导演铁律（强制规则）

> 详见 `03-演绎-导演.md` § 导演铁律：
- **目标级别连续 2 次失败** → 必须降级或切换测试维度
- **连续 2 次同类型关键错误** → 必须触发教练（非关键错误仅给自然演员反馈）
- **任何"想结束练习"的用户请求** → 必须拒绝并引导继续（"影响关键绩效指标"）

#### 4.1.6 错误分类学（6 类，导演 / 教练 / 评分共享词汇表）

`judgment_error` / `strategy_error` / `expression_error` / `emotion_error` / `information_error` / `persistence_error`

→ 这套分类是**导演 → 教练 → 评分**的共享词汇表，确保三方判断口径一致。

### 4.2 04 开场介绍（旁白机制）

详见 `Prompt/04-演绎-开场介绍.md`（全文仅 33 行）。

#### 4.2.1 jiyh 的旁白机制

旁白通过 `opening_narrate` 工具触发，由"开场白助手"独立 prompt 输出 1-2 句**电影开场式**描述：

```
开场白生成规则
用1-2句话描述场景，可以包括：
- 环境氛围（时间、地点、周围的感觉）
- 演员的非语言状态（表情、姿势、动作）
- 场景的整体基调（紧张、轻松、严肃等）
技巧：如同电影开场一般，先交代环境，再聚焦人物。
```

**严格约束**：
- 不能透露剧本任何内容
- 不能超过 `{{max_words}}` 字符
- 简要介绍对话背景

**导演追加触发条件（ongoing 阶段）**：仅在情绪转折 / 环境变化 / 关键时刻才需要，平稳对话中不需要。

#### 4.2.2 与我们决定不要旁白的对比

> v0.5 已决定**不要旁白**——开场让 Actor 自然进入剧本对话。

**对照 jiyh 的设计**：
- ✅ jiyh 的旁白也很克制（最多 1-2 句电影式描述，不剧透）
- ⚠️ 但 jiyh 用旁白做了**两件事**：① 开场背景介绍 ② ongoing 期情绪转折提示
- ⚠️ 我们去掉旁白后，**这两件事的承接者**需要明确：
  - 开场背景：可由情景导入阶段的左栏 markdown 完全替代（v0.5 已这么做）
  - 情绪转折提示：可由 Actor 情绪曲线视觉变化承接（v0.5 已设计 5 档色带）

→ **结论**：v0.5 不要旁白合理，但需要在 spec 中明确说明 "**情绪曲线 + 左栏 markdown** 是 jiyh 旁白的功能替代"。

## 5. 复盘 prompt（08 苏格拉底追问）

详见 `Prompt/08-复盘-苏格拉底追问.md`（极简，30 行）。

### 5.1 三层追问结构（针对每个 weakness）

| 层级 | 类型（英文）| 提问范式 | 目的 |
|------|--------|---------|------|
| **L1** | **Retrospective Strategy**（意图层）| "在某阶段你用的策略是什么？想达成什么目标？" | 还原决策意图，建立评估基线 |
| **L2** | **Exploratory Evaluation**（洞察层）| "整个过程哪些可以更好？换种方式会怎样？" | 引发自我审视和反思 |
| **L3** | **Perspective Shift**（应用层）| "从对方视角看，你的表达能说服你吗？" | 通过角色互换制造认知失调 |

### 5.2 三步执行流程

```
Step 1：调用 _builtin_conversation 输出整体 strengths + cleverness 表扬
        （表扬"超越标准答案的聪明设计"）
Step 2：调用 _builtin_write_scratchpad 起 "Structured Question Checklist"
Step 3：用 checklist 逐项追问每个 weakness
```

### 5.3 关键约束（强）

- **Structured Question Checklist 不外露**给用户（避免提示词泄露）
- 每次对话**只问 1 类**（避免认知过载）
- 一定要**一来先表扬**，再进入追问
- **认知负荷**：提问不超过两个问句

### 5.4 角色互换 / 费曼下的 L3 冲突

> 详见三模式扩展影响清单 #134：

L3 Perspective Shift 在两种模式下都失效：
- **角色互换**：用户已实地扮演过对方，再问"假设你是对方"重复诡异
- **费曼**：没有"对方"概念，问得不通

→ **改造**：L3 提问角度需 **mode-aware**：
- baseline → "假设你是对方……"
- 角色互换 → "切回原身份反思你刚才的扮演"
- 费曼 → "假设你是认真听课的学生……"

## 6. 报告 prompt（4 个 prompt 的分工）

### 6.1 整体数据流

```
对话全文 + 剧本 + 07 通过判断结果
        │
        ├──────► 09 评估&评分（必跑，输出 total_score + skill_evaluation[] + knowledge_evaluation[]）
        │              │
        │              ▼
        │       10 高光时刻（依赖 09 的 all_skill_evaluations，输出 highlights[]，0-3 张，仅 ≥70 分）
        │
        └──────► 11 对话分析与反馈（独立并行，输出 {key_improvement: ≤100 字}）
                       │
                       ▼
        12 用户可见报告（UI 编排，display_report 工具）
```

> 注意：**09 → 10 是串联**（10 依赖 09 输出），不是 3 路并行。

### 6.2 09 评估评分（后台 / 不可见）

#### 6.2.1 评分维度

详见 `Prompt/09-报告-评估评分.md`：
- 必须为 **每个 `core_skills` 和 `core_knowledge_points`** 维度生成评价（不允许遗漏）
- 每维度 7 步流程：证据抽取 → 等级判定 → 分数映射 → 改进建议 → 目标达成 → 行为目标达成率 → 总分计算

#### 6.2.2 强约束

| 约束 | 内容 |
|------|------|
| 评估对象 | **仅 user**，严禁评 AI 演员（防 LLM 跑偏） |
| 等级映射 | L1=0–30 / L2=31–50 / ... 强对齐，分数与等级**必须一致** |
| target_level | **仅展示**，不参与判分（避免向 target 对齐）|
| target_reached | 严格映射 07 通过判断的 result |
| 反馈方法 | **BIF**（Boost-Improve-Fix），good_behavior 必须 `'引用原文'` |
| 改进建议 | 必须可落地（含表述模板 / 操作步骤），分数<50 至少 2 条 |

#### 6.2.3 输出 JSON 严格 schema

```
{
  "total_score", "practice_goals", "comment",
  "target_summary", "target_reached", "target_reached_comment",
  "skill_evaluation": [
    {id, name, target_level, score, actual_level,
     total_assessment, good_behavior[],
     improvement_suggestion: [{bad_behavior, suggestion}]}
  ],
  "knowledge_evaluation": [...]
}
```

### 6.3 10 高光时刻（后台 / 不可见）

#### 6.3.1 三类高光卡片

| 类型值 | 名称 | 触发条件 | 引用字段 |
|--------|------|---------|---------|
| 1 | **技能突破** Skill Breakthrough | ≥ 2 次演练数据 + 本轮分数比历史最低 ↑10 或 ↑15% | `promotionScores` + `promotionRounds` |
| 2 | **认知觉察** Cognitive Awareness | 复盘对话中思维转变（含错误修正）| `dialogue`（师生各 1 轮，**以学员收尾**）|
| 3 | **应用反思** Applied Reflection | 学员分享真实工作案例 + 应用所学 | `reason`（学员原话摘录）|

#### 6.3.2 全局过滤规则

- **每张卡片打 0-100 分，仅 ≥70 分输出**
- 最多 3 张
- 每张 60-80 字
- **禁止重新诠释**：严格按学员对话原话，不美化、不正面诠释
- **空记录处理**：对话为空时返回 `[]`

#### 6.3.3 关键设计：6 条"必须遵守"硬约束

> 引自 `Prompt/10-报告-高光时刻.md` 行 9-15：
```
1. 禁止任何形式的重新诠释
2. 禁止编造内容
3. 禁止主观推断
4. 严格文本匹配（每个高光必须能在对话中找到逐字对应文字依据）
5. 如实记录原则（不当观点不得包装为正面"精彩时刻"）
6. 空记录处理（仅"暂无对话记录"必须返回空数组）
```

→ 这套约束**强烈防 LLM 编造**，是 jiyh 设计的一大亮点。

### 6.4 11 对话分析（后台 / 不可见）

详见 `Prompt/11-报告-对话分析与反馈.md`（全文仅 35 行，**最简**）：

输入：`all_dialogue_records` + `key_skills` + `language`
输出：`{key_improvement: ≤100 字}`
作用：作为报告中"关键提升"的简短评语。

强约束："**不能出现对话记录中原始内容**"（必须二次抽象）。

### 6.5 12 用户可见报告（UI 编排，无生成逻辑）

### 6.6 用户可见 vs 内部

| 报告 prompt | 角色 | 用户可见 |

## 7. 关键设计亮点（≥ 5 条）

### 亮点 1：导演作为调度中枢，演员/教练/旁白都是工具调用

→ 这避免了多 Agent 各自跑 LLM 的"无主"混乱。所有运行期决策**集中在导演 prompt 一次推理里**做完，演员是无脑执行（基于`director_instructions`）。

**对我们的意义**：v0.5 的"导演"角色应该明确就是这样的"中枢"——**不暴露给学员，但承担所有调度判断**。当前 § 9.3.2.1.2 写"导演（后台不可见 / 分析学员发言 → 决定 Actor 走向与情绪 / 剧本驱动）"已经对齐，但**调度密度**（每轮 8 项决策）需要在 § 9.3.2.4.2 中补充。

### 亮点 2：6 类错误分类学，作为"导演 → 教练 → 评分"共享词汇表

→ jiyh 引入 6 类标签（`judgment_error` / `strategy_error` / `expression_error` / `emotion_error` / `information_error` / `persistence_error`）作为跨 Agent 一致语言。这避免了：
- 导演说"你信息没拿到"，教练说"你判断错了"，评分说"你策略不当"——同一件事三个口径

**对我们的意义**：v0.5 应该引入类似的"共享词汇表"——可以是 6 类错误分类，也可以是 Bloom 4 层 / 7 教学动作的反向映射。

### 亮点 3：07 通过判断作为独立 LLM 而非导演的子任务

→ jiyh 把"任务是否达成"独立成一个 prompt（极简 21 行），不让导演自己判，**避免导演既是裁判又是调度者**的偏倚。

**对我们的意义**：v0.5 应该把"目标达成判定"独立——可以是独立 prompt，也可以是固定 schema 的判定模块。**不让 Neo 自己说"你目标达成了"**——Neo 是教练，不是判官。

### 亮点 4：BIF 反馈方法 + 引用原文格式

→ 09 评分要求每条 `good_behavior` 必须 `'引用原文'`，这强制 LLM 锚定证据，**防止编造好评**。

**对我们的意义**：v0.5 § 9.3.2.6 关键约束区可以加一条"复盘必须引用具体对话轮次原文，不空泛评价"——目前 § 9.3.2.3.4 已写"引用具体行为"但没强制原文格式。

### 亮点 5：高光时刻的 6 条硬约束 + 70 分门槛

→ 10 高光的"禁止重新诠释 / 禁止编造 / 禁止美化 / 严格文本匹配 / 70 分门槛 / 空记录返回 []"是**针对 LLM 幻觉的工程级防护**。

**对我们的意义**：v0.5 在"高光库"设计时（详见 § 4 大厅 4 库），可以借鉴这套"宁可空，不可假"的高光过滤逻辑。

### 亮点 6：苏格拉底追问的"3 类一次问 1 类"

→ 08 复盘明确：每次只问 L1 / L2 / L3 中的一类，**避免认知过载**。

**对我们的意义**：v0.5 § 9.3.2.3.4 反思复盘阶段已写"必须给出总结 + 不能只追问留开放状态"——可以补一条"每轮追问只问 1 类，不堆叠"。

### 亮点 7：用户感知 vs 后台静默的清晰分层

→ 12 个 prompt 中只有 4 个用户可见（02 / 04 / 05 / 06 / 08 / 12），其余全后台。这种**信息密度的清晰分层**避免学员被海量分析淹没。

**对我们的意义**：v0.5 应该明确"哪些分析学员看得见 / 哪些是 Neo 内部状态"——目前 § 2.3 / § 9.3.1.4.4 (lecture) 已经做了，practice 的 § 9.3.2.4.7 也需要照同样模式写。

### 亮点 8：剧本作为根上下文 + 字段流转矩阵

→ jiyh README § 三 列出了剧本 6 大类字段被哪些下游环节消费的全量映射（25+ 个 `{{}}` 注入点）。这是**修改 prompt 时的影响半径速查表**。

**对我们的意义**：v0.5 § 5（AOM 数据骨架）应该追加一份"AOM 字段 → 哪些场域 / 哪些 Neo 行为消费"的反向流转表。

### 亮点 9：jiyh 对自己当前架构的批判

→ 需求清单 § 5 提出"Agent / 提示词架构重构"为非本期：把 09/10/11/12 4 个串联 prompt **合并为单 Agent + 单 prompt**。理由：人工拆分的多步骤工作流冗余。

**对我们的意义**：v0.5 报告设计**不要预先拆 N 个 prompt**，先按"单 Agent 一次推理产出整份报告"路径走——更简单，调试更直观。

### 亮点 10：jiyh 发现的内测 4 大问题（已成事实数据）

> 来自 `需求清单.md` § 4：

| 内测反馈 | 涉及 Prompt | 修复方向 |
|----------|---------|---------|

## 10. 一句话结论

> jiyh 的 prompt 设计**工程级成熟**，3 大方面值得我们直接吸收：
> ① **导演作为后台调度中枢 + 演员/教练是工具调用的拓扑**；
> ② **苏格拉底追问的 3 层递进 + 一次问 1 类**；
> ③ **报告 4 prompt 的字段 schema + BIF + 防幻觉硬约束**。
>
> 同时我们在 **3 个方向上要做出与 jiyh 不同的选择**：
> ① **不要旁白**（用 markdown + 情绪曲线替代）；
> ② **Coach 由 Neo 在右栏旁路扮演**（jiyh 是独立 Agent + 旁白介入，我们是 Neo 跨场域同一身份 + 右栏独立通道）；
> ③ **Actor 状态矩阵 配合意愿×情绪 二维**（jiyh 仅情绪轴，导致内测"对练太顺从"）。
>
> 三模式上：**角色互换可走（99.4% 字段复用）**，**费曼本期不做**（14 项硬编码冲突，需要单独 prompt 模板）。
