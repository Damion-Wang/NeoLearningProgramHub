### 9.3.3 recap（小结）

> **总览**：recap 是学员在 Course 学习中**随时可入**的内容反思场域 —— 对 Course 内容学习情况的小结，目的是**增强学习效果**（不是评价）。内容由三个层面组成：**知识层面回顾 + 表现层面回顾 + 下一步建议**。学员首次完整态进入触发 milestone 锁定首聊报告永久保留；之后产生的新数据通过自由复习层覆盖式更新（每次覆盖整层为当下最新综合）。

**4 数据状态 × 4 Neo 行为态对应总览**：

| 进入数据状态 | Neo 行为态 | Walk-through | 待办任务产出 | milestone |
|------------|----------|-------------|------------|-----------|
| 空白态（0 个关联 Activity 完成）| **预览态** | 不做 | ❌ | ❌ |
| 部分态（部分关联 Activity 完成）| **预览态** | 不做 | ❌ | ❌ |
| 完整态首次（关联 Activity 全部完成 + 首次进入）| **首聊态** | 完整 5 块 | ✅ 1-3 条入大厅待办 | ✅ 触发 |
| 自由复习态（milestone 已触发后再进入）+ 有变化 | **回访态** | 不重讲 / 主动指出变化点 | ❌（已在首聊态产出）| 已完成 |
| 自由复习态 + 无变化 | **复习态** | 不主动 / Neo 自主判断是否开场 | ❌ | 已完成 |

> **关键设计**：milestone 是一次性事件——完整态首次进入触发后状态永久转入"自由复习态"。首聊报告锁定为"首聊层 5 块"永久保留；自由复习态产生的新内容统一进入独立的"自由复习层"区块（**每次更新覆盖整层为当下最新综合**——因为更新触发点不统一，无法按时间叠加）。详见 § 9.3.3.2.6 自由复习层。

#### 9.3.3.1 业务定义

##### 9.3.3.1.1 一句话定位

> recap（小结）是学员在 Course 学习中**随时可入**的反思场域 —— 通过查看 Neo 给出的当前反思报告 + 与 Neo 对话挖掘洞察，把"做过什么"沉淀为"看见什么 + 接下来怎么用"。

recap 同时承担 **三重价值**：

| 价值层次 | 学员视角 | 项目视角 |
|---------|---------|---------|
| **知识层面回顾** | 把 Course 内学过的关键概念 / 框架重新整合，结合自己学习时提出的真实场景做内化 | 让"学过"沉淀为"内化"——核心交付能力 |
| **表现层面回顾** | 看到 Neo 对自己学习行为的整体判断（互动行为 + 6 维画像变化 + 高光与卡点）—— 自己看不到的视角 | 沉淀可举证的学员成长轨迹 |
| **下一步建议** | 在 Neo 的引导下，从反思转向具体下一步行动（应用计划自动入大厅待办）| 让学习闭环到工作场景，避免学完即忘 |

##### 9.3.3.1.2 三场域差异 + 内容反思 vs 项目进度边界

| 场域 | 学员主任务 | 粒度 |
|------|----------|------|
| **lecture** | 概念吸收（输入）| SCO 级 |
| **practice** | 行为养成（输出）| Activity 级，含 Activity 内复盘阶段 |
| **recap** | 反思整合 + 行动转向 | 跨多个关联 Activity 的反思整合 |

**与 practice Activity 内复盘阶段的边界**：

- **practice 复盘**（详见 § 9.3.2.3.4）= 单次对练后的行为复盘——针对那一次演练的具体决策点
- **recap** = 跨多个关联 Activity 的反思整合——综合 lecture 学习行为 + practice 表现 + Neo 跨场域观察，呈现学员当前的整体状态

两者不重复，是 Activity 级 vs Course 级两层叠加。

**内容反思 vs 项目进度边界**：

recap 是**内容维度**的反思场，**不承担项目进度仪表盘职能** ——

| 属于 recap | 不属于 recap |
|-----------|------------|
| Course 内容主题与关联 Activity 的知识回顾 | 距项目结营还有几周 / 时间轴位置 |
| 学员在不同 Activity 的互动行为画像 | "已完成 X/Y 个 Activity"项目进度数字 |
| Neo 在关联 Activity 内对学员 6 维画像的判断 + 改变 | 全项目 6 维画像的差分对比 |
| 关联 Activity 内的具体高光与卡点片段 | Quiz 命中率 / 通关率（Quiz 不一定有对错，详见 § 9.3.1.3.4）|
| Neo 给学员的具体行动建议（待办任务）| 项目阶段性 KPI / 业绩指标 |

**项目进度类信息由大厅 § 9.2 承担**（项目总览区 / 时间轴 / 进度数字 / 待办清单），recap 不重复也不交叉。

##### 9.3.3.1.3 进入路径 + 增量更新机制

recap Activity 出现在**项目整体内容目录中**（与 lecture / practice 平级，三场域 icon 差异化区分）。学员有**两种进入路径**：

| 进入路径 | 含义 |
|---------|------|
| **从大厅内容目录随时进入** | 学员在 Course 学习的**任意时刻**通过大厅内容目录 / 课程目录抽屉点入 recap —— 不要求前序完成度，不强制顺序 |
| **从上一个 Activity 自然跳转进入** | 学员完成 recap 前一个 Activity 后，通过"完成 Modal · 继续下一 Activity"按钮自然过渡进入 |

**首次进入的双路并存** —— 默认是 Activity 完成 Modal 自然过渡，但学员主动从大厅内容目录提前进入也允许（按数据状态自适应）。

**完整态首次进入的判定**：

- **关联 Activity 全部完成** + **学员首次进入 recap** = 完整态首次（触发 Neo 首聊态行为：完整 Walk-through + 产出应用计划入大厅待办）
- 进入路径不影响判定（不论从大厅内容目录点入还是从上一个 Activity 跳转进入，只要满足"前序全完成 + 首次"就触发 milestone）

**报告增量更新机制**：

- 报告以"关联 Activity 完成"事件为单位后台增量重新生成；学员进入时直接看最新缓存
- milestone 触发瞬间锁定首聊层，永久保留
- milestone 后，新数据通过"自由复习层"覆盖式更新（详见 § 9.3.3.2.6）
- 学员侧零等待，后台静默完成

##### 9.3.3.1.4 学员进入时的 4 种数据状态

| 状态 | 含义 | 报告呈现 |
|------|------|---------|
| **空白态** | 关联 Activity 内 0 个完成 | 不生成报告，引导"先做点什么再回来" |
| **部分态** | 部分关联 Activity 完成（milestone 未触发）| 仅基于已完成数据生成；明确标注"仅基于已完成的 X/Y 个前序 Activity" |
| **完整态** | 所有关联 Activity ≥ 1 次完成 + milestone 未触发的窗口 | 完整报告——这是即将触发 milestone 的状态，等学员完整走完首聊 Walk-through 即触发 |
| **自由复习态** | milestone 已触发后的所有再进入 | **报告分层呈现**：首聊层（永久独立）+ 自由复习层（每次覆盖式更新）|

> **数据状态 → Neo 行为态 严格对应**详见顶部总览表与 § 9.3.3.4.3。

##### 9.3.3.1.5 milestone 完成定义

recap 的"完成"采用 **milestone 概念**：

- **milestone 触发** = **完整态首次进入** → Neo 首聊态完整 Walk-through → 对话有实质性反思 → Neo 一次性产出应用计划入大厅待办
- 仅"完整态首次进入"才能触发 milestone（部分态 / 空白态进入即使完整对话也不计 milestone）
- milestone 触发瞬间锁定首聊报告（永久独立保留）
- 触发后状态永久转入"自由复习态"——后续再进入按 Neo 行为态切换（回访态 / 复习态）
- **报告增量更新不退回 milestone 未完成** —— Subbar 显示"📊 报告有更新"小红点提示

学员部分态 / 空白态下进入也允许阅览当下报告 / 与 Neo 闲聊（Neo 预览态行为，详见 § 9.3.3.4.3），但不计 milestone。

#### 9.3.3.2 Course 个人报告（核心交付物）

> **作用**：报告是 Neo 引导对话的"道具"——写到 70%，留 30% 给 Neo Chat 展开。报告内容承载 recap 的三层价值（知识回顾 + 表现回顾 + 下一步建议）。

##### 9.3.3.2.0 报告全景

**首聊层 5 块结构**（milestone 触发瞬间锁定，永久保留）：

| # | 块 | 承载层面 | 内容 | 形态 | 数据来源 |
|---|----|--------|----|------|---------|
| ① | **内容锚点** | 知识层面回顾 | Course 主题 + 关键概念 / 框架 + 学员在场域内提出的真实场景串联 + 学员表现一句概括 | 纯 markdown | Course AOM 元数据 + 关联 Activity 概览 + 学员场景描述（来自 lecture FEEDBACK_COLLECT 自由文本 / Neo Chat 对话 / practice 演练上下文）|
| ② | **走过的轨迹** | 表现层面回顾 | 学员在不同关联 Activity 的互动行为画像 | markdown + 1 张图 | 全场域日志：每关联 Activity 的全类型互动总数（含 Quiz 答题 / practice 中栏 / Neo Chat）|
| ③ | **6 维画像** | 表现层面回顾 | 有变化点 → 变化点列表 + 嵌触发证据 / 无变化 → 6 维当前快照 | markdown + 条件性 1 张图 | Neo 在关联 Activity 内记录的画像变化数据（详见 § 9.2.1 + § 9.3.1.4.6 / § 9.3.2.4.6）+ 6 维当前值 |
| ④ | **关键证据** | 表现层面回顾 | 高光 1-2 条 + 卡点 1-2 条（具体行为片段引用）| 纯 markdown 引用块 | 关联 Activity 内：高光库子集 + practice 复盘待改进点 + Neo 跨场域感知卡点模式 |
| ⑤ | **待办任务** | 下一步建议 | Neo 产出的 1-3 条具体行动建议（应用计划），列在报告里同时自动入大厅待办清单 | 纯 markdown 列表 | Neo 基于报告全数据 + 跨场域记忆 + 学员场景的高度个性化生成 |

**报告全长**：首聊层 700-1000 字 + 嵌入图表 1-2 张 / 学员阅读时长 2-3 分钟 / 完整解读体验（含 Neo Walk-through 对话）8-10 分钟。

##### 9.3.3.2.1 ① 内容锚点（首聊层）

| 维度 | 说明 |
|------|------|
| **是什么** | 一段 80-150 字的 markdown：Course 主题 + 关键概念 / 框架 + 学员在场域内提出的真实场景串联 + 学员表现一句概括 |
| **目的** | 让学员一秒回到"这门课在讲什么 + 我跟它的连接"——核心是**知识层面回顾**，结合学员自己的场景做内化锚点 |
| **形态** | 纯 markdown，无图 |
| **数据来源** | Course AOM 元数据（主题 / 关键概念）+ 关联 Activity 列表 + 学员在场域内主动提出的场景描述（由 Neo 跨场域记忆调取）|
| **状态自适应** | 空白态：仅 Course 主题一句 / 部分态：Course 主题 + 已学概念 + 已提场景串联 / 完整态：完整知识回顾 + 学员场景串联 + 表现一句话 |
| **可变性** | milestone 前每次刷新；milestone 触发时锁定为首聊层；之后**永久不变**（学员场景已在首聊时整合） |

**示例骨架**（占位）：

> 这门课讲的是 **{Course 主题}**，关键概念是 **{概念 1 / 概念 2 / 概念 3}**。你在学习中提到过 **{学员真实场景 1，如"我跟同事在 KPI 拆解上有分歧"}**，这正好是本课要解决的问题——{概念 N} 在这种场景下能 {落地用法}。从 {Activity X / Activity Y} 走过来，你整体上 {表现节奏一句概括}。

##### 9.3.3.2.2 ② 走过的轨迹（首聊层）

| 维度 | 说明 |
|------|------|
| **是什么** | 学员在本 Course 关联 Activity 中的**学习行为画像**——哪些 Activity 互动最多 / 哪些最少 |
| **目的** | 让学员看到自己在内容上的**投入分布**——在哪里花了时间 / 在哪里走过场。这是**内容反思素材**，不是项目进度仪表盘 |
| **形态** | markdown 一段（50-100 字解读）+ 图 1（互动行为对比柱图，详见 § 9.3.3.2.0 图表清单）|
| **数据来源** | 全场域日志：每关联 Activity 内学员的全类型互动数（不区分类型，避免跨类型比较失真）。**不记 token 消耗**（跨类型 Activity token 差异巨大会误导）|
| **状态自适应** | 空白态：不显示 / 部分态：仅显示已完成 Activity 的对比 + 标注"仅含 X/Y" / 完整态：完整对比图 |
| **可变性** | milestone 前每次刷新；milestone 触发时锁定为首聊层 |

**markdown 解读要点**：
- 指出投入差异最大的 1-2 个对比（"你在 Activity A 互动最多 / 在 Activity B 几乎没问问题"）
- 不预设"哪种好哪种差"——这是行为事实陈述，让学员自己反思

##### 9.3.3.2.3 ③ 6 维画像（首聊层）

| 维度 | 说明 |
|------|------|
| **是什么** | Neo 对学员 6 维度（详见 § 9.2.1 B.2.B）的判断：**有变化点 → 显示变化点列表 + 嵌触发证据**；**无变化点 → 显示 6 维当前快照** |
| **目的** | 让学员看到 Neo 在本 Course 中**对自己的判断**——这是学员看不到的视角，是 1v1 老师的真功夫 |
| **形态** | markdown 解读 + 图 2（6 维画像图，变化点 / 快照二选一，详见 § 9.3.3.2.0 图表清单）|
| **数据来源** | Neo 在 lecture / practice 关联 Activity 内观察到学员某维度有实质变化时记录的"画像变化数据"（详见 § 9.2.1 + § 9.3.1.4.6 / § 9.3.2.4.6）+ 6 维当前值 |
| **状态自适应** | 空白态：不显示 / 部分态：基于已完成 Activity 的变化点（可能为空 → 显示当前快照）/ 完整态：变化点列表 / 当前快照 二选一 |
| **可变性** | milestone 前每次刷新；milestone 触发时锁定为首聊层 |

**变化点呈现规则（每个变化点自含触发证据）**：

```
> Y 维度（如"主动追问能力"）↑ 3→4
> 证据：在 Activity {X} 第 {N} 轮，你 {具体行为片段}——这是这门课希望你练出来的关键动作
```

每个变化点都嵌入触发证据（具体到 Activity / 对话片段），让学员看到"凭什么这么说"。

**无变化点的兜底**：

```
> 本 Course 中 Neo 对你的画像没有触发实质变化。当前 6 维快照：
> [6 维当前值进度条]
```

让学员看到"我现在的状态"，避免"白学了"的失望感。

##### 9.3.3.2.4 ④ 关键证据（首聊层）

| 维度 | 说明 |
|------|------|
| **是什么** | 把抽象洞察落到**具体行为片段**——高光 1-2 条 + 卡点 1-2 条 |
| **目的** | 让学员**认得出**自己——洞察必须基于学员能识别的具体场景 |
| **形态** | 纯 markdown 引用块（每条 = 标题 + 引用片段 + Neo 一句话留白）—— 无图 |
| **数据来源** | **严格限定关联 Activity 内**。高光 = 关联 Activity 内高光库子集（高光卡片是 Neo 在各场域的通用能力，详见 § 9.2.1，recap 取与本次关联范围相符的子集）；卡点 = 关联 practice 复盘的待改进点 + Neo 跨场域感知到的反复出现的卡点模式。**不依赖 Quiz 错题**（Quiz 不一定有对错） |
| **状态自适应** | 空白态：不显示 / 部分态：基于已完成 Activity 内的证据；数量不足时**宁可空着也不补关联外** / 完整态：完整高光 + 卡点 |
| **可变性** | milestone 前每次刷新；milestone 触发时锁定为首聊层 |

**与大厅高光区的关系（透明性对齐）**：

- 高光卡片本身是 Neo 在各场域（lecture / practice / 大厅）的通用能力产物（详见 § 9.2.1）
- recap ④ 块呈现的高光是**大厅高光区的子集** —— 取关联 Activity 范围内的高光
- 学员在大厅高光区能看到完整集合，recap 仅按本次关联范围筛选

**呈现示例**（占位）：

> **🌟 高光｜在 Activity A 你做对了一件事**
>
> > 你在第 N 轮主动追问对方的真实顾虑，没有被表面的拒绝挡住。这是这门课希望你练出来的关键动作。

> **⚠️ 卡点｜你反复栽在同一个地方**
>
> > 在 Activity B 第 X 轮 / Activity C 第 Y 轮，你都是先认同对方再说自己的观点，但最后**忘记把自己的立场说清楚**。

##### 9.3.3.2.5 ⑤ 待办任务（首聊层）

| 维度 | 说明 |
|------|------|
| **是什么** | Neo 产出的 **1-3 条具体行动建议（应用计划）**——直接列在报告里同时自动入大厅待办清单 |
| **目的** | recap 的**下一步建议**层落地——把反思转化为可执行的小任务，闭环到学员真实工作场景 |
| **形态** | 纯 markdown 列表（每条 = 1 句简洁可执行的小任务）—— 无图 |
| **数据来源** | Neo 基于报告 ①-④ 全数据 + 跨场域记忆 + 学员场景的**高度个性化生成**——同 Course 不同学员产出的待办任务差异明显 |
| **生成时机** | **仅在首聊态走完 Walk-through 后生成一次**；后续 milestone 不再变 |
| **流向** | 同步加入大厅"待办清单"（沿用 § 9.2.2.1 (1) E + § 9.2.1 C.3.4 大厅待办清单机制）；后续追踪 / 学员勾选 / Neo 跟进 全部在大厅待办流程内 |
| **学员参与** | 不参与撰写、不需确认——Neo 直接产出，直接入待办 |
| **可变性** | milestone 触发瞬间生成 + 锁定首聊层；**永久不变**（即使后续报告增量更新、自由复习态再进入也不再生成新待办）|

**呈现示例**（占位）：

```markdown
### ⑤ 待办任务

基于这门课，我建议你接下来一周做这几件事——已加入大厅待办清单：

1. {具体行动 1}（如"下周一跟 X 部门同事的会上，试一次先表达自己立场再求共识"）
2. {具体行动 2}
3. {具体行动 3}

→ [跳转大厅待办查看]
```

**后续追踪**（与大厅协同，详见 § 9.3.3.8.4）：

- 学员可在大厅待办主动点掉
- 学员点掉时 Neo 跟上来问情况
- Neo 评估后再决定是否允许点掉
- **完成不反映到 recap 报告**（闭环在大厅）

##### 9.3.3.2.6 自由复习层（独立区块，覆盖式更新）

> **作用**：milestone 触发后，自由复习态产生的所有新内容统一收录在本区块——独立于首聊层 5 块结构，**不打散到 5 块里**，**每次更新覆盖整层**。

**位置**：在报告末尾（首聊层 ①②③④⑤ 之后），用 markdown 标题 `## 自由复习追加` 作分界。

**内容范围**：

| 内容类型 | 说明 |
|---------|------|
| Neo 在自由复习态对话中产生的**当下观察** | 学员重进 recap 跟 Neo 聊天时，Neo 基于截至当下所有自由复习态对话产生的综合观察 |
| **截至当下的画像变化点** | milestone 后 Neo 在 lecture / practice 内新写入的画像变化（如学员重练 practice 后某维度变化）|
| **截至当下的新高光 / 卡点** | milestone 后产生的新关联 Activity 内高光 + practice 重练带来的新卡点 |
| **新关联 Activity 总结**（如有）| KGP 后期追加新关联 Activity 时，Neo 对该新 Activity 内学员表现的综合回顾 |

**更新机制（★）**：

- **每次更新覆盖整层**：自由复习层不是按时间叠加，而是**每次更新触发时基于截至当下的所有数据重新生成一份当下最新综合，整体覆盖上次内容**
- **理由**：自由复习态的更新触发点不统一（学员可能 Activity 1 聊两句、Activity 2 聊半小时，产生两次更新），按时间叠加会让内容割裂；用"覆盖式整体重生成"保证学员每次看到的是**当下最新一份完整综合**
- **不保留历史版本**：学员看不到"上一次的自由复习层"，也无法对比
- **首聊层永远不变**：自由复习层的覆盖式更新不影响首聊层（首聊层永久独立保留）

**呈现形态**：

- 自由复习层是**一份当下最新的综合内容**，不分块（区别于首聊层的 5 块结构）
- markdown 形态自由——可含小标题 / 引用块 / 列表 / 图表（如有新画像变化点可附图）
- 不显示更新时间轴 / 不显示"上次追加 vs 这次追加"分割

**学员侧呈现**：

```markdown
## 首聊回顾（永久保留）

### ① 内容锚点
...

### ② 走过的轨迹
...

### ③ 6 维画像
...

### ④ 关键证据
...

### ⑤ 待办任务
...

---

## 自由复习追加（截至当下最新）

- Neo 综合观察：...
- 截至当下的画像变化点：...
- 截至当下的新高光 / 卡点：...
```

**Neo Chat 历史**：在 Activity 内不被清空——学员可翻 Chat 历史回看跟 Neo 聊过的具体对话内容（不在自由复习层重复呈现）。

**笔记复用**：学员"想留住某段感动"通过笔记功能记录（笔记悬浮球，沿用 § 9.2.3）。

##### 9.3.3.2.7 生成 / 更新 通用规则

| 维度 | 规则 |
|------|------|
| **触发（milestone 前）** | 关联 Activity 完成事件触发后台增量重新生成；新版本覆盖旧版本（不留旧快照）|
| **触发（milestone 触发瞬间）** | 当下报告内容固化为"首聊层"，永久独立保留 |
| **触发（milestone 后）** | 自由复习层每次更新触发时整层覆盖式重生成（基于截至当下所有数据综合）；详见 § 9.3.3.2.6；首聊层永远不变 |
| **缓存** | 学员进入 recap 时直接看最新缓存（首聊层 + 自由复习层叠加视图）|
| **学员侧零等待** | 后台静默完成，学员无感 |
| **不提供"刷新"按钮** | 自动更新即可，按钮反而暴露机制 |
| **修改性** | 学员侧**不可修改**（整体一次性生成）|
| **导出性** | 场域内不可导出（沿用 § 9.3.5 跨场域统一规则）；大厅"我的报告"主题可导出 |
| **图表准入** | 三条全过才放图：① 对学员有内容反思价值 ② 范式稳定（可套模板）③ 数据来源清晰。本期 ≤ 2 张（图 1 互动行为对比 / 图 2 6 维画像）|

| 状态 | 触发 | 视觉提示 |
|------|------|---------|

| **报告最后更新时间** | 默认 | Subbar 显示"📊 X 分钟前更新" |
| **报告有更新红点** | 距上次进入有增量更新 | Subbar 红点；进入后 Neo 完成回访态开场后清除 |
| **关联范围徽章** | 默认 | 报告头部显示"本次关联 Activity：[X / Y / Z]"（学员侧明示，详见 § 9.3.3.2.0）|
| **部分态徽章** | 学员处于部分态 | 报告头部显示"⚠️ 仅基于已完成的 X/Y 个前序 Activity" |

```
│ 完成更多前序 Activity 后报告会自动更新                  │
└───────────────────────────────────────────────────────┘
```

避免学员误以为是完整评估。

#### 9.3.3.4 对话场景编排

> **设计理由**：recap 是融合的对话场景（看报告 + 与 Neo 聊 + Neo 给行动建议三件事在同一屏幕同时进行），不是分阶段的内容序列。recap 内部行为是产品级统一规则，KGP 不干预。

##### 9.3.3.4.1 场景结构

单一连续对话流：

```
学员进入 recap Activity
  ↓
Neo 按当前数据状态判断行为态（预览 / 首聊 / 回访 / 复习）
  ↓
学员 ↔ Neo 自由对话（融合看报告 + 反思挖掘 + 接收行动建议）
  ↓
首聊态走完 Walk-through 后 Neo 自动产出应用计划 → 入大厅待办清单
  ↓
完成 milestone（学员主动结束 / Neo 引导收尾）
  或退出（断点续播保存）
```

##### 9.3.3.4.2 KGP 在 recap 上的能力边界

KGP 只控制两件事：

| KGP 控制项 | 含义 |
|----------|------|
| ① **位置** | 把 recap Activity 摆放在 AOM 的什么位置（Course 内某点 / Course 末尾 / 项目末尾，详见 § 6.7）|
| ② **关联范围** | 决定 recap 关联**哪些前序 Activity** 作为数据源 —— 关联范围由 KGP 自由控制，**不强制与 Course 绑定**。本期典型场景是关联同 Course 内全部前序 Activity，但**功能上不限制 KGP 跨 Course 关联或仅关联部分**（为未来课程编排玩法留口）|

recap 内部行为（Walk-through 节奏、反思引导、应用计划产出、完成判定标准、报告生成规则）= 产品级统一规则，**KGP 不干预**。

##### 9.3.3.4.3 Neo 行为四态切换

学员可能多次进入同一个 recap Activity（随时打开机制）。Neo 根据**进入时的数据状态 + 距上次发生了什么**自动切换行为：

| 行为态 | 触发数据状态 | Walk-through | 主动开场 | 应用计划产出 | milestone | Neo 行为细节 |
|------|------------|-------------|--------|------------|-----------|----------|
| **预览态** | 空白态 / 部分态 | ❌ 不做 | 应答 + 提醒"部分状态" | ❌ | ❌ | Neo 不主动 walk-through；应答学员提问 + 提醒"这份报告是部分状态，多做几个再过来会看得更准" |
| **首聊态** | 完整态首次进入 | ✅ 完整 5 块 | ✅ | ✅ 走完后产出入大厅待办 | ✅ 触发（一次性事件，触发后状态转入自由复习态）| 完整逐块引导学员看报告，详见 § 9.3.3.5.4 Walk-through 节奏 |
| **回访态** | 自由复习态 + 距上次有"实质性变化" | ❌ 不重讲全程 | ✅ 主动指出变化点 | ❌（已在首聊态产出）| 已完成 | "自然温和 · 先建连接再深入"——主动指出变化点 + 引用上次对话 + 询问近况 |
| **复习态** | 自由复习态 + 无实质变化 | ❌ 不主动 | ⚠️ Neo 自主判断（可问可不问）| ❌ | 已完成 | 主体被动答疑 —— 学员问什么 Neo 答什么 |

##### 9.3.3.4.4 行为四态切换的数据信号

Neo 综合多个信号自主判断当前应该用哪种态（spec 不规定固定阈值）：

**关联 Activity 内信号**（限定 KGP 配的关联范围）：

- 距上次会话是否有新关联 Activity 完成
- 关联 Activity 内是否有新高光卡片

**全场景信号**（不限定来源）：

- 6 维画像是否有明显变化（含大厅 / 跨 Course 来源）
- 学员在大厅或其他场景是否有新对话
- 学员上次在 recap 内跟 Neo 聊到哪
- 上次 Walk-through 走到哪一块

##### 9.3.3.4.5 待办任务产出 → 大厅待办清单

待办任务（应用计划）的产出形态、生成时机、内容规则详见 § 9.3.3.2.5（首聊层 ⑤ 待办任务）。本节仅说明**对话场景中的产出节奏**与**与大厅协同的后续追踪**。

**对话场景节奏**：

- 首聊态 Walk-through 走完 ①-④ 后，Neo 主动收尾 → 直接列出 1-3 条待办任务 → 同步加入大厅待办清单 → milestone 触发完成
- 学员侧不需手动确认；Neo 产出即定型

**与大厅协同的后续追踪**：

- 学员可在大厅待办清单**主动点掉**待办项（永远不限制总量）
- 学员点掉时 **Neo 跟上来问情况**（"做完了吗？怎么做的？"）
- Neo 评估学员回应后**再决定是否允许点掉**（即点掉需 Neo 确认）
- **完成不反映到 recap 报告** —— 闭环在大厅待办区完成（详见 § 9.3.3.8.4）

#### 9.3.3.5 Neo 行为（反思态）

##### 9.3.3.5.1 反思态定位

Neo 在 recap 与大厅 / lecture / practice 共用同一身份（同 voice / 方法论 / 记忆，详见 § 3.2 + § 9.3.0）。在 recap 中以 **反思态** 出现 —— 区别于 lecture Tutor 态 / practice Coach 态。

反思态的核心姿态：**引导学员看见自己 + 不替学员说总结**。

##### 9.3.3.5.2 7 动作分工

| 动作 | 权重 | recap 中的体现 |
|------|------|---------------|
| **反思** | 主场 | 引导学员看见自己的行为模式与变化轨迹 |
| **追问** | 主场 | 苏格拉底式追问，挖出学员自己的洞察 |
| **讨论** | 主场 | 与学员就报告内容平等对话 |
| **解惑** | 辅助 | 学员对报告或概念有疑问时回答 |
| **考核** | 辅助 | 报告里的 6 维画像 + 高光 + 卡点 隐含评估反馈 |
| **讲授** | 轻量补漏 | 学员忘了某概念 Neo 可补，**不主动**讲新内容 |
| **练习** | — | 不出现（→ practice）|

##### 9.3.3.5.3 Walk-through 节奏（仅首聊态）

Neo 在首聊态主动逐块引导学员看报告：

```
1. Neo 第 1 句：定位锚点
   "我们一起看看这份小结。先问你一件事：
    你看完第 ① 节内容回顾，最让你想起的场景是哪个？"

2. 学员回应 → Neo 引导到第 ② 节走过的轨迹
   "那我们看看你在不同环节的投入分布……"

3. 引导到第 ③ 节 6 维画像
   "我注意到你在 [Activity X] 之后 [Y 维度] 上来了，
    具体证据是 [嵌入证据片段]……你认这个评价吗？"

4. 引导到第 ④ 节高光 + 卡点
   "再看几个关键瞬间……"

5. 引导到第 ⑤ 节待办任务（=应用计划）
   "基于以上这些，我给你列了几件下一步可以试的事——
    我已经把它们加到你的大厅待办清单里了。"
```

**关键点**：
- 首聊态是一次性事件——走完即触发 milestone，不重复
- 学员中途退出 walk-through → 重进时从中断点续 walk-through，不从头
- **没有"重新听 walk-through"机制** —— 首聊态完成后通过 Neo 自由问答覆盖学员需求

##### 9.3.3.5.4 留白式追问规范

Neo 追问的核心：**不替学员说总结，引发学员自陈**。

| 类型 | 示例 |
|------|------|
| 引发观察 | "你看到这条洞察吗？想到了什么？" |
| 引发反思 | "为什么你觉得这维度低？" / "这个模式是不是熟悉的？" |
| 引发联想 | "这跟你上周在大厅说的那件事有关系吗？" |
| 引发承诺 | "你觉得接下来一周能做的一件事是什么？"（首聊态收尾）|

##### 9.3.3.5.5 跨场域记忆调用

Neo 在 recap 中**主动**引用学员的跨场域信息——让"老师记得你"成为体感：

| 引用对象 | 来源 |
|---------|------|
| 学员在 lecture 中提到的真实场景 | lecture FEEDBACK_COLLECT / Neo Chat 对话 |
| 学员在 practice 演练里的具体行为 | 关联 practice 演练对话 |
| 学员在大厅 Neo Chat 里说过的话 | 跨场域 Neo 记忆系统 |

引用准确性信任 § 9.4.1 反幻觉机制。

##### 9.3.3.5.6 必收尾约束

Neo 追问之后**必须**给收尾——不能只追问留开放状态（沿用 practice 复盘 § 9.3.2.6.5）。

收尾形态：

- 首聊态收尾 = Neo 给观察 + 留言 + 产出应用计划入大厅待办
- 回访态收尾 = Neo 总结变化点 + 引导学员说反思
- 复习态收尾 = 学员问什么 Neo 答什么后给收尾

#### 9.3.3.6 学员行为与核心交互

##### 9.3.3.6.1 输入方式

| 方式 | 交互 |
|------|------|
| **右栏 Neo Chat 输入框** | 学员与 Neo 对话（打字 / 语音）—— recap 内学员只跟 Neo 对话，不写应用计划 |

##### 9.3.3.6.2 核心交互一览

recap 多数交互沿用 lecture 设计（详见 § 9.3.1.5.2）：

| 交互 | 说明 |
|------|------|
| **随时进入** | recap 整个 Activity 入口对前序完成度无锁——可随时进；4 状态自适应 |
| **防挂机** | 默认 5 分钟无操作 → 系统弹提醒（沿用 lecture，可在 program config 调整阈值）|
| **学员主动提问 Neo** | 全程右栏 Chat 可问 Neo |
| **抽屉锚点跳转** | 学员点抽屉 Tab1 报告 5 块锚点 → 课件区滚动到对应块 |
| **Activity 间跳转** | 通过抽屉 Tab2 主动跳转其他 Activity（弹窗确认 + 触发断点续播保存）|
| **断点续播** | 中途退出 → 系统记录当前对话上下文 + Walk-through 进度；下次进入自动恢复 |
| **笔记悬浮球** | 屏幕边缘常驻（沿用 § 9.2.3）—— 学员"想留住此刻"通过笔记功能记录 |

##### 9.3.3.6.3 "随时进入"特殊机制

学员处于不同状态进入 recap 时：

| 状态 | Neo 第一句 |
|------|----------|
| **空白态** | "我们还没什么可聊的——先去做点什么再回来吧。" |
| **部分态** | "这份小结现在只能看到你走过的一部分。多做几个再过来会看得更准。" |
| **完整态首次** | 进入完整 Walk-through |
| **自由复习态 + 有变化** | "自从上次见你，你又做了 X，我看到 Y 维度有变化……" |
| **自由复习态 + 无变化** | Neo 自主判断是否开场 |

##### 9.3.3.6.4 重看 / 重进

完成 milestone 后再进入 recap = 行为四态切换决定行为：

| 进入场景 | Neo 行为 |
|---------|---------|
| 报告无更新 | 复习态——Neo 自主判断是否开场，主体被动答疑 |
| 报告有更新（Subbar 红点提示）| 回访态——Neo 主动指出变化点 |

报告随关联 Activity 完成自动更新（增量），学员侧无感。

学员"想再听一遍 walk-through"的需求**不通过独立按钮覆盖**——首聊态是一次性结构化讨论，剩下通过 Neo 自由问答覆盖需求。

#### 9.3.3.7 关键约束

按主题分 4 类：

##### 9.3.3.7.1 反思设计原则

| 约束 | 说明 |
|------|------|
| **不替学员总结洞察** | Neo 引导学员说，不替学员说——防止报告 / 对话变成 Neo 的"评分书"。学员反思必须由学员**自陈**，Neo 用追问引发，不强行总结 |
| **不预设标准答案 / 不给"应该"** | 沿用 § 2.6.2 + § 9.3.1.6.1 —— Neo 不告诉学员"你应该这样想 / 这样做"，引导学员自己得出 |
| **显性证据限定关联 Activity / 隐性背景不限定** | 报告内可被学员看到具体来源的证据类（高光 / 卡点 / 对练具体行为 / 6 维画像变化点）严格限定 KGP 配的关联 Activity 内；Neo 综合产物（自由复习层观察 / 待办任务依据 / 跨场域记忆）**不限定来源**（限定反而失真）。**不依赖 Quiz 错题**（Quiz 不一定有对错，详见 § 9.3.1.3.4） |

##### 9.3.3.7.2 报告产物约束

| 约束 | 说明 |
|------|------|
| **不可修改 / 场域内不可导出** | 整体一次性生成 / 学员侧不可改；milestone 前每次覆盖更新 / milestone 触发后首聊层永久保留 + 自由复习层每次覆盖式更新。场域内不导出（沿用 § 9.3.5）；大厅"我的报告"主题可导出 |
| **不承担项目进度信息** | 详见 § 9.3.3.1.2 边界—— 时间锚 / 完成度 / 项目周数 / 阶段 KPI 等由大厅 § 9.2 承担 |
| **与 team report 边界** | recap = 学员侧反思（个人 × Course / Course 末更新 / 学员不可改）；team report = 管理端阶段性对外汇报（团队 × 项目周期 / 月度或项目末 / HITL 编辑）。数据可能共享（如 6 维画像），呈现完全分开（详见 § 10.3）|

##### 9.3.3.7.3 体验机制约束

| 约束 | 说明 |
|------|------|
| **取消 SCO 锁定** | recap 整个 Activity 入口可随时进；不存在 SCO 序列，与 lecture / practice 严格顺序锁定不同 |
| **6 维画像可见性** | 学员侧 recap 报告露出（含变化点 / 当前快照）；管理端 D-5.0 不露出（与 § 9.3.2.3.3 状态矩阵决策一致——管理端可视化由 § 10 后续决策）|

##### 9.3.3.7.4 通用沿用

- **防挂机**：默认 5 分钟无操作触发（沿用 § 9.3.1.6.4）
- **数据持久化**：推荐每个会话 / Walk-through 节点完成时落库（沿用 § 9.3.1.6.6）
- **隐私三层**：沿用 § 2.6.3（学员之间默认完全隔离 / 跨学员引用需同意 / 管理端例外）

#### 9.3.3.8 完成与跨场域衔接

##### 9.3.3.8.1 Activity 完成判定（milestone 概念）

| milestone 状态 | 含义 | 学员看到 |
|---------------|------|---------|
| **未完成** | 学员还没在"完整态首次"下走完 Walk-through —— 含两种情况：① 关联 Activity 未全部完成（部分态 / 空白态进入即使完整对话也不计 milestone）② 关联 Activity 已全完成但学员尚未首次进入 | recap Activity 抽屉显示"未完成"；下次"完整态首次"进入触发 milestone 完成 |
| **已完成 milestone** | 完整态首次进入触发首聊态完整 Walk-through + Neo 评估有实质性反思 + 应用计划已产出并入大厅待办；首聊报告锁定为首聊层永久保留 | recap Activity 抽屉标"已完成"；下次进入按行为四态切换（自由复习态 → 回访 / 复习）|
| **已完成但报告有更新** | 已完成 milestone 后报告增量更新 | 不退回未完成；Subbar 显示"📊 报告有更新"；下次进入触发回访态 |

完成判定的具体标准 —— Neo 自主判断"完整 Walk-through + 实质性反思 + 应用计划产出"三件事（不规定固定轮数门槛）。

##### 9.3.3.8.2 完成 Modal

学员首次完成 milestone 时弹完成 Modal：

```
┌─────────────────────────────┐
│                        [X]  │
│         ✅                    │
│   Course 小结完成！            │
│  你已完成「{Course 小结名}」   │
│  应用计划已入大厅待办（3 项）   │
│                              │
│  [回到大厅]                   │
│  [自由复习]                   │
│  [继续下一 Activity]            │
└─────────────────────────────┘
```

按钮行为沿用 lecture / practice 3 选项设计（详见 § 9.3.1.7.2 + § 9.3.2.7.2）。

> **"自由复习"统一概念**：lecture / practice / recap 完成 Modal 里的"自由复习"按钮 + recap "自由复习态"状态，三者指向同一概念——**Activity 流程结束之后，学员允许回到该场域针对 Activity 内容和已经发生过的事情和 Neo 进行探讨**。

回到大厅：默认进大厅首页（项目总览区已含待办清单），不为 recap 跳转打破默认体验。

##### 9.3.3.8.3 重进 = 行为四态切换

完成后再进入 → Neo 按行为态判断（详见 § 9.3.3.4.3）：

- 报告有更新（有实质变化）→ 回访态
- 报告无更新（无实质变化）→ 复习态

milestone 未完成前再进入：

- 仍处于空白 / 部分 / 完整态 → 对应预览态 / 首聊态行为
- 中途退出后重进 → 从断点续 walk-through（不从头）

##### 9.3.3.8.4 应用计划的后续追踪

应用计划入大厅待办后，在大厅待办流程内闭环（不在 recap 内回访）：

```
学员在大厅待办主动点掉（永远不限制总量）
   ↓
Neo 跟上来问情况（"做完了吗？怎么做的？"）
   ↓
学员回复
   ↓
Neo 评估后再决定是否允许点掉
```

剩余服务期内同样适用。

> **应用计划完成不反映到 recap 报告** —— 闭环在大厅待办区完成；不污染 recap 报告分层结构。

##### 9.3.3.8.5 跨场域衔接

> 全局跨场域机制详见 § 9.3.4 跨场域协同（Truth Source）；本节仅从 recap 侧视角列入口与出口。

| 衔接 | 数据流 |
|------|--------|
| **关联 Activity（lecture / practice）→ recap** | 报告数据流：高光库子集 / 6 维画像变化 / 学员互动行为日志 / practice 复盘待改进点 / Neo 跨场域感知 全部进入报告生成（详见 § 9.3.3.2 各块数据来源）|
| **recap → 大厅** | 完成状态 + 报告进入大厅"我的报告"主题（§ 9.2.2.3）；应用计划进入大厅待办清单（§ 9.2.2.1 (1) E）|
| **recap → 管理端** | team report 拉最新 recap 数据；数据共享但呈现完全分开（详见 § 10.3）|

> **与大厅的边界**：项目进度类信息由大厅 § 9.2 承担，recap 不承担也不重复（详见 § 9.3.3.1.2 + § 9.3.3.7.6）。

管理员账号登录 → 默认进入 Operation 看板。

#### 10.1.4 HR vs 项目运营 vs admin 协同

| 角色 | 主要行为模式 |
|------|------------|
| **admin** | 项目根账号——开营前配置全权 + 创建管理员账号 + 锁定服务周期 |
| **HR** | 不定期统观——看板巡视 + 阶段性 team report 生成 + 对外汇报 |
| **项目运营** | 日常执行——异常学员发现 / 学员名单变更 / 催学规则配置 / 报告生成执行 |

本期 3 类管理员**共用同一管理端**，权限差异详见 backlog（待 § 10 详细设计阶段）。

### 10.2 看板 Operation

#### 10.2.1 业务定义

管理员日常监控项目运行健康度的场域。三维度监控 + 热力图 + Ora 双交互。

#### 10.2.2 三维度监控

| 维度 | 内容 |
|------|------|
| **项目维度** | 项目整体健康度 / 进度 / 完成率 |
| **人员维度** | 学员个体进度 / 异常学员 / 活跃度分布 |
| **内容维度** | 各 Course 学习情况 / 卡点分布 / Quiz 通关率 |

#### 10.2.3 热力图

学员行为热力可视化——每个学员每日活跃度色块 / 每个 Course 学员通过率色块。

#### 10.2.4 Ora 双交互模式
| 模式 | 触发 | 行为 |
|------|------|------|
| **看板被动呈现** | 默认呈现 | 三维度数据 + 热力图 + Ora 自动生成的项目健康度解读 |
| **Ora 问答** | 管理员主动提问 | "为什么 W4 脱训突增？" / "张三这周参与度怎样？" → Ora 用证据回答 |

> **本期不做**：看板点击 → Ora 实时解释——这是未来形态（详见 § 3.3.4）

#### 10.2.5 异常学员处置

按 § 6.6.2：

- **平台层**：Ora 看板 / 问答 中发现异常学员（脱训 / 进度落后 / Quiz 频繁失败）
- **线下层**：项目运营**线下跟进**该学员（电话 / 微信 / 1on1 沟通）
- **平台不强制干预**——干预靠人 / 渠道完成

#### 10.2.6 颜色条 / 排序 / 详情页

- **颜色条标识健康度**（绿 / 黄 / 红）
- **异常自动排顶**（异常项目 / 异常学员 自动排在顶部）
- **详情页"建议动作"** —— Ora 给出建议动作（详见 § 3.3.5）

### 10.3 团队报告生成 Team Report-Gen

> 这是 § 10 的**核心模块**——管理端的主要交付能力，承担中期反馈 / 半程 review / 结项报告 / 剩余服务期内的新报告 全部生成场景（详见 § 6.7 + § 6.8）。

#### 10.3.1 业务定义

管理员产出对外可举证报告的场域——Ora 拉全程数据 + 多轮对话生成初稿 + 管理员 HITL 编辑 + 导出 PDF / Word。

#### 10.3.2 引擎核心 · 每管理员独立报告库
- **每个管理员**有**独立的报告库**——存放该管理员生成 / 编辑的全部 team report
- **里程碑触发时**给每个管理员一份**独立草稿副本**
- **本期不做协同分享 / 不做实时多人编辑** —— 需协同时由管理员**导出后线下同步**

> 这个设计简化了多人协同的复杂度——研发只需保证每个管理员的报告库独立，不需做实时协同编辑。

#### 10.3.3 prompt 模板由"关联动作 / 评估要点"驱动
- 管理员**不需要选模板，也不手写 prompt**
- 管理员在创建 team report（或里程碑配置）时**填"关联动作 / 评估要点"**（如"评估学员中期参与度 + 概念掌握度"）
- 系统据此**自动生成**对应的 prompt 模板

> 本期 v0.5.0 优先做 **"结项报告 prompt 模板"** —— 用作里程碑结营时的默认 prompt。其他模板（中期 / 半程 / 自定义）后续逐步加（详见 § 6.7.1）。

#### 10.3.4 多轮对话生成 + HITL 编辑

生成流程：

```
里程碑到期触发 / 管理员手动触发
   ↓
Ora 拉项目全程数据
   ↓
Ora 与管理员多轮对话（确认范围 / 学员维度 / 数据筛选 / 文本风格）
   ↓
Ora 生成初稿
   ↓
管理员 HITL 编辑（增删改 / 调整结构 / 改文案）
   ↓
导出 PDF / Word
```

#### 10.3.5 与里程碑的关联

详见 § 6.6.3 + § 6.7.1：

- 里程碑到期当天 → Ora 自动生成 team report 草稿（每管理员一份副本）
- 站内消息提前 3 天每日提醒管理员
- Ora 在草稿生成后即时推送通知
- 过期未处理 → 提醒保留在列表（不堆累 / 不重复推送 / 不升级渠道）

#### 10.3.6 模板含义澄清
按 § 6.7.1：

> 本节"模板"指的是 **prompt 约束的写作规则**（即给 Ora 的"写作 brief / 文章骨架 / 核心要素清单"），**不是固定文章模板或 HTML 模板**。
>
> Ora 基于模板的写作约束 + 项目实际数据 + 关联动作 / 评估要点 → 生成具体文本。

#### 10.3.7 本期不做

- ❌ **学员个人结营报告** —— 学员小结全部在阶段 5 学习期内通过 Course 个人小结 Activity 完成（详见 § 6.7）
- ❌ **多模板自由切换 UI** —— 本期仅"结项报告 prompt 模板"
- ❌ **协同分享 / 实时多人编辑** —— 见 § 10.3.2

### 10.5 消息 Message

#### 10.5.1 业务定义

管理员对学员定向沟通的场域——支持手动发送富文本消息 + 系统自动催学消息的统一查看与跟踪。

#### 10.5.2 GUI 与交互

| 子模块 | 功能 |
|-------|------|
| **手动定向发送** | 项目运营从学员名单中**手动选人** + 编辑富文本 + 发送 |
| **发送记录** | 手动 + 自动催学消息**统一查看**——含对象 / 时间 / 内容 / 投递日志 |

#### 10.5.3 与催学规则的关系

按 § 6.6.5：

- **全员催学** = 通过催学规则自动触发（系统层）
- **异常学员定向催** = 通过本模块**手动选人**发送（运营层）
- 两者的发送记录在本模块**统一呈现**

#### 10.5.4 投递日志

- 完整记录：谁收到 / 是否查看 / 是否点击链接
- 本期**不支持学员回应**（学员无法在催学消息上回复）

#### 10.5.5 本期不做

- ❌ **邮件 / IM / SMS 渠道**（仅站内）
- ❌ **消息已读率 dashboard**（投递日志后台可查，前端不展示）
