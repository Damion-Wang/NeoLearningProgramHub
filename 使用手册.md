## 常见任务怎么做

### 查阅最新 spec

```
路径：00-public/1-product/spec/design/
- learner/[已审阅]XX-name.md   学员端模块（7 个）
- management/[已审阅]XX-name.md 管理端模块（5 个）
- common/[已审阅]XX-name.md     通用规则（3 个）
- common/03-ai-brand.md         AI 品牌（新增）
- learner/07-discovery-library.md 发现库（新增）
- requirements-v0.4.0.md         总纲
```

约定：
- 文件名带 `[已审阅]` 前缀 = 已通过审阅
- 章节标题 `反向补充：XXX` = 来自 demo 实践的补充

### 修改 spec

直接编辑 `[已审阅]XX-name.md`，**不要删原内容**。新设计追加章节：

```markdown
## 反向补充：XXX

[新内容]
```

### 看历史决策（"为什么这么设计"）

按时间顺序：
1. `00-public/5-process/journal/` 工作日志（按日期）
2. `00-public/5-process/debates/` 设计辩论（按议题轮次）
3. `00-public/4-decisions/` ADR（架构决策记录，含上下文+方案对比+结论）

### 让 Claude 帮你做事

新对话开始时 Claude 会自动触发 `session-opener` 显示项目状态。

常见指令：
- "**做个 demo**" → Claude 启动原型构建工作流
- "**改 spec 里 XX**" → Claude 直接编辑对应文件
- "**5 角色辩论一下 XXX**" → 触发 multi-round-debate Skill
- "**完成了**" → Claude 触发 completion-gate 反问 4 件套
- "**推 git**" → Claude 触发 git-push-guard 三联检查

约定：
- 复杂任务先让 Claude 写规划 md，确认后再执行
- 大规模修改用并行 Agent
- UI 改动要求浏览器截图证据

## 项目内重要约定

### 命名约定

| 前缀/后缀 | 含义 |
|---------|------|
| `[已审阅]` | spec 已通过审阅 |
| `XX-name.md`（无前缀） | 待审阅或新增 |
| `00-`/`01-`/`02-`... | 一级目录用 `0X-` 前缀排序 |
| `DM_Temp/` | DM 持续在编工作区（与 00-public 区分） |
| `02-temp/` | 公共临时区 |
| `.[name]/` | 隐藏目录（.git/.claude） |

### 路径约定

- `00-public/` 是「正式交付」，进入这里 = 已审阅
- `DM_Temp/` 是「持续在编」，内容流转：`DM_Temp/ → 审阅 → 00-public/`
- `02-temp/` 是「短期暂存」，季度末归档到 `02-temp/archived-YYYY-Q*/`
- `01-ref/` 是「永久参考」，不删不改不归档

### 版本约定

- spec 主版本：`requirements-vX.Y.Z.md`（vX 主升级跨季度，vY 月度，vZ 修订）
- 原型版本：`demo v1.0 / v1.1 / ...`（在 PROGRESS.md 记录）
- 历史版本不删除，旧版仍可在原位置访问

### 品牌约定

- ✅ 用 **睿学**（中文）/ **AI TUTOR**（英文）
- ❌ 禁用「AI 老师」（已弃用）
- AI 角色：用户视角合并 Leo+Neo→**Neo**；Ora 保留为管理端独立身份

## 怎么贡献

### 新建文件

- 放对位置（参考目录速查表）
- 按命名约定（如新 spec 加 `XX-name.md`）
- 同目录加/更新 `README.md`（说明本目录用途）

### 审阅别人的 spec

- 通过审阅 → 文件名加 `[已审阅]` 前缀
- 有疑问 → 在 `02-temp/` 或对应文件追加 `审阅意见：XXX` 章节

### 归档已完成的工作

- DM 工作完成后 → 转入 `00-public/` 对应区
- 临时辩论/规划 md → 归档到 `00-public/5-process/journal/YYYY-MM-XXX/`
- 季度末 → `02-temp/` 整体归档为 `02-temp/archived-YYYY-Q*/`

## 协作规范（Claude Code）

### 触发 Skill 的关键词

| 你说什么 | Claude 触发什么 |
|---------|---------------|
| 新 session 第一句话 | session-opener（自动） |
| "完成了"/"搞定"/"done" | completion-gate（4 件套反问） |
| "推 git"/"commit"/"提交" | git-push-guard（三联检查） |
| "5 角色辩论"/"做个辩论"/方案分歧 | multi-round-debate |

### 跳过 Skill

- "**先跳过**"/"**emergency**" → 绕过本次强制检查
- 但 Claude 会在下次相同时机再次提醒

### 多人协作

- 不同人改不同区：DM 主要在 `DM_Temp/`，其他人不动
- 共同 review：在 `02-temp/` 写 review.md，@对应人
- 重要决策落到 `00-public/4-decisions/` ADR

## 常见问题 FAQ

### Q1：spec 和原型不一致以哪个为准？

按时间：晚的为准。原型构建过程涌现的设计**反向补充**到 spec（章节标 `反向补充：XXX`）。

### Q2：DM_Temp 和 00-public 怎么区分？

- DM_Temp = 在编草稿，DM 一人改
- 00-public = 已审阅交付，所有人共用

DM 工作完成后转入 00-public。

### Q3：00-public 已审阅文件可以再改吗？

可以。修改后保持 `[已审阅]` 前缀，新增内容用 `反向补充：XXX` 章节。

### Q4：原型怎么跑起来？

```bash
cd DM_Temp/prototype && python -m http.server 8765
# 浏览器打开 http://localhost:8765/build/01-login.html
```

### Q5：找不到某个文件？

试试这些：
1. `find . -name "*关键词*"`
2. 看 `00-public/` 各 README.md
3. 问 Claude："找一下关于 XXX 的文件"

### Q6：要存的临时文件放哪？

- 自己用：`DM_Temp/`
- 共享：`02-temp/`
- 不要放项目根目录！

### Q7：怎么清理项目？

让 Claude 触发 `git-push-guard`（推 git 前自动跑），或显式说"清理一下项目"。
