# AI 老师 -- 产品需求说明文档

**版本：** v2.0
**日期：** 2026-04-13
**来源：** 产品架构白板图 + R1/R2 辩论汇总 + 创始人回应 + ADR-001~006 + 豆包聊天记录 + AOM/Prosona/SIDE 技术分析
**范围：** 6月培训项目的产品规划（MVP）
**项目规模：** 30-40人学员 + 1-2 HR + 1-2运营，6个月周期 [创始人补充]

## 1. 产品定义

### 一句话定义 [已确认]

> AI 老师是一个可编排的 AI 原生培训项目交付引擎，通过自适应的 AI 动作序列（授课/辅导/测评/对练/萃取），将企业培训从"不可验证的课程完成"升级为"可举证的行为改变"。

### 产品愿景 [已确认]

- **解决什么问题：** 传统企业培训无法验证行为改变，学员完课率低、HR 无法向 CEO 举证培训效果、运营人力成本高
- **为谁解决：** 企业培训场景下的四类角色——学员（学习者）、HR（项目发起人和验收者）、运营（项目交付人）、内容制作者（KGP）
- **怎么解决：** 通过 AI 原生的教学引擎，以半结构化内容脚本驱动自适应教学、辅导、对练、测评全流程，统一数据采集并自动生成可举证的行为改变报告

## 2. 角色与端口

### 2.1 四个角色定义 [已确认]

| 角色 | 归属 | 职责 | 核心关注点 |
|------|------|------|-----------|
| **学员** | 客户企业 | 学习者，完成培训任务，在辅导中解决真实工作问题 | 学到有用的东西、花最少时间完成任务、获得可带走的工具和方法 |
| **HR** | 客户企业 | 项目发起人和验收者，关注成果和汇报 | 向 CEO 举证培训效果、了解学员学习过程、生产汇报素材 |
| **运营** | 供应商侧（我们团队派出） | 全程辅助项目交付，远期目标产品自运营 | 高效管理大规模学员、及时发现和处理异常、保障项目交付质量 |
| **内容制作（KGP）** | 供应商侧 | 课程内容生产（脚本/剧本/SCO/测评框架），不是软件端口用户 | 制作高质量的教学内容、定义自适应规则、设计对练和测评剧本 |

### 2.2 三个产品端口 [已确认] [ADR-001]

学员端、HR 端、运营端是**同一个 Web 应用**，通过权限控制不同角色看到的入口和功能。

| 端口 | 主要用户 | 布局 | 备注 |
|------|---------|------|------|
| **学员端** | 学员 | 左侧看板 + 右侧 Chat UI（辅导对话）；教学模块全屏切换 | HR 可切换为学员身份体验课程 |
| **HR 端** | HR | 待确认具体布局 | 含报告智能体交互式使用 |
| **运营端** | 运营人员 | 标准管理后台 + 右侧运营 Agent | 运营 Agent 与学员端辅导模块完全独立 |

### 2.3 各角色的使用场景 [已确认]

**学员：**
- 首次进入系统，了解培训项目和学习任务
- 在辅导模块中提出真实工作问题，获得针对性建议
- 进入教学模块完成授课、对练、测评任务
- 工作中实践后回来讨论效果
- 查看进度、带走工具卡片/表格
- 在消息中心接收运营下发的消息和催学提醒 [创始人补充]

**HR：**
- 学习期随时通过 Agent 了解过程发生了什么
- 生产汇报素材（给 CEO 的 PPT、月度报告等）
- 查看项目健康度看板
- 收到群体级预警摘要
- 结项期查看报告、生产文件包
- 切换为学员身份体验课程

**运营：**
- 开营前导入名单、配置参数、下发任务
- 学习期看进度、催学、监控异常、处理预警
- 下发临时任务（如"用一句话金句总结今天学到了什么"） [创始人补充]
- 使用报告智能体批量生成报告
- 结项期整理成果、输出文件包

## 3. 产品架构总览

### 3.1 学员端架构 [已确认]

辅导模块作为主入口（ADR-003），四种教学模块全屏切换，所有数据统一存入 Database。

**辅导模块（主界面/入口）：**
- 定位：模拟长期陪伴的 1v1 辅导教练
- 右侧：Chat UI 对话界面
- 左侧看板内容：学习进度、课程结构、个人笔记、可带走内容（学习卡片/工具表格/工具文本）、待办任务/作业、消息中心 [创始人补充]
- 可带走内容来源：部分 KGP 预制 + 部分 AI 根据学员情况实时生成

**四种教学模块（全屏切换，可随时返回辅导）：** [创始人补充]
- 授课模块（课程演绎模块）
- 对练模块（Roleplay）
- 测评模块（访谈/量表）
- 个人报告模块（报告 + 解析） [创始人补充]

四种模块之间有三重隐性关联：
- **数据关联：** 授课中的 Quiz 结果影响对练的场景选择，测评结果改变授课路径
- **内容关联：** 四种模块围绕同一知识点/主题运转，共享内容上下文
- **序列关联：** 有固定或推荐的先后顺序（如先授课再对练最后测评，Course 结束后触发个人报告）

### 3.2 HR 端架构 [已确认]

- 报告智能体（交互式使用）
- 学习进度查看
- 项目健康度看板
- Agent 交互（了解过程发生了什么、生产汇报素材）
- 群体级预警摘要接收
- 切换为学员身份体验课程

### 3.3 运营端架构 [已确认]

- 标准管理后台 + 右侧运营 Agent
- 预警看板（5类主动预警）
- 运营工具（名单导入、参数配置、任务下发、催学管理）
- 运营 Agent（查询细节、辅助决策，与学员端辅导模块完全独立）
- 报告生成（批量模式）
- 临时任务下发（单向群聊模式：运营看所有人提交，学员只看运营下发） [创始人补充]

## 4. 数据模型

### 4.1 内容层级结构 [已确认] [数据验证]

AOM（Activity Object Model）采用五层嵌套结构。需求文档中的术语与 AOM 中的对应关系如下：

```
Project → Module(=Course) → Activity → SCO(scoFlow) → Segment
                                          ↑ 每种内容形式 = 独立 SCO
                                          ↑ 版本半预制，自适应切换
                                          ↑ 无 segmentList 的 SCO 使用 sco 对象(quiz等)
```

**术语对齐说明：** [数据验证]
- **Project：** 一个培训项目（如"横向协作：销售内测"），AOM 顶层结构
- **Module（需求文档中称 Course）：** 项目下的课程单元（如"横向协作"）。AOM 中字段名为 `modules`，包含 `moduleName`、`moduleDescription`、`activities` [数据验证]
- **Activity：** 课程下的教学活动（如"课前互动""PIN法三步走"），AOM 中包含 `activityType`（lecture/drill/assessment/report）、`directScoContent`、外部资源引用等
- **SCO（Shareable Content Object）：** 最小可交互学习单元，对应 AOM 中 `directScoContent.scoFlow[]` 数组的每个元素，每个 SCO 支持多版本，通过 Tag 标记版本属性
- **Segment：** SCO 下的内容片段（PPT 页面级别），对应 `segmentList[]` 中的每个元素，是最小内容粒度 [数据验证]

### 4.2 授课 AOM 与对练 AOM 的结构差异 [数据验证]

| 维度 | 授课 AOM | 对练 AOM |
|------|----------|----------|
| **activityType** | `lecture` | `roleplay` |
| **directScoContent** | 完整的 scoFlow 结构（内联 PPT/Quiz/segmentList/interactionPoint） | `null`（不走段落驱动逻辑） |
| **外部资源引用** | 无（内容内联） | `resourceId` + `artifactId` + `competencyArtifactId` 三件套 |
| **competencyGoals** | 空数组（能力评估非授课主要关注点） | 完整的能力目标（含 keyBehaviors 和 rubric） |
| **interactionPoint** | 每个 SCO 都有（blueprint/rule/description/aiTutorDialogue/interactionType） | 无（通过剧本替代） |
| **内容驱动方式** | 按段落播放 + 在交互点执行规则 | 在行为框架内自由对话（剧本+Agent 组合驱动） |

### 4.3 AOM 关键字段说明 [数据验证]

#### 4.3.1 interactionPoint（每个授课 SCO 都有）

| 字段 | 类型 | 说明 |
|------|------|------|
| `blueprint` | string | 教学设计意图（给 AI 读的"教师手册"，包含教育学原理和设计决策的理由） |
| `rule` | string | 交互触发/结束规则（如"播放完毕"或复杂的通关条件描述） |
| `description` | string | 内容摘要（面向系统的功能描述） |
| `aiTutorDialogue` | array | AI 导师预设对话（当前多为空数组，对话行为由 AI 动态生成） |
| `interactionType` | string | 交互类型标识：`NARRATION`（自动播放）或 `CHAT`（多轮对话） |

#### 4.3.2 自适应相关字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `separateTag` | string | 分流标签（如"防御拖延型""利益博弈型""情绪对抗型"），定义该内容面向哪种学员类型 |
| `groupKey` | string | 版本槽位标识，同一 groupKey 的多个 segment/SCO 是同一位置的不同自适应版本 |
| `tutorMode` | number | AI 导师模式开关（0=标准，1=自适应），标记该 SCO 需要根据学员标签进行内容选择 |
| `passConditions` | object | 通关条件（`description` + `rules[]`），纯自然语言定义，由 LLM 解释执行 |

#### 4.3.3 对练 AOM 的外部资源引用三件套

| 字段 | 说明 |
|------|------|
| `resourceId` | 指向资源服务中的剧本资源（Markdown/JSON 剧本文件） |
| `artifactId` | 指向 artifact 服务中的剧本产物（结构化 JSON 版剧本） |
| `competencyArtifactId` | 指向能力评估标准的 artifact（含 rubric 的详细评估矩阵） |

### 4.4 SCO 粒度规则 [已确认]

每种内容形式 = 一个独立的 SCO：

| 内容形式 | SCO 示例 |
|---------|---------|
| PPT 讲解 | 一段知识点讲解（含 PPT 素材） |
| Quiz | 一组测验题（问答/选择/判断/连线） |
| 案例视频 | 一段案例演示视频 |
| 小测评 | 一个自适应或代入感测评 |
| 小解答 | 一段基于学员问题的解答 |
| 对练场景 | 一个 Roleplay 对话场景 |
| 访谈流程 | 一个 BEI 或其他访谈流程 |
| GUI 量表 | 一个表单式量表（如 MBTI 类） |

每个 SCO 支持多版本（半预制），通过自适应规则切换版本。版本差异可以是难度差异，也可以是角度差异（如同一知识点的不同案例/不同讲法）。[创始人调整]

### 4.5 Segment 层字段说明 [数据验证]

| 字段 | 类型 | 说明 |
|------|------|------|
| `segmentId` | string/null | 片段ID（如"1.1"，"5.1"） |
| `segmentType` | string | 片段类型：`ppt`（PPT 幻灯片）或 `asvideo`（伪视频/3D 数字人） |
| `name` | string | 片段名称 |
| `script` | string | 讲解脚本/旁白文案 |
| `description` | string | 片段描述 |
| `separateTag` | string | 分流标签（用于自适应内容版本选择） |
| `groupKey` | string | 分组键（同一 groupKey 的多个 segment 为同一内容的不同版本） |
| `content` | object | 内容数据（ppt 类型包含 `pptUrl`；asvideo 类型包含场景列表、帧列表、音频URL、字幕文本等） |

## 5. 核心架构原则

### ADR-001: 三端合一 Web 应用 [已确认]

学员端、HR 端、运营端是同一个 Web 应用，通过权限控制不同角色看到的入口和功能。

- 理由：HR 需切换学员身份体验；减少开发维护成本；统一数据层
- 后果：需要设计好权限和视图切换逻辑

### ADR-002: 预制菜模式 [已确认]

第一版是"预制菜"——Project/Course/Activity/SCO 全部由供应商（KGP）预制交付，管理员不配置课程结构。

- 理由：6月时间紧；内容质量由专业团队把控；HR 不具备课程设计能力
- 未来方向：迭代为 Agent 平台模式，让企业可自助

### ADR-003: 辅导模块作为学员端主入口 [已确认]

辅导模块（1v1 AI 教练对话）作为学员端主界面。经创始人市场验证：学员对正式培训抵触越来越多，陪伴式辅导本身是学员真实需要的。

- 布局：左侧看板 + 右侧 Chat UI
- 教学模块全屏切换进入，可随时返回辅导
- 核心原则：**AI 主动但不阻塞** -- 辅导对话 always available，不是 mandatory path

### ADR-004: 半结构化内容驱动架构 [已确认] [创始人调整]

采用"半结构化内容驱动"架构：

- **平台**提供执行引擎（播放器、对话引擎、数据采集、报告生成等）
- **内容**（KGP 制作的脚本/剧本）控制具体行为：
  - 授课脚本：讲解节奏、小测评时机、自适应分支规则
  - 对练剧本：角色设定、情绪表达范围、行为边界
  - 访谈剧本：追问策略、目标信息、评估标准
  - 自适应规则：SCO 版本切换条件（不固定权重，由内容设计者根据教学目标定义）
- **平台级行为**（例外）：催学、预警、报告生成等运营功能由平台控制
- 核心价值：**同一引擎 + 不同脚本 = 不同培训产品**

**来自 AOM/代码分析的直接证据：** [数据验证]
- `passConditions`（通关条件）以自然语言写在数据中，由 LLM 解释执行
- `solution`（AI 行为准则）以自然语言写在数据中，直接指挥 AI 的对话行为
- `blueprint`（教学设计意图）以自然语言写在数据中，包含教育学原理
- `separateTag` + `groupKey` 实现自适应分流，平台不需要预知分流类型名称
- 对练剧本的 `behavioral_patterns` 定义 AI 的递进式反应链，替代授课中的 blueprint/rule/solution 三元组

### ADR-005: 记忆工程三级管道 [已决定] [创始人补充]

本期实现完整的三级记忆管道：

1. **课堂内记忆**：授课/对练/测评中采集的数据（学员输入的真实场景、行为表现、测评结果）实时汇总到统一 Database
2. **大厅记忆**：辅导智能体可读取课堂数据，引用学员在教学模块中的表现来给出个性化辅导
3. **管理端记忆**：报告智能体读取全部数据（课堂+辅导），生成 HR 可用的报告

- 理由：内测发现"情感错付"问题——学员分享了真实问题但后续没有回应；辅导模块如果不能引用课堂数据，就只是通用问答机器人，失去"陪伴感"；HR 端报告需要全链路数据才能生成有说服力的行为改变证据
- 后果（正面）：学员感受到"AI 记得我说过什么"，辅导质量大幅提升，报告数据完整
- 后果（负面）：技术复杂度增加，需要设计数据结构和访问权限；LLM 上下文管理挑战
- 详见：`decisions/005-memory-engineering.md`

### ADR-006: 辅导知识库用渐进式披露而非 RAG [已决定] [创始人补充]

辅导知识库采用"渐进式披露"的文件层级检索方式：

- 知识库按主题分层级目录
- 每个层级有 README 文件，描述该层包含的内容索引
- AI 像读文件系统一样逐层匹配，从顶层 README 开始，逐步深入找到相关内容
- 不使用 Vector Augmented Generation（传统 RAG）

- 理由：创始人明确要求"整个里面不要用到传统RAG"；渐进式披露更符合半结构化内容驱动的架构原则；检索过程可追溯、可调试，不是黑盒；与 KGP 生产的内容包结构天然兼容
- 后果（正面）：检索过程透明，结果可控，与内容体系对齐
- 后果（负面）：对知识库目录结构和 README 质量要求高；大规模知识库下的检索效率需要验证
- 详见：`decisions/006-no-rag-progressive-disclosure.md`

### 辅导智能体角色优先级 [创始人调整]

1. **首先是辅导** -- 陪伴式回答真实问题
2. **其次才是督学** -- 推荐任务、催促进度
3. **还要教会用户** -- 引导学员学会使用工具、知道能问什么问题

## 7. 学员端详细行为规格

### 7.1 辅导模块行为

#### 7.1.1 主动提问真实工作问题

- **触发方式：** 主动（学员随时在 Chat UI 中输入问题）
- **User Behaviour：** 学员在辅导对话中提出与培训主题相关的真实工作问题，例如"我下周要给团队做一次绩效反馈谈话，但有个下属总是很防御，怎么办？"
- **Agent/System Behaviour：**
  - AI 基于课程内容和学员提供的上下文给出针对性建议，不推回"通用理论" [已确认]
  - AI 用学员描述的情境生成具体建议（例如"裁员后团队焦虑的情况下，正向反馈的关键不是表扬业绩，而是认可留下来的人的情感付出。比如你可以这样说..."）[已确认]
  - 如果学员问到 AI 确实无法回答的企业内部信息（如"我们公司的绩效考核周期是什么"），AI 坦诚告知并提供管理视角参考："这个信息我目前没有，建议你确认一下公司的具体规定。不过从管理实践角度，我可以帮你思考[相关话题]..." [已确认]
  - 后台异步标注：AI 识别学员主动提问涉及的能力维度，记录求助行为和问题摘要 [已确认]
  - 如果问题与某个课程/活动高度相关，AI 可自然推荐"关于这个话题，[课程名]有一套实用的方法框架，要不要看看？"
  - 辅导智能体的知识检索采用渐进式披露方式（ADR-006），而非传统 RAG [创始人补充]
- **期望感受：** "这个 AI 教练真懂我的问题，给的建议很实用"
- **本期范围：** 是（6月 MVP）

#### 7.1.2 接受 AI 推荐进入教学模块

- **触发方式：** 被动（AI 根据对话内容或学习进度推荐）
- **User Behaviour：** 学员在辅导对话中看到 AI 推荐的课程/活动，点击推荐卡片进入教学模块
- **Agent/System Behaviour：**
  - AI 根据对话上下文或学习进度，自然推荐相关的授课/对练/测评活动
  - 推荐以可点击的任务卡片形式呈现（标题 + 预计时长 + 进度状态）
  - 学员点击后 0.3 秒过渡动画切换至教学模块全屏界面
  - AI 不强制要求学员必须进入，推荐是建议不是命令
- **期望感受：** "AI 推荐的很合理，正好是我需要的"
- **本期范围：** 是

#### 7.1.3 自主选择课程进入（通过左侧看板或课程目录）

- **触发方式：** 主动（学员自行浏览和选择）
- **User Behaviour：** 学员通过左侧看板的课程结构导航，或通过教学模块内隐藏的课程目录，自主选择要学习的内容并进入
- **Agent/System Behaviour：**
  - 左侧看板展示完整的项目结构（Course → Activity → SCO），标注进度状态
  - 学员点击任意活动直接进入对应教学模块，无需经过辅导对话 [已确认]
  - 系统记录学员的自主选择行为，供后台分析
- **期望感受：** "我可以自己决定学什么，不需要每次都跟 AI 聊"
- **本期范围：** 是

#### 7.1.4 实践任务反馈（工作中试了方法后回来讨论）

- **触发方式：** 主动（学员自发回来分享实践经验）
- **User Behaviour：** 学员在实际工作中尝试了课程中学到的方法后，回到辅导模块分享体验，讨论效果，寻求改进建议。例如"我上次按你说的方法跟小王谈了，效果不太好，他还是很防御"
- **Agent/System Behaviour：**
  - AI 调取之前的对话记忆（通过记忆工程三级管道 ADR-005），理解学员说的是哪次建议和哪个场景 [创始人补充]
  - AI 根据学员描述的实践结果，提供进一步的分析和改进建议
  - 后台标注：记录学员的实践行为和反馈，作为行为改变的证据
  - 如有相关课程内容可进一步学习，AI 自然推荐
- **期望感受：** "AI 记得我之前的问题，能帮我复盘和改进"
- **本期范围：** 是

#### 7.1.5 查看进度/课程结构/待办/工具卡片

- **触发方式：** 主动（学员浏览左侧看板）
- **User Behaviour：** 学员在左侧看板中查看个人学习进度、项目课程结构、待办任务/作业、以及已获得的可带走内容（学习卡片、工具表格、工具文本）
- **Agent/System Behaviour：**
  - 看板实时更新学习进度数据
  - 课程结构以树形展示（Project → Course → Activity），标注完成状态
  - 可带走内容区域展示已解锁的工具卡片和表格（来源：KGP 预制 + AI 实时生成）
  - 待办区域展示未完成的任务和截止时间
- **期望感受：** "一目了然知道学了什么、还差什么、拿到了什么"
- **本期范围：** 是

#### 7.1.6 消息中心 [创始人补充]

- **触发方式：** 被动（运营下发消息时自动出现在左侧看板的消息中心）
- **User Behaviour：** 学员在左侧看板的消息中心中查看运营下发的消息和催学提醒
- **Agent/System Behaviour：**
  - 消息中心展示运营下发的临时任务（如"用一句话金句总结今天学到了什么"）
  - 展示催学提醒消息
  - 学员可直接在消息中心内提交临时任务的回复
  - **单向群聊模式**：运营可以看到所有人提交的内容，学员只看到运营下发的消息（看不到其他学员的提交） [创始人补充]
- **期望感受：** "知道运营关注我的学习，有参与感"
- **本期范围：** 是

#### 7.1.7 首次进入引导

- **触发方式：** 被动（系统自动触发，仅首次进入）
- **User Behaviour：** 学员首次进入系统后，看到简短的引导层和 AI 的欢迎消息，在 10 秒内可以开始第一个学习任务或自由互动
- **Agent/System Behaviour：** [已确认]
  - 2-3 秒 overlay 引导层：介绍项目名称、AI 辅导教练身份、界面布局说明
  - overlay 消失后，AI 第一条消息（简短，不超过 30 字）："[姓名]你好，你今天有 [N] 个任务待完成。"
  - 对话区域下方直接出现可点击的任务卡片（标题 + 预计时长 + 进度状态）
  - 学员可选择：(A) 直接点卡片进入教学模块；(B) 在对话中打字互动；(C) 在左侧看板自行浏览
  - **核心原则：AI 主动但不阻塞** [创始人调整]
  - 辅导智能体在首次交互中还要引导学员学会使用工具、知道能问什么问题（角色优先级：辅导 > 督学 > 工具引导）[创始人调整]
- **期望感受：** "很清楚这是什么、能干什么，我可以马上开始"
- **本期范围：** 是

#### 7.1.8 再次进入

- **触发方式：** 被动（系统自动识别非首次进入）
- **User Behaviour：** 学员再次打开系统，看到 AI 的欢迎回来消息和当前任务
- **Agent/System Behaviour：**
  - AI 消息："欢迎回来。你上次完成了[X]，今天的任务是[Y]。" + 任务卡片 [已确认]
  - 不再重复自我介绍
  - 如果上次在教学模块中途退出，提供恢复选择（详见 7.2 断点续做规则）
- **期望感受：** "AI 记得我上次学到哪了"
- **本期范围：** 是

#### 7.1.9 所有任务完成后

- **触发方式：** 被动（系统检测到所有任务已完成）
- **User Behaviour：** 学员完成所有任务后打开系统
- **Agent/System Behaviour：**
  - AI 消息："你目前的任务都完成了。有什么想聊的吗？或者可以看看你的学习笔记。" [已确认]
  - 左侧看板高亮笔记和可带走内容
  - 辅导对话仍然可用，学员可随时提问
- **期望感受：** "任务完成了，但 AI 教练还在，随时可以聊"
- **本期范围：** 是

### 7.2 授课模块行为

#### 7.2.1 听 PPT 讲解

- **触发方式：** 脚本驱动（授课脚本控制讲解顺序）
- **User Behaviour：** 学员在授课界面中观看 AI 配合 PPT 进行的知识点讲解，可以翻页、暂停
- **Agent/System Behaviour：**
  - AI 老师按授课脚本执行 SCO 序列，讲解带有 PPT 的知识点
  - SCO 内容为半预制（KGP 预制文本 + AI 个性化润色/补充）
  - 支持多版本 SCO，AI 根据自适应规则选择合适版本（ADR-004）
  - 后台异步标注学员在此环节的能力维度信号
- **期望感受：** "讲得清晰易懂，感觉是为我定制的"
- **本期范围：** 是

#### 7.2.2 看案例视频

- **触发方式：** 脚本驱动（授课脚本控制播放时机）
- **User Behaviour：** 学员在授课过程中观看案例演示视频，支持播放/暂停/进度拖动
- **Agent/System Behaviour：**
  - 系统在授课脚本指定的时机播放预制的案例视频 SCO
  - 视频播放完毕后，AI 可引导学员讨论案例要点或进入下一个 SCO
  - 两种视觉形态并存：数字人小窗+PPT 和 全屏伪视频（3D 数字人全身画面） [技术现状]
- **期望感受：** "案例很真实，跟我的工作场景很像"
- **本期范围：** 是

#### 7.2.3 做小测评（自适应/代入感）

- **触发方式：** 脚本驱动（授课脚本控制测评时机）
- **User Behaviour：** 学员在授课过程中完成简单的小测评，测评基于当前课程内容，用于自适应判断或增加代入感
- **Agent/System Behaviour：**
  - 小测评 SCO 嵌入在授课流程中，由授课脚本控制出现时机
  - 测评结果用于两个目的：(1) 自适应引擎判断后续 SCO 版本选择；(2) 增加学员代入感（如"想象你是这个场景中的管理者..."）
  - 测评结果自动映射到能力维度
- **期望感受：** "不是考试，是帮我理解自己的情况"
- **本期范围：** 是

#### 7.2.4 做 Quiz（问答/选择/判断/连线）

- **触发方式：** 脚本驱动（授课脚本控制 Quiz 时机和类型）
- **User Behaviour：** 学员在授课过程中完成各种类型的 Quiz 小测验——问答题、选择题、判断题、连线题
- **Agent/System Behaviour：**
  - Quiz 类型多样，目的各异：引发思考、带入个人场景、检验阶段性理解、场景化迁移
  - 本期支持 4 种交互类型：问答题、选择题、判断题、连线题 [已确认]
  - Quiz 结果自动映射到能力维度，作为自适应 L2 的判断信号之一
  - AI 对 Quiz 结果给出即时反馈（正确/部分正确/可以改进的地方）
  - 技术实现：QuizTaskHandler 采用苏格拉底式启发（最多5轮），支持展示/不展示答案两种模式，完成后异步生成高光分析、探索分析、个报碎片等 [技术现状]
- **期望感受：** "不是无聊的考试，每个题都让我想到了自己的情况"
- **本期范围：** 是

#### 7.2.5 插嘴提问 + 探讨

- **触发方式：** 主动（学员在授课过程中随时打断提问）
- **User Behaviour：** 学员在授课过程中对某个知识点有疑问或想法，随时在对话中提问或发表评论
- **Agent/System Behaviour：** [已确认]
  - **与当前内容相关的问题：** AI 解答学员疑问，但不剧透后续内容，不偏离主题
  - **与当前内容不相关的问题：** AI 简要回应后引导回主题，例如"这个问题很好，我们在[后续课程]会详细讨论。先继续当前的话题——"
  - AI 解答完毕后自然衔接回授课流程
  - 后台记录学员的提问，作为辅导模块的上下文记忆
  - 注：架构图中"Activity答疑"即指授课中的插嘴提问功能，不是独立模块 [创始人补充]
- **期望感受：** "随时可以打断问问题，AI 会认真回答但不跑题"
- **本期范围：** 是

#### 7.2.6 主动回顾（要求重讲）

- **触发方式：** 主动（学员主动要求重新讲解某个内容）
- **User Behaviour：** 学员对某个知识点没听懂或想再听一遍，主动要求 AI 重新讲解
- **Agent/System Behaviour：** [已确认]
  - AI 用不同的方式/角度重新讲解该知识点（可能切换到同一 SCO 的另一版本）
  - 补充讲解完毕后，AI 自然拉回授课主线
  - 本期由学员主动触发，AI 不主动判断是否需要重讲
- **期望感受：** "没听懂可以重来，AI 还会换个方式讲"
- **本期范围：** 是（学员主动触发）

#### 7.2.7 获得工具卡片/表格

- **触发方式：** 脚本驱动 + 主动（授课脚本预设发放时机，或学员主动索取）
- **User Behaviour：** 学员在授课过程中获得实用的工具卡片或表格模板，可收藏到左侧看板的"可带走内容"区域
- **Agent/System Behaviour：**
  - 工具卡片/表格来源：KGP 预制 + AI 根据学员情况实时生成 [已确认]
  - 预制内容在授课脚本指定的时机自动发放
  - AI 生成内容基于学员的真实问题和对话上下文
  - 内容自动保存到左侧看板"可带走内容"区域
- **期望感受：** "学完就能拿到工具，回去直接用"
- **本期范围：** 是

#### 7.2.8 小解答（基于真实问题和课程记忆）

- **触发方式：** 脚本驱动（授课脚本在适当时机插入）

### 7.3 对练模块行为 [技术现状]

#### 7.3.0 对练模块整体结构 [数据验证] [技术现状]

对练 Activity 内部采用四阶段结构，每个阶段可理解为独立的 SCO：

1. **导学（Guide）**：情景导入——展示角色信息、场景背景、对练目标、所需技能/知识点
2. **演绎（Deduction）**：Roleplay 对话——AI 扮演角色与学员对话练习
3. **复盘（Review）**：苏格拉底反思——退出角色，以辅导教练身份引导回顾
4. **报告（Report）**：对练报告——生成多维度评分报告（可选，`excludeReport: 1` 时跳过） [数据验证]

**技术实现：导演-演员分离模式（Director → Actor → Narrator）** [技术现状]

演绎阶段采用多角色协作架构：
- **Director（导演/主 LLM）**：解读剧本、编排每轮对话的场景转换/情绪变化/行动指令
- **Actor（演员/独立 LLM 调用）**：按照导演指令和剧本人设进行角色表演
- **Narrator（旁白/独立 LLM 调用）**：输出开场白和沉浸式场景描述
- **Coach（教练）**：在用户困难时给予引导（已实现但当前被禁用）

角色间通过消息 Tag 权限系统实现信息隔离（用户看不到导演的编排消息，演员看不到旁白的消息等），确保各角色不会"出戏"。

**对练剧本是深度结构化的：** [数据验证]
- 角色心理画像（70+ 字段）：显性目标/隐性目标、激怒点/软化点、底线、可协商项、真实需求等
- 递进式行为反应规则（`behavioral_patterns`）：如"一上来先冰山态度" → "用户施压则翻旧账" → "用户用拉则情绪由冷硬转委屈"
- 防剧透指令（`hidden_situation`）：限制 AI 不能主动透露的方案
- 4 阶段能力触发图谱（`skill_trigger_map`）：每阶段含阶段目标、决策点列表、所需技能/知识点
- L1-L5 五级评估标准（`skill_ladders`）：AOM 中 rubric 定义 L1/L3/L5 三级，剧本扩展为完整五级

**当前只实现 Roleplay 一种玩法**——角色互换/费曼/辩论/多人均未实现 [技术现状]

#### 7.3.1 导学（情景导入）

- **触发方式：** 脚本驱动（进入对练 Activity 后自动触发）
- **User Behaviour：** 学员进入对练模块后，首先看到场景介绍、角色分配和对练目标
- **Agent/System Behaviour：** [技术现状]
  - 导学分为三步依次展示：
    1. **背景介绍（bg-intro）**：展示用户角色+演员角色信息、业务场景背景、数字人形象
    2. **目标介绍（goal-intro）**：展示对练主目标和分项目标
    3. **技能介绍（skill-intro）**：展示所需核心技能和知识点
  - 每步以卡片形式发送到前端，AI 用简洁语言向学员总结关键信息
  - 通过 `shown_stages` 状态追踪已展示阶段，确保依次完成
- **期望感受：** "清楚知道接下来要做什么、扮演什么角色"
- **本期范围：** 是

#### 7.3.2 Roleplay 对话（演绎）

- **触发方式：** 脚本驱动（对练剧本控制场景设定）+ 学员选择进入
- **User Behaviour：** 学员进入演绎阶段后，与 AI 模拟的角色进行对话练习。AI 扮演特定角色（如难搞的下属、挑剔的客户等），学员练习应对
- **Agent/System Behaviour：**
  - **玩法类型：** 真实场景演练、角色互换（换位思考）、费曼学习法（AI 扮学生）、辩论 [已确认]
  - 本期仅支持单人对练、仅支持 Roleplay 一种玩法 [已确认] [技术现状]
  - AI 角色的设定、情绪表达范围、行为边界完全由对练剧本控制（ADR-004）[创始人调整]
  - AI 情绪初始状态与场景设定一致 [已确认]
  - AI 情绪可被学员行为影响：有效沟通 → 情绪缓和；忽视/粗暴 → 情绪升级一档 [已确认]
  - 情绪升级天花板：无论学员表现多差，不超过"中度" [已确认]
  - 每次对练预绑定 1-2 个能力维度；AI 实时标注结构化行为观察点 [已确认]
  - 对练过程中 AI 保持角色一致性，不中途退出角色
  - **演绎流程：** [技术现状]
    1. Narrator 生成沉浸式场景描述（旁白开场）
    2. Actor 输出角色开场白（附带数字人 TTS 配置）
    3. 每轮对话：Director 分析用户发言 → 生成导演指令（场景转换/情绪变化/行动指令） → Actor 根据指令表演 → 输出进度百分比
    4. 结束时：调用目标达成评估（0/1 二值判断）
  - **进度追踪：** Director 在每轮对话后评估进度百分比，实时发送到前端 [技术现状]
- **期望感受：** "像真的在跟人沟通，有压力但不恐惧"
- **本期范围：** 是

#### 7.3.3 苏格拉底反思（复盘）

- **触发方式：** 脚本驱动（对练剧本控制，在演绎结束后自动触发）
- **User Behaviour：** 演绎结束后，学员接受 AI 的反思引导，回顾自己在对练中的表现

- **期望感受：** "AI 的反馈很有价值，帮我看到了自己没注意到的地方"
- **本期范围：** 是

#### 7.3.4 对练报告（可选） [技术现状]

- **触发方式：** 脚本驱动（演绎+复盘结束后自动触发，可通过 `excludeReport: 1` 跳过）

### 7.4 测评模块行为

#### 7.4.0 测评模块整体结构 [创始人补充]

测评模块采用两段结构：

1. **测（收集）**：通过对话式访谈或 GUI 量表收集学员信息
2. **评（解读，可选）**：对话式解读测评结果，学员可和 AI 讨论

当前 Prosona 中无独立的测评 Handler，测评能力通过 Quiz 评分机制 + 对练目标评估拼凑实现。产品规划中的 BEI、量表、学员画像等需要新开发测评 Handler。 [技术现状]

#### 7.4.1 对话式访谈（BEI 等）

- **触发方式：** 脚本驱动（访谈剧本控制追问策略和流程）
- **User Behaviour：** 学员在测评模块中与 AI 进行对话式访谈，AI 引导学员讲述真实的工作案例和经历
- **Agent/System Behaviour：**
  - 测评类型和策略完全由内容剧本控制（ADR-004）[创始人调整]
  - 可能的测评类型：BEI（行为事件访谈）、学员画像、学员困惑收集、学员高光经验萃取 [创始人调整]
  - BEI 时机由内容设计决定，不固定 [已确认]
  - **BEI 追问策略（5步流程）：** [已确认]
    1. 开放式提问引出案例
    2. STAR 完整性检查（后台自动判断 Situation/Task/Action/Result 是否齐全）
    3. 针对性追问缺失元素（自然语气，不像审讯）
       - 特别关注：学员说"我们"时追问"你个人的作用"；Action 模糊时追问具体动作
    4. 回避行为识别：连续 2 次模糊/转移话题 → 不再纠缠，切换到新案例
    5. 同一案例最多追问 3 轮；不完整的案例记录完整度评分（X/4），案例不完整本身就是评估信息
  - 追问路径围绕能力维度展开，学员 STAR 案例结构化存储 [已确认]
- **期望感受：** "像跟一个专业的教练对话，引导我理清思路，而不是被审问"
- **本期范围：** 是

#### 7.4.2 GUI 量表

- **触发方式：** 脚本驱动（测评剧本控制量表类型和时机）
- **User Behaviour：** 学员在测评模块中完成表单式的量表测评（类似 MBTI），通过选择题或滑块完成评估
- **Agent/System Behaviour：**
  - 量表类型和内容由测评剧本控制（ADR-004）
  - GUI 表单式交互（非对话式），提供标准化的选项供学员选择
  - 结果自动映射到能力维度
  - 完成后可展示即时结果摘要（取决于测评剧本设计）
- **期望感受：** "简单直接，快速完成"
- **本期范围：** 是

#### 7.4.3 测评解读（可选） [创始人补充]

- **触发方式：** 脚本驱动（测评剧本控制是否启用解读环节）
- **User Behaviour：** 学员完成测评后，进入解读对话，可以和 AI 讨论测评结果

### 7.5 个人报告模块行为 [创始人补充] [技术现状]

个人报告是第四种学员端教学模块，和授课/对练/测评平级。暂定每个 Course 结束后固定触发，未来可从辅导模块随时进入。

#### 7.5.1 模块整体结构

个人报告模块采用两段结构：

1. **报告生成（渐进式加载）**：AI 汇聚全部学习数据，并行生成多维度报告
2. **报告解析（对话式解读）**：学员可以和 AI 讨论报告内容（如"你为什么给我这么低分？"）

#### 7.5.2 报告生成

- **触发方式：** 脚本驱动（Course 结束后自动触发）
- **User Behaviour：** 学员完成一个 Course 的全部活动后，看到报告生成页面，等待报告加载完成
- **Agent/System Behaviour：** [技术现状]
  - **渐进式加载体验（情感化文案设计）**：
    - 加载过程中分阶段显示不同文案："正在从对话中提炼你的高光时刻" → "正在为你构建专属的能力地图" → "为你准备一份专属的成长总结，这份特别的礼物，马上就好"
    - 加载时间较长（30s-180s），通过情感化文案缓解等待焦虑
  - **报告生成引擎（8 大模块并行）**：[技术现状]
    - 独立 LangGraph 子图，分五层执行：
      - 第一层（并行 6 节点）：基础信息、高光时刻、能力成长、认知转化、学习表现、技能提升
      - 第二层：成长表现（汇总特质/转变/优势）
      - 第三层（并行 2 节点）：综合评估、行动建议
      - 第四层：行动摘要
      - 第五层：聚合结果并持久化
    - 数据来自学习过程中实时生成的异步碎片（trait 学习特质碎片、cognitive 认知转化碎片、competency 能力评估碎片）
    - 碎片在 SCO 对话完成后、Quiz 答题闭环后、对练活动完成后异步生成
  - **报告累积下发**：分三段逐步下发到前端
    - Part 1：学习表现 + 认知转化 + 技能提升 + 能力成长
    - Part 2：Part 1 + 综合评估 + 成长表现 + 高光时刻
    - Part 3：Part 1+2 + 行动建议
  - **三步写入机制**：创建 Artifact → 关联用户项目 → 结构化保存 [技术现状]
- **期望感受：** "等待是值得的，报告非常个性化和有深度"
- **本期范围：** 是

#### 7.5.3 报告解析 [创始人补充]

## 10. AI/系统行为规格

### 10.1 双轨运行 [已确认]

- **前台：** 正常教学/辅导交互（授课、对话、对练、测评）
- **后台：** 异步标注能力维度和表现信号
- 采用方案 B：异步双次 LLM 调用，后台标注用轻量级模型 [已确认]
- **关键约束：** 后台标注不得干扰前台对话的自然性；学员的操作性问题就回答操作性问题，不为采集数据而改变对话方向 [已确认]
- 各模块标注方式：
  - **授课模块：** Quiz 结果自动映射到能力维度；AI 在自然对话中适度追问以收集行为证据（仅在话题自然涉及时）
  - **对练模块：** 每次对练预绑定 1-2 个能力维度；KGP 定义"结构化行为观察点清单"；AI 实时标注观察点
  - **辅导模块：** AI 识别学员主动提问涉及的能力维度，记录求助行为和问题摘要
  - **测评模块：** 追问路径围绕能力维度展开，STAR 案例结构化存储

### 10.2 AI 质量控制四层架构 [已确认]

#### 第一层：预防（内容护栏）

- 授课模块知识点讲解使用半预制 SCO，偏离空间小
- AI 的 system prompt 中写入"核心知识点断言清单"和"绝对不能说错的内容清单"（KGP 制作时提供）
- 对练模块 AI 角色的 system prompt 定义行为边界（允许表达/禁止表达矩阵）

#### 第二层：实时检测（异常信号采集）

- **5 类异常信号：** [已确认]
  1. 学员连续 3 轮表达困惑/不满
  2. AI 回答与知识点断言清单冲突（后台轻量级分类器校验，分"无冲突/可能冲突/明确冲突"三级，仅"明确冲突"触发预警）
  3. 对练中 AI 脱离角色设定（后台 classifier 判断）
  4. 单次对话时长异常（过长或过短）
  5. 学员主动投诉 AI 行为

#### 第三层：分级响应 [已确认]

| 级别 | 条件 | 处理方式 |
|------|------|---------|
| **轻度异常** | 异常信号 1-2 项，非严重 | AI 下一轮自动修正："关于刚才的问题，更准确的说法是..." |
| **中度异常** | 多项异常信号或单项严重 | 推送运营预警队列，运营查看上下文后决定处理方式；运营可直接修改知识点清单，AI 后续回答即时更新 |
| **重度异常** | 系统判断 AI 输出可能严重偏离 | 将 AI 切换到"安全模式"（仅使用预制内容回答，不做实时生成）；同时紧急推送运营+技术团队 |
| **违规内容** | AI 输出歧视/暴力/威胁等内容 | 暂停会话；紧急推送全团队 |

#### 第四层：事后修复 [已确认]

- 修正后主动通知受影响学员："关于之前我提到的[话题]，我纠正一下——更准确的说法是[正确内容]。抱歉给你造成困惑。"
- HR 端"项目健康度"看板更新异常事件和处理状态
- 严重事故主动同步 HR

### 10.3 催学自动化机制

详见第 9.5 节。核心要点：

- 规则自动执行，运营只管重要情况 [创始人调整]
- 价值导向消息内容
- 三级升级机制

### 10.4 报告智能体两种模式 [已确认]

| 模式 | 使用者 | 流程 | 特点 |
|------|--------|------|------|
| **HR 交互式** | HR | 提出需求 → Agent 确认 → 生成初稿 → 预览 → 标注修改 → 迭代 → 下载 | Agent 记忆偏好，越来越少问 |

## 11. 自适应体系

### 11.1 L1: LLM 实时响应 [已确认]

- 已实现的能力
- 每次 AI 交互都是自适应的——AI 根据当前对话上下文实时调整响应
- 覆盖所有模块（辅导/授课/对练/测评）

### 11.2 L2: SCO 多版本 Tag 切换 [已确认] [创始人调整] [数据验证]

- 本期只做 Activity 层的 SCO 版本切换（不做 Project 层的 Activity 顺序调整）
- 判定规则由内容脚本定义，不固定权重 [创始人调整]
- 版本差异不只是难度问题，可能是角度问题（如同一知识点的不同案例/不同讲法）[创始人调整]
- 具体规则由 KGP 根据教学目标定义，后续讨论 [创始人调整]

**L2 的技术实现方式：** [数据验证]

1. **标签输出（课前诊断）**：课前 CHAT 类型的 SCO 通过多轮对话收集学员信息，`passConditions` 中的打标规则将对话结果转化为学员类型标签（如"防御拖延型""利益博弈型""情绪对抗型"）
2. **标签消费（内容分发）**：后续 SCO 的 `separateTag` 定义面向哪种学员类型，`groupKey` 标识同一槽位的不同版本，`tutorMode=1` 标记需要分流的 SCO
3. **匹配机制**：系统读取学员标签，在同一 `groupKey` 的多个 segment/SCO 中选择 `separateTag` 匹配的版本播放

**完整数据链路：** [数据验证]

```
课前互动 CHAT (passConditions.rules 中的打标规则)
    ↓ 输出学员类型标签（如"防御拖延型"）
    ↓ 存储到 student_profile (Redis + State 逐层传播)
    ↓
后续教学 SCO (tutorMode=1)
    ↓ 读取学员类型标签
    ↓ 匹配 separateTag
    ↓ 从同 groupKey 的多个版本中选一个播放
```

**降级策略：** [技术现状]
- 学员跳过课前诊断 → 不筛选，全量展示
- LLM 分析失败 → 同上
- LLM 超时 → 使用兜底类型
- 旧课程包无课前诊断 → 全量展示

**关键设计原则**：AOM 中没有显式的"变量传递"字段，上下文传递通过 `separateTag` 和 `groupKey` 的隐式约定实现——这是一种**内容驱动的弱耦合机制**。分流的类型名称完全由内容定义，平台不需要预知这些类型。 [数据验证]

### 11.3 对学员透明切换 [已确认]

- 不显示"你被切换到了简单/高级版本"
- AI 自然过渡："让我换个角度来看这个问题" 或 "给你举一个更贴近实际情况的例子"
- 用教学策略的话术代替难度的显性切换——学员感受到"AI 懂我"，而不是"我被降级了"

### 11.4 未来 L3: Activity 顺序调整 [待确认]

- 在 Project 层面调整 Activity 的先后顺序
- 根据学员画像和学习表现动态重排学习路径

### 11.5 更高层面（远期）

- 生产临时文件（如根据学员特定问题临时生成额外材料）

## 12. 闭环反馈机制 [创始人补充]

### 12.1 内测痛点：情感错付

内测中发现的关键问题：学员在课前认真分享了真实工作问题（如"我跟设计部的张楠关系很僵"），但后续课程完全没有回应这些问题——学员投入了真实情感却得不到呼应，产生"情感错付"的负面体验。 [创始人补充]

### 12.2 闭环反馈 SCO（P1）

**形态**：一种特殊的 SCO 类型，在课前采集学员真实场景，在课后用学到的知识复盘学员的真实问题。 [创始人补充]

**与小测评 SCO 的区别**：小测评是即时的知识检查，闭环反馈是跨时间的场景复用。

**实现方式**：当前可用记忆系统（ADR-005）+ Quiz 暂时替代——课前 CHAT 采集学员场景存入记忆系统，课后 Quiz 或小解答 SCO 引用之前采集的场景数据进行复盘。看实际效果再决定是否需要独立的闭环反馈 SCO 类型。 [创始人补充]

## 13. 记忆工程 [创始人补充] [技术现状]

### 13.1 三级记忆管道（ADR-005）

本期实现完整的三级记忆管道，解决跨模块数据流转问题：

```
┌────────────────────────────────────────────────────────────┐
│ 第一级：课堂内记忆                                          │
│ 授课/对练/测评中采集的数据                                   │
│ (学员输入的真实场景、行为表现、测评结果、Quiz回答)            │
│                    ↓ 实时汇总到统一 Database                 │
├────────────────────────────────────────────────────────────┤
│ 第二级：大厅记忆                                            │
│ 辅导智能体可读取课堂数据                                     │
│ ("你在对练中说了XX"、"上次你提到的小王的问题")               │
│                    ↓ 流转给管理端                            │
├────────────────────────────────────────────────────────────┤
│ 第三级：管理端记忆                                          │
│ 报告智能体读取全部数据（课堂+辅导）                          │
│ 生成 HR 可用的报告（全链路数据汇聚）                         │
└────────────────────────────────────────────────────────────┘
```

### 13.2 内测痛点与设计动机

- **"情感错付"问题**：学员在课前分享真实问题，后续课程没有回应——这是记忆工程要解决的核心痛点 [创始人补充]
- **辅导模块失去陪伴感**：如果辅导智能体不能引用课堂数据，就变成了通用问答机器人
- **报告数据不完整**：HR 端报告需要全链路数据才能生成有说服力的行为改变证据

### 13.3 各模块对记忆工程的依赖

| 消费方 | 依赖的记忆级别 | 使用方式 |
|--------|-------------|---------|
| **授课小解答** | 第二级（大厅记忆） | 调取学员之前在辅导中提过的真实问题，结合当前知识点给出个性化解答 |
| **辅导模块** | 第二级（大厅记忆） | 引用课堂表现："你在对练中的表现说明你已经掌握了X，但Y还需要加强" |
| **闭环反馈 SCO** | 第一级（课堂记忆） | 课后引用课前采集的真实场景进行复盘 |
| **报告智能体** | 第三级（管理端记忆） | 汇聚全部数据生成学习旅程证据链 |
| **个人报告** | 第一级+第二级 | 汇聚学习过程中的异步碎片（trait/cognitive/competency） |
| **对练复盘** | 第一级（课堂记忆） | 引用演绎阶段的具体对话进行苏格拉底追问 |
