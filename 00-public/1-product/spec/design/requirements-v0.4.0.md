## 1. 产品定义与愿景

AI 老师是面向企业培训场景的 AI 原生学习陪伴系统。通过"大厅 + 四间场域"的空间模型，让 30-40 名基层管理者在项目周期内（典型 6 个月，可配置 1 个月~1 年），获得 1v1 AI 教练（Leo）的持续陪伴、结构化课程（Neo）的自适应学习、沉浸式对练（Actor）的行为训练，最终产出可举证的行为改变报告。

**核心比喻**：平台是钢琴，内容是乐谱，AI 是演奏者。更换乐谱（AOM 内容包），就能交付完全不同主题的培训项目。

**核心价值**：
- 对学员：Leo 随时在身边，带着真实场景辅导学会并迁移知识
- 对管理员：基于全过程数据的行为改变证据报告，Ora 辅助撰写
- 对运营：系统自动执行可配置的催学规则，Ora 解读数据

## 2. 角色与端口

### 2.1 两种角色

| 角色 | 归属 | 核心诉求 |
|------|------|---------|
| **学员** | 客户企业 | 解决真实工作问题，完成培训任务 |
| **管理员** | 客户企业 HR / 供应商运营（统一角色） | 管理项目全程、监控进度、生成报告 |

> KGP（内容制作）不是软件端口角色，是内容供应链。

### 2.2 两个端口

| 端口 | 角色 | 说明 |
|------|------|------|
| **学员端** | 学员 | 大厅 + 四间场域（Lecture/Practice/Inquiry/Report） |
| **管理端** | 管理员 | 首页 Operation + 报告生成 + 消息 + 配置 |

本期只有一个管理端，HR 和运营统一为"管理员"角色，使用同一端口、同一界面。端口切换只有"学员端 / 管理端"两个选项。

## 3. 产品全生命周期

### 3.1 全流程概述

```
企业签约下单
    ↓
[后台] 商品平台下发带有 AOM 内容的商品 → 创建项目实例 → 创建 admin 账号
    ↓
[交付] 交付人员用 admin 登录管理端 → 项目内容预览（只读）→ 确认内容正确
    ↓
[运营] 运营用 admin 启动开营流程（配置模块）
    → 项目基本信息 → 学员名单导入 → 角色分配 → 催学规则 → 平台个性化
    ↓
[开营] 确认开营 → 运营在平台外发送登录链接 → 学员端开放登录
    ↓
[学习期] 学员学习（项目周期内）/ 管理员监控+报告+督学
    ↓
[结项] 结项报告 → 项目关闭
```

### 3.2 开营前后的系统状态

| 状态 | 管理端可用模块 | 学员端 |
|------|-------------|--------|
| **开营前** | 仅「配置」可用，其他三个导航锁定 | 不可登录 |
| **开营后** | 四个模块全部解锁 | 可登录使用 |

开营后所有配置项本期均可编辑（重要修改的安全验证机制放未来迭代）。

### 3.3 项目规模

- 单项目学员数：30-40 人
- 管理员账号：1-3 个
- 项目周期：1 个月 ~ 1 年（管理员配置）
- 第一个内容主题：基层管理者（8 个 Course）

## 4. AI 角色架构

本产品有五个 AI 角色，分布在学员端和管理端：

### 4.1 学员端：Leo + Neo + Actor + 导演

| 角色 | 前台名称 | 类型 | 位置 | 驱动方式 |
|------|---------|------|------|---------|
| **Leo** | Leo | Coach（教练） | 大厅 | 统一 Database + 自身判断 |
| **Neo** | Neo | Tutor（老师） | 四间场域 | 统一 Database + AOM 脚本 + 自身判断 |
| **Actor** | 具体人物（如"张总"） | 演员 | 对练场域 | 仅遵循剧本 |
| **导演** | 不暴露给学员 | 后台控制 | 对练场域 | 仅遵循剧本 |

**Leo 核心价值**：带着真实场景辅导学员学会并迁移知识。角色优先级：辅导 > 课程推荐 > 工具笔记引导 > 督学。基于 Bloom 认知层级判断——概念层推课给 Neo，应用层和迁移层自己引导。

**Neo 核心价值**：按 AOM 脚本结构化教学，实时感知学员课堂学习状态（听没听懂、认知旅程、什么条件有利于学习）。Neo 可访问统一 Database 跨 Course 引用数据，但关注点聚焦在当前 Activity 主题。在对练场域中承担情景导入、答疑与辅导、反思复盘。

**Actor**：对练场域中独立于 Neo 的角色扮演对象，由剧本驱动。与 Neo 分开对话窗口（左栏模拟对话+右栏 Neo 辅导）。Actor 仅遵循剧本，不访问统一 Database。

**导演**：对练场域的后台角色，不暴露给学员。分析学员每轮发言，决定 Actor 的场景走向和情绪变化。导演仅遵循剧本，不访问统一 Database。

### 4.2 管理端：Ora

| 角色 | 前台名称 | 位置 | 驱动方式 |
|------|---------|------|---------|
| **Ora** | Ora | 管理端首页+报告页 | 统一 Database（全局视角） |

**Ora 核心价值**：数据解读、对比洞察、报告辅助生成与 HITL 编辑、证据追溯、隐性知识发现。首页和报告页共享记忆。

### 4.3 角色间关系

```
统一 Database
  ├── Leo（读写，Program 全量视角）
  ├── Neo（读写，Activity 为主+学员画像）
  ├── Ora（读取，全局视角）
  │
  （Actor 和导演不访问 Database，仅遵循剧本）
  ├── 导演（读取剧本 → 指挥 Actor）
  └── Actor（接受导演指令 → 角色扮演）
```

- **Leo/Neo/Ora** 共享统一 Database，但引用时互不提及。Leo 和 Neo 都可跨 Course 读取数据，区别在于关注点（Leo 全局陪伴视角，Neo 当前 Activity 教学视角）
- **Actor/导演** 仅遵循剧本，不访问统一 Database，不知道学员的历史数据
- Ora 可读取所有数据（含 Leo 辅导原话），本期不做脱敏
- **所有 AI 角色（含 Leo/Ora）都是同一会话长上下文**——Leo 在大厅的对话历史持续积累不重置；Ora 在首页和报告页的对话上下文跨页面保持
- **轻量级 AI 点位**：除五个核心角色外，系统中还有一些轻量级 LLM 调用（如催学消息润色），不算独立 AI 角色

### 4.4 关键机制说明

**InsightCapture 分层**：高光瞬间的感知和后台记录是 P0（用于报告生成）；前台推送卡片让学员收藏是 P3。

**报告证据标注**：报告中的"高光表现"和"待改进点"由报告生成时 AI 回顾全部对话记录后标注，不是实时标注。

**数据粒度分层**：底层存储为 SCO 级（含每道 Quiz 的回答、每轮对话记录），管理端 UI 展示为 Activity 级。学员端报告可引用 SCO 级证据。

**L2 自适应标签系统**：授课场域的 `separateTag`/`groupKey` 和对练场域的 `ASSESSMENT_TAG SCO` 是同一套机制——通过判定环节给学员打标签，后续 SCO 根据标签选择对应的 Segment 版本。标签作用域限定在 Activity 内。
- Neo 在对练场域的答疑与辅导以剧本和导演指令为上下文，但 Neo 本身可访问 Database

## 5. 两端布局

### 5.1 学员端布局

**常显导航**（个人 + 社区[置灰]）+ **左栏** + **右栏**

- 大厅：左栏=看板区（项目概览+课程目录+热力图+工具库+笔记库+待办+社区占位），右栏=Leo 对话
- 授课场域 Lecture：左栏=目录+课件展示，右栏=Neo 对话
- 对练场域 Practice：左栏=演员头像+背景信息+模拟对话，右栏=Neo 答疑与辅导
- 调研场域 Inquiry（P3）：说明双栏→核心单栏沉浸→确认双栏
- 报告场域 Report：左栏=报告内容，右栏=Neo 对话

### 5.2 管理端布局

**常显导航**（首页/报告/消息/配置）+ 内容区

- 首页/报告：双栏（左栏内容 + 右栏 Ora 对话）
- 消息/配置：单栏全幅（纯 GUI，无 Ora）

### 5.3 通用栏（两端统一）

| 元素 | 功能 |
|------|------|
| **Logo** | 可被管理端"平台个性化"替换为客户企业 LOGO |
| **提醒🔔** | 点击展开消息面板（站内消息，两端统一） |
| **头像👤** | 通用功能入口：账号管理/个人设置/帮助(新窗口)/端口切换(学员端↔管理端)/退出 |

## 6. 数据模型（AOM 五层）

```
Project（培训项目）
  └── Course（课程/模块，共 8 个）
       └── Activity（活动：授课/对练/调研/报告，15-25分钟）
            └── SCO（最小学习内容单元）
                 └── Segment（SCO 的版本，用于自适应切换）
```

**SCO 类型**：`NARRATION`（PPT讲解）、`VIDEO`（案例视频）、`QUIZ`（互动测验）、`QA`（小解答）、`FEEDBACK_COLLECT`（闭环-课前采集）、`FEEDBACK_REVIEW`（闭环-课后复盘）、`ASSESSMENT_TAG`（打标判定，纯后台）

## 8. 跨模块系统

### 8.1 内容驱动 [ADR-004]

平台是引擎，内容是燃料。所有教学行为由 AOM 脚本/对练剧本/调研剧本控制，平台不硬编码。Actor 行为由剧本控制，Neo 答疑与辅导基于自身判断（以剧本和导演指令为上下文）。

### 8.2 记忆工程 [ADR-005]

三级记忆管道，本期全做：
1. **课堂内采集**：授课/对练/调研中的所有对话 + Quiz 回答 + 行为观察
2. **大厅辅导引用**：Leo 读取课堂数据，跨 Course 记忆打通
3. **管理端汇聚**：Ora 拉取全链路数据，辅助报告生成和数据解读

统一 Database，各角色按权限和视角读取。半结构化记录。

### 8.3 自适应体系

| 层级 | 机制 | 本期 |
|------|------|------|
| L1 | 每次对话交互，LLM 实时调整 | 做 |
| L2 | ASSESSMENT_TAG 打标 + Segment 版本切换 | 做 |
| L3 | Activity 顺序调整 | 未来 |
| L4 | 动态内容生成 | 未来 |

### 8.4 闭环反馈

FEEDBACK_COLLECT（课前采集）→ 授课中 referenceSlots 引用 → FEEDBACK_REVIEW（课后复盘）→ Leo 辅导引用 → 报告呈现。通过 memory_id 关联。

### 8.5 复习模式

三种复习方式：
- **自由复习**（本期做）：进入已学 Activity，和 Neo 答疑讨论+翻看之前页面
- **辅导回顾**：Leo 自然语言讲解巩固（本期做）→ 调取笔记工具讲解（P3）→ 调取课堂素材讲解（未来）
- **重学**（未来）：hall 看板点击重学某个 Activity

### 8.6 AI 质量控制

本期做基础版本：后台查找溯源 + 技术调试后门。AI 质量数据不放学员端/管理端。

### 8.7 举证机制

路径个性化，评估维度标准化。AI 双轨运行（前台教学不受后台采集干扰）。报告按能力维度汇聚全模块证据，生成行为改变证据链。

## 9. 核心设计哲学

1. **AI 不做"判官"**——所有学员输入都有价值，不判断真假
2. **内容驱动行为**——平台不硬编码教学逻辑，由脚本控制
3. **Leo/Neo 分工明确**——Leo 做陪伴辅导，Neo 做结构化教学，互不混淆
4. **LUI 优先于 GUI**——优先用对话式交互（如首次登录 Leo 自我介绍而非 overlay 引导）
5. **需求感知式推荐**——Leo 基于学员需求感知引出课程推荐，不硬推进度
6. **SCO 锁定机制**——未播放到的 SCO 锁定，学员只能回顾已学完内容
7. **防挂机**——5 分钟无操作提醒/暂停
8. **价值导向催学**——催学消息告诉学员"来了有什么好处"，而非"你多久没来了"
