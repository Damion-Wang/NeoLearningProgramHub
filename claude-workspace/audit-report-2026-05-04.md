## 二、4 维 fan-out 核心发现

### 维度 A · PM 功能合理性（8 问题 / 7 冗余 / 5 缺失）

**关键发现**：
- § 9.3.3 recap + § 9.3.2 practice + § 10 共 4000 行 / 把"一次对练 + 一次小结 + 一份团队报告"推到了**研发不做完出不了第一版**的密度
- 严重度高 2 项：**6 维画像强制暴露给学员** / **recap 双层报告机制过度复杂**
- 5 项关键缺失：今天该学什么 CTA / HR 纵向追踪学员能力 / 督学强度开关 / 应用计划放弃兜底 / admin 账号转交流程

### 维度 B · Tech Lead 研发可理解度（10 问题 / 5 概念模糊 / 5 接口不清）

**关键发现**：
- **软层**（业务/Persona/动作/感知）写得非常细致 / 可直接对齐 Agent 实现意图
- **硬层缺口集中**：AOM schema 不锁字段 / 三级记忆只到概念无读写时机 / KGP 接口契约缺位 / Database 共享 schema 模糊
- 概念模糊：**milestone** 三层混用（项目级 / recap 内首聊触发 / 报告中心触发）/ Course-Module 命名 / 管理员-HR-运营 / Truth Source 含义
- 总评："研发能上手做 GUI / Persona prompt / 看板 widget，**但做不了数据层与记忆层架构选型**"

### 维度 C · New Reader 可读性（10 卡点 / 4 章节顺序 / 4 引用过密 / 10 修订）

**关键发现**：
- **282 处跨章引用** / 平均每 24 行 1 次跳转
- 10 个具体卡点：开篇无导航 / § 1.4 引第 3 章 / § 2.3 forward refs / § 4.2 用 AOM 但定义在 § 5 / § 6.1.2 vs § 6.1.3 阶段命名不一致 / 5-6 级嵌套
- 总评："**作者对设计有完整骨架，但新读者需要先看骨架地图再进细节，目前是直接被砸进满汁的肉里**"
- 最低成本提升：加 § 0 阅读指南 + 术语表 + § 5 AOM 一句话提前到 § 1 + 删 forward refs banner，**这 4 件事就能把"读得懂"的门槛降一半**

### 维度 D · Doc Architect 丛书拆分（3 方案 / 推荐方案 1）

**关键发现**：
- **6831 行单文件已超出主流 PRD 单文件可维护阈值**（业界标杆 Stripe/AWS spec 单文件通常 2-3k 行上限）
- 推荐 **方案 1 · 按角色端口拆**：
  - **Book A · 总册**（§ 1-§ 8 + § 9.1 + § 10.1）≈ 2200 行
  - **Book B · 学员端**（§ 9.2-§ 9.4 + 补章）≈ 2800 行
  - **Book C · 管理端**（§ 10.2-§ 10.5）≈ 1700 行
- 时点：**D-5.0 freeze 后 / D-6.0 启动前 间隙做**
- 拆分顺序：先抽 C → 再 B → A 自然剩
- 跨册引用方案：A = truth source / B+C 单向引用 A / 不双写

## 四、DM 反向讨论 · 关键反转 4 项

DM 视角（基于 CLAUDE.md 11 条 + 第一性原理 + 三层文档定位）挑战 Debate 结论，**4 项反转**：

### 反转 1 · 议题 2 · 保留双层 + milestone（不砍）

> "双层正是逼学员产生掌握度自评的关键钩子——砍掉评估层等于把 Neo 退化成课程小结生成器。Debate 砍双层是把研发省事当成产品决策。"

- **DM 立场**：保留双层 + milestone / 真正要砍的是 spec 里**讨论性废话**
- **理由**：Neo 第一性 = 教学效果 / 双层服务"反思动作"教学钩子

### 反转 2 · 议题 4 · 议题作废（事实性误读）

> "Debate Agent 把'Home'误读为学员端 Home—— § 10.2 Home 是**管理端首页** / Ora 本身就是管理端 AI / 没 Ora 入口 = HR 进来看不到搭档。"

- **DM 立场**：议题作废重提 / 当前 spec 现状已合理
- **理由**：Ora 第一性 = HR 搭档 / 管理端 Home 必须有 Ora 主区

### 反转 3 · 议题 5 · 拒绝盲砍 / 先映射

> "5 路砍 2 路这种折中是典型群体思维——看起来理性其实没人真去问'这 5 路分别对应 HR 哪个真实工作流'。砍掉的是 HR 真实在用的路径，折中 = 残废。"

- **DM 立场**：先做"5 路 × HR 工作流映射表" / DM 看完再拍 / 不接受 Debate 盲砍
- **理由**：Ora 第一性 = 可举证 ROI / 每条路径必须能回答"HR 拿这条路径去给老板举证什么"

### 反转 4 · 议题 6 · 留 03-full 附录（不拆）

> "CLAUDE.md 14 条明写'03-full = 唯一持久产品 spec 产出文档'。拆出去就违反三层文档定位。研发要查技术对接，翻附录就是了 / 一个文件搜索成本最低。"

- **DM 立场**：留在 03-full 附录 / 不拆独立文件
- **理由**：同源单值原则 / 反对文档碎片化

### DM 总评

> "Debate 5 角色看起来挺周全，实际栽在两点：**一是把研发省事当成产品决策**（议题 2 / 6 都是研发视角偷渡进 PM 决议）；**二是把 Home 概念搞混**（议题 4 是事实性误读不是观点分歧）。第一性原理是**总判断器**——Neo 是 1v1 老师 / Ora 是 HR 搭档 / 任何决议先过这两关 / 过不了的不管 5 个 Agent 多一致都得推翻。"
