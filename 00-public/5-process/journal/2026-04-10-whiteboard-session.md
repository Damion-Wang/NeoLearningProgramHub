# 设计日志 2026-04-10: 白板产品架构确认

## 讨论了什么

基于创始人的白板图，逐项确认 AI 老师 6月版本的产品架构。覆盖了模块划分、角色端口、交互形态、数据流向、生命周期等核心问题。

## 关键发现

1. **辅导模块是入口而非平级模块** — 最初以为辅导和授课/对练/测评是平级的四个模块，实际辅导是"大厅"，教学模块是"教室"
2. **三端实际是一个 App** — 学员/HR/运营共用同一 Web 应用，权限控制入口。HR 可切换为学员体验课程
3. **运营是供应商派出的人** — 不是客户内部的人。这意味着运营端的设计要考虑"代操作"模式
4. **报告智能体贯穿学习期** — 不只是结项时用，HR 在学习过程中就可以随时通过 Agent 了解情况、生产素材
5. **学员可带走学习资产** — 学习卡片、工具表格、工具文本（预制+AI生成），存在辅导模块左侧看板
6. **三种教学模块有三重关联** — 数据关联+内容关联+序列关联，不是独立的三个工具
7. **本期不做的运营功能明确** — 布置作业、评比、奖项、开营结营都在产品外

## 决策

- ADR-001: 三端合一 Web 应用
- ADR-002: 第一版预制菜模式
- ADR-003: 辅导模块作为学员端主界面

## 经验教训

- **用 AskUserQuestion 工具逐步确认**比一次性列出所有问题效率高得多——每轮3个问题，聚焦一个主题
- **先确认角色和端口再画泳道图**是正确顺序——角色定义不清会导致泳道图反复修改
- **白板图的解读不能靠猜**——第一次我把"辅导/对练/测评"误读为 SCO 类型，实际是完全不同的含义。必须逐项确认
- **产品设计的讨论从"不覆盖什么"开始**比从"覆盖什么"更高效——先砍掉不做的，剩下的就是必须做的
