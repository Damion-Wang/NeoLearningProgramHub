## 用途

防止用户（和 Claude 自己）在"看起来做完了"时过早收工。强制走 4 项核验门控：测试 / 文档 / commit / 决策记录，把"软完成"升级为"硬完成"。

核心价值：
- 抵抗乐观估计偏差
- 把隐性 TODO 显性化
- 保证产物有可追溯证据

## 何时触发

**用户语言信号**（任一命中即触发）：
- 中文：完成 / 搞定 / 做完 / 做好了 / 弄好了 / 齐活了 / OK 了 / 行了
- 英文：done / finished / completed / wrapped up / good to go

**上下文限定**（避免误触）：
- 用户在指代某个具体任务的完成（不是闲聊"我吃完饭了"）
- 通常出现在一段工作之后，对话主题是"在做某事"

**不触发场景**：
- 用户问"X 完成了吗？"（这是反问，非声明）
- 用户在描述别人 / 历史完成

## 4 件套反问

### 检查 1：测了吗？

判定逻辑：
- 本次任务涉及 UI 改动 → 必须有 Playwright 截图证据
- 涉及代码逻辑 → 必须有功能验证（运行 / 测试输出）
- 涉及文档 → Read 一遍确认渲染无误（链接 / 图片 / 表格）
- 涉及配置 → 重启服务或重新载入验证

如未测试：
> 检测到 [UI/逻辑/文档] 改动但未见验证证据，建议立刻执行 [具体动作]

### 检查 2：文档同步了吗？

判定逻辑：
- 涉及 spec 改动 → `00-public/product-V-0.2.0-D-4.0/spec/` 或 `product-V-0.2.0-D-5.0/plan/` 下相关文件已更新？
- 涉及流程 / 规范 → `CLAUDE.md` / `README.md` 已更新？
- 涉及目录新增 → 该目录的 README 已创建？
- 涉及 API / 接口 → 接口文档已更新？

如未同步，列出待补清单：
> 待同步文档：
> - [ ] spec/design/xxx.md
> - [ ] 该目录 README.md

### 检查 3：commit 了吗？

判定逻辑：
- `git status` 是否干净
- 如有未提交变更 → 列出文件清单
- 如未提交 → 提示走 git-push-guard skill 检查后提交

```bash
git status --short
```

未 commit 输出：
> 检测到 N 个未提交变更：
> - M  file1.md
> - ?? file2.md
> 建议先执行 git-push-guard 检查后提交

### 检查 4：决策记录了吗？

判定逻辑：
- 本次涉及关键产品 / 架构决策？（架构选型、方案 A/B、信息架构）
- 如是 → `00-public/4-decisions/` 是否新增 ADR 文件？
- 如否 → 跳过此项

ADR 缺失输出：
> 本次涉及 [决策点]，建议在 `00-public/4-decisions/` 新增 ADR：
>   - 文件名：ADR-NNN-[决策简写].md
>   - 模板：背景 / 决策 / 后果 / 替代方案

## 标准流程

### Step 1：识别任务

从最近 5-10 轮对话上下文识别本次"完成"指代的任务，一句话总结。

### Step 2：扫描证据

并行执行：
- `git status --short`（commit 检查）
- `git diff --stat HEAD`（看改了哪些文件，推断范围）
- 检查 `00-public/4-decisions/` 最近 3 天内新增（决策检查）
- 回看本次会话 Playwright 调用记录（测试检查）
- 回看本次 Edit / Write 调用，提取文档相关变更（文档检查）

### Step 3：渲染门控报告

按"输出格式"输出。

### Step 4：等待用户选择

- 全部通过：恭喜并标记完成
- 有未达成：用户可选「逐项补完」或「强制完成」

### Step 5：跳过日志

若用户回"强制完成"：
- 追加到 `02-temp/skipped-checks.md`
- 格式：`[YYYY-MM-DD HH:mm] 任务: xxx | 跳过项: [测试/文档/commit/决策] | 原因: xxx`
- 下次同类任务触发时优先提醒

## 输入 / 输出

**输入**：用户的"完成"声明 + 上下文中本次任务的范围

**输出**：1 个 markdown 门控报告 + 0/1 个跳过日志条目

## 输出格式

```
完成门控检查

任务：{从上下文识别的任务一句话}
范围：{改动了 N 个文件 / 涉及 [UI/逻辑/文档/配置]}

------
[1/4] 测试：{Playwright 截图证据 / 功能验证 / 缺失说明}
[2/4] 文档：{spec/CLAUDE.md/README 同步情况}
[3/4] Commit：{git status 状态}
[4/4] 决策：{ADR 记录情况；不涉及则显示 N/A}

------
{若任一未达成}
检测到 X 项未达成，建议先补完再声明完成

补完顺序（推荐）：
1. {优先级最高的一项}
2. {次}
3. {次}

仍要标为完成请回复「强制完成」（将记录到 02-temp/skipped-checks.md）

{若全部达成}
完成核验通过！

下一步建议：
- 若本会话告一段落 → 写入 02-temp/next-step.md
- 若准备推送 → 执行 git-push-guard skill
```

**示例**（部分通过）：
```
完成门控检查

任务：创建 4 个项目级 Skill 文件
范围：新增 5 个文件（4 SKILL.md + 1 README.md）

------
[1/4] 测试（未通过）：未读取已写入文件验证 frontmatter 格式
[2/4] 文档（通过）：README.md 索引页已创建
[3/4] Commit（未通过）：5 个未提交变更（git status 输出）
[4/4] 决策（N/A）：本次不涉及架构决策

------
检测到 2 项未达成

补完顺序：
1. Read 验证 4 个 SKILL.md 的 frontmatter 解析正确
2. 走 git-push-guard 后提交

仍要标为完成请回复「强制完成」
```

## 跳过机制

`02-temp/skipped-checks.md` 格式：
```markdown
# Skipped Checks Log

| 日期时间 | 任务 | 跳过项 | 原因 |
|----------|------|--------|------|
| 2026-04-26 15:30 | 创建 4 Skill | 测试 | 用户判定 frontmatter 已肉眼检查 |
```

## 注意事项

- 不要把"完成"识别得太宽：用户说"早饭吃完了"不触发
- 4 项中 N/A 也算一种状态，不计入未达成
- 强制完成后不要抱怨，简洁记录即可
- 该 skill 不修改正在做的工作产物，只做核验
- 如某项检查有歧义（比如改了 1 行 README，算文档同步吗），从严判定为未达成让用户决定
