# A5 · 调研结论同步 spec 的 SOP

> **决策日期**：2026-04-26
> **依据**：A4 三件套节奏（每个 v0.x 触发一次同步）+ CLAUDE.md 核心约束 #6（spec 改动方式：直接编辑 [已审阅] 文件，新设计追加"反向补充：XXX"章节）

---

## 同步原则

1. **不改原审阅内容**：保留原 [已审阅] 文件全部既有章节
2. **追加反向补充章节**：新发现按"反向补充：XXX"或"调研补充：XXX"前缀加章节
3. **第一性原理优先**：调研结论与 spec 现状冲突时，先看是否违反第一性原理；不违反就追加，违反就升级 spec
4. **保留可追溯**：每次同步注明"来源：调研 vX.X / 0424 决策 / Round-N 辩论"
5. **版本号递增**：spec v0.4.0 → v0.5.0（重大）/ v0.4.1（小补充）

## 全生命周期 SOP 同步 → 不入 spec，独立成文

SOP 是**项目交付方法论**，不是产品需求。建议归档到：
- `00-public/2-business/` 或 `00-public/4-decisions/`
- 文件名：`project-lifecycle-sop-v1.0.md`

但 SOP 中的"产品要做什么"部分要回流到 spec：
- "开营前 admin 配置项目" → spec/design/management/03-project-config.md
- "结项后学员账号只读" → 0424 决策 S-12 → requirements § 3.2 反向补充
- "持续价值追踪报告" → 0424 决策 S-14 → management/02-report-gen.md 反向补充

## 同步动作 checklist

每次触发同步时执行：

- [ ] 确认本轮调研产出版本（v0.1/v0.2/v1.0）
- [ ] 列出受影响 spec 文件清单
- [ ] 按 7 类映射规则逐文件补充
- [ ] 每处补充加来源注释（"来源：调研 vX.X · YYYY-MM-DD"）
- [ ] 检查是否需要升级 ADR（如行为原则变化，加 ADR-011/012/...）
- [ ] 更新 PROGRESS.md（追加同步记录）
- [ ] 更新 requirements-v0.4.0 附录 changelog（v0.4.1/v0.4.2/v0.5）
- [ ] 触发 git-push-guard 三联检查 + commit

## 冲突处理预案

| 冲突类型 | 处理 |
|---------|------|
| 调研结论 ≠ spec 现状，且不违反第一性原理 | 反向补充章节追加新表述，原表述加注"v0.4.0 设计，调研建议升级"|
| 调研结论 ≠ spec 现状，且违反第一性原理 | 启动 multi-round-debate，5 角色辩论后决策（必走 round-N 归档）|
| 调研结论之间矛盾（B1 vs B2 vs B3）| DM 拍板，写决策记录到 4-decisions/ |
| 调研结论不影响 spec（仅影响项目交付）| 不入 spec，归档到 SOP / 知识库 |
