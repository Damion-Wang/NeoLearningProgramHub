## § 1 · 议题 1 · 服务周期 vs 项目周期 UI 呈现

### 1.1 各角色第一轮发言

**PM**：

我倾向 UI 层**默认隐藏服务周期**，只显示项目周期。

具体方案：
- 学员端 Topbar / 大厅看板：只显示"项目第 X 周 / 共 24 周"或"项目剩余 X 天"
- 当项目周期结束后到服务周期结束期间：横幅提示"项目已完成，但你的账号还可使用 X 天，欢迎复习"
- 管理端：HR 能在配置页看到完整服务周期 + 项目周期两个字段
- 学员从来不主动看到"服务周期"概念（避免认知负担）

**UX**：

PM 方向对。**学员认知模型只承载一个时间锚**——多了 confusing。

但设计细节：
- 大厅看板进度条：项目结束前显示"项目进度 X%"，结束后**变成"复习期 · 剩余 X 天"**（颜色变浅）
- Neo 在项目结束当天主动提示"恭喜完成项目！剩余 X 天可继续复习/应用"——这是个仪式感时刻
- 不要在用户多个地方重复展示剩余天数（造成时间压力）

**学员代表**：

我作为学员关心的是：
- 项目结束 ≠ 学习结束（DM 强调）。所以**不要让我感觉"项目结束了你还赖着干嘛"**。
- 仪式感是关键——结项汇报 / 完成证书 / 个人报告 应该在项目结束当天清晰呈现
- 但**不希望被"剩余 X 天"压力提醒**，特别是每天看到都让我焦虑

**HR 代表**：

HR 想看到完整二元模型。
- 管理端项目配置页：明确两个字段（服务周期 / 项目周期）+ 两者对比图
- HR 在续约谈判时需要知道"还剩多少服务期"
- 学员端 UI 我同意 PM——简化呈现

**业务专家**：

参考企业培训实践：
- LinkedIn Learning toC：完成课程后无限期复习（无服务周期概念）
- 企业 LMS：通常"项目结束 = 课程关闭"，没有 servive 期概念
- 睿学的"二元模型"是**新设计**，市场没参照

担心：学员可能困惑"项目结束了，我现在还能做什么？"。**Neo 在项目结束当天的提示极其重要**——这是教育用户认知的关键节点。

### 1.3 挑战者反驳

**挑战者**：

我反驳"项目结束当天的仪式感"。

仪式感听起来好听，但实际效果可能反向——学员看到"项目完成"会**心理上画句号**，反而不再回来用平台。仪式感成了"再见"信号。

## § 3 · 议题 3 · 授课 7 题型 GUI

### 3.1 各角色第一轮发言

**PM**：

7 题型（单选/多选/填空/连线/判断/排序/问答）的 GUI 实现是 spec gap。当前 02-lecture-zone.md 提了"选择题/填空题/判断题在看板区作答；问答题在对话区"——只覆盖 4 种。

7 题型的看板区/对话区分配：

| 题型 | 看板区 | 对话区 |
|------|--------|--------|
| 单选 | ✅ 标准选项 | ❌ |
| 多选 | ✅ 复选框 | ❌ |
| 填空 | ✅ 输入框 | ❌ |
| 判断 | ✅ 是/否按钮 | ❌ |
| **连线** | ?（拖拽？）| ?（文字描述？）|
| **排序** | ?（拖拽？）| ?（文字描述？）|
| 问答 | ❌ | ✅ |

连线和排序在看板区是**典型 PC 拖拽交互**。chat UI 上可能很糟糕。

**UX**：

连线和排序是**触屏 / PC 都难做的题型**。
- 连线：左侧概念 / 右侧定义 → 学员在 mobile 上拖拽是噩梦
- 排序：5 个步骤的正确顺序 → mobile 拖拽也不好

UX 建议：
- 连线 / 排序 **降级为多选+填空**：连线 → "概念 X 对应哪个定义？A/B/C/D"；排序 → "下面 5 个步骤，第 1 步是？第 2 步是？..."（5 道连续选择）
- 真正的拖拽交互留 V2 支持（投资大但价值小）

**学员代表**：

我作为学员玩过 mobile 上的连线题——80% 都难用。降级为多选我能接受。

但排序是真有价值题型——"管理流程 5 步先后顺序"是认知考查。多选展开成 5 题太啰嗦。

**HR 代表**：

HR 不关心具体题型 GUI。

**业务专家**：

教育领域的"题型完整性" 不应该决定 GUI——而应该决定**学习目标的可考查性**。
- 单选 = 识别（lowest Bloom）
- 多选 + 填空 = 理解
- 判断 = 应用
- **连线 = 关系建立**（identification of relationships）—— 用多选可勉强替代
- **排序 = 程序性知识**（procedural knowledge）—— 多选很难替代

所以排序应该**保留拖拽**或换个表达方式（"哪个先发生？"逐步问）。

### 3.2 主持人识别冲突

**主持人**：

冲突点：
- PM/UX："连线 / 排序" 都降级为多选/填空
- 学员代表/业务专家：排序应保留（有教育价值）
- 连线降级共识，但排序有争议

挑战者反驳。

### 3.3 挑战者反驳

**挑战者**：

我反驳"7 题型完整性"。

为什么是 7？因为 0424 决策定的。但 0424 决策是基于功能列表，不是教学价值。**实际本期 8 Course（FLM 管理学）需要哪些题型？**
- 管理学很少考"5 步排序"——"5 步排序"是程序性知识题型，FLM 内容偏分析 + 应用，不偏程序
- 连线：FLM 也少（管理学 = 关系思考但不严格"对应连接"）

**真实需求 = 单选 + 多选 + 填空 + 问答 4 种就够 90% 场景**。判断 / 连线 / 排序 是冗余设计。

我建议**砍 3 题型**，本期只做 4 种。Spec 加注"7 种是上限，本期实际用 4 种"。

## § 4 · 议题 4 · 对练逃生通道 GUI（精简）

### 4.1 各角色发言（精简）

**PM**：50 题决策已定"对练学员连续 2 次'我不会'触发逃生通道"。GUI 上：

**UX**：方案合理。"Actor 暂停"动画可以用"灰度 + 慢动作"（视觉暗示"剧情暂停"，类似游戏的 pause 状态）

**挑战者**：

我反驳"逃生通道"概念本身。

对练核心价值是**练习 + 反思**。学员卡死 → 给 Neo 讲授 = **学员逃避**了练习。本质上等同于退出对练去看课件。

### 4.3 自由讨论

**PM 反驳挑战者**：

但学员卡死时**强行让他练 = 挫败感 + 流失**。教育心理学研究（Dweck 实验）显示挫败 + 没出口 = 学员放弃。

业务专家折中方案（1 次 Actor 提示 + 2 次 Neo 讲授）很好——平衡练习 vs 不挫败。

**业务专家**：

加更精确：
- 学员说"我不会"第 1 次 → Actor 在情境内给提示（"也许你可以试试 PIN 法的'P'部分先问问对方"）
- 学员说"我不会"第 2 次 → Neo 主动出声："要不要我暂停对练，跟你讲讲怎么用 PIN 法？"

**挑战者**：

## § 5 · 议题 5 · Ora 看板 GUI（精简）

### 5.1 各角色发言（精简）

**PM**：

轮 1 决定 Ora 本期"被动 review，不主动推送"。GUI 设计：
- HR / SE 看 Ora 看板时，如何"看出"项目异常？
- 没主动推送 = 没红色警示 / 没消息通知 = 异常隐藏在数据里

我建议看板 GUI 加"异常标识"（不主动推送，但**视觉上明显**）：
- 项目卡片左侧颜色条：绿（健康）/ 黄（注意）/ 红（异常）
- 红色项目自动排序到顶部（不推消息但视觉吸引）
- 点击红色项目 → 详情页显示具体异常原因（脱训 / 进度 / 情绪信号 等）

**UX**：

方案合理。"颜色条 + 自动排序"= 被动 review 但视觉引导。
- 颜色条用谨慎的"暖色 + 柔和"调（不是震耳欲聋的红，避免 HR 焦虑）
- 异常详情页要给"建议动作"（如"建议给学员 X 发催学消息"）—— 但**不替 SE 决定执行**

**学员代表**：学员不感知 Ora 看板 GUI。

**HR 代表**：

HR 视角 OK。但我担心：
- 红色项目排序到顶部 = 我每天看到的全是红色项目 → 焦虑
- 应该有"过滤"（隐藏已处理的红色项目）
- 红色 → 黄色 → 绿色 的状态变化 HR 看得到（"我处理了，状态变了"）

**业务专家**：

参考企业 SaaS 看板：HubSpot / Salesforce 都用"颜色 + 优先级排序"。被动 review 但视觉强引导是工业标准。

加一条：**Ora 自动生成"日报"** = 不是"主动推送消息"，是"被动呈现的总结报告"。每天 HR 登录看到一份"今日异常 5 条"——既是被动也是高效。

### 5.2 挑战者反驳

**挑战者**：

我反驳"颜色条 + 排序 = 被动 review"。

加颜色 + 自动排序 + 日报 = **本质上还是主动推送**，只是换了 UI 形态。DM 说"本期仅看板，不主动推送"——你们绕了一个弯。

如果真要"被动 review"，应该是：
- 看板上数据原样呈现，不加任何视觉强引导
- HR 自己看出来异常（这才是"被动"的真实形态）
- 要不要"看出"问题，是 HR 的责任

但这真的对 HR 友好吗？... 也许 DM 说的"被动 review"实际意思是介于两者之间。

### 5.3 自由讨论

**PM 反驳挑战者**：

挑战者点出了一个**词的边界**问题。"被动" vs "主动"是个连续光谱：
- 主动推送（消息 / 邮件）= 推（push）
- 颜色条 + 排序 = 显（display）
- 原始数据 = 静（static）

DM 反馈的"本期仅看板，不主动推送"——明显反对"消息推送"。但反对"颜色条 + 排序"吗？没明说。

这是个 **DM 互动问题**——我作为 PM 不能擅自定义"被动"。

**业务专家**：

参考工业实践 SaaS 看板都用"颜色 + 排序"，这是 UI 设计的基础。**完全不用颜色 = 看板可用性极差**。我相信 DM 反对的是 push（消息 / 邮件），不是 display（颜色 / 排序）。

**学员代表**：无关。

**HR 代表**：

HR 视角，颜色 + 排序 + 日报 = **我能干活**。完全不用颜色 = 我每天看一堆灰色数字猜哪个有问题。前者明显合理。

**挑战者**：

业务专家的"工业实践"反驳合理。我接受"颜色 + 排序 = display 是被动 review 的合理形态"。但**日报算不算 push** 边界模糊——这要 DM 拍。

### 5.4 主持人裁定（含 DM 待确认）

**主持人**：

议题 5 部分共识 + 待 DM 拍：

**初步结论**：
- ✅ 颜色条 + 自动排序 + 详情页"建议动作" → 都属 display，不属 push，本期可做
- ❓ Ora 自动生成"日报"是 push 还是 display？需 DM 拍
