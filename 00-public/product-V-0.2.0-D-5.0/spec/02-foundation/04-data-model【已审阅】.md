## 5 · 数据模型 AOM

> 本章从产品视角回答 3 个问题：① AOM 是啥（本质 / 价值 / 与产品的关系）② 演绎什么效果（学员 / 管理员看到什么、Neo 怎么演绎）③ 自适应机制（本期支持什么）。
>
> **章节边界**：本章描述产品视角与生产规则，**不锁定具体字段** —— AOM 字段不是固定 schema，会跟随业务演进调整。**不包含** KGP 内容生产规则与数据反馈循环（属内容侧自主管理 / 不需专门规范文档）。

### 5.1 AOM 是啥

#### 5.1.1 一句话定义

> **AOM** = Activity-Oriented Model（活动导向模型）—— **教学剧本 + 学习日记的数据骨架**：KGP 内容生产平台的产物，由 Neo 作为**演绎的文档提示词**遵从执行；同时作为学习日记的结构,记录学员实时学习历程。

#### 5.1.2 双重身份 · 数据 + 提示词

AOM 在文件形式上是 JSON，但在角色上有两层：

| 身份 | 用途 | 谁在执行 |
|------|------|---------|
| **数据** | 流程控制 / 内容播放 / 模式切换 | 平台代码（解析 + 渲染）|
| **提示词** | 教学设计意图 / AI 行为准则 / 通关判定 / 收尾话术 | AI 大模型（解释 + 演绎）|

> **核心洞察**：AOM 把"AI 行为脚本"写在内容数据里——**内容设计师用自然语言"编程"AI 的行为，平台无需为每种教学场景写代码**。这是"内容驱动平台"的真正含义。

**Neo 自由权衡 vs AOM 约束的边界**：

- **AOM 约束的部分**：教什么 / 演什么 / 怎么打标 / 怎么判通关 —— Neo 必须遵从
- **Neo 自由权衡的部分**：实时对话风格 / Voice / 立场判断 —— 来自 § 2 教学方法论 + § 3 Persona

> **AOM 决定 Neo 大部分行为** —— 大厅辅导的背景知识、授课的演绎流程、对练的剧本与复盘原则、小结的解读 context，全部从 AOM 中读取。

#### 5.1.3 双重用途 · 演绎脚本 + 学习旅程记录（两套独立文件）

AOM 在生命周期上不只是"生产时被读取"——它有两层用途，**对应两套独立的文件**：

| 文件类型 | 角色 | 谁生产 | 谁消费 | 跨学员共享 |
|---------|------|-------|--------|-----------|
| **AOM 模板**（演绎脚本）| 教学剧本 | KGP 生产 | Neo 按其教学；学员看到产品表现 | ✅ 共享（一套模板服务多个学员）|
| **AOM 学习实例**（学习旅程记录）| 单学员的实际学习轨迹 | 平台运行时为每个学员生成 | 大厅 Neo 召回记忆 / Ora 聚合分析 / 报告生成 / 反馈给 KGP | ❌ 每学员独立 |

**两套文件共用同一层级骨架**（Project / Module / Activity / SCO / asset），但**字段填充时机和填充方不同**：

| 层级位置 | AOM 模板里写什么 | AOM 学习实例里写什么 |
|---------|----------------|---------------------|
| Project | 项目背景 / 能力目标 / 学员画像假设 / 行业热词 | 学员真实画像 / 已学完 Module 列表 / 能力进度分布 |
| Module | Module 主题 / 关键概念图 / 教学目标 | 学员在该 Module 的完成度 / 卡点节点 |
| Activity | activityType / 学习目标 / 知识点 / SCO 流编排 | 学员实际完成的 SCO 序列 / 用时 / 触发的标签 |
| SCO | 教学意图 / AI 行为准则 / 通关条件 / 演示素材引用 | 学员答题记录 / 对话片段 / 行为观察 / 通关结果 |
| asset | PPT / 视频 / 剧本素材 + separateTag | 学员实际看到的 asset 版本（按标签匹配的结果）|

**关键关系**：

- **生产链路**：KGP → 写 AOM 模板 → 平台演绎
- **运行链路**：学员学习 → 平台写 AOM 学习实例 → 反馈到 Database
- **闭环**：AOM 学习实例的聚合数据反馈到 KGP → 优化 AOM 模板的下一个版本（数据反馈循环规则属 KGP 内容生产侧，已整章移出本 spec）

> 简单说：**AOM 模板是"教学说明书"，AOM 学习实例是"学员的学习日记"**——两者共用一套结构骨架，但是两份不同的文件。

### 5.2 5 层结构（产品视角速览）

```
Project / Program（培训项目，1 个 = 1 期培训如"基层管理者培训"）
  └── Module / Course（课程主题，对应一个大场景 / 大挑战 / 大授课主题）
       └── Activity（独立闭环教学活动，15-25 分钟）
            └── SCO（最小学习内容单元）
                 └── asset（演绎素材，本期支持多版本切换）
```

> **注**：asset = 演绎素材（历史名 Segment / v0.5.0 起统一改名）。

#### 5.2.1 术语映射 · 技术侧 vs 用户/内容侧

不同视角的命名约定：

| 5 层位置 | 技术侧（spec/design / 数据模型描述）| 用户侧 + 内容侧（用户场景 / KGP 沟通）|
|---------|-----------|--------------|
| 顶层 | **Project** | **Program**（即 NeoLearningProgram）|
| 2 层 | **Module** | **Course** |
| 3 层 | **Activity** | Activity（统一）|
| 4 层 | **SCO** | SCO（统一）|
| 底层 | **asset** | asset（统一）|

**用在哪**：

- **§ 5（数据模型）用技术侧术语**：Project / Module / asset
- **§ 4 / § 7 / § 8（用户场景）用用户侧术语**：Program / Course / asset
- Activity / SCO / asset 在两侧都统一使用，不混淆

#### 5.2.2 SCO ↔ asset 的对应关系

- **默认情况**：1 SCO ↔ 1 asset（无版本切换时）
- **版本切换场景**：1 SCO ↔ N assets（每个 asset 不同 separateTag，按学员标签选其一）
- **不会出现**：1 SCO 下挂多个相同 separateTag 的 asset（这是 KGP 内容生产的规则）

#### 5.2.3 各层产品意义（一句话）

| 层 | 产品意义 |
|----|---------|
| **Project** | "这一期培训是给谁的、教什么领域"——决定 Neo 整体语气 / 行业用词 / 大厅辅导知识体系 / **教练背景知识**（§ 5.3.1 领域知识工具库 + 辅导场景库 + 案例库 / 在 Project 级提供）|
| **Module** | "这门 Course 解决一个什么大主题"——决定 Module 内 Activity 的连贯性 / 跨 Activity 召回 |
| **Activity** | "学员一次完整的学习单元"——决定本次学习场域类型（授课 / 对练 / 小结）+ 学习目标 |
| **SCO** | "Neo 一次具体的交互"——决定 Neo 这一刻讲什么 / 问什么 / 演示什么 |
| **asset** | "同一个 SCO 的不同呈现版本"——支持千人千面（不同标签学员看不同案例）|

#### 5.2.4 SCO 类型清单（本期 9 类 · 按场域分组）

承上：SCO 是 § 5.2 5 层结构的第 4 层（学员一次具体的交互单元），本期支持 9 类 SCO 类型，**按场域分组**——lecture 内 SCO 类型 4 类（含 FEEDBACK 双头配对）/ practice 内 SCO 类型对应 4 阶段 / recap 内 SCO 类型 1 类：

| SCO 类型 | 一句话定义 | 场域 |
|---------|-----------|------|
| **SLICE** | PPT / HTML 知识切片演绎（图文 + Neo 口播） | lecture |
| **VIDEO** | 动画 / 案例视频演绎（音视频 + Neo 衔接） | lecture |
| **QUIZ** | 互动测验（选择 / 填空 / 自由作答 + 通关条件） | lecture |
| **FEEDBACK_COLLECT + FEEDBACK_REVIEW**（双头配对）| 采集 / 复盘对话（开放式问答 + 写入个人记忆）—— 双头配对：collect 在前采集场景 / review 在后复盘 / 同一 memory_id 串联 | lecture |
| **PRACTICE_INTRO** | 对练导入（场景设定 / Actor 人设 / 任务说明） | practice |
| **PRACTICE_DRILL** | 对练演练（Actor 拟真对话 + Neo 旁路答疑） | practice |
| **PRACTICE_REVIEW** | 对练复盘（Neo 苏格拉底追问 + 必给总结） | practice |
| **PRACTICE_REPORT** | 对练报告（评分 + 高光 + 提升建议 + Neo 解读） | practice |
| **RECAP_DIALOGUE** | recap Walk-through（Neo 引导 + 报告 + 自由复习） | recap |

> **不做独立 ASSESSMENT_TAG SCO 类型**（v0.4 设计取消）——打标融入互动 SCO 的通关条件 / passConditions，学员无感。

### 5.3 演绎什么效果 · AOM 驱动的产品表现

> 本节说清楚：**KGP 在 AOM 中提供什么内容**，**AOM 内容如何驱动 5 个场域的产品表现**。
>
> **章节边界**：本节仅列 KGP 在 AOM 里写什么内容素材——**不写** Neo 能力（如关键词检索 / 推荐 / 答疑话术 / 复盘追问）/ 关联 Activity 上下文 / 产品形态（详见 § 9 学员端各章）。

#### 5.3.1 大厅辅导（Project 级 + Module 级 context）

**KGP 在 AOM 中提供的内容素材**（Project 级）：

| 内容素材 | 说明 |
|---------|------|
| **领域知识工具库** | 工具 / 框架清单（如基层管理主题：冰山模型 / GROW 教练法 / RACI 矩阵 / SMART 目标 等）|
| **辅导场景库** | 学员可能遇到的工作场景 + 对应辅导策略 |
| **案例库** | 按能力维度组织的真实场景案例 |

**驱动产品表现**：

- 学员在大厅向 Neo 提工作问题（"我下属总不主动汇报，怎么办？"）
- Neo 基于 AOM 知识工具库 + 场景库 + 案例库生成辅导

> 检索映射 / 推荐 Activity / 自适应导航 等 = Neo 能力，不是 AOM 内容。

#### 5.3.2 授课演绎（lecture Activity）

**KGP 在 AOM 中提供的内容素材**：

| 内容素材 | 说明 |
|---------|------|
| **流程脚本** | SCO 序列编排 |
| **演示素材** | PPT 图片 / 动画视频 / 关联资源 |
| **Quiz 通关条件** | passCondition + solution 字段（灵活运用承担考核 / 思考引发 / 反思激起 等多种用途）|
| **前端演示事件** | PPT 关键点高亮 / 进度提示 / 动画触发等演示事件配置 |

**驱动产品表现**：

- Neo 按 SCO 流逐步讲授，PPT 配合解说 / 案例视频 / 互动 Quiz

> 答疑风格 / 学员超纲提问处理 = Neo 能力，不是 AOM 内容。

#### 5.3.3 对练演绎（practice Activity）

**对练 Activity = 4 阶段**（导入 / 演练 / 复盘 / 报告，KGP 可任意 1-4 组合）。各阶段 KGP 在 AOM 中提供：

| 阶段 | KGP 在 AOM 中提供 |
|-----|------------------|
| **导入** | 场景文案 / Actor 人设 / 学员任务 / 知识点回顾 |
| **演练** | **Actor 人设 + 演绎逻辑**（触发条件 / 软化点 / 底线 / 情绪天花板）+ **成功标准**（注：**没有 Actor 对白库**——Actor 对白由 Neo / Director 实时生成，AOM 仅给人设 + 演绎逻辑）|
| **复盘** | **必问点**（KGP 标定的复盘关键点位）|
| **报告** | **评分维度**（competencyGoals）+ **评分阶梯**（skill_ladders / 技能阶梯）|

**驱动产品表现**：

- 学员与 Actor 拟真对话，Neo 旁观介入，Director 后台编排

> 状态矩阵 / 三角色协作 / Neo 不能做 5 条 / 复盘追问话术 = Neo 能力或产品形态，不是 AOM 内容。

#### 5.3.4 小结演绎（recap Activity）

**recap Activity = 单一对话场景**（不分 SCO，详见 § 9.3.3）。KGP 在 AOM 中**仅控制两件事**：

| KGP 可控项 | 内容 |
|----------|------|
| **placement** | recap Activity 在 AOM 中的位置 |
| **关联 Activity 范围** | 哪几个前序 Activity 作为 recap 数据源 |

**驱动产品表现**：

- Neo 在首聊态走完 Walk-through 后基于关联 Activity 数据综合生成报告
- 报告 5 块结构（内容锚点 / 走过的轨迹 / 6 维画像 / 关键证据 / 待办任务）+ 自由复习层 = 产品形态，详见 § 9.3.3

> Course 主题 / 关键概念清单 = 关联 Activity 已携带的上下文，不需 AOM 在 recap 节点单独提供。

#### 5.3.5 未来扩展 Activity 类型

以下 Activity 类型本期不实现，列为未来扩展：

| 未来 Activity 类型 | 产品含义 | 优先级 |
|-------------------|---------|--------|
| **inquiry**（调研）| 学员深度自我画像与反馈采集（对话访谈 / 量表）| 未来 |
| **assignment**（作业）| 学员独立完成的应用任务 · **未来优先做"案例拆解"作为第一种作业类型** | 未来 |
| **sandbox**（沙盘）| 沉浸式模拟场景的多人 / 多回合演练 | 更远未来 |

> 历史 v0.4.0 文档中 inquiry / assessment 出现作为 P3 优先级——v0.5.0 明确**不在本期 scope 内**。

### 5.4 自适应机制

> **本节 scope 严格界定**：本期支持什么 / 不支持什么 / 未来支持什么。

#### 5.4.1 自适应分层（L1-L4 全景）

| 层级 | 含义 | 本期实现 |
|------|------|---------|
| **L1** | 每次对话交互 LLM 实时调整（Neo 自由权衡）| ✅ 完整支持 |
| **L2** | 学员标签 + asset 多版本切换 | ✅ 本期支持（**仅在 lecture 实现**）|
| **L3** | Activity 顺序动态调整 | 🟡 **本期部分支持**——大厅辅导可推荐学员先上某个 Activity（轻量调整）；完整 L3（基于学员表现自动推断学习路径）= 未来 |
| **L4** | 动态内容生成 | ⏳ 未来 |

#### 5.4.2 本期支持

##### 1）asset 多版本切换

同一个 SCO 的演绎 asset（PPT / 视频）可以有多个版本，按学员标签切换。

**实际示例**（横向协作 PIN 法 SCO 5）：

- 课前互动诊断 → 学员被打上标签（"防御拖延型" / "利益博弈型" / "情绪对抗型"）
- 后续的"视频案例与解析" SCO → 同一位置 3 个 asset 版本：
  - 防御拖延型 → 看《迟到的数据报告》
  - 利益博弈型 → 看《争夺的预算池》
  - 情绪对抗型 → 看《越界的临时需求》

##### 2）大厅辅导推荐 Activity

Neo 在大厅可以根据学员状态 / 提问 / 进度，**推荐学员先上某个 Activity**。

#### 5.4.3 本期不支持与未来计划

| 项 | 本期 | 未来计划 |
|----|------|---------|
| **SCO 多版本切换** | ❌ 不做（同一 Activity 内不能有"高难版 SCO" vs "低难版 SCO"） | ✓ 计划 |
| **必修 / 选修编排** | ❌ 不做（所有 Activity 顺序固定，学员不可自选路径）| ✓ 计划 |
| **动态生成内容** | ❌ 不做（AOM 是 KGP 提前生产，平台不动态生成新内容）| ✓ L4 |
| **L3 自适应**（Activity 顺序动态调整）| 🟡 部分支持（仅大厅辅导推荐，详见 § 5.4.2） | ✓ 完整 L3 计划 |
| **大厅辅导知识体系跨 Project 可插拔** | ❌ 不做 | ✓ 计划（学员从基层管理 Course Pack 切到销售 Course Pack 时知识体系自动切换——参考 coaching-skills 项目的 KNOWLEDGE_TOOLS 知识工具调用模式）|

### 5.5 Course Pack 清单 + 未来可插拔

#### 5.5.1 当前 demo Course 清单（暂用 · 待定）

按"基层管理者"主题暂列 8 个 Course：

| # | Course 名 | 一句话介绍 | 关键工具 / 方法 |

| 7 | **向上影响** | 与上级的有效沟通和资源争取——结构化向上汇报、坏消息怎么报、寻求支持的话术 | 向上汇报结构 / 坏消息汇报法 / 资源支持沟通 |
| 8 | **过程管控** | 团队工作过程的监控与调整——从"不知道下属在做什么"到"清晰节奏 + 可控风险" | 1on1 面谈 / 3P 会议 / 任务跟踪 / 绩效面谈 |
