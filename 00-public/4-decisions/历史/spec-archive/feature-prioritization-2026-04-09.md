## 2. Must Have — MVP（2个月内交付）

> **目标**: 基于"基层管理者·横向协同"主题完成可售卖的完整培训项目  
> **交付日期**: 6月初  
> **核心逻辑**: 学员端覆盖完整 AI 动作链（授课→对练→辅导→测评→萃取→报告），管理端实现项目运营和效果举证闭环

### 2.1 学员端功能

#### M-01: AI授课（交互式讲解）

| 维度 | 说明 |
|------|------|
| **描述** | 基于 KGP 内容的交互式 AI 讲解，替代传统视频课的知识传递环节。支持多轮对话式教学，学员可随时提问、要求举例或换种方式讲解 |
| **依赖** | KGP 内容生产（外部依赖）；LLM 基础设施 |
| **工作量** | **S** — 已有 demo，需产品化打磨交互体验和内容适配 |
| **构建顺序** | Sprint 1（第1优先） |
| **优先级理由** | 授课是整个学习链路的起点，没有知识输入就没有后续的练习、测评、萃取。已有 demo 基础，产品化成本低 [Data] |

#### M-02: AI对练（模拟场景演练）

| 维度 | 说明 |
|------|------|
| **描述** | AI 扮演特定角色（同事、下属、客户等），与学员进行情境化对话演练。支持横向协同场景中的沟通技能练习，提供实时反馈和改进建议 |
| **依赖** | M-01 AI授课（学员需先学习基础知识后才能对练） |
| **工作量** | **S** — 模块已有，需场景化适配和评分机制完善 |
| **构建顺序** | Sprint 1（第1优先，可与授课并行） |
| **优先级理由** | 对练是产出 Kirkpatrick L3 行为数据的核心模块。82%企业仅做到满意度评估 [Data]，AI对练产出的行为数据是"效果举证"的关键武器。已有模块，改造成本低 [Data] |

#### M-03: AI辅导（个性化指导）

| 维度 | 说明 |
|------|------|
| **描述** | 基于学员在授课和对练中暴露的弱点，AI 提供针对性的一对一辅导。包括知识点深入讲解、错误模式分析、改进策略建议 |
| **依赖** | M-01 AI授课、M-02 AI对练（需要学员行为数据作为辅导输入） |
| **工作量** | **M** — 需要设计辅导策略引擎和个性化推荐逻辑 |
| **构建顺序** | Sprint 2 |
| **优先级理由** | 辅导是自适应学习的核心体现——"千人千面"而非"千人一面"。63%员工认为培训内容与工作需求脱节 [Data]，AI辅导直接解决这个问题 |

#### M-04: AI测评（能力评估）

| 维度 | 说明 |
|------|------|
| **描述** | 多维度能力评估，包括知识掌握度测试、情境判断测试、对练表现评分。产出可量化的能力指标，支持前测/后测对比 |
| **依赖** | M-01 AI授课、M-02 AI对练（评估需要覆盖授课和对练中的能力维度） |
| **工作量** | **M** — 需要设计评估模型、出题策略和评分标准 |
| **构建顺序** | Sprint 2（与AI辅导并行） |
| **优先级理由** | 测评是效果举证链路的数据基础。没有前测后测的对比数据，"效果可量化"就是空话。直接支撑 HR 的政治生存需求：用数据说话 [Data] |

#### M-05: AI报告（学习成果报告）

| 维度 | 说明 |
|------|------|
| **描述** | 自动生成面向学员的个人学习报告，包括能力雷达图、学习轨迹、薄弱点分析、改进建议。是学员侧的学习闭环 |
| **依赖** | M-04 AI测评（需要测评数据作为报告素材）；M-02 AI对练（行为数据） |
| **工作量** | **M** — 需要设计报告模板、数据聚合逻辑和可视化组件 |
| **构建顺序** | Sprint 3 |
| **优先级理由** | 学员报告是"学习闭环"的最后一环，让学员看到自己的成长轨迹。51%员工对培训失去信心 [Data]，可视化的成长证据是重建信心的关键 |

#### M-06: AI萃取（作业/知识提炼）

| 维度 | 说明 |
|------|------|
| **描述** | AI 引导学员将所学知识与自身工作场景结合，完成结构化的知识萃取和行动计划输出。支持文字、语音等多种输入方式 |
| **依赖** | M-01 AI授课（需要课程内容框架作为萃取引导） |
| **工作量** | **M** — 需要设计萃取引导流程和结构化输出模板 |
| **构建顺序** | Sprint 2（与辅导/测评并行） |
| **优先级理由** | 萃取是知识迁移的关键环节——"学以致用"的桥梁。培训迁移率平均仅10%-20% [Data]，AI萃取通过引导式输出促进知识内化 |

#### M-07: 自适应（L1+L2 产品化）

| 维度 | 说明 |
|------|------|
| **描述** | L1交互级：LLM 实时响应用户输入，动态调整讲解深度和方式。L2 SCO级：基于 Quiz 结果进行分支判定，为不同水平学员推送不同版本的学习内容（多版本 SCO + tag 机制） |
| **依赖** | M-01 AI授课、M-04 AI测评（L2 分支依赖测评/Quiz 结果） |
| **工作量** | **L** — L1 已实现；L2 目前是 hack 实现，产品化需要设计通用的分支编排机制 [Data] |
| **构建顺序** | Sprint 2-3（贯穿，L1 在 Sprint 1 已天然包含，L2 产品化在 Sprint 2-3） |
| **优先级理由** | 自适应是 AI 老师区别于视频课的核心差异化。"千人一面"是客户高频痛点词 [Data]。L2 产品化是 MVP 可售卖的底线——没有自适应就只是"换了皮的视频课" [Opinion] |

### 2.2 管理运营端功能

#### M-08: 运营工具

| 维度 | 说明 |
|------|------|
| **描述** | 培训项目运营管理工具集，包括：项目创建与配置、学员分组与分配、自动催学（替代微信群手动催促）、进度追踪、学习提醒 |
| **依赖** | 学员端基础功能（M-01 ~ M-06，需要有学习数据可管理） |
| **工作量** | **L** — 功能面广，涉及项目管理、学员管理、自动化催学等多个子模块 |
| **构建顺序** | Sprint 2-3（与学员端功能并行推进） |
| **优先级理由** | 培训运营人员是产品的日常使用者和最强内部推动者。"每个月十几个项目在跑，光催学就占了一半工作时间" [Assumption]。运营工具直接解决其核心 JTBD：自动化催学、减少手动操作 |

#### M-09: 报告解读 Agent

| 维度 | 说明 |
|------|------|
| **描述** | 面向培训管理员的 AI Agent，可以用自然语言查询和解读培训数据。例如："这批学员最弱的能力维度是什么？""跟上一批比有什么变化？"替代手动导出 Excel 分析的流程 |
| **依赖** | M-05 AI报告（学员端报告数据）；M-11 Dashboard（数据基础设施） |
| **工作量** | **M** — LLM Agent 开发，核心是数据查询和自然语言解读能力 |
| **构建顺序** | Sprint 3 |
| **优先级理由** | 直接解决 HR "效果不可证"的致命痛点（痛点排名第1 [Data]）。HR 需要的不只是数据，是**可直接用于向管理层汇报的解读**。"数据统计全靠手工：从平台导出Excel → 手动汇总 → 制作PPT" [Data]——报告解读 Agent 一键替代这个痛苦流程 |

#### M-10: Dashboard（管理看板）

| 维度 | 说明 |
|------|------|
| **描述** | 培训项目管理看板，包括：项目整体进度概览、学员完成率/活跃度、能力分布热力图、前后测对比、异常预警（如：长时间未登录学员） |
| **依赖** | 学员端功能（需要产出学习数据）；数据聚合层 |
| **工作量** | **M** — 数据可视化组件开发，需要设计数据聚合逻辑和展示层 |
| **构建顺序** | Sprint 2-3 |
| **优先级理由** | Dashboard 是培训运营人员的日常工作台，也是 HR 向上汇报的数据来源。没有 Dashboard，运营人员就回到"手动导 Excel"的老路 [Data] |

#### M-11: Harness 报告生成

| 维度 | 说明 |
|------|------|
| **描述** | 基于 Harness 工程理念，根据管理员需求定制化生成项目结项报告。支持模板选择、数据维度配置、自动图表生成，输出可直接用于向业务部门汇报的结项报告 |
| **依赖** | M-10 Dashboard（数据来源）；M-09 报告解读 Agent（解读能力复用） |
| **工作量** | **M** — 报告模板系统 + 数据聚合 + PDF/PPT 输出 |
| **构建顺序** | Sprint 3-4 |
| **优先级理由** | 结项报告是培训项目的**最终交付物**，是 HR 向业务部门和管理层证明价值的直接武器。64%企业因"效果不可视"被砍预算 [Data]——Harness 报告让效果可视化 |

## 3. Should Have — v1.1（第3-4个月）

> **目标**: 扩展交付渠道和自适应能力，为规模化销售做好准备  
> **交付日期**: 8月  
> **核心逻辑**: MVP 验证后，解决客户选型准入门槛（企微集成、移动端）和产品深度（L3自适应、高级分析）

#### S-01: 移动端适配优化

| 维度 | 说明 |
|------|------|
| **描述** | 学员端全面移动端适配，确保在手机/平板上的 AI 授课、对练、测评等核心体验流畅。支持碎片化学习场景 |
| **依赖** | MVP 学员端功能全部完成 |
| **工作量** | **M** — 响应式适配 + 移动端交互优化（如语音输入、触控操作） |
| **构建顺序** | Sprint 5 |
| **优先级理由** | 7月中旬里程碑要求学员端移动端交付 [Data]。学员数字素养高，日常使用微信、抖音、B站 [Estimate]，移动端是自然使用场景。企业微信是核心学习入口 [Data]，移动端不适配 = 体验断裂 |

#### S-02: 企业微信深度集成

| 维度 | 说明 |
|------|------|
| **描述** | 与企业微信深度集成：消息推送/催学提醒、应用内嵌入学习页面、单点登录、学习进度同步到企微工作台 |
| **依赖** | S-01 移动端适配（企微场景本质就是移动端）；M-08 运营工具（催学消息触发） |
| **工作量** | **L** — 企微开放平台 API 对接、应用审核、消息模板、SSO |
| **构建顺序** | Sprint 5-6 |
| **优先级理由** | 企业微信/钉钉集成在客户选型标准中排名第4 [Data]，是大客户的**准入门槛**。"无需跳出即可完成学习" [Data]——如果要求学员打开独立 App 或网页，使用率会断崖式下跌 [Opinion] |

#### S-03: L3 Activity 级自适应编排

| 维度 | 说明 |
|------|------|
| **描述** | Activity 级别的智能编排：根据学员的整体学习表现，AI 自动调整后续的学习活动序列（如：跳过已掌握模块、增加薄弱模块的练习、推荐补充内容） |
| **依赖** | M-07 自适应 L1+L2（L3 是在 L2 基础上的升级）；M-04 AI测评（需要能力评估数据驱动编排） |
| **工作量** | **XL** — 全新能力，需要设计 Activity 编排引擎、决策规则、学习路径可视化 |
| **构建顺序** | Sprint 6-7 |
| **优先级理由** | L3 是 AI 老师自适应能力的"皇冠"——真正的个性化学习路径。但 MVP 阶段 L1+L2 已经足够差异化，L3 是锦上添花。brainstorm 中已标注"L3是全新探索，可能在2个月内无法达到可售卖标准" [Opinion]，放到 v1.1 更稳妥 |

#### S-04: 增强型 Dashboard 分析

| 维度 | 说明 |
|------|------|
| **描述** | Dashboard 能力升级：跨项目对比分析、学员群体画像、能力趋势预测、自定义报表、数据导出（Excel/PDF） |
| **依赖** | M-10 Dashboard（基础版）；多个项目运行后产生的数据积累 |
| **工作量** | **M** — 在基础 Dashboard 上叠加高级分析和自定义能力 |
| **构建顺序** | Sprint 6 |
| **优先级理由** | MVP Dashboard 满足基本运营需求，但规模化后客户需要跨项目视角和趋势分析。增强分析能力直接提升 HR "用数据说话"的深度 |

## 4. Could Have — H2（下半年）

> **目标**: 产品线扩展（Neo Course 规模化、AI组织培训）和运营效率提升  
> **核心逻辑**: 在 NeoLearning 标杆客户验证后，启动规模化产品线

#### C-01: Neo Course 平台功能

| 维度 | 说明 |
|------|------|
| **描述** | 帮助企业将已有视频课快速 AI 化的生产平台。包括：视频课程上传 → AI 自动分析 → 生成互动检测(Check/Quiz) → 智能答疑 → Agent 学习报告。首批升级4门自评课 |
| **依赖** | MVP 核心 AI 能力（授课、测评、报告）作为底层引擎 |
| **工作量** | **XL** — 独立产品线，涉及视频解析、内容自动生成、平台化架构 |
| **构建顺序** | Sprint 8-10 |
| **优先级理由** | Neo Course 是规模化增长引擎（目标200-300份，贡献600-900万 [Data]），但依赖 NeoLearning 先验证核心 AI 能力。7月底上线目标 [Data] |

#### C-02: AI组织培训内容

| 维度 | 说明 |
|------|------|
| **描述** | 帮助企业适应"人与AI协同"新工作模式的标准化培训内容。包括：AI 工具使用、AI 协作思维、AI 安全与合规等课程模块 |
| **依赖** | MVP 平台能力；AI组织培训的课程内容 KGP 生产 |
| **工作量** | **L** — 平台能力复用，主要工作量在内容生产和课程设计 |
| **构建顺序** | Sprint 7-9 |
| **优先级理由** | 目标100家企业 x 50账号 x 800-1,000元/账号 = 500万收入 [Data]。6月底 Online MVP 目标 [Data]。市场需求明确：33.3%企业处于AI转型观望中 [Data] |

#### C-03: 批量操作

| 维度 | 说明 |
|------|------|
| **描述** | 管理端批量操作能力：批量创建项目、批量导入/分配学员、批量催学、批量生成报告、批量数据导出 |
| **依赖** | M-08 运营工具（基础运营能力）；M-11 Harness 报告生成 |
| **工作量** | **M** — 在现有功能基础上增加批量处理层 |
| **构建顺序** | Sprint 8 |
| **优先级理由** | 单项目运营 MVP 足够，但当客户同时运行多个培训项目时（"每个月十几个项目同时跑" [Assumption]），批量操作变为刚需。属于规模化运营效率问题 |

#### C-04: 高级分析

| 维度 | 说明 |
|------|------|
| **描述** | 深度分析能力：学员能力成长预测模型、培训ROI量化计算、组织能力热力图、与业务指标（如销售业绩、客户满意度）的关联分析 |
| **依赖** | S-04 增强型 Dashboard；足够的数据积累（至少3-6个月运营数据） |
| **工作量** | **XL** — 需要数据科学/ML 能力，以及与客户业务系统的数据打通 |
| **构建顺序** | Sprint 10+ |
| **优先级理由** | 高级分析是 Kirkpatrick L4（业务成果层）的终极实现，仅25.6%企业能做到 [Data]。但需要足够数据积累，且需与客户业务系统集成，复杂度极高。放到 H2 后期或更晚 [Opinion] |

#### W-02: 完全自助平台

| 维度 | 说明 |
|------|------|

## 6. 推荐构建顺序

### Sprint 规划总览

> 说明：每个 Sprint 为2周。以下规划基于 vibe coding 的 3-5x 开发效率假设。

```
Sprint 1（第1-2周）  ▎基础交互层 — 学员能"学"和"练"
Sprint 2（第3-4周）  ▎深度学习层 — 学员能"被辅导"、"被评估"、"做萃取"
Sprint 3（第5-6周）  ▎闭环+管理 — 报告闭环 + 管理端核心
Sprint 4（第7-8周）  ▎打磨+交付 — L2产品化 + Harness报告 + 联调测试
━━━━━━━━━━━━ MVP 交付线（6月初）━━━━━━━━━━━━
Sprint 5（第9-10周） ▎移动端 + 企微集成启动
Sprint 6（第11-12周）▎企微集成完成 + 增强Dashboard + L3启动
Sprint 7（第13-16周）▎L3 Activity编排 + AI组织培训内容
━━━━━━━━━━━━ v1.1 交付线（8月）━━━━━━━━━━━━
Sprint 8-10（H2）    ▎Neo Course平台 + 批量操作 + 高级分析
```

### Sprint 详细计划

#### Sprint 1（第1-2周）— 基础交互层

| 功能 | 工作量 | 交付物 |
|------|--------|--------|
| M-01 AI授课 | S | 可交互的 AI 讲解模块（横向协同主题） |
| M-02 AI对练 | S | 场景化对练模块（横向协同沟通场景） |
| 基础数据模型搭建 | M | Project → Course → Activity → SCO 数据结构 |

**Sprint 1 验收标准**: 学员可以完成"听AI讲 → 跟AI练"的基本学习流程。

#### Sprint 2（第3-4周）— 深度学习层

| 功能 | 工作量 | 交付物 |
|------|--------|--------|
| M-03 AI辅导 | M | 基于学习弱点的个性化辅导模块 |
| M-04 AI测评 | M | 前测+后测+过程评估能力 |
| M-06 AI萃取 | M | 结构化知识萃取和行动计划输出 |

| M-08 运营工具（启动） | L（部分） | 项目创建、学员分配、自动催学 |

**Sprint 2 验收标准**: 学员可以完成"学→练→被辅导→被评估→做萃取"的完整学习动作链。

#### Sprint 3（第5-6周）— 闭环 + 管理端

| 功能 | 工作量 | 交付物 |
|------|--------|--------|
| M-05 AI报告 | M | 学员个人学习报告（能力雷达图、学习轨迹） |
| M-09 报告解读 Agent | M | 管理员可用自然语言查询培训数据 |
| M-10 Dashboard | M | 项目进度概览、学员完成率、能力分布 |
| M-08 运营工具（完成） | L（部分） | 进度追踪、学习提醒、异常预警 |

**Sprint 3 验收标准**: 学员有完整的学习报告；管理员有 Dashboard 和 Agent 可以查看和解读数据。

#### Sprint 4（第7-8周）— 打磨 + 交付

| 功能 | 工作量 | 交付物 |

| M-07 自适应 L2（完成） | L（部分） | 通用分支编排机制产品化 |
| M-11 Harness 报告生成 | M | 可定制化的项目结项报告 |
| 全链路联调测试 | M | 完整的"横向协同"项目可走通 |
| 天使客户内测 | — | 60家天使客户中选取首批体验 |

**Sprint 4 验收标准**: "基层管理者·横向协同"培训项目可完整交付，包括学员端全流程 + 管理端运营和报告。可进入天使客户内测。

## 8. 风险与缓解

| 风险 | 影响 | 概率 | 缓解措施 | 标签 |
|------|------|------|----------|------|
| KGP 内容生产阻塞 | 所有学员端功能无内容可用 | 高 | KGP 与产品开发并行推进；先用"横向协同"单一主题验证，降低内容范围 | [Data] |
| L2 产品化比预期复杂 | 自适应能力在 MVP 中缺失 | 中 | L2 先做最小可用版本（2-3个分支路径），后续迭代完善通用编排 | [Opinion] |
| 2个月内 11 个功能 MVP 交付风险 | 质量不达标或延期 | 中高 | vibe coding 3-5x 效率是核心信心来源；但需严格控制每个功能的 MVP 范围，避免过度设计 | [Opinion] |
| 企微集成审核周期不可控 | v1.1 延期 | 中 | 提前提交企微应用审核；MVP 阶段先用 H5 链接方式 | [Estimate] |
| Neo 产品形态未定导致连锁阻塞 | KGP 方法论无法确定，影响内容生产 | 高 | 本文档的 MVP 范围定义即为 Neo 产品形态的第一刀——先锁定 AI 项目教练形态 | [Data] |

## Flags

**Red Flags:**
- 2个月内交付 11 个 Must Have 功能（学员端7个 + 管理端4个），即使有 vibe coding 加持，范围仍然极大。建议每个功能严格定义 MVP 最小范围，避免功能蔓延 [Opinion]
- KGP 内容生产是学员端所有功能的前置依赖，如果内容不就绪，产品开发就是空转 [Data]
- Neo 产品形态未定 → KGP 阻塞的连锁反应仍未解除，本文档的 MVP 定义需要尽快与创始团队对齐确认 [Data]

**Yellow Flags:**
- L2 自适应从 hack 到产品化的工作量可能被低估（当前是 Quiz SCO 后硬编码 tag 判定 [Data]，通用化需要重新设计） [Opinion]
- Harness 报告生成依赖链最长（需要 Dashboard + 报告 Agent + 学员数据全部就绪），是最容易延期的功能 [Opinion]
- 企微集成涉及第三方平台审核流程，时间线不完全可控 [Estimate]
- AI组织培训（C-02）的6月底 MVP 目标 [Data] 与本文档将其列为 Could Have 存在冲突，需与商业化团队确认优先级 [Opinion]
