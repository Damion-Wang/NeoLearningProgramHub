### 9.3.3 recap（小结）

> **总览**：recap 是 Course 内的"反思镜场域" —— 学员可在 Course 学习中**随时进入**查看 Neo 对自己学习的当前判断 + 与 Neo 对话挖掘洞察。报告以"前序 Activity 完成"为单位**增量更新**，Neo 根据上次会话与新数据**动态切换行为态**（首聊 / 回访 / 复习），首聊态走完 Walk-through 后由 Neo 一次性产出应用计划自动加入大厅待办清单。
>
> 本节按 **业务定义 / 形态 / 对话场景 / 报告与数据 / Neo 行为 / 学员行为 / 关键约束 / 完成与跨场域 / 本期不做** 9 个维度展开。
>
> **与 lecture / practice 8 子节结构差异**：① 多出 § 9.3.3.4 一整节专写"报告"——报告是 recap 的核心交付物 ② **取消 SCO 编排**改为"对话场景编排"——recap 是融合的对话场景，不是结构化的内容序列 ③ **Neo 行为按三态切换**（首聊 / 回访 / 复习）—— 多次进入 + 报告会变是 recap 独有 ④ **整个 Activity 入口可随时进入**，与 lecture / practice 严格 SCO 顺序锁定不同 ⑤ 完成判定有 milestone 概念 + "数据更新不退回未完成"。

#### 9.3.3.1 业务定义

##### 9.3.3.1.1 一句话定位

> recap（小结）是学员在 Course 学习中**随时可入**的反思场域 —— 通过查看 Neo 给出的当前快照报告 + 与 Neo 对话挖掘洞察，把"做过什么"沉淀为"看见什么、要做什么"。

recap 同时承担 **三重价值**：

| 价值层次 | 学员视角 | 项目视角 |
|---------|---------|---------|
| **反思整合** | 把 Course 内零散的 lecture / practice 体验整合为一份当下的"我看见自己" | 关键交付能力——让"学过"沉淀为"内化" |
| **轨迹镜像** | 看到 Neo 对自己跨场域的综合判断（6 维画像）—— 自己看不到的视角 | 沉淀可举证的学员成长轨迹 |
| **行动转向** | 在 Neo 的引导下，从反思转向具体下一步行动（应用计划自动入大厅待办）| 让学习闭环到工作场景，避免学完即忘 |

##### 9.3.3.1.2 三场域差异 + 与 practice 复盘的边界

| 场域 | 学员主任务 | 粒度 |
|------|----------|------|
| **lecture** | 概念吸收（输入）| SCO 级 |
| **practice** | 行为养成（输出）| Activity 级，含 Activity 内复盘阶段 |
| **recap** | 反思整合 + 行动转向 | 跨多个关联 Activity 的反思整合 |

**与 practice Activity 内复盘阶段的边界**：

- **practice 复盘**（详见 § 9.3.2.3.4）= 单次对练后的行为复盘——针对那一次演练的具体决策点
- **recap** = 跨多个关联 Activity 的反思整合——综合 lecture 学习行为 + practice 表现 + Neo 跨场域观察，呈现学员当前的整体状态

两者不重复，是 Activity 级 vs Course 级两层叠加。

**内容反思 vs 项目进度边界（★）**：

recap 是**内容维度**的反思场，**不承担项目进度仪表盘职能**——

| 属于 recap | 不属于 recap |
|-----------|------------|
| 学员开营时的目标 + 本 Course 主题的关联 | 距离项目结营还有几周 / 时间轴位置 |
| 学员在不同 Activity 的互动行为 / 投入度 | "已完成 X/Y 个 Activity"项目进度数字 |
| Neo 在关联 Activity 内对学员 6 维画像的判断 + 改变 | 全项目 6 维画像的差分对比（缺乏"为什么变"的叙事支撑）|
| 关联 Activity 内的具体高光与卡点片段 | Quiz 命中率 / 通关率（Quiz 不一定有对错，详见 § 9.3.1.3.4）|
| Neo 基于学员真实状态的师长式留言 | 项目阶段性 KPI / 业绩指标 |

**项目进度类信息由大厅 § 9.2 承担**（项目总览区 / 时间轴 / 进度数字 / 待办清单），recap 不重复也不交叉——这是 recap 与大厅的核心边界。

##### 9.3.3.1.3 "随时打开 + 增量更新"机制（★）

recap 区别于 lecture / practice 的关键设计：

- **随时打开**：学员可在 Course 学习的**任意时刻**通过抽屉课程目录进入 recap—— 不要求前序 Activity 完成度，不强制顺序
- **增量更新**：报告以"前序 Activity 完成"事件为单位后台增量重新生成；学员进入 recap 时直接看最新缓存
- **不保留历史快照**：每次更新覆盖旧版本，学员不能"看上次的版本"
- **学员侧零等待**：增量更新在后台静默完成，学员进入即看

##### 9.3.3.1.4 学员进入时的 4 种状态

学员进入 recap 时所处的状态决定报告呈现 + Neo 行为：

| 状态 | 含义 | 报告呈现 |
|------|------|---------|
| **空白态** | 关联 Activity 内 0 个完成 | 不生成报告，引导"先做点什么再回来" |
| **部分态** | 部分关联 Activity 完成 | 仅基于已完成数据生成；明确标注"仅基于已完成的 X/Y 个前序 Activity" |
| **完整态** | 所有关联 Activity ≥ 1 次完成 | 完整报告 |
| **重练态** | 某 practice 重练 N 次（已完整）| 报告反映最新一次完整重练 + N 次趋势（与 § 9.3.2.3.5 一致） |

##### 9.3.3.1.5 成功标准

recap 的"完成"采用 **milestone 概念**：

- **milestone 完成** = 学员首次完整走完 Neo Walk-through + 对话有实质性反思 + Neo 一次性产出应用计划入大厅待办
- 完成后再进入 = 自动进入回访态或复习态（详见 § 9.3.3.3.3）
- **报告有更新不退回未完成** —— 已完成的 milestone 不被推翻，只是 Subbar 显示"📊 报告有更新"小红点提示

学员"未完成"也允许产出报告 / 学员阅览 / 与 Neo 闲聊，但不计 milestone。

#### 9.3.3.2 界面布局形态

recap 沿用 § 4.2 学员端"4 场域统一布局" —— Topbar + 全幅课件区 + 右侧 Neo Chat + 左侧覆盖式抽屉 + 顶部 Subbar。

##### 9.3.3.2.1 单一布局（两栏）

```
┌─────────────────────────────────────────────────────────────────────┐
│ Topbar（项目名 / Course / Activity / 学习计时）                       │
├─────────────────────────────────────────────────────────────────────┤
│ Subbar（路径面包屑 / 📊 报告最后更新时间 / 数据更新小红点）             │
├──┬──────────────────────────────────────┬───────────────────────────┤
│抽│  课件区（左侧）                       │  Neo Chat（右侧）          │
│屉│                                      │                           │
│  │  · 报告 markdown                      │  · Neo 头像 + 名称         │
│  │    ① Course 锚点                      │  · 按当前态开场：           │
│  │    ② 走过的轨迹                       │    - 首聊态：完整 Walk-through│
│  │    ③ 6 维画像 · 当前快照              │    - 回访态：主动指出变化点  │
│  │    ④ 关键证据                         │    - 复习态：自主判断是否开场 │
│  │    ⑤ Neo 留言                         │  · 留白式追问引导反思        │
│  │  · 嵌入图表（最多 2-3 张）              │                           │
│  │                                      │  [输入框：和 Neo 聊聊……]   │
└──┴──────────────────────────────────────┴───────────────────────────┘
```

与 lecture / practice 布局的差异：

- **不分阶段切布局**（与 practice 4 阶段不同布局相反）—— 全程单一两栏
- **课件区承载报告而非学习内容** —— 报告是 Neo 引导对话的"道具"

##### 9.3.3.2.2 共用元素

**顶部 Subbar**：

| 元素 | 说明 |
|------|------|
| 路径面包屑 | 项目 / Course / Activity 当前位置 |
| 📊 报告最后更新时间 | 让学员知道报告新鲜度 |
| "报告有更新"小红点 | 距上次进入 recap 后报告增量更新过 → 显示红点；学员进入并 Neo 完成回访态开场后清除 |

**左侧覆盖式抽屉**（沿用 § 9.3.1.2.2 lecture 抽屉设计）：含两 Tab——

- **Tab1：当前 Activity 目录** —— recap 没有 SCO 序列，Tab1 改为呈现"报告 5 块骨架"作为锚点跳转（学员点击某块 → 课件区滚动到对应位置）
- **Tab2：整个课程目录** —— 当前 Course 的 Activity 列表（沿用 lecture / practice）

**右侧 Neo Chat**：常驻，承载 Neo 的反思态对话——Walk-through / 留白追问 / 跨场域记忆调用。

**笔记悬浮球**：屏幕边缘常驻，跨场域形态一致（详见 § 9.2.3）。

##### 9.3.3.2.3 状态指示

recap 4 个状态指示元素（与 lecture / practice 不同的部分加 ★）：

| 状态 | 触发 | 视觉提示 |
|------|------|---------|
| **学习计时** | 默认 | Topbar 右侧呈现本 Activity 累计时长 |
| **报告最后更新时间** ★ | 默认 | Subbar 显示"📊 X 分钟前更新" |
| **报告有更新红点** ★ | 距上次进入有增量更新 | Subbar 红点；进入后 Neo 完成回访态开场后清除 |
| **部分态徽章** ★ | 学员处于部分态 | 报告头部显示"仅基于已完成的 X/Y 个前序 Activity" |

> **不做的状态**：
> - **当前 SCO / 阶段进度** —— recap 不存在 SCO 序列，没有"X/N 进度"概念
> - **完成度百分比** —— milestone 是布尔值，不显示进度条

##### 9.3.3.2.4 部分态进入时的特殊呈现

部分态进入是 recap 独有的场景。报告头部明确标注：

```
┌───────────────────────────────────────────────────────┐
│ ⚠️ 仅基于已完成的 2/5 个前序 Activity 生成              │
│ 完成更多前序 Activity 后报告会自动更新                  │
└───────────────────────────────────────────────────────┘
```

避免学员误以为是完整评估。

#### 9.3.3.3 对话场景编排

> **设计理由**：recap 是融合的对话场景（看报告 + 与 Neo 聊 + Neo 给行动建议三件事在同一屏幕同时进行），不是分阶段的内容序列。强行套 SCO 会扭曲体验。recap 内部行为是产品级统一规则，KGP 不干预。

##### 9.3.3.3.1 场景结构

单一连续对话流：

```
学员进入 recap Activity
  ↓
Neo 按当前态开场（首聊 / 回访 / 复习 三态切换）
  ↓
学员 ↔ Neo 自由对话（融合看报告 + 反思挖掘 + 接收行动建议）
  ↓
首聊态走完 Walk-through 后 Neo 自动产出应用计划 → 入大厅待办清单
  ↓
完成 milestone（学员主动结束 / Neo 引导收尾）
  或退出（断点续播保存）
```

##### 9.3.3.3.2 KGP 在 recap 上的能力边界

KGP 只控制两件事：

| KGP 控制项 | 含义 |
|----------|------|
| ① **位置** | 把 recap Activity 摆放在 AOM 的什么位置（Course 内某点 / Course 末尾 / 项目末尾，详见 § 6.7）|
| ② **关联范围** | 决定 recap 关联**哪些前序 Activity** 作为数据源 —— 关联范围由 KGP 自由控制，**不强制与 Course 绑定**。本期典型场景是关联同 Course 内前序 Activity，但功能上不限制 KGP 跨 Course 关联或仅关联部分（为未来课程编排玩法留口）|

recap 内部行为（Walk-through 节奏、反思引导、应用计划产出、完成判定标准、报告生成规则）= 产品级统一规则，**KGP 不干预**。

##### 9.3.3.3.3 Neo 多次进入差异化（★ 三态切换）

学员可能多次进入同一个 recap Activity（随时打开机制）。Neo 根据"距上次会话发生了什么"动态切换行为：

| 态 | 触发条件 | Neo 行为 |
|----|---------|---------|
| **首聊态** | 学员首次进入 recap / 上次没聊完中途退出 | 完整 Walk-through 报告 5 块（按当前数据可用度）+ 走完后产出应用计划入大厅待办 |
| **回访态** | 再次进入 + 距上次有"实质性变化" | 不重讲全程；主动指出**变化点**（如"自从上次见你，你又做了 X，画像里 Y 维度往上走了"）+ 引用上次对话 + 询问近况 |
| **复习态** | 再次进入 + 无实质变化 | **Neo 自主判断是否开场**（可问可不问）；主体被动答疑——学员问什么 Neo 答什么 |

##### 9.3.3.3.4 三态切换的数据信号

Neo 综合多个信号自主判断当前应该用哪种态（spec 不规定固定阈值）：

**关联 Activity 内信号**（限定 KGP 配的关联范围）：

- 距上次会话是否有新关联 Activity 完成
- 关联 Activity 内是否有新高光卡片

**全场景信号**（不限定来源）：

- 6 维画像是否有明显变化（含大厅 / 跨 Course 来源）
- 学员在大厅或其他场景是否有新对话
- 学员上次在 recap 内跟 Neo 聊到哪
- 上次 Walk-through 走到哪一块

##### 9.3.3.3.5 学员对"重新 Walk-through"的主动权

学员通过对话表达诉求（如"我们再过一遍" / "重新讲一下吧"）—— Neo 自然响应。**不提供独立按钮** —— 贴合"老师"体感（通过对话表达，不通过 GUI 按钮）。

##### 9.3.3.3.6 应用计划产出（★ 重要）

**应用计划由 Neo 产出**（不是学员手写承诺），是 Neo 基于报告全数据 + 跨场域记忆给学员的"建议下一步行动"：

| 维度 | 规则 |
|------|------|
| **形态** | 1-3 条具体小任务粒度的行动建议 |
| **生成时机** | **仅在首聊态走完 Walk-through 后生成一次** |
| **后续不更新** | 即使报告增量更新、学员重进 recap，应用计划不再变（与报告增量更新解耦）|
| **流向** | 生成后**自动加入大厅"待办清单"**（沿用 § 9.2.2.1 (1) E + § 9.2.1 C.3.4 大厅待办清单机制）|
| **后续追踪** | 勾选完成 / Neo 反馈追问 全部在大厅待办流程内，recap 内不再回访 |

学员侧在 recap 报告内只看到"应用计划已加入大厅待办清单 ✓ {N 项}"+ 跳转大厅链接，**不参与撰写、不需确认**。

##### 9.3.3.3.7 recap 在 AOM 中的位置

由 KGP 决定（典型末尾、可半程、可项目末，详见 § 6.7）。

#### 9.3.3.4 Course 个人报告 · 数据与生成（★ 核心节）

> **作用**：报告是 Neo 引导对话的"道具" —— 写到 70%，留 30% 给 Neo Chat 展开。本节定义报告生成机制 + 数据来源 + 内容骨架 + 状态自适应规则。

##### 9.3.3.4.1 生成机制

| 维度 | 规则 |
|------|------|
| **触发** | 每次"前序 Activity 完成"事件触发后台增量重新生成 |
| **缓存** | 学员进入 recap 时直接看最新缓存 |
| **不保留历史快照** | 每次更新覆盖旧版本；学员不能"看上次的版本" |
| **学员侧零等待** | 后台静默完成，学员无感 |
| **不提供"刷新"按钮** | 自动更新即可，按钮反而暴露机制（保护"老师真实"感）|

##### 9.3.3.4.2 报告骨架 · 5 块结构

```
① Course 锚点（学员目标 + 时间锚 / 30-50 字）
② 走过的轨迹（部分态主显，完整态压缩 / 50-100 字）
③ 6 维画像 · 当前快照 ★ 灵魂（含可视化 / 80-120 字）
④ 关键证据（高光 + 卡点 + 「我以为 vs 实际」偏差 / 200-400 字）
⑤ Neo 留言（班主任谈话 / 80-150 字）

总长 500-800 字 + 1-3 张图 / 阅读时长 2-3 分钟
```

每块的设计意图：

| 块 | 解决什么问题 |
|----|------------|

##### 9.3.3.4.5 状态自适应矩阵（★）

5 块在 4 种状态下的呈现规则：

| 块 | 空白态 | 部分态 | 完整态 | 重练态 |
|----|--------|--------|--------|--------|
| ① Course 锚点 | 显示 | 显示 | 显示 | 显示 |
| ② 走过的轨迹 | "你还没开始走" | 主显（"已完成 X/Y" + 投入度）| 压缩为一句 | 含 N 次重练折线 |
| ③ 6 维画像 | 不显示 | 数据足时显示，部分维度可能空缺 | 完整呈现 | 完整呈现 |
| ④ 关键证据 | 不显示 | 部分显示（按数据可用性）| 完整 | 完整（含跨次对比）|
| ⑤ Neo 留言 | "我们等你迈出第一步" | 基于现有数据的轻量留言 | 完整班主任谈话 | 含跨次对比的留言 |

##### 9.3.3.4.6 6 维画像在报告中的呈现

**直接呈现"当前快照"** —— 不澄清跨场域数据来源（6 维画像本就是综合产物，澄清反而失真）。

具体形态由 PM 决策点 D1 锁定（候选：数值表 / 进度条 / 雷达图 / 文字描述 + 关键变化）。

##### 9.3.3.4.7 关键证据块的内容选择规则（含作用域）

关键证据是"**显性证据**" —— 学员能在报告里看到具体来源——所以**严格限定 recap 关联 Activity 内**，关联外的不进。

| 子块 | 内容 | 数据源 | 数量 |
|------|------|--------|------|
| **高光** | 学员表现亮眼的具体片段 | 关联 Activity 内高光库 | 1-2 条 |
| **卡点** | 学员反复栽的地方（模式）| 关联 practice 复盘 + 关联 lecture Quiz 错题模式 | 1-2 条 |
| **「我以为 vs 实际」偏差** | 学员自评 vs 实际表现的差距 | KGP 在 lecture FEEDBACK_COLLECT 设结构化"自评"字段 | 0-2 条（PM 决策点 D4）|

**降级规则**：关联内数据不足时**宁可空着也不补关联外** —— 保护"老师真实"感（不编造 / 不硬塞）。

##### 9.3.3.4.8 Neo 留言的生成原则

Neo 留言是"**隐性背景**" —— 基于全数据 + 跨场域记忆 + 学员开营目标的"班主任谈话"，**不限定数据来源**（含大厅 / 跨 Course）：

| 维度 | 规则 |
|------|------|
| **口吻** | 师长式——"我注意到 / 我担心 / 我希望" |
| **内容** | 表达观察 + 担心 + 期待；不是评分总结 |
| **作用** | 为后续 Neo Chat 对话提供锚点（学员读完想接话）|
| **生成性** | 一次性生成，不可改 |

##### 9.3.3.4.9 报告 diff 标识机制

报告生成时记录 vs 上版的变化点：

| 用途 | 内容 |
|------|------|
| 给 Neo 判断回访态用 | 哪些数据是新的 / 哪些块有变化 |
| Subbar 红点提示用 | 是否有更新 |
| 学员侧 | 不需看到详细 diff —— 只看到"报告有更新"的红点提示 |

##### 9.3.3.4.10 数据降级原则

数据不足时的默认行为：

- **不硬塞**：数据不够不强凑内容
- **不编造**：不让 Neo 编造没发生过的事
- **写占位说明**：明确告诉学员"这部分还没有足够的数据"
- **保护"老师真实"感**：宁可少说，不说错

##### 9.3.3.4.11 历史快照保留规则

本期不保留历史快照 —— 报告是当下视图，每次更新覆盖。

学员看不到"上次的版本"，也无法对比。这是简化设计；如未来有强诉求再做版本管理。

#### 9.3.3.5 Neo 行为（反思态）

##### 9.3.3.5.1 反思态定位

Neo 在 recap 与大厅 / lecture / practice 共用同一身份（同 voice / 方法论 / 记忆，详见 § 3.2 + § 9.3.0）。在 recap 中以 **反思态** 出现 —— 区别于 lecture Tutor 态 / practice Coach 态。

反思态的核心姿态：**引导学员看见自己 + 不替学员说总结**。

##### 9.3.3.5.2 7 动作分工

| 动作 | 权重 | recap 中的体现 |
|------|------|---------------|
| **反思** | ★ 主场 | 引导学员看见自己的行为模式与变化轨迹 |
| **追问** | ★ 主场 | 苏格拉底式追问，挖出学员自己的洞察 |
| **讨论** | ★ 主场 | 与学员就报告内容平等对话 |
| **解惑** | ✓ 辅助 | 学员对报告或概念有疑问时回答 |
| **考核** | ✓ 辅助 | 报告里的 6 维画像 + 高光 + 卡点 隐含评估反馈 |
| **讲授** | — / 轻量补漏 | 学员忘了某概念 Neo 可补，**不主动**讲新内容 |
| **练习** | — | 不出现（→ practice）|

##### 9.3.3.5.3 三态行为规范

| 态 | Walk-through | 主动开场 | 应用计划产出 |
|----|-------------|--------|------------|
| **首聊态** | 完整 5 块 | ✅ | ✅ 走完后产出入大厅待办 |
| **回访态** | 不重讲全程 | ✅ 主动指出变化点 | ❌（已在首聊态产出）|
| **复习态** | 不主动 | ⚠️ Neo 自主判断（可问可不问）| ❌ |

##### 9.3.3.5.4 Walk-through 节奏（首聊态）

Neo 主动逐块引导学员看报告：

```
1. Neo 第 1 句：定位锚点
   "我们一起看看这份小结。先问你一件事：你看完第 ② 节走过的轨迹，
    最让你意外的是什么？"

2. 学员回应 → Neo 引导到第 ③ 块 6 维画像
   "那为什么你觉得这维度低？我帮你看具体证据……"
   （引用第 ④ 块里的片段）

3. 走完 5 块 → Neo 转到留言
   "最后想跟你说一段话……"

4. Neo 主动收尾产出应用计划
   "基于这些，我建议你接下来一周做这几件事……
    我已经把它们加到你的待办清单里了。"
```

不让学员自己默读完事 —— 报告是道具，对话是主角。

##### 9.3.3.5.5 留白式追问规范

Neo 追问的核心：**不替学员说总结，引发学员自陈**。

| 类型 | 示例 |
|------|------|
| 引发观察 | "你看到这条洞察吗？想到了什么？" |
| 引发反思 | "为什么你觉得这维度低？" / "这个模式是不是熟悉的？" |
| 引发联想 | "这跟你上周在大厅说的那件事有关系吗？" |
| 引发承诺 | "你觉得接下来一周能做的一件事是什么？"（首聊态收尾）|

详见 § 9.3.3.7.1 不替学员总结洞察 ★ 核心约束。

##### 9.3.3.5.6 跨场域记忆调用（★）

Neo 在 recap 中**主动**引用学员的跨场域信息——让"老师记得你"成为体感：

| 引用对象 | 来源 |
|---------|------|
| 学员开营时说过的目标 | 开营 Activity / project config |
| 学员在 lecture FEEDBACK 里的自陈 | lecture FEEDBACK_COLLECT |
| 学员在 practice 演练里的具体行为 | 关联 practice 演练对话 |
| 学员在大厅 Neo Chat 里说过的话 | 跨场域 Neo 记忆系统 |

##### 9.3.3.5.7 必收尾约束

Neo 追问之后**必须**给收尾——不能只追问留开放状态（沿用 practice 复盘 § 9.3.2.6.5 同口径）。

收尾形态：

- 首聊态收尾 = Neo 给观察 + 留言 + 产出应用计划
- 回访态收尾 = Neo 总结变化点 + 引导学员说反思
- 复习态收尾 = 学员问什么 Neo 答什么后给收尾

#### 9.3.3.6 学员行为与核心交互

##### 9.3.3.6.1 输入方式

| 方式 | 交互 |
|------|------|
| **右栏 Neo Chat 输入框** | 学员与 Neo 对话（打字 / 语音）—— recap 内学员只跟 Neo 对话，不写应用计划（应用计划由 Neo 产出）|

##### 9.3.3.6.2 核心交互一览

recap 多数交互沿用 lecture 设计（详见 § 9.3.1.5.2）：

| 交互 | 说明 |
|------|------|
| **随时进入** ★ | recap 整个 Activity 入口对前序 Activity 完成度无锁——可随时进；4 状态自适应（详见 § 9.3.3.1.4）|
| **防挂机** | 默认 5 分钟无操作 → 系统弹提醒（沿用 lecture，可在 program config 调整阈值）|
| **学员主动提问 Neo** | 全程右栏 Chat 可问 Neo |
| **抽屉锚点跳转** | 学员点抽屉 Tab1 报告 5 块锚点 → 课件区滚动到对应块 |
| **Activity 间跳转** | 通过抽屉 Tab2 主动跳转其他 Activity（弹窗确认 + 触发断点续播保存）|
| **断点续播** | 中途退出 → 系统记录当前对话上下文 + Walk-through 进度；下次进入自动恢复 |
| **笔记悬浮球** | 屏幕边缘常驻（沿用 § 9.2.3）|

##### 9.3.3.6.3 "随时进入"的特殊交互

学员处于不同状态进入 recap 时的差异：

| 状态 | Neo 第一句 |
|------|----------|
| **空白态** | "我们还没什么可聊的——先去做点什么再回来吧。" |
| **部分态** | "这只是基于你已完成的部分，回头多做几个再来看，画像会更准。" + 进入 Walk-through 已有部分 |
| **完整态首次** | 进入完整 Walk-through |
| **完整态回访** | "自从上次见你，你又做了 X……" |
| **完整态复习** | 自主判断是否开场（可问可不问）|

##### 9.3.3.6.4 部分态进入时的引导

Neo 不强求学员"先去完成所有前序"再回来 —— 学员可以在部分态下听 Neo 讲已有数据，并自由对话。但 Neo 会**明示报告的不完整性**，避免学员误以为是完整评估。

##### 9.3.3.6.5 应用计划呈现

Neo 在首聊态走完 Walk-through 后产出应用计划，自动入大厅待办清单。学员侧在 recap 报告内看到：

```
✅ 应用计划已加入大厅待办清单（3 项）
   → 跳转大厅查看
```

学员**不编辑、不确认** —— Neo 直接产出 + 直接入待办。后续追踪 / 勾选完成 / Neo 反馈追问全部在大厅待办清单流程内（沿用 § 9.2.2.1 (1) E + § 9.2.1 C.3.4 完成反馈机制）。

##### 9.3.3.6.6 学员主动请求"重新 Walk-through"

学员通过对话表达诉求（如"我们再过一遍" / "重新讲一下吧"）—— Neo 自然响应，重新进入 Walk-through 流程。**不提供独立按钮** —— 贴合"老师"体感。

##### 9.3.3.6.7 重看 / 重进

完成后再进入 recap = 三态切换决定行为：

| 进入场景 | Neo 行为 |
|---------|---------|
| 报告无更新 | 复习态——Neo 自主判断是否开场，主体被动答疑 |
| 报告有更新（Subbar 红点提示）| 回访态——Neo 主动指出变化点 |
| 学员明确说"重新过一遍" | Walk-through 重新走（但不再产出新应用计划）|

报告随前序 Activity 完成自动更新（增量），学员侧无感。

#### 9.3.3.7 关键约束

##### 9.3.3.7.1 不替学员总结洞察（★ 核心）

Neo 引导学员说，不替学员说 —— 防止报告 / 对话变成 Neo 的"评分书"。学员的反思必须由学员**自陈**，Neo 用追问引发，不强行总结。

##### 9.3.3.7.2 不预设标准答案 / 不给"应该"

沿用 § 2.6.2 + § 9.3.1.6.1 同口径 —— Neo 不告诉学员"你应该这样想 / 这样做"，引导学员自己得出。

##### 9.3.3.7.3 报告不可修改

整体一次性生成，学员侧不可改；下次更新覆盖。

##### 9.3.3.7.4 报告导出规则

场域内不导出（沿用 § 9.3.5.6 跨场域统一规则）；大厅"我的报告"主题可导出。

##### 9.3.3.7.5 与管理端 team report 边界

| 维度 | recap Course 个人报告 | 管理端 team report |
|------|--------------------|-------------------|
| **场景** | 学员侧反思场域 | 管理端阶段性对外汇报 |
| **粒度** | 个人 × Course | 团队 × 项目周期 |
| **数据** | 可能共享（如 6 维画像）| 可能共享 |
| **呈现** | 学员视角的师长谈话 | HR / 业务部视角的客观汇报 |
| **修改** | 学员不可改 | HITL 编辑（详见 § 10.3）|

数据可能共享但呈现完全分开。

##### 9.3.3.7.6 数据降级原则

不硬塞 / 不编造 / 写占位说明（呼应 § 9.3.3.4.10）。

##### 9.3.3.7.7 6 维画像可见性

学员侧 recap 报告露出（含趋势）；管理端 D-5.0 不露出（与 § 9.3.2.3.3 状态矩阵决策一致 —— 管理端可视化由 § 10 后续设计决策）。

##### 9.3.3.7.8 取消 SCO 锁定

recap 整个 Activity 入口可随时进；不存在 SCO 序列，与 lecture / practice 严格顺序锁定不同。

##### 9.3.3.7.9 Walk-through 主动权

学员通过对话表达"重新走" —— 不提供独立按钮。

##### 9.3.3.7.10 显性证据限定关联 Activity

报告内可被学员看到具体来源的"证据类"内容（高光 / 卡点 / Quiz 错题模式 / 对练具体行为）严格限定 KGP 配的关联 Activity 内；关联外的不进。

##### 9.3.3.7.11 隐性背景不限定来源

6 维画像 / Neo 留言 / 跨场域记忆 = Neo 综合产物，**不限定数据来源**（限定反而失真）；6 维画像直接呈现"当前快照"，不澄清来源。

##### 9.3.3.7.12 应用计划由 Neo 产出

学员不参与撰写应用计划；Neo 在首聊态走完后产出，自动入大厅待办；后续不再生成 / 不再更新（与报告增量更新解耦）。

##### 9.3.3.7.13 防挂机 + 数据持久化与隐私

- **保存频率**：推荐每个会话 / Walk-through 节点完成时落库（沿用 § 9.3.1.6.6）
- **防挂机**：默认 5 分钟无操作触发
- **隐私三层**：沿用 § 2.6.3（学员之间默认完全隔离 / 跨学员引用需同意 / 管理端例外）

#### 9.3.3.8 完成与跨场域衔接

##### 9.3.3.8.1 Activity 完成判定（milestone 概念）

| milestone 状态 | 含义 | 学员看到 |
|---------------|------|---------|
| **未完成** | 学员还没首次完整走完 Walk-through | recap Activity 抽屉显示"未完成"；下次进入 Neo 接首聊态 |
| **已完成 milestone** | 首次完整 Walk-through + Neo 评估有实质性反思 + 应用计划已产出并入大厅待办 | recap Activity 抽屉标"已完成"；下次进入按三态切换 |
| **已完成但报告有更新** | 已完成后报告增量更新 | 不退回未完成；Subbar 显示"📊 报告有更新"；下次进入触发回访态 |

完成判定的具体标准由 PM 决策点 D5 锁定（候选：Neo 自主判断 / 简单轮数门槛 / 综合 Neo 判断 + 最低轮数）。

##### 9.3.3.8.2 完成 Modal

学员首次完成 milestone 时弹完成 Modal：

```
┌─────────────────────────────┐
│                        [X]  │
│         ✅                    │
│   Course 小结完成！            │
│  你已完成「{Course 小结名}」   │
│  应用计划已入大厅待办（3 项）   │
│                              │
│  [回到大厅]                   │
│  [自由复习]                   │
│  [继续下一 Activity]            │
└─────────────────────────────┘
```

按钮行为沿用 lecture / practice 3 选项设计（详见 § 9.3.1.7.2 + § 9.3.2.7.2）。

##### 9.3.3.8.3 重进 = 三态切换

完成后再进入 → Neo 按三态判断（详见 § 9.3.3.3.3）：

- 报告有更新 → 自然进入回访态
- 报告无更新 → 复习态

##### 9.3.3.8.4 应用计划的后续追踪

应用计划入大厅待办后，沿用 § 9.2.2.1 (1) E + § 9.2.1 C.3.4 待办清单的"完成反馈机制"：

```
学员勾选完成
   ↓
Neo 对应用计划提问（"做完了吗？怎么做的？"）
   ↓
学员回复
   ↓
Neo 评估并勾掉（本期不对回复内容做深度评估）
```

剩余服务期内同样适用 —— 不需要 recap 内额外回访机制。

### 10.1 管理端总览

#### 10.1.1 管理员旅程

```
登录管理端
   ↓
进入首页（看板 Operation）
   ↓
日常巡视 → Ora 看板（被动呈现）+ Ora 问答（主动对话）
   ↓

   ↓

```

#### 10.1.2 管理端模块清单

| 模块 | 类型 | 一句话 |
|------|------|--------|
| **首页 Operation** | 监控 | Ora 看板 + Ora 问答 + 异常学员发现 |
| **团队报告生成 Team Report-Gen** | 交付 | Ora 拉数据 + 多轮对话 + HITL 编辑 + 导出 |

#### 10.1.3 入口

管理员账号登录 → 默认进入 Operation 看板。

#### 10.1.4 HR vs 项目运营 vs admin 协同

| 角色 | 主要行为模式 |
|------|------------|
| **admin** | 项目根账号——开营前配置全权 + 创建管理员账号 + 锁定服务周期 |
| **HR** | 不定期统观——看板巡视 + 阶段性 team report 生成 + 对外汇报 |
| **项目运营** | 日常执行——异常学员发现 / 学员名单变更 / 催学规则配置 / 报告生成执行 |

本期 3 类管理员**共用同一管理端**，权限差异详见 backlog（待 § 10 详细设计阶段）。

### 10.2 看板 Operation

#### 10.2.1 业务定义

管理员日常监控项目运行健康度的场域。三维度监控 + 热力图 + Ora 双交互。

#### 10.2.2 三维度监控

| 维度 | 内容 |
|------|------|
| **项目维度** | 项目整体健康度 / 进度 / 完成率 |
| **人员维度** | 学员个体进度 / 异常学员 / 活跃度分布 |
| **内容维度** | 各 Course 学习情况 / 卡点分布 / Quiz 通关率 |

#### 10.2.3 热力图

学员行为热力可视化——每个学员每日活跃度色块 / 每个 Course 学员通过率色块。

#### 10.2.4 Ora 双交互模式
| 模式 | 触发 | 行为 |
|------|------|------|
| **看板被动呈现** | 默认呈现 | 三维度数据 + 热力图 + Ora 自动生成的项目健康度解读 |
| **Ora 问答** | 管理员主动提问 | "为什么 W4 脱训突增？" / "张三这周参与度怎样？" → Ora 用证据回答 |

> **本期不做**：看板点击 → Ora 实时解释——这是未来形态（详见 § 3.3.4）

#### 10.2.5 异常学员处置

按 § 6.6.2：

- **平台层**：Ora 看板 / 问答 中发现异常学员（脱训 / 进度落后 / Quiz 频繁失败）
- **线下层**：项目运营**线下跟进**该学员（电话 / 微信 / 1on1 沟通）
- **平台不强制干预**——干预靠人 / 渠道完成

#### 10.2.6 颜色条 / 排序 / 详情页

- **颜色条标识健康度**（绿 / 黄 / 红）
- **异常自动排顶**（异常项目 / 异常学员 自动排在顶部）
- **详情页"建议动作"** —— Ora 给出建议动作（详见 § 3.3.5）

### 10.3 团队报告生成 Team Report-Gen

> 这是 § 10 的**核心模块**——管理端的主要交付能力，承担中期反馈 / 半程 review / 结项报告 / 剩余服务期内的新报告 全部生成场景（详见 § 6.7 + § 6.8）。

#### 10.3.1 业务定义

管理员产出对外可举证报告的场域——Ora 拉全程数据 + 多轮对话生成初稿 + 管理员 HITL 编辑 + 导出 PDF / Word。

#### 10.3.2 引擎核心 · 每管理员独立报告库
- **每个管理员**有**独立的报告库**——存放该管理员生成 / 编辑的全部 team report
- **里程碑触发时**给每个管理员一份**独立草稿副本**
- **本期不做协同分享 / 不做实时多人编辑** —— 需协同时由管理员**导出后线下同步**

> 这个设计简化了多人协同的复杂度——研发只需保证每个管理员的报告库独立，不需做实时协同编辑。

#### 10.3.3 prompt 模板由"关联动作 / 评估要点"驱动
- 管理员**不需要选模板，也不手写 prompt**
- 管理员在创建 team report（或里程碑配置）时**填"关联动作 / 评估要点"**（如"评估学员中期参与度 + 概念掌握度"）
- 系统据此**自动生成**对应的 prompt 模板

> 本期 v0.5.0 优先做 **"结项报告 prompt 模板"** —— 用作里程碑结营时的默认 prompt。其他模板（中期 / 半程 / 自定义）后续逐步加（详见 § 6.7.1）。

#### 10.3.4 多轮对话生成 + HITL 编辑

生成流程：

```
里程碑到期触发 / 管理员手动触发
   ↓
Ora 拉项目全程数据
   ↓
Ora 与管理员多轮对话（确认范围 / 学员维度 / 数据筛选 / 文本风格）
   ↓
Ora 生成初稿
   ↓
管理员 HITL 编辑（增删改 / 调整结构 / 改文案）
   ↓
导出 PDF / Word
```

#### 10.3.5 与里程碑的关联

详见 § 6.6.3 + § 6.7.1：

- 里程碑到期当天 → Ora 自动生成 team report 草稿（每管理员一份副本）
- 站内消息提前 3 天每日提醒管理员
- Ora 在草稿生成后即时推送通知
- 过期未处理 → 提醒保留在列表（不堆累 / 不重复推送 / 不升级渠道）

#### 10.3.6 模板含义澄清
按 § 6.7.1：

> 本节"模板"指的是 **prompt 约束的写作规则**（即给 Ora 的"写作 brief / 文章骨架 / 核心要素清单"），**不是固定文章模板或 HTML 模板**。
>
> Ora 基于模板的写作约束 + 项目实际数据 + 关联动作 / 评估要点 → 生成具体文本。
