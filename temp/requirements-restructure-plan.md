# 产品需求文档重构计划

**日期：** 2026-04-13
**背景：** 基于创始人豆包对话记录中的核心产品思维，结合全部技术分析（Prosona/AOM/对练/辅导Demo），重新审视 requirements.md 该怎么改。

---

## 一、当前文档的根本问题

当前 requirements.md v2.0 有一个结构性问题：**它是从商业分析(startup-design)自顶向下推导出来的，而不是从创始人的产品心智模型生长出来的。**

具体表现：
1. 文档从"角色与端口"开始，但创始人的思维起点是**"大厅与教室"**——辅导是大厅，教学模块是教室，学员在两者之间自由穿梭
2. 行为规格来自辩论(debate)推演，但实际上很多行为**已经在代码中有了更成熟的实现**（coaching-skills的13步流水线、对练的导演-演员分离）
3. 技术和产品混在一起——"双轨运行"、"异步标注"这些是技术实现方案，不应该出现在产品需求文档中
4. 记忆工程、闭环反馈等创始人在豆包对话中反复强调的核心概念，在文档中是"补丁式"插入的，不是有机结构

## 二、创始人在豆包对话中的核心心智模型

重新读一遍豆包对话，创始人的产品思维有一个非常清晰的层次：

### 第一层：空间隐喻

```
┌─────────┐    ┌─────────────────────────┐
│  大厅    │ ←→ │       教室（多间）        │
│ (辅导)   │    │  授课室 | 对练室 | 考场   │
│          │    │                         │
│ 看板     │    │  个人报告室              │
│ 消息     │    │                         │
│ 笔记     │    │  ...更多教室（未来）      │
│ 工具     │    │                         │
└─────────┘    └─────────────────────────┘
```

- 大厅是学员的"家"——永远可以回来，有一个了解你一切的教练
- 教室是一次性进入的——完成任务后回到大厅
- 每间教室有不同的"教学法"——授课是演绎式，对练是沉浸式，测评是访谈式

### 第二层：内容驱动

产品的灵魂不在平台，在内容。AOM 是"乐谱"，Prosona 是"钢琴"。

```
KGP生产 → AOM数据包（半结构化乐谱）→ Prosona执行（演奏）→ 学员体验
```

- 授课脚本定义了AI老师讲什么、怎么讲、何时互动
- 对练剧本定义了AI角色的性格、行为边界、评估标准
- 测评剧本定义了追问策略、通关条件
- **平台只是执行器，内容才是产品**

### 第三层：记忆连贯

创始人最在意的（内测痛点驱动）：

```
课前：学员分享真实工作困境
  ↓
课中：AI用这些困境举例、用知识点解答
  ↓
课后：复盘"你之前说的那个问题，现在你会怎么处理？"
  ↓
辅导：随时引用"你在对练中说过..."
  ↓
报告：基于全过程数据生成行为改变证据
```

如果这条记忆链断了（"情感错付"），整个产品就变成了另一个"刷课"系统。

### 第四层：多端协同

```
学员端                    管理端
  辅导(大厅)               HR视角：看报告、问Agent、生产汇报素材
  教学(教室)               运营视角：看预警、催学、批量报告、临时任务
  个人报告                 共用：报告智能体
```

三端合一Web应用，权限区分。

## 三、新文档结构提案

基于创始人的心智模型，我建议这样重组：

```
产品需求文档 v3.0

第一部分：产品全景（给任何人3分钟看懂）
  1. 产品定义与愿景
  2. 空间模型：大厅与教室
  3. 角色与端口

第二部分：内容驱动架构（理解"乐谱与钢琴"的关系）
  4. 数据模型：AOM 五层结构
  5. 内容如何驱动行为（ADR-004 的具体化）
  6. 自适应体系（内容层面的版本切换）

第三部分：学员端（按空间组织，不按功能列表）
  7. 大厅：辅导模块
     7.1 辅导教练（Persona-Skill架构，基于coaching-skills demo）
     7.2 左侧看板（进度/课程结构/消息中心/笔记/工具/待办）
     7.3 从大厅到教室的导航（推荐+自选+课程目录）
  
  8. 教室一：授课模块
     8.1 授课的演绎流程（AOM驱动，SCO序列执行）
     8.2 SCO类型详解（PPT讲解/案例视频/小测评/小解答/Quiz）
     8.3 授课中的互动（插嘴提问/主动回顾/获得工具卡片）
     8.4 教室内导航（隐藏课程目录，多层折叠）
  
  9. 教室二：对练模块
     9.1 对练四阶段（导学→演绎→复盘→报告[可选]）
     9.2 导演-演员分离架构
     9.3 剧本如何控制行为（角色心理画像/行为反应规则/评估标准）
     9.4 当前玩法（Roleplay）与未来玩法（角色互换/费曼/辩论/多人）
  
  10. 教室三：测评模块
     10.1 两段结构（测+评）
     10.2 对话式测评（BEI/学员画像/困惑收集/高光萃取）
     10.3 GUI量表（类MBTI交互）
  
  11. 教室四：个人报告模块
     11.1 生成阶段（渐进式加载+情感化文案）
     11.2 解析阶段（学员与AI讨论报告）
     11.3 技术现状（Prosona LangGraph子图，8模块并行）

第四部分：管理端
  12. HR视角
  13. 运营视角
  14. 报告智能体（共用）
  15. 运营Agent

第五部分：跨模块系统
  16. 记忆工程（三级管道：课堂→大厅→管理端）
  17. 闭环反馈（课前采集→课后复盘，解决"情感错付"）
  18. AI质量控制（预防→检测→响应→修复）
  19. 催学机制（规则自动+重要人工）
  20. 运营预警（5类预警+看板）

第六部分：约束与边界
  21. 设计决策索引（ADR-001~006）
  22. 技术现状与差距（已实现/已设计/待开发）
  23. 项目规模与周期（30-40人，6个月）
  24. 迭代方向
  25. 开放问题
```

## 四、与当前 v2.0 的关键差异

| 维度 | v2.0 | v3.0 提案 |
|------|------|----------|
| **组织逻辑** | 按功能类型（行为规格/数据模型/自适应） | 按空间隐喻（大厅/教室一/教室二...） |
| **技术与产品** | 混在一起 | 分离——"产品要什么"在主体，"技术怎么做"在第六部分 |
| **内容驱动** | 作为一条ADR提及 | 升级为独立章节，AOM结构详解 |
| **记忆工程** | 补丁式插入 | 升级为跨模块系统的核心章节 |
| **辅导模块** | 列出行为清单 | 基于coaching-skills demo的架构描述 |
| **对练模块** | 泛泛的behaviour描述 | 基于实际代码的导演-演员架构+剧本结构 |
| **来源标注** | [已确认]/[待确认]/[创始人调整] | 去掉——v3.0是综合决策后的产物，不需要标注来源 |
