## 1 · 角色与端口

> 本章定义 v0.5.0 的端口架构（学员端 / 管理端 / 产品运营端）和对应角色。
>
> **关键澄清**：本期涉及两层"运营"——**项目运营**（客户 / 服务侧，用管理端）vs **产品运营**（睿学供应商侧，用产品运营端）。两者完全不同，全文统一使用此命名。

### 1.1 端口与角色总览

> **术语对照（议题 2 决议 · 维度区分）**：
> - **管理员** = **角色权限名** / = HR + 项目运营（管理端用户的统称） / 角色维度
> - **admin** = **项目根账号** / 1 个 / 开营前可改 / 开营后锁定 / 账号维度
> - **两者不在同一维度，互不包含**：admin 是账号类型 / 管理员是角色权限。admin 账号默认持有"管理员"角色权限 + 项目根级 admin 权限；HR / 项目运营账号只有"管理员"角色权限（无 admin 权限）
> - **不能称 admin 是"广义管理员"**——维度不同；当区分子角色时显式写 admin / HR / 项目运营

#### 1.1.1 3 个端口（端口视角）

| 端口 | AI 角色 | 当前状态 |
|------|---------|---------|
| **学员端** | Neo | 完整产品化 |
| **管理端** | Ora | 完整产品化 |
| **产品运营端** | 无 GUI | **短期非产品化**——研发后台动作 |

#### 1.1.2 4 类角色对照（角色视角）

| 角色 | 工作范围 | 属于哪侧 |
|------|---------|---------|
| 学员 | 自己的学习（6 个月项目周期）| 客户员工 |
| HR | 单个项目的统观（整体状态把握 / 阶段评估 / 对外汇报）| 客户管理者 |
| **项目运营** | 单个项目的执行（催学 / 跟进 / 异常处置）| 客户 / 服务侧 |
| **产品运营** | 跨项目的产品级运营（SKU / 数据 / 监控 / 迭代）| 睿学供应商侧 |

> **角色 ↔ 端口映射**：学员→学员端 / HR + 项目运营→管理端（共用）/ 产品运营→产品运营端。

#### 1.1.4 跨端关系简述

- **学员端 ↔ 管理端**：通过统一 Database 间接协作（Neo 写 → Ora 读，详见 [02-foundation/01-personas § 1.1.3]）
- **管理端 ↔ 产品运营端**：管理端只读自身项目的配置；产品运营端可见全部租户数据（最高权限）
- **一个用户可能有多个角色**：HR 既是管理员也可能是学员（培训内容包含管理课程时），通过端口切换实现；账号体系可独立 / 可关联

### 1.2 学员端

**用户**：学员（企业员工，参加培训项目的 1v1 学习对象）

**AI 角色**：**Neo**——唯一顶层 AI 角色，跨大厅 + 3 个学习场域统一身份（详见 [02-foundation/02-methodology § 1] / [02-foundation/01-personas § 1.2]）。Neo 形象个性化详见 [04-global/01-user-global § 1.3] 个人设置。

**通用约束**：3 个学习场域（授课 / 对练 / 小结）作为 **Activity 颗粒度**遵从 AOM 内容编排——具体教什么 / 练什么 / 反思什么由内容侧驱动，不是 Neo 自由发挥（AOM 数据模型详见 [02-foundation/04-data-model § 1]）。大厅是统一中枢，不属于学习场域，由 Neo 承担教练职能。

**主要场景**（一句话产品功能定义）：

- **大厅 Hall**——学员在产品中的统一中枢与陪伴入口；承载项目周期总览、课程目录浏览、学习成果汇集（4 库），同时由 Neo 在此承担 **辅导 / 课程推荐 / 督学** 3 项核心教练职能（辅导 = 跨课程应用咨询；课程推荐 = 基于学员当前状态推下一步学习方向；督学 = 按学习计划主动提醒与跟进）
- **授课 Lecture**——按 AOM 脚本驱动的结构化教学场域；通过 Neo 讲授 + 互动 Quiz + 即时答疑，让学员系统性掌握每个 Course 的知识点
- **对练 Practice**——让学员在拟真场景中"动手做"的练习场域；通过角色扮演 + AOM 剧本驱动，把课堂知识转化为可迁移到真实工作的行为
- **小结 Recap**——学员个人维度的学习反思场域；按 AOM 编排，Neo 引导学员看见自己的成长轨迹与卡点模式。**本期实现形式 = Course 个人报告（每门 Course 结束）+ Neo 解读对话**

> **未来扩展场域**：inquiry（调研）/ assignment（作业，首先做案例拆解）/ sandbox（沙盘）等——本期不实现（详见 [02-foundation/04-data-model § 1.3.5]）。

**入口**：学员账号登录

→ 详见 [05-learner/01-overview § 1] 学员端（详细场景展开）

> **场域 UI 设计统一**：学员端各场景（大厅 + lecture / practice / recap）在 UI 设计规范层面统一管理——共用元素（Topbar / 抽屉 / Neo Chat 形态 / 笔记悬浮球 / 状态指示 等）抽出为设计规范，各场域在规范下自行布局（如 [05-learner/04-practice § 1] practice 的三栏例外）。

### 1.3 管理端

**用户**：管理员——含两类角色：

- **HR**——侧重**统观**（项目整体状态把握 / 阶段性效果评估 / 对外汇报与报告交付）
- **项目运营**——侧重**执行**（项目进度跟进 / 催学动作 / 异常学员处置 / 中期反馈）

**关键说明**：本期 HR 和项目运营**共用同一个管理端**，**不分账号类型**。两者的差异通过工作分工自然区分——同一份看板、同一份报告，HR 和项目运营各取所需。

**AI 角色**：

- **Ora**——管理端唯一 AI 角色（详见 [02-foundation/01-personas § 1.3]）

**主要场景**（一句话产品功能定义）：

- **首页 Home**——管理员日常监控项目运行健康度的场域；通过项目 / 人员 / 内容三维度 + 热力图 + Ora 解读，让 HR / 项目运营快速识别异常并定位干预点
- **报告中心 Report Center**——管理员产出对外可举证报告 + 历史报告归档查看 + 报告库管理 的场域；Ora 拉全程数据 + 多轮对话生成初稿 + 管理员 HITL 编辑，输出多模板报告（中期 / 结项 / 自定义）供 HR 向业务部门交付
- **项目配置 Program Config**——项目准备期与变更期的设置场域；管理项目信息 / 学员名单 / 角色分配 / 催学规则 / 平台个性化 / 内容预览 / 开营确认，是项目从启动到收尾的"控制台"
- **消息中心 Message**——管理员对学员定向沟通的场域；支持手动发送富文本消息 + 发送记录跟踪

**入口**：管理员账号登录（HR / 项目运营共用）

→ 详见 [06-management/01-overview § 1] 管理端（详细场景展开）

### 1.4 产品运营动作（短期非产品化）

**用户**：睿学供应商侧的产品运营 / 研发 / 运维

**实现方式**：

- **本期不做产品化 GUI**——通过研发同学的后台动作完成
- 后台动作 = 直接操作数据库 / 服务配置 / AI 模型部署管道
- 未来产品化路径：作为 v1.x 后期工作，逐步从"研发动作" → "运维 GUI" → "产品运营 GUI"

**4 项核心职责**：

#### 4.4.1 服务 SKU 申请与下发

- **触发场景**：客户签单后开通服务实例
- **本期实现**：研发手动执行（创建项目实例 / 分配学员名额 / 配置 Course Pack / 创建管理员账号）
- **未来形态**：商品平台下单 → 自动触发 SKU 下发流水线

#### 4.4.2 跨项目数据底座

- **职责**：管理多客户 / 多项目的数据存储与隔离（租户级隔离）
- **本期实现**：研发负责数据库 schema / 租户分区 / 备份策略
- **未来形态**：数据底座 dashboard 让产品运营自助查询

#### 4.4.3 平台监控与分析

- **职责**：平台健康度 / API 性能 / 异常告警 / Token 使用统计
- **本期实现**：研发用通用监控工具（日志系统 / 告警群 / 基础 dashboard）
- **未来形态**：集成到产品运营端控制台

#### 4.4.4 产品优化迭代

- **职责**：AI 模型版本迭代 / Soul 调整 / Course Pack 内容包更新 / 灰度发布
- **本期实现**：研发手动部署 + AB test 通过命令行 / 配置文件
- **未来形态**：产品运营端有"模型管理 / Soul 编辑 / 内容包发布"控制台

**入口**：研发后台（不暴露给客户 / 学员 / HR / 项目运营）
