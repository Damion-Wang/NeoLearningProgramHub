## § 6.5 program config 详细设计 backlog

### 1.1 项目里程碑（Milestone）详细配置

#### 里程碑分类详细规则

**3 类里程碑**：

| 类型 | 含义 | 数量约束 |
|------|------|---------|
| **定时里程碑** | 按周期重复生成的里程碑，仅 3 种模板：单周 / 双周 / 单月 | 模板触发后系统按周期自动生成，数量取决于项目周期长度 |
| **自定义里程碑** | admin / 项目运营自由插入的任意日期 + 名称里程碑 | **上限 10 个** |
| **阶段里程碑** | 项目阶段性质的里程碑——本期仅 2 种：中期 + 结营 | 每种各 1 个 |

#### 阶段里程碑细则

| 阶段里程碑 | 必选 / 可选 | 是否可改 | 默认时机 |
|-----------|-----------|---------|---------|
| **结营** | 必选（不可删）| 可改名 + 调时间 | 项目周期结束日 |
| **中期** | 可选（创建项目时由 admin 勾选）| 创建后可改名 + 调时间 + 删除 | 项目周期中点 |

> 未来扩展更多阶段里程碑（开营 / 半程 / 收官等），本期仅做中期 + 结营。

#### 约束规则

- 里程碑只能在项目周期内（不能在剩余服务期）
- 里程碑分布完全自由（不要求均匀）
- 同一天**可以**有多个里程碑（未来考虑合并 / 去重逻辑）
- 不设禁用状态——要么有要么删
- 不设"完成"状态——只有"是否到期"两态
- 数量无上限（除自定义里程碑上限 10 外）
- 结营外都可配置

#### 里程碑关联动作 · 管理员侧

| 动作 | 触发时机 | 说明 |
|------|---------|------|
| **生成 team report 草稿** | 里程碑当天 | 默认生成草稿，prompt 由该里程碑配置时填的"关联动作 / 评估要点"驱动生成 |
| **每个管理员一份副本** | 同上 | 每个管理员独立报告库 + 独立草稿副本，本期不做协同分享 / 不做实时多人编辑——需要协同时由管理员导出线下同步 |
| **Ora 提醒管理员** | 草稿生成后立即 | Ora 推送站内通知，含 team report 副本跳转链接 |
| **站内消息预提醒** | 里程碑前 3 天每日 | 与 Ora 提醒为两套不同机制——站内消息提前提醒，Ora 在草稿生成后追加 |
| **接收人** | 项目全部管理员（admin / HR / 项目运营）| 不区分谁负责，运营侧自行协调 |
| **过期未处理** | 提醒保留在列表，不堆累、不重复推送、不升级渠道 | 管理员任意时间打开 team report 副本继续处理 |

#### 里程碑关联动作 · 学员侧（轻量）

| 动作 | 触发时机 | 说明 |
|------|---------|------|
| 大厅倒计时 | 全程 | 学员在大厅看到"距下个里程碑 X 天"（明文可见的里程碑）|
| 里程碑列表 | 全程 | 学员可在大厅进度区看到项目时间轴上明文可见的里程碑 |
| 里程碑到期推送 | 当天 | 仅站内消息（产品化语言："项目过半啦"等，不出现"里程碑"运营术语）|
| Neo 督学话术调整 | —— | **本期 Neo 不感知里程碑**（Neo 督学功能后期再做）|
| 学员主动触发 | —— | 学员**不能**主动触发里程碑动作（运营控制）|

### 1.2 催学规则详细配置

#### 对象

- **本期只做"全员催学"**——所有学员都收到
- **异常学员**：系统标识但不自动催，由项目运营**手动选人**发送站内消息

#### 异常学员判定（系统标识）

两个信号并行（任一触发即标识为异常）：

| 信号 | 默认值 | program config 可调 |
|------|--------|---------------------|
| **脱训信号** | 连续 X 天未登录 | X 可调 |
| **进度落后信号** | 落后项目平均进度 X%（按 Activity 粒度计算）| X 可调 |

#### 渠道

- **本期仅站内消息**——邮件 / 钉钉 / 企业微信 / 飞书 / SMS 全部本期不做（看后续渠道集成成本逐步加）

#### 文案模板

- 支持**模板变量**：`{学员姓名}` / `{个人进度}` / `{项目平均进度}` / `{下一里程碑名}` / `{距里程碑天数}` / `{距结营天数}` 等
- 推送时自动填充

#### 触发类型

| 触发类型 | 说明 |
|---------|------|
| **里程碑绑定型** | 选某个里程碑当天触发（与里程碑关联）|
| **脱训信号型** | 学员被标识为脱训时触发 |
| **进度落后信号型** | 学员被标识为进度落后时触发 |
| 手动触发型 | 运营手动选人发送（异常学员场景）|

#### 调性

- **仅"提醒 / 警示"** —— 鼓励 / 表扬不做催学（这部分由 Neo 在大厅交互中自然表达）

#### 频率上限

- **不设上限**——运营自负责防骚扰

#### 投递日志

- **完整记录**：谁收到 / 是否查看 / 是否点击链接
- **本期不支持回应**（学员无法在催学消息上回复）

### 1.4 program config 全阶段功能矩阵

> 此矩阵汇总每阶段需要 program config 提供的具体功能，作为 § 10.4 program config 详细设计的输入清单。

| 阶段 | program config 功能 / 字段 |
|------|---------------------------|
| **1 商机识别** | **公司画像 context 字段**（可空置）|
| **2 试运行体验** | 试运行项目创建（独立项目实例）/ 简化 schema / 数据隔离 |
| **3 签单** | 项目实例初始化（学员上限 / 内容范围 签单即锁定 + 服务周期默认 1 年待开营锁定）/ **创建 admin 账号**触发 |
| **4 开营组织** | 项目信息编辑（项目名默认自动生成 / **服务周期默认 1 年合同锁定** / 项目周期默认 = 服务周期 - 1 周 / 项目描述 / 客户名 / 公司画像 context 全部选填）/ **批量导入人员名单**（必填，默认学员角色）/ **角色分配**（学员 / 管理员 多选 checkbox）/ 催学规则配置（选填，默认站内）/ 平台个性化（选填）/ 内容预览（仅查看）/ Token 用量统计（后台默认开启，无开关）/ **❌ 不做学员分组标签** / **❌ 预览模式本期不做** / **"确认开营"按钮**（连锁触发：账号开放 + 系统消息 + 状态切换 + 轻量化开营）|
| **5 学习期** | Ora 看板 + Ora 问答 / 异常学员发现（→ 线下跟进）/ **里程碑配置**（详见 § 6.6.3）/ **学员名单 / 项目周期变更**（严格确认）/ **催学规则**（详见 § 6.6.5）/ **平台个性化**（详见 § 6.6.6）/ **team report 生成入口**（每管理员独立报告库 / 不分享）/ **小结 Activity 由 AOM 编排** / ❌ 内容范围不可改 / ❌ 变更日志本期不做 |
| **6 结营** | 结营触发器 / **team report 结项报告生成**（基于结项 prompt 模板）/ "已结营"状态切换 |
| **7 剩余服务期** | 状态显示"剩余服务期内" / 配置只读 / 学员端完整继续学习能力 / **AOM 学习实例 phase 标记**（区分结营前 / 后记录）/ team report 生成入口持续可用 / **服务到期前 1 个月自动提醒** |
| **8 归档** | 归档状态展示 / 产品运营端手工归档接口 / **数据交付包导出工具**（含管理端产物 / 不含原始学习记录）|

**关键设计原则**：

1. **服务周期开营后锁定**——开营前 admin 可通过特殊安全措施修改服务周期；**确认开营后任何阶段都不可改**
2. **学员上限 / 内容范围签单即锁定**——签单后这两项不可改（区别于服务周期 / 项目周期开营后才锁定）
3. **项目周期可调但严格确认**——变更需要 HR 确认 + 销售签字 + 二次弹窗
4. **里程碑驱动**——项目里程碑（默认必选结营 + 可选中期 + 自由插入上限 10）触发 team report 草稿生成 + 站内消息提醒（前 3 天）+ Ora 当天提醒（详见 § 6.6.3）
5. **"确认开营"是分水岭**——之前 program config 全量可编辑；之后进入只读 + 变更管理状态；连锁动作（账号开放 + 系统消息 + 状态切换）
6. **admin 账号 vs 管理员账号 严格区分**——admin 是项目根账号（阶段 3 创建）；管理员账号是 admin 在阶段 4 创建的子账号（HR / 项目运营）
7. **team report 是核心交付引擎**——中期反馈 / 半程 review / 结项报告 / 剩余服务期内的新报告 全部通过 team report 自定义功能实现，本期优先做"结项报告 prompt 模板"
8. **个人结营报告本期不做**——学员个人小结全部在阶段 5 学习期内通过 Course 个人小结 Activity 完成
9. **变更日志本期不做**——未来再加（变更操作的可追溯性后续支持）
10. **数据交付分层**——管理端产物可导出；原始学习记录不导出（防反推产品功能）

## § 10 管理端 详细设计输入

### 2.1 team report 引擎

- **每个管理员独立报告库 + 独立草稿副本**
- 不做协同分享 / 不做实时多人编辑
- 需要协同时由管理员导出线下同步
- prompt 由里程碑配置时填的"关联动作 / 评估要点"驱动生成（不是选模板 / 不手写）

### 2.2 Ora 双交互模式

- **看板被动呈现**——三维度 + 热力图 + Ora 自动生成的健康度解读（本期不做"看板点击 → Ora 实时解释"）
- **问答主动对话**——管理员主动向 Ora 提问（"为什么 W4 脱训突增？"）

### 2.3 异常学员处置流程

- 平台层：Ora 看板 / 问答 中发现异常学员
- 线下层：项目运营线下跟进（电话 / 微信 / 1on1 沟通）
- 平台不做学员个体的强制干预动作

### 2.4 投递日志查询入口

- 谁收到 / 是否查看 / 是否点击链接
- 本期不支持学员在催学消息上回复

## § 9 学员端详细设计 backlog

> **背景**：2026-04-28 DM 决策——§ 9 学员端先过大纲（outline B 框架），具体内容后续在详细设计阶段再展开。本节存放从 plan/05 中拆出的旧版详细内容草稿，作为后续 § 9 详细设计的参考素材。

### 6.1 学员端总览（旧草稿）

#### 学员旅程（旧草稿）

```
登录学员端
   ↓
进入大厅（默认主页 · Neo 持续陪伴）
   ↓
按 Course 序列学习 → 进入 lecture / practice / recap Activity
   ↓
学习中产生笔记（悬浮球贯穿）
   ↓
回到大厅 → 4 库回顾 + Neo 跨课程辅导 + 看自己的进度
   ↓
项目周期内反复（直到结营）
   ↓
项目结营后进入剩余服务期 → 完整继续学习能力（详见 § 6.8）
```

#### 学员端模块清单（旧草稿）

| 模块 | 类型 | 一句话 |
|------|------|--------|
| **大厅 Hall** | 中枢 | Neo 教练 + 看板 + 4 库 + 里程碑倒计时 |
| **lecture** | 学习场域 | Neo 按 AOM 脚本结构化教学 |
| **practice** | 学习场域 | Actor 角色扮演 + Neo 旁观（双栏布局）|
| **recap** | 学习场域 | Course 个人报告 + Neo 解读对话 |
| **笔记悬浮球** | 跨场域 | 贯穿大厅 + 学习场域 |

> **未来扩展场域**：inquiry（调研）/ assignment（作业）/ 沙盘 等本期不做（详见 § 5.4.5）。

#### 入口

学员账号登录 → 默认进入大厅。

### 6.2 大厅 Hall（旧草稿）

#### 大厅是学员端的统一中枢

大厅承担 4 个核心职能：

1. **项目陪伴入口** —— Neo 在大厅持续陪伴
2. **学习导航** —— 课程目录 / 当前进度 / 下一步推荐
3. **学习成果汇集** —— 4 库回顾
4. **里程碑可视化** —— 倒计时 + 进度区里程碑标记

#### Neo 在大厅的 3 项核心职能

详见 § 4.2 大厅描述。本节展开"在大厅 GUI 上的呈现"：

| 职能 | GUI 呈现 |
|------|---------|
| **辅导**（跨课程应用咨询）| 大厅右侧 Neo Chat 抽屉常驻——学员遇到工作问题随时与 Neo 对话 |
| **课程推荐** | 大厅看板上"接下来推荐"卡片 + Neo 主动消息推荐 |
| **督学** | Neo 主动 Chat 消息（如"你已 X 天没进入授课场域"）+ 站内催学消息（详见 § 6.6.5）|

> **本期 Neo 不感知里程碑**（详见 50 题 Q22）—— Neo 督学话术调整 / 里程碑感知 留作未来。

#### 大厅看板

- **项目概览** —— 学员当前项目名 / 项目周期 / 距结营天数
- **课程目录** —— 已开放 / 进行中 / 已完成 Course 列表
- **热力图** —— 学员个人学习行为可视化（每日活跃度热力）
- **待办** —— 接下来要做的 Activity / 未读消息提醒
- **里程碑倒计时** —— "距下个里程碑 X 天"（**仅明文可见的里程碑**，详见 50 题 Q23）
- **里程碑列表** —— 项目时间轴上明文可见的里程碑（产品化语言，不出现"里程碑"运营术语）

#### 4 库 GUI

按 v0.4.0 + 50 题决策（覆盖 0424 C-05 决策 = 4 tab 水平排列）：

| 库 | 内容 |
|----|------|
| **成果库** | 学员产出物（Course 个人报告 / 项目结营报告 / 反馈采集结果 / 调研结果未来）|
| **发现库** | AI 推送的高光（Neo 在学习中标记的"做得好的瞬间"）|
| **工具库** | AOM 内容包提供的方法论工具卡片（Project 级知识工具库，详见 § 5.3.2.1）|
| **笔记库** | 学员自己的笔记（悬浮球记录的产物）|

**4 库交互**：

- 大厅看板**4 tab 水平排列 + 数量显示**（取代 v0.4 早期"2×2 网格"方案）
- 点击 tab → 右侧抽屉滑出该库内容
- 库内可搜索 / 筛选 / 跳转源头

#### 复习与回顾

学员在大厅可：

- 重看 Course 历史（已完成 Course 重新进入 lecture / practice / recap）
- 重练对练（重做 practice Activity，产生新记录但与原记录区分）
- 回顾 4 库（详见 § 9.2.2.3 (笔记库)）
- 与 Neo 对话讨论历史学习内容

> **结营后**（剩余服务期）的复习行为 → 学习记录与结营前做明显标记区别（详见 § 6.8）。

### 6.3 lecture（旧草稿）

#### 业务定义

按 AOM 脚本驱动的结构化教学场域。Neo 通过 SCO 流逐步讲授 + 互动 Quiz + 即时答疑，让学员系统性掌握每个 Course 的知识点。

#### GUI 与交互

布局：默认两段（Topbar + 主内容区）+ 右侧 Neo Chat 抽屉。

主内容区按 SCO 类型呈现：

- **SLICE**（PPT 讲解）—— PPT 图片 + Neo 配音讲解 + 进度提示
- **VIDEO**（动画视频）—— 视频播放器 + 字幕 + 角色对白
- **QUIZ**（互动测验）—— 题目展示 + 学员作答 + Neo 评判 + 引导反馈
- **FEEDBACK**（采集 / 复盘）—— Neo 与学员对话采集 / 反馈

#### 关键约束

- **SCO 锁定**——学员必须按序完成（不可跳过 / 不可乱序）
- **防挂机**——学员长时间不互动时 Neo 主动追问 / 暂停播放
- **闭环反馈**——FEEDBACK_COLLECT 与 FEEDBACK_REVIEW 通过 memory_id 关联（详见 § 5）
- **不给答案**（教学态约束）—— Neo 引导方向，不直接给答案（详见 § 2.6.2）
- **不剧透**——不提前告诉学员后续结论 / 答案 / 学习路径终点

#### 前端演示事件

AOM 内容包可指定 SCO 内的前端演示事件（详见 § 5.3.2.3）：

- PPT 关键点高亮
- 动画触发（如概念图逐项展开）
- 进度提示（如"这是本 Activity 第 3 / 5 个 SCO"）
- Quiz 答错的视觉反馈

### 6.4 practice（旧草稿）

#### 业务定义

让学员在拟真场景中"动手做"的练习场域。通过 Actor 角色扮演 + AOM 剧本驱动 + Neo 旁观介入，把课堂知识转化为可迁移到真实工作的行为。

#### GUI 与交互（双栏布局）

布局：**双栏例外布局**——对练是唯一与默认布局不同的场域，左 60-70% Actor 主区 + 右 30-40% Neo 旁观抽屉

- **左栏（60-70%）**：Actor 角色扮演主区——学员与 Actor 对话
- **右栏（30-40%）**：Neo 旁观抽屉——Neo 默认沉默 / 卡死或学员主动求助时介入

#### 子 Agent · Actor / 导演

两个对练专属子 Agent（不算顶层 AI 角色，详见 § 4.1）：

| 子 Agent | 角色 | 学员可见 |
|---------|------|---------|
| **Actor** | 角色扮演演员（按 AOM 剧本扮演场景中的"对手"——下属 / 跨部门同事 / 客户 等）| ✅ 主交互对象 |
| **导演** | 后台 Agent（控制 Actor 行为 + 剧本进度 + 触发条件）| ❌ 学员不可见，仅后台 |

**关键约束**（详见 § 2.6.2）：

- **Actor 与 Neo 独立** —— Neo 看得到 Actor 对话，**Actor 看不到 Neo 存在**
- **Actor 不参与逃生** —— 学员卡死时仅 Neo 介入，Actor 不会"破坏角色"提示

#### 学员卡死时介入逻辑

按 50 题决策 + § 2.6.2：

| 情况 | 介入方 | 介入内容 |
|------|--------|---------|
| 学员卡死（长时间不发言 / 重复无效输入）| Neo | Neo 在右栏主动提问"我看你卡住了，要不要我帮你梳理一下？" |
| 学员主动求助（点击 Neo 抽屉发问）| Neo | Neo 答疑 / 给思路（不直接给答案）|
| 卡死多次仍不前进 | Neo | 切讲授态——Neo 暂停 Actor 对话，进入辅导讲解模式 |

#### 对练 Activity 由多个 SCO 组成

详见 § 5.4.3：

- **前情导入 SCO**（场景 / Actor 人设 / 学员任务）
- **角色扮演 SCO**（与 Actor 对话）
- **复盘 SCO**（Neo 引导反思 + 追问）—— Neo 主导

#### 关键约束

- **不剧透**（详见 § 2.6.2）—— Neo 不提前告诉学员剧本走向 / 演员意图
- **Actor 行为完全由 AOM 剧本驱动**（详见 § 5.4.3）

### 6.5 recap（旧草稿）

#### 业务定义

学员个人维度的学习反思场域。**本期实现形式 = Course 个人报告 + Neo 解读对话**（详见 § 4.2 / § 5.4.4）。

#### 关键澄清

- 小结 **属于学习期**（不属于结营动作）—— 详见 § 6.7
- 小结 Activity 的位置由 **AOM 编排决定**——典型每个 Module 末尾配 1 个小结
- 小结的本期实现形式 = 每门 Course 学完触发的 Course 个人报告 +（项目结营时的）项目结营个人报告

#### GUI 与交互

布局：默认两段（Topbar + 主内容区）+ 右侧 Neo Chat 抽屉。

主内容区呈现 Course 个人报告——由多个 SCO 组成（详见 § 5.4.4）：

- **内容回顾 SCO** —— Course 关键知识点 / 案例 / 工具卡片回顾
- **表现复盘 SCO** —— Neo 引导反思学员在前 Activity 的表现模式 / 卡点
- **应用计划 SCO** —— 学员写下行为承诺（下次工作中具体做什么）

Neo 在右栏伴随对话——引导反思 + 追问（详见 § 2.4.7 反思动作）。

#### 关键约束

- **解读 + 反问**——Neo 不替学员评价，引导学员自己看见自己（详见 § 2.6.2）
- **轻量讲授**允许——反思中学员忘记某概念时 Neo 可补充（详见 § 2.5 矩阵备注）
- **不替学员总结洞察**——必须学员先说，Neo 后反映

### 6.6 笔记悬浮球（旧草稿）

#### 业务定义

笔记是学员随时记录的快速产出工具——**悬浮球**形态贯穿大厅 + 3 学习场域。

#### GUI 与交互

- 悬浮球默认在右下角（可拖拽位置 / 不可隐藏）
- 点击 → 弹出笔记面板
- 支持：打字 / 粘贴 / 截图 / 关联当前学习内容（自动记录学员当前位置：Course / Activity / SCO）

#### 笔记归档

笔记归档到大厅 4 库的**笔记库**（详见 § 9.2.2.3 (笔记库)）—— 详见 § 4.2 学员端"§ 9.2 vs § 9.2.3 关系"说明：

- **§ 9.2.2.3 我的笔记** = 存储与管理（含笔记库）
- **§ 9.2.3 笔记悬浮球** = 录入与记录功能
- 两者**协作不重复**

#### 本期不做

- ❌ **AI 整理 / 摘要笔记** —— 看后续
- ❌ **笔记导出** —— 学员可以在剩余服务期前自行复制（手动），本期不做"批量导出"功能
- ❌ **跨学员共享笔记** —— 学员之间默认隔离（详见 § 2.6.4）

### 6.7 跨场域记忆与 Neo 召回（旧草稿）

#### 学员端 Neo 跨场域召回机制

Neo 是同一个 AI 角色 跨大厅 + 3 场域统一身份（详见 § 2.5 / § 3.2）。

跨场域召回的产品体现：

| 场景 | 学员体验 |
|------|---------|
| 学员在大厅说过"我下属总不主动汇报" | 进入"辅导反馈" Course lecture 时 Neo 一句"你提到过下属汇报问题，我们来看看怎么解决" |
| 学员在 lecture A 答错某 Quiz | 进入 practice 时 Neo 旁观 + 复盘强化该知识点 |
| 学员上次小结写的"下次直接说事实不绕" | 下次 practice Neo 在右栏提醒"你上次承诺直接说事实"|

#### 反幻觉

详见 § 3.4.1 反幻觉：

- 找不到证据时实诚拒答（"我没印象，你提醒一下？"）
- 不为"显得懂"编造记忆
- 不主动调用低置信度记忆

#### 与管理端的边界

学员的所有学习记录 → 统一 Database → 管理端 Ora 聚合后可见（详见 § 3.1.3）。

学员端 **看不到** 管理端 Ora 自有 Database 的内容（详见 § 3.1.3 视图 B）—— Neo 在学员侧只调用学员个人记忆 + 学员聚合数据，不调用管理员侧数据。

### 7.8 Neo 大厅模式 内部实现细节（研发参考）

> 2026-04-28 重构 § 9.2.1 B 节后,Neo 大厅模式的"内部能力清单"从 spec 正文移出。spec 正文（§ 9.2.1 B.1）改为站在学员视角描述"7 教学动作的大厅演绎"；本节归档原 B.1 的 8 项内部技术细节,作为研发实现参考。

#### 7.8.1 ContentSkill 边界（Bloom 4 层）

Neo 在大厅允许轻量知识注入,但**注重应用 + 迁移引导而非概念教学**：

| 学员认知层级 | Neo 大厅策略 |
|------------|------------|
| **概念层**（不知道是什么）| 不讲概念,**主动推荐进入对应 Activity** |
| **理解层**（知道但没理解透）| 简要提示,**引导回顾课堂已讲内容** |
| **应用层**（理解但不会用）| **结合学员真实工作场景**做应用指导 |
| **迁移层**（会用但不会迁移）| **帮学员在新场景中灵活运用**已学方法 |

**前提条件**：Neo 要明确知道学员学了什么 / 没学什么——读取 AOM 课程结构 + 场域内教学记录作为上下文。

#### 7.8.2 4 种人格模式（学员无感切换）

Neo 根据对话上下文在 4 种模式间无缝切换,**学员无感知**。

| 模式 | 核心身份 | 切换信号 |
|------|---------|---------|
| **MENTOR**（导师）| 结果导向的管理辅导者 | 负面情绪触发（最高优先级）|
| **STRATEGIST**（战略家）| 系统思维管理顾问 | 规划意图 + 非焦虑状态 |
| **CHALLENGER**（挑战者）| 直言不讳的质疑者 | 高信心 / 兴奋 + 对话轮次 > 2 |
| **OBSERVER**（观察者）| 清晰的镜子 | 反思 + 深入阶段 |

> 4 模式启动强度由 § 9.2.1 B.2.B 感知图谱"挑战接纳意愿" / "表达开放意愿" 调度。

#### 7.8.3 5 种对话策略

Neo 根据信号 + 意图自动选：

| 策略 | 说明 | 使用场景 |
|------|------|---------|
| **TELL**（分享）| 直接提供信息 / 经验 | 学员明确求知 / 工具推荐 |
| **ASK**（苏格拉底提问）| 用问题引导学员自己找答案 | 默认策略 / 引导反思 |
| **REVIEW**（回顾串联）| 串联已学内容,建立知识连接 | 跨课程引用 / 回顾巩固 |
| **PLAN**（结构化规划）| 引导制定可执行行动计划 | 对话进入行动阶段 |
| **FRAMEWORK_GUIDED**（框架引导）| 用方法论框架引导分析 | 复杂问题需结构化分析 |

#### 7.8.4 6 种对话阶段

Neo 感知对话所处阶段,自动调整提问节奏和脚手架深度：

| 阶段 | Neo 行为 |
|------|---------|
| OPENING | 建立连接,了解来意 |
| EXPLORING | 展开话题,收集信息 |
| DEEPENING | 追问本质,洞察根因 |
| INSIGHT | 认知突破,确认新理解 |
| ACTION | 制定计划,落地承诺 |
| CLOSING | 总结收获,布置行动 |

#### 7.8.5 6 项核心技能

| 技能 | 说明 |
|------|------|
| **知识工具注入** | 学员消息含知识关键词时,自动检索相关工具 / 方法论注入对话上下文 |
| **经验洞察捕捉** | 实时扫描经验碎片（9 模式匹配,具体 9 模式定义待研发实现）,会话末批量提取最多 3 条洞察 → 高光瞬间生成推荐笔记卡片 |
| **行动计划检测** | 自动识别"我下周试试" / "我打算..." → **推送到代办清单**（呈现位置见 § 9.2.2.1 (3) E）|
| **课堂记忆引用** | 跨 Activity / 跨 Course 精准引用学员历史表现（与 § 9.2.1 B.3 跨课程记忆引用 一致）|
| **复盘引导** | 5 级深度结构化复盘（对应 Bloom 层级,具体 5 级定义待研发实现）|
| **认知链提问** | Bloom 层级递进提问（澄清→深入→反思→假设→行动）—— 同链上不增加认知负荷 |
