### 10.2 首页 Home

> 首页是管理员进入管理端的默认页 —— 类似学员端 § 9.2 大厅的结构：**看板区**（项目监控 + 数据呈现）+ **Ora 区**（管理端 Agent 辅导）。
>
> **首页只看 / 不干预**：管理员通过看板发现异常 → 干预动作走 § 10.5 消息中心（手动催学）+ § 10.4 项目配置（调规则）+ 线下跟进。

#### 首页 = 2 顶层模块

```
首页 Home
├── 看板区（详见 § 10.2.2）── 信息呈现 + 3 维度（项目 / 人员 / 内容）
│   ├── 10.2.2.1 项目维度（6 widget · 总览条 / 时间线 / 项目级聚合 / 活跃度 / 登录状态 / 健康度热力图）
│   ├── 10.2.2.2 人员维度（4 widget · 学员列表 / 标杆排名 / 个体详情 / 多选发消息）+ 进度态 chip 计算逻辑
│   └── 10.2.2.3 内容维度（3 widget · 课程目录看板 / 学习时长分布 / 内容维聚合卡）
└── Ora 区（详见 § 10.2.3）── Home Ora 辅导 = 信息呈现 + 证据解读
    └── 3 能力（呈现 / 解读 / 钻取）+ 4 类卡片产出
```

#### 入口与模块关系

- **入口**：管理员账号登录 → 默认进入首页 Home（详见 § 10.1.4）
- **与其他模块的关系**：
  - 跳 § 10.3 报告中心 → **Report Ora** 接力（信息共享 / 上下文带过去）
  - 跳 § 10.4 项目配置 / § 10.5 消息中心 → **纯 GUI 操作**（无 Ora 参与）
- **看板与 Ora 的关系**：共享底层数据，但**操作上独立**——看 widget 是看板独立体验 / 问 Ora 是 chat 独立体验，本期不做强联动（详见 § 10.2.3.C.1）

#### 10.2.1 业务定义

管理员日常监控项目运行健康度的场域 —— 通过看板呈现 **项目 / 人员 / 内容 三维度数据** + **Ora 辅助解读 / 问答**，让 HR / 项目运营快速识别异常。

**与学员端 § 9.2 大厅 的对应关系**：

| 学员端 | 管理端 |
|--------|--------|
| § 9.2 大厅 = 中枢层（学员入口） | § 10.2 首页 = 管理端入口 |
| § 9.2.2 看板（4 区）| § 10.2.2 看板区（3 维度）|
| § 9.2.1 Neo 辅导（A/B/C 三层）| § 10.2.3 Ora 区（A/B/C 三层）|
| § 9.2.3 笔记悬浮球 | —（管理端无）|

#### 10.2.2 看板区

> 看板区按 **项目 / 人员 / 内容** 三维度组织，每个维度承载若干 widget。

##### 10.2.2.1 项目维度

6 widget（顶部总览条 + 时间线 + 4 卡组）：

**(1) 项目总览条**（顶部 · 项目级 KPI 集中呈现 · 整行）

整行条带承载 6 个项目级 KPI 信息点：

| 信息点 | 示例 |
|------|------|
| 项目名 + 周次 | "基层管理者培训项目 · W6 / 26 · 第 6 周" |
| 距下里程碑 | "距下里程碑 42 天（W12 复盘）" |
| 本周活跃 chip | "↑12% 本周活跃"（环比小箭头）|
| 需关注 chip | "⚠ 6 需关注"（hover 弹学员名单：4 风险 + 2 沉默）|
| 项目完成率 | "28%" 大字 + "项目完成率"标签 |
| 学员总数 | "24" + "学员总数"标签 |

价值：管理员开屏第一眼看到的项目驾驶舱总览 / 浓缩三维度核心信号。

**(2) 项目周期时间线**（整行 · 与学员端 § 9.2.2.1 B 同款）

- **形态**：横向时间条
- **承载信息**：项目周期 + 距结营天数 + 距下里程碑天数 + 5 关键节点（开营 / 中期 / 当前 / 复盘 / 结营）
- **里程碑标记**：标在时间轴上
- **复用**：与学员端 § 9.2.2.1 B 项目时间轴 widget 同款（管理端版本同形态、同数据 / 不同视角）

**(3) 项目级聚合卡**（合并原进度卡 + 复合卡 · 4 数字 + 日/周/月 trend tab）

| # | 数字 | 含义 | 示例 |
|---|------|------|------|
| 1 | **学习时长** | 项目全员累计学习小时数 | 95h（↑18%）|
| 2 | **Neo 互动** | 累计次数 | 298 次（↑12%）|
| 3 | **笔记 / 高光 / 待办** | 全员累计 3 类卡片产出 | 34 · 22 · 8（↑8%）|
| 4 | **人均互动 / 人均学时** | 人均互动次数 / 人均学时 | 12.4 次 · 4.0h（↑12%）|

- 每个数字带 trend 箭头（环比上升 / 下降）
- 顶部 **日 / 周 / 月 Tab 切换**对比窗口（默认 周 / 沿用 0.4.0）
- 价值：项目级总和 + 趋势观察 / 对外汇报核心数据 / 与学员端"个人学习收获卡"风格同源

**(4) 学员活跃度卡**（含日 / 周 / 月 3 Tab 切换）

- **形态**：KPI 数字 + 环比箭头 + 顶部 3 Tab
- **3 Tab 切换数据**：
  - **日**：当天 vs 昨天
  - **周**：当周 vs 上周（默认 / 沿用 0.4.0）
  - **月**：当月 vs 上月
- **指标**：日均活跃天数 / 人 + 环比箭头（上升绿 / 下降红）
- **价值**：参与度风向标 / 掉了能感知

**(5) 学员登录状态分布卡**（占位 · 仅看登录维度）

- **形态**：3 chip 横向并列
- **3 状态定义**（沿用 0.4.0）：
  - 🟢 **活跃** = 3 天内有登录
  - 🟡 **风险** = 3-7 天未登录
  - 🔴 **沉默** = 超过 7 天未登录
- **价值**：识别"该催学员"，驱动到 § 10.5 消息中心（手动催学）

> **占位说明**：本卡只看登录状态，不反映学习进度落后等学习侧信号。学习进度类信号未来可独立成卡（"学习进度分布"）。

**(6) 项目健康度热力图**（3 圈嵌套合一图）

- **形态**：4 周 × 7 天 色块网格 / 每个色块用 **3 圈嵌套** 表达 3 维数据
- **3 圈含义**（外 → 内）：
  - **外圈** = 当日活跃人数（边框深浅）
  - **中环** = 当日学习时长总数（中环颜色）
  - **内核** = 当日 Neo 互动次数（中心点颜色）
- **悬停交互**：弹卡片显示当天三维度详细数字（活跃 X 人 / 学习 Y 小时 / 互动 Z 次）
- **粒度**：按日计入
- **价值**：单图同时承载 3 个信号（人数 + 时长 + 投入深度）/ 识别学习节律 + 投入质量
- **参考**：组件可类比学员端 § 9.2.2.1 A 热力图的"双层色"扩展为"3 圈"

##### 10.2.2.2 人员维度

4 widget（基于 spec 6 widget 改造 · 学习进度状态融入表格 chip / 个人报告改行内展开）。按管理员对人员维度的 3 类核心需求组织：**A · 找需要关注的人**（异常发现）+ **B · 找做得好的人**（标杆识别）+ **C · 看具体学员**（个体诊断）。

> **本节不重复异常学员快速面板**——前面项目维度 § 10.2.2.1 (5) 登录状态分布卡已承担"异常学员发现"职能；人员维度更侧重通过列表 + 详情看具体学员。

**(1) 学员列表表格**（主体 widget · **7 列**）

| 列 | 内容 |
|---|------|
| 学员 | 头像 + 姓名 + 角色 chip |
| 进度 | 进度条 + 完成率% **（可排序）**|
| 最近登录 | 相对时间（如"今天 09:12" / "5 天前"）**（可排序）**|
| 进度态 | chip（领先 / 跟上 / 落后 · 详见 (2) 计算逻辑）|
| 学时 | 累计学习时长（如 12h）|
| 互动 | Neo 互动量（如 320 次）|
| 操作 | 行末 chevron / 点击展开行内 (3) 个体详情 |

**支持工具**：
- **查询**：顶部搜索框 / **按姓名 + 角色 模糊匹配**（本期无部门维度）
- **筛选**：3 项筛选 chip（**登录状态** · 沿用 § 10.2.2.1 (5) 定义 / **进度态** / **进度区间**：< 30% / 30-60% / 60-90% / ≥ 90%）
- **分页**：**不需要分页**（人数 ≤ 100 / 每次最多展开 1 个 / 高度可控）

> **登录状态不作表内列**——通过 § 10.2.2.1 (5) 登录状态分布卡呈现 + 表格筛选器筛选 / 不重复占列。

**(2) 进度态 chip 计算逻辑**（**非独立 widget · 仅供 (1) 表格"进度态"列计算**）

基于群体平均进度 ± 阈值 · 边缘偏向"跟上"。

阈值由 § 10.4.2.4 催学规则配置（**配置即源 / 多处使用**）。

```
变量：
  M = 群体平均进度 = 项目内全部学员的"已完成 Activity 数"平均值
  n = 学员个体已完成 Activity 数

阈值（沿用 § 10.4.2.4 催学规则配置）：
  领先阈值（上界）= M × 进度领先阈值（默认 1.20）
  落后阈值（下界）= M × 进度落后阈值（默认 0.80）

判定（边界含等号 → 偏向"跟上"）：
  领先 = n > 上界          （严格大于上界）
  跟上 = 下界 ≤ n ≤ 上界   （含两端边界）
  落后 = n < 下界          （严格小于下界）
```

**举例**（M = 16）：
- 学员完成 20 → 领先（1.25 > 1.20）
- 学员完成 19.2 → 跟上（等于 1.20 / 含等号）
- 学员完成 13 → 跟上（0.81 > 0.80）
- 学员完成 12 → 落后（0.75 < 0.80）

> **风险记录**（本期不做）：
> - **项目早期边缘问题**：当 M < 3 时（项目刚开营 / 多数人没开始），±20% 阈值过严 / 可能误判 → 本期接受 / 不做早期保护
> - **项目末期边缘问题**：当 M ≥ N × 0.85 时（接近完成），"领先"判定难触发 / 全部归"跟上" → 本期接受 / 不做末期保护

**(3) 标杆排名 widget**（B 类需求 · 2 独立卡 并列）

**2 张独立卡 横向并列**（不 Tab 切换 / 视觉双卡同时呈现）：

- **卡 1 · 按进度 Top 5**：人均完成 X / Y Activity（仅管理端可见）
- **卡 2 · 按互动 Top 5**：累计 Neo 互动次数（仅管理端可见）

每张卡格式：

```
┌─────────────────────────────────┐
│ Top 5 · 按进度                  │
├─────────────────────────────────┤
│ 1. 张三   95%  完成 30/32        │
│ 2. 李四   92%  完成 29/32        │
│ 3. ...                           │
│ 4. ...                           │
│ 5. ...                           │
└─────────────────────────────────┘
```

> **沿用 0.4.0 锁定**：本期排名仅"进度 + 互动"两类 / 不卷能力维度 / 学员端不可见

**(4) 个体详情面板**（C 类需求 · 点击学员行 → **行内展开** Course 进度 / 6 维 / 报告）

行内展开内容：

| 区块 | 内容 |
|------|------|
| **基本信息** | 姓名 / 角色 / 入项目时间 |
| **进度详情** | 复用学员端 § 9.2.2.2 内容目录区组件（Course → Activity 列表 / 状态 chip / 类型 zone）/ 视觉一致 |
| **6 维画像快照** | 复用 § 2.3.3 / § 9.2.1 B.2.B.4 雷达图组件（6 维 × 5 等级）/ **本节是 widget 实现 Truth Source / 可见性约束遵循 § 9.3.3.9.4** |
| **报告列表** | 行内呈现学员**所有 recap 报告 + 所有对练报告**（不开弹窗 / 不开新页）：<br>· **仅完成态 recap 可见**（非完成态不露出 / 与 § 3.4.4.2 / § 9.3.3.2.10 管理端可见性条一致）<br>· 点击某报告 → **新开页面**展示该报告内容（多个报告各自独立 tab / 关闭即可）|
| **CTA 按钮：给该学员发消息** | 跳 § 10.5 消息中心 / 自动填充收件人 / 等待管理员输入消息内容 |

> 不含"学员级活跃热力图"——本期不放（项目维度 § 10.2.2.1 (6) 项目级 3 圈热力图已承担群体活跃信号）。
> 不含"Neo 转述的关键观察"——本期不放。

**(5) 多选批量发消息**（本期不规划 / 待续）

> ⏳ 本期不做 / 留待后续。当前仅支持单学员发消息（CTA 按钮 2 触发）。未来可支持选中多行 → 批量发消息（如批量催学）。

##### 10.2.2.3 内容维度

**核心管理员需求**：内容跑得怎么样？哪个 Course / Activity 是卡点？哪些内容真正激活了学员（高光产出）？投入时长是否合理？

本节 3 个 widget：(1) 课程目录式内容看板（A+B 合并） / (2) 学习时长分布 / (3) 内容维度复合卡。

砍：完成率柱状图（融入目录看板）/ 内容互动热度排行 / 内容卡点 watchlist / 内容覆盖率（高度信息冗余 / 价值不足以撑独立 widget）。

**(1) 课程目录式内容看板**

形态：可折叠的两层目录树（不是图表 / 是表格 + 树）
排序：按 KGP 设定的 **Course 顺序 + Activity 顺序**（即与学员端 § 9.2.2.2 内容目录区一致），不再做"完成率升序"等其他排序——按目录序就是管理员最自然的查找方式。

**第一层 · Course 行**（默认全部展开）

| 列 | 内容 |
|----|------|
| Course 名 | 文字（点击折叠/展开本 Course 下的 Activity 子行）|
| 状态 chip | **未开始**（无人开始）/ **进行中**（有人开始但未全员完成）/ **已完成**（全员完成）|
| Activity 数 | 该 Course 内 Activity 总数 |
| 已开始 n/N | 至少完成 1 个 Activity 的学员数 / 项目总学员数 |
| Course 完成率 | 完成本 Course **全部** Activity 的学员数 / 项目总学员数 % |
| 平均耗时 | 已开始学员的平均累计耗时（分钟）|

> Course 状态 chip **不依赖项目里程碑或 Course 起止时间**——纯按"是否有学员开始 / 是否全员完成"判定，更直接表达 Course 实际进展。

**第二层 · Activity 行**（缩进展示在 Course 下）

| 列 | 内容 |
|----|------|
| Activity 名 | 文字 |
| 类型 chip | lecture / practice / recap |
| **状态分布** | 100% 堆叠条（**灰=未开始 / 黄=进行中 / 绿=已完成**）|
| **已完成** | Z 人（Z%）—— 主数字单独成列，避免堆叠条上挤数字 |
| 平均耗时 | 完成该 Activity 学员的平均耗时（分钟）|
| **高光** | "K 张 [📚库]" —— 数字 + 高光库入口图标 |

**Activity 高光库 · 二级面板**

新增交互——点击 Activity 行的高光数 / 库图标 → 弹出"该 Activity 高光库"面板。

交互层级（三层 modal 链）：

```
L1 主页 内容看板表格
  └─ 点击 Activity 行的"高光"列
     └─ L2 该 Activity 高光库面板（弹出 modal 不新开页）
        ├─ 内容：所有学员在该 Activity 下产生的高光卡集合
        ├─ 分组：按学员人员分类（学员 A / 学员 B / 学员 C ...）
        ├─ 头部：总数 / 涉及学员数 / 产出最多的 Top 3 学员
        └─ 点击单张高光卡
           └─ L3 卡片完整内容（modal 内 overlay 展开 / 不新开页）
              └─ 关闭即回 L2，再关即回 L1
```

**与人员维度个人报告库的关键差异**：

| 维度 | 个体详情面板报告列表（§ 10.2.2.2 (4)）| Activity 高光库（本节）|
|------|----------------------------|---------------------|
| 触发 | 点击学员 → 学员个人报告库 | 点击 Activity → 该 Activity 高光库 |
| 分组 | 按报告类型（recap / 对练）| 按学员人员 |
| 单条点击 | **新开页面**（报告长 / 需独立页对比）| **modal 内展开**（高光卡短 / 单条信息密度低）|
| 退出 | 关闭新页 | 关闭 modal |

**不做**：
- 排序切换（按目录序就是最自然的查找方式 / 信息量本身可控）
- 列筛选器（如需查特定 Activity，浏览器 Ctrl+F）
- 翻页（项目内 Course 数 ≤ 10 / Activity 数 ≤ 100，垂直滚动可承载）
- "看卡在此学员"操作按钮（候选 D 已砍 / 异常学员发现职能由项目维度 § 10.2.2.1 (5) 承担）

**(2) 学习时长分布**

形态：堆叠柱状图（每根柱 = 1 门 Course / 柱内按 Activity 颜色分段）
- 横轴：Course 名（按 KGP 目录序）
- 纵轴：总学习时长（小时）
- 每根柱内段：该 Course 内各 Activity 的时长占比（不同色段）

**主指标**：每门 Course 全员投入总时长（柱高）

**副指标**（hover / 侧栏小字显示）：
- 该 Course 人均时长 = 总时长 / 已开始该 Course 学员数
- 该 Course 占项目总时长的 %

**Activity 颗粒呈现**：
- 鼠标 hover 某个色段 → tooltip 显示 Activity 名 + 该 Activity 总时长 + 平均耗时
- 不强制显式显示 Activity 名（避免拥挤）

**与 widget 1 的关系**：
- widget 1 看"完成率"维度（**多少人完成了**）
- widget 2 看"投入时长"维度（**投入了多少时间**）
- 两者交叉判断："完成率高但时长少 = 内容偏简单"，"完成率低但时长多 = 内容偏难"

**不做**：
- 时间区间筛选器（看的是项目至今累计 / 不需要"近 7 天"等切片）
- 切换为饼图 / 树状图（堆叠柱状已足够表达 Course + Activity 双层）
- 横轴改为 Activity 粒度（Activity 多于 Course 数倍 / 柱状会太密 / Activity 级时长在 widget 1 表格已有）

**(3) 内容维聚合卡**（4 数字 + 日/周/月 tab + trend chip · 与项目级聚合卡同源）

形态：4 数字 + 日 / 周 / 月 Tab 切换对比窗口 + 每个数字带 trend 箭头（环比上升 / 下降）

| # | 数字 | 表达方式 | 含义 |
|---|------|---------|------|
| 1 | 平均完成 Activity 数 / 总数 | "X.X / N" 大字 + trend | 项目内学员的均值水平（学员们平均完成多少 Activity）|
| 2 | 最多完成 Activity 数 / 总数 | "Y / N" 大字 + trend | 项目内的标杆水平（学霸完成了多少 Activity）|
| 3 | 平均完成耗时 | "M min/Activity" + trend | 学员每个 Activity 平均耗时 |
| 4 | 高光产出总数 | "K" 张 + trend | 全员至今高光卡总数（来自 § 2.6 5 类卡片产出 > 高光卡）|

> 不展示"总 Course 数 / 总 Activity 数"作为独立数字——它们是静态信息，融入 1 / 2 的"X.X / N"和"Y / N"分母即可，配合"平均完成"和"最多完成"做比例呈现，比单独固定展示更具洞察。

**砍**：
- "整体完成率 %"（数学上等于 1 / N，重复）
- "内容覆盖率（被互动 Activity / 总 Activity）"（信息价值低）
- "项目总学习时长"（与项目维度复合卡重复）
- "提问总数 / 推送点击数"（候选 C 已砍 / 数据点也不放复合卡）

#### 10.2.3 Ora 区（Home 场景）

> **章节定位**：本节聚焦 **Home 看板上 Ora 的能力**——Home 场景下 Ora 的核心任务 / 3 动作 / 边界。

> **不在本节范围**：Ora 整体角色定位 / Voice / Persona / 双 Agent 实例 / 4 类卡片 Truth Source / HR 拍板 / 不直接联系学员 等共性 → 详见 § 10.1.6 + § 3.3 + § 3.4.4。

> **设计原则**：Ora 比 Neo 简单——把握第一性目标（可举证 ROI 报告）+ Ora 职责边界（HR 拍板 / 不教学 / 不直接联系学员）即可，本节不重复抄整体共性。

##### 10.2.3.A Home Ora 核心任务

**Home Ora = 信息呈现 + 证据解读**

把管理员想知道的信息呈现出来 / 后续决策由管理员自己判断。

| 是 | 不是 |
|----|------|
| ✓ 响应管理员提问 / 在 chat 中按问题组织数据 | ❌ 主动 push 警示弹窗 |
| ✓ 解释数据背后的逻辑（基于已有数据）| ❌ 臆造外部信息（如"疑似公司大会冲突"）|
| ✓ 拉证据 raw 带 memory_id 追溯 | ❌ 给具体方案 / 替管理员决策 |
| ✓ 引导跳目标模块完成后续动作 | ❌ 在 Home 内出长篇产出 |

**3 动作组合**：呈现 / 解读 / 钻取——**层级递进**：数字 → 解释 → 证据。

**4 类卡片产出**（沿用 Neo 5 类卡片产出体系 / Ora 砍 1 类）：askUserQuestion / 证据卡 / 报告引用卡 / 学员快照卡（详见 § 10.2.3.B.4）。

> **与 Neo 7 动作的关系**：Neo 7 动作（讲授 / 解惑 / 讨论 / 考研 / 练习 / 追问 / 反思）来自"模拟 1v1 线下授课老师"的第一性原理类比；Ora 不教学、不替学员行动，所以动作集**显著简单**。Home Ora 3 动作覆盖"看 + 问 + 追"基本闭环，足够支撑 Home 场景。

##### 10.2.3.B 3 能力 + 4 类卡片

###### 10.2.3.B.1 呈现（Present）· 响应"我想看什么"

**定义**：管理员通过 chat 提问 → Ora **按问题动态组织数据呈现**（区别于 widget 已经显示的内容 / 不是 widget 的重述）。

**典型场景**：

| 管理员问 | Ora 呈现 |
|---------|---------|
| "本周项目整体情况" | 三维度核心数字 + 简短描述（如"完成率 65% / 活跃 22 人 / 卡点 2 个 Activity"）|
| "Top 3 风险学员是谁" | 列出 3 名学员 + 各自当前状态 + 关键风险信号 |
| "Course 3 进展如何" | 该 Course 的 Activity 完成情况 + 卡点 Activity + 平均耗时 |

**数据来源边界**：仅来自 Ora 上下文 6 维（§ 3.4.4.2）—— 不引入外部信息（公司日程 / 节假日 / 季度目标 等不在系统采集范围内的信息不可引用）。

###### 10.2.3.B.2 解读（Interpret）· 响应"为什么这样"

**定义**：管理员追问"为什么" → Ora **基于已有数据解释背后逻辑**。

**典型场景**：

| 管理员问 | Ora 解读 |
|---------|---------|
| "为什么本周完成率下降" | "下降集中在 Course 3 Activity 7—— 12 名学员卡在该 Activity / 平均尝试 3 次未过 / Neo 感知到挫败信号 4 次。"|
| "为什么张三登录频率下降" | "张三最后一次登录 5 天前 / 上次会话中 Neo 识别『信念主动性』下降信号 / Course 2 Activity 3 反复尝试 3 次未过。"|

**约束**：
- ✅ 解释来自 Ora 已有上下文的数据（§ 3.4.4.2）
- ❌ 不臆造外部因素（"疑似公司季度大会冲突" / "可能因为节假日影响"——除非这些信息在 program config 中已配置）
- 原则：**不能保证有的信息 → 不能给的能力**

###### 10.2.3.B.3 钻取（Drill-down）· 响应"证据在哪"

**定义**：管理员追问"证据呢 / 给我看 raw" → Ora **拉具体证据带 memory_id 追溯**。

**典型场景**：管理员对"张三落后"的解读不放心 → 问"具体证据"

Ora 钻取返回：
- **登录日志**：W1 日均 45min / W2 日均 28min / W3 日均 12min（带时间戳）
- **Activity 完成详情**：Course 2 Activity 3 尝试 3 次（每次 timestamp + 答题轨迹 memory_id）
- **Neo 感知信号片段**：3 次"挫败信号"（每条带 memory_id 可点击查 raw chat）
- **6 维画像快照**：本周 vs 上周变化（"信念主动性" -0.3 / "认知挑战度" +0.2）

**机制沿用**：§ 3.3.4 转述 + memory_id 追溯——所有 raw 引用都可点击查 source。

###### 10.2.3.B.4 4 类卡片产出（Home Ora 推送形态）

> **4 类卡片完整定义** → § 10.1.6.2（Truth Source）。本节仅说明 Home 场景下的具体推送形态。

| 卡片 | Home 场景下的推送形态 |
|------|------------|
| **askUserQuestion 卡** | "你关心 Top 风险还是 Top 优秀？" / "看本周还是本月？" / 给 2-4 选项让管理员选 |
| **证据卡** | 单条关键证据 + memory_id 追溯链接（如某次 Neo 感知"挫败信号"原始 chat / Activity 答题轨迹），由钻取（B.3）触发 |
| **报告引用卡** | 引用 § 10.3 历史 team report 章节（如"该问题在 W4 中期报告 § 3.2 已分析"）+ CTA"打开报告" |
| **学员快照卡** | 单学员关键状态总览（进度 / 风险 / 6 维画像快照 / 最近行为）+ CTA"展开个体详情面板"（联动 § 10.2.2.2 (4)）|

**关键约束**：4 类卡片**均在 Ora 响应管理员提问中附带产出**——管理员发起对话 → Ora 答 + 附带卡片，**不独立主动 push**（与 § 10.2.3.C.2 一致）。

##### 10.2.3.C 4 条边界（Home 场景特化）

###### 10.2.3.C.1 不与 widget 强联动（本期）

- Ora 与看板共享**底层数据**——Ora 知道 widget 呈现的内容
- 但**不做"点击 widget → Ora 解读该数据"**的强联动交互
- widget 不是 Ora 的触发器；Ora 不是 widget 的注释机
- 管理员看 widget 是看板独立体验 / 问 Ora 是 chat 独立体验 / 共享底层数据但操作上独立

> 后续版本（V0.6.0 +）若开放联动，再单独决策。

###### 10.2.3.C.2 不主动 push / 不警示弹窗

- **异常浮现是看板自身职能**——红黄绿 chip / 异常学员置顶 / 卡点 Activity 高亮 全部在 widget 内呈现
- Ora **不重复主动 push 警示卡** / 不弹窗 / 不顶部 banner
- Ora 是被动响应模式（chat 提问 → Ora 答），不是主动巡视模式

###### 10.2.3.C.3 不臆造外部信息

- Ora 上下文范围（§ 3.4.4.2 Ora 6 维）以外的信息**不进入推理**
- ❌ 公司日程 / 季度大会 / 行业事件 / 节假日 等外部信息（除非已在 program config 中配置）
- 原则：**不能保证有的信息 → 不能给的能力**——避免幻觉式解读

###### 10.2.3.C.4 不出方案 / 不替管理员决策

- Home Ora = 信息呈现 + 证据解读
- ❌ 不主动给"建议你怎么做"方案
- ❌ 不引导决策路径（"你应该跳到 Message 起草" 不是 Ora 主动给）
- 后续决策动作（发消息 / 改配置 / 生成报告）**由管理员自己判断**——管理员决定后通过 Home 左侧导航跳目标模块：
  - 跳 § 10.3 报告中心 → **Report Ora** 接力协作（生成 / 编辑报告）
  - 跳 § 10.4 项目配置 / § 10.5 消息中心 → **纯 GUI 操作**（无 Ora 参与，管理员直接编辑配置 / 起草消息）

#### 10.2.4 系统约束

##### 10.2.4.1 数据更新机制 + 跨管理员一致性 ★

**核心矛盾**：单学员行为实时更新没问题，**几十名学员同时实时推数据 → 数据爆炸 + 看板抖动**，影响管理员阅读。

**本期方案**：**按小时定时更新 + 手动更新动作**——具体频率由研发决定（PM 不锁数字）。

**适用范围**：本机制**仅适用 Home 看板数据**——
- ✅ Home 看板 widget 数据（学员行为聚合 / Neo 感知数据 / 高光数等）
- ❌ § 10.4 项目配置编辑 / § 10.5 消息发送 等**写入操作**（写入立即生效 / 不走小时刷新）
- 具体 database 更新机制由研发决定（CLAUDE.md "不要问技术实现"原则）

**跨管理员一致性**：分两层——

| 层级 | 内容 |
|------|------|
| **数据视图一致** | 同项目下所有管理员（HR + 项目运营）**看到同样的 Home 看板数据**；Ora 对话历史按管理员账号独立存储 |
| **刷新时序一致** | **定时更新** → 全部管理员页面**同步刷新**（看到同一时刻的快照）；**手动更新**（任一管理员触发）→ 其他管理员页面**不会立即跟随刷新**，受 Web 端推送限制 / 建议在**下次动作**触发刷新（如手动 F5 / 切换页面 / 发消息）|

**为什么不做实时推**：
- Web 端 WebSocket 全员同步会增加研发复杂度 / 本期不做
- 数据滞后小时级足够支撑管理员"看 + 决策"节奏
- 学员侧的实时性由 Neo 承担，与管理端的"准实时"是两套节奏

**研发空间**：
- 具体定时频率（30 min / 60 min / 自定义）由研发决定
- 手动更新触发后是后端立即重算 vs 等下次定时窗口，由研发决定
- 跨管理员同步机制（轮询 / SSE / 下次请求触发）由研发决定

##### 10.2.4.2 项目生命周期下的 Home 状态

| 阶段 | Home 状态 |
|------|----------|
| **开营前** | **Home 入口不开放**——管理员账号登录后仅 § 10.4 项目配置 入口可用（详见 § 10.1.4）|
| **开营初期 / 数据未积累** | 数据**如实展示为 0**——不做"数据积累中"占位文案 / 不做"等待数据"动画 |
| **项目结束后**（结营触发后）| 数据**持续更新**（学员仍可在剩余服务期内学习 / 产生数据）；Home 顶部显示**结营状态锚点** chip（如"已结营 · W17"）作为视觉提示 |

##### 10.2.4.3 跨模块衔接

> 跨管理员一致性已并入 § 10.2.4.1，本节聚焦"跳模块时上下文如何传"。

**跳 § 10.3 Report Center 衔接**：
- 管理员从 Home 跳报告中心时，**Home Ora 当前对话上下文 + 关注的数据切片**通过双 Agent 信息共享传给 **Report Ora**
- Report Ora 接力时可直接引用管理员在 Home 提的问题 / 看的数据，避免重复对话
- 共享机制沿用 § 3.3.4 / § 3.4.4 双 Agent 数据共享原则

**Ora 记忆累积**：
- 管理员与 Home Ora 的对话**纳入 Ora 记忆**（按管理员账号 + 项目维度持久化）
- 跨会话调用：管理员下次进 Home，Home Ora "记得"上次问过什么、关注哪些学员
- 跨模块共享：Report Ora 同样可读这部分记忆（如管理员上周在 Home 问过"张三为什么落后" → 本周生成报告时 Report Ora 可主动呈现张三相关分析）
- 不做：跨项目记忆共享（Ora 跨项目对比能力本期不做，详见 § 10.2.4.4）

**跳 § 10.4 / § 10.5 衔接**：纯 GUI 操作，不带 Ora 上下文（详见 § 10.2.3.C.4）。

##### 10.2.4.4 本期不做

| 项 | 说明 |
|----|------|
| **跨项目对比** | CLAUDE.md 提及 Ora 未来跨项目对比能力——**本期 Home 仅看单项目** / 不做同期项目对比 widget / 不做跨项目记忆 → 留 V0.6.0+ |
| **数据时效具体目标值** | 不锁"X 分钟内呈现"等技术指标 → 沿用 CLAUDE.md "不要问技术实现"原则 / 研发决定 |
| **管理端实时推送** | 不做 WebSocket / SSE 强实时同步（详见 § 10.2.4.1）|

#### 10.2.5 GUI 布局

**整体框架**：管理端通用框架 = 左侧导航栏（4 模块直达）+ 右侧主内容区。Home 主内容区采用 **看板区主体 + Ora chat 侧栏** 双栏布局。

**布局形态**：

```
┌──────┬──────────────────────────────────────┬──────────────┐
│      │  Home 看板区（主体 / 占 ~70-75%）       │              │
│ 左侧 │ ┌────────────────────────────────────┐ │  Ora chat   │
│ 导航 │ │ § 10.2.2.1 项目维度（6 widget）     │ │  侧栏       │
│      │ ├────────────────────────────────────┤ │ (~25-30%)   │
│ Home │ │ § 10.2.2.2 人员维度（4 widget）     │ │             │
│ Rpt  │ ├────────────────────────────────────┤ │  对话历史    │
│ Cfg  │ │ § 10.2.2.3 内容维度（3 widget）     │ │  +          │
│ Msg  │ └────────────────────────────────────┘ │  输入框      │
│      │                                       │              │
└──────┴──────────────────────────────────────┴──────────────┘
```

**Ora chat 区**：

| 维度 | 设计 |
|------|------|
| **位置** | Home 主内容区右侧 / 与看板区**并列**（不重叠 / 不悬浮）|
| **占比** | ~25-30% 屏幕宽度 / 默认常驻 |
| **默认态** | **常驻展开**——管理员开屏即可看到 chat 区 / 输入框可立即提问 |
| **可折叠** | 提供折叠按钮（对位 § 9 学员端 chat 区折叠）/ 折叠后看板区扩展至全宽 |
| **滚动** | chat 历史向上滚动 / 输入框固定底部 |

**看板区**：

- **三维度纵向排布**：项目维度 / 人员维度 / 内容维度 自上而下排列
- **每个维度内部 widget 排布**：横向 grid（按 widget 重要性 + 信息密度）
- **滚动**：整体向下滚动（不分页）/ 顶部 sticky 维度切换 tab（可选）

**响应式约束**：
- 本期主要面向 PC 端（管理员主要使用场景）
- 移动端 / 平板端的具体适配由原型阶段决策
- spec 不锁具体断点 / px / rem 数值
