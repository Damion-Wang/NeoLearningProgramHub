# AI老师 — 终止/转向标准 (Kill Criteria)

**v2 - 基于 requirements v0.3.3 更新于 2026-04-13**

**阶段:** Phase 8 — Validation  
**项目:** ai-teacher  
**日期:** 2026-04-13  
**目的:** 定义明确的、可测量的条件，在这些条件下应当终止当前路线或执行根本性转向  
**标签说明:** [Data] = 已验证数据, [Estimate] = 推算值, [Assumption] = 待验证假设, [Opinion] = 主观判断  
**变更说明:** 基于 v0.3.3 完整产品定义，天使客户验证标准更具体（与完整产品形态对齐），新增内容制作能力终止标准

---

## 使用原则

**Kill Criteria不是悲观主义，而是纪律。**

没有预定义的终止标准，团队会陷入"承诺升级"陷阱——投入越多越不愿放弃，即使数据已经明确告诉你方向错了。以下每一条标准都对应一个核心假设；如果假设被否定，继续前进就是在赌博，而非在创业。 [Opinion]

**触发Kill Criteria ≠ 放弃业务。** 它意味着当前路线的关键假设不成立，需要：
1. 暂停当前开发/销售活动
2. 分析否定数据的根因
3. 决定：(a) 调整假设后重试 (b) 转向新方向 (c) 终止项目

## Kill Criteria 清单

### KC1: 天使客户付费意愿不足

> **如果在15-20家天使客户访谈中，少于3家（<20%）明确表示愿意为AI培训产品单独付费（即不将其视为绚星现有平台的免费升级），则终止当前独立产品路线。**

| 维度 | 说明 |
|------|------|
| 对应假设 | CP5: HR愿意为此付费；M2(risk): 客户不将AI视为免费升级 |
| 对应实验 | E1: 天使客户深度访谈 |
| 测量方式 | 访谈Q13（标配升级 vs 独立产品）+ Q14（预算有无）+ Q15（决策权限） |
| 判定时间 | 访谈完成后（约第3周） |
| 标签 | [Assumption] |

**如果触发：**
- 转向方向A：将AI老师功能嵌入绚星LMS平台作为增值模块（功能级而非产品级）
- 转向方向B：改为面向培训服务商（而非企业HR）的工具，让他们用AI老师交付项目
- 不应做的事：降价到接近免费来"证明"客户愿意用——这只证明了免费的东西有人用，不验证商业模式

---

### KC2: 定价天花板过低

> **如果Van Westendorp测试显示最优定价点（OPP）低于800元/账号，且超过60%的受访者认为1,600元/账号"太贵"，则当前收入模型不成立。**

| 维度 | 说明 |
|------|------|
| 对应假设 | PR1: 1,600-1,800元定价可接受 |
| 对应实验 | E2: Van Westendorp定价测试 |
| 测量方式 | Van Westendorp四条曲线交叉点 + 1,600元处的"太贵"累积频率 |
| 判定时间 | 与E1同步（约第3周） |
| 标签 | [Assumption] |

**为什么是800元：** 800元/账号是AI组织培训的入门定价 [Data — brief]。如果NeoLearning高端产品的OPP不高于入门产品，说明客户不认为高端产品有溢价价值。

**如果触发：**
- 转向方向A：采用"低价格+高频次"策略（800元/账号 x 更大规模）
- 转向方向B：按项目包定价（而非按人头），隐藏单价敏感性
- 转向方向C：按效果付费模式（53.4%企业偏好 [Data]）——基础费+效果奖金

---

### KC3: AI测评无法产出可区分的L3行为证据

> **如果AI测评原型的盲测中，专家无法可靠地区分AI行为评估报告与通用完课数据报告（区分准确率<60%），或5位天使客户HR对报告的"汇报可用性"评分低于2.5/5.0，则AI老师的核心差异化不成立。**

| 维度 | 说明 |
|------|------|
| 对应假设 | AI1: AI测评能产出可信L3证据；AI5: 报告质量足以汇报 |
| 对应实验 | E3: AI测评原型测试 |
| 测量方式 | 盲测区分率 + HR可信度评分 |
| 判定时间 | E3完成后（约第4周） |
| 标签 | [Assumption] |

**为什么这是生死线：**
> v0.3.3设计了完整的举证链路（FEEDBACK_COLLECT→referenceSlots→FEEDBACK_REVIEW→报告证据链），资源分配建议中"举证+报告"占40%。如果AI测评的产出与"视频课完课率+选择题成绩"无本质区别，这40%的投入将被浪费，整个产品退化为"更好的线上课体验"——无法支撑1,600元溢价。 [Opinion]

**如果触发：**
- 转向方向A：引入Human-in-the-loop——AI产出初步数据+人工培训师审核评估
- 转向方向B：放弃"AI测评"作为核心差异化，转向"AI高效率内容交付"定位
- 转向方向C：聚焦AI对练单点（得助智能已验证市场），而非全链路培训交付

---

### KC4: Concierge MVP学员参与度低于传统模式

> **如果Concierge MVP中学员完课率低于60%（传统视频课约50%），或学员满意度低于3.5/5.0（传统培训约3.5-3.8），则AI互动式教学并未比传统方式更好。**

| 维度 | 说明 |
|------|------|
| 对应假设 | AI2: AI教学显著提升完课率；AI4: 学员感到千人千面 |
| 对应实验 | E4: Concierge MVP |
| 测量方式 | 系统完课率数据 + 学员满意度调研 |
| 判定时间 | E4完成后（约第7周） |
| 标签 | [Assumption] |

**v0.3.3新增评判维度：**
- 辅导教练引用率：回大厅时辅导教练的必然引用是否自然？学员是否感受到"AI记得我"？
- 闭环反馈完整度：课前采集→课后复盘的闭环是否完成？学员在课后复盘中是否愿意用所学知识分析自己的问题？
- 三栏布局满意度：学员是否觉得三栏布局便于使用？对话区宽度调整是否有人使用？

**如果触发：**
- 诊断根因：是内容问题（横向协同场景设计不够吸引人）？交互问题（AI对话体验不佳）？辅导问题（教练回复质量不够）？闭环问题（记忆链路断裂导致"情感错付"未解决）？布局问题（三栏布局学习曲线高）？
- 不同根因对应不同转向

### KC5: 天使客户无法完成完整的横向协作Course（v0.3.3更新）

> **如果60%天使客户（即3/5家以上）的参训学员中，完成完整横向协作Course（含辅导→授课→对练→测评→报告全流程）的比例低于40%，则产品的完整交付模式不可行。**

| 维度 | 说明 |
|------|------|
| 对应假设 | AI2+AI4+PA2+PA5: 完整学习旅程的可行性 |
| 对应实验 | E4: Concierge MVP |
| 测量方式 | 学员完成全部Activity（授课+对练+测评+报告）的完成率 |
| 判定时间 | E4完成后（约第7周） |
| 标签 | [Assumption] |

**v0.3.3背景：** 横向协作是MVP第一个交付的Course（§3.4）。一个完整的Course包含多个Activity（授课+对练+测评），加上Course结束后的个人报告。如果大多数天使客户的学员无法走完这条完整路径，说明产品的交付链路过长或复杂度过高。

**为什么是"60%客户的40%学员"：**
- "60%客户"确保结论不是个别客户的特殊情况
- "40%学员"是传统培训完课率（~50%）的下限——如果AI互动式教学连传统培训的最低完课率都达不到，完整交付模式不成立
- 注意：这里测量的不是简单的视频观看完成率，而是包含对练和测评在内的全流程完成率，标准已经比传统完课率更宽容

**如果触发：**
- 方案A：缩短Course长度——将8个Course的内容压缩，减少每个Course的Activity数量
- 方案B：降低"完成"标准——允许学员跳过对练或测评，只完成授课即算完成
- 方案C：改变交付节奏——从"6个月自主学习"改为"运营主导的阶段性推进"，增加人工干预

### KC6: 首项目NPS严重不达标

> **如果首个完整项目的HR端NPS低于0（贬损者>推荐者），或Sean Ellis测试中"非常失望"比例低于15%，则产品-市场匹配度严重不足。**

| 维度 | 说明 |
|------|------|
| 对应假设 | 整体PMF假设 |
| 对应实验 | E5: 首项目NPS调查 |
| 测量方式 | NPS计算 + Sean Ellis问题 |
| 判定时间 | E5完成后（约第8周） |
| 标签 | [Assumption] |

**如果触发：**
- 深入分析贬损者（NPS 0-6分）的具体反馈
- 如果贬损原因集中在产品体验→迭代改进后再测
- 如果贬损原因集中在"不需要这个东西"→根本性反思产品定位

### KC7: 母公司客户基础跌破可行性阈值

> **如果绚星CRM数据分析显示活跃客户数（过去12个月有续费或购买行为）低于800家，或目标客群（2,000+人企业）低于400家，则内嵌分发优势的基础被严重侵蚀。**

| 维度 | 说明 |
|------|------|
| 对应假设 | MT5: ≥600家客户有AI采购意向；BM1: 近零CAC |
| 对应实验 | CRM数据分析（非外部实验） |
| 测量方式 | CRM导出：客户活跃度+企业规模分布 |
| 判定时间 | 本周可完成 |
| 标签 | [Data — 母公司客户从3,439降至2,358] |

**如果触发：**
- 评估建立独立销售能力的成本和时间
- 评估PLG（产品驱动增长）渠道的可行性
- 重新测算CAC和单位经济模型

### KC8: 开发进度严重偏离里程碑（v0.3.3更新）

> **如果截至5月底（Week 6），以下三个核心模块中任一个无法达到内测可用状态，则六月版本时间线不可行：(1) AI授课模块（含PPT自主发挥+Quiz+闭环反馈SCO）(2) AI对练模块（含导演-演员分离+至少1个剧本）(3) 三栏布局框架（含看板↔对话双向联动基础版）。**

| 维度 | 说明 |
|------|------|
| 对应假设 | TM2: 六月版本P0范围可交付；P7(risk): v0.3.3范围过大 |
| 对应实验 | 内部开发进度（非外部实验） |
| 测量方式 | 5月底三个核心模块的交付完成度 |
| 判定时间 | 5月底 |
| 标签 | [Assumption] |

**v0.3.3变更：** 原KC7仅看"授课和测评两个模块"。v0.3.3的产品形态要求增加三栏布局框架作为第三个必达模块——因为三端统一三栏布局（ADR-007）是v0.3.3的基础架构，没有三栏布局，整个产品的交互框架不存在。

**为什么看这三个而非全部P0：** 
- AI授课是学员端核心体验
- AI对练是v0.3.3中最复杂的模块（导演-演员分离+剧本控制），也是能力评估的核心来源
- 三栏布局是三端的基础容器，没有它其他功能无处安放
- 测评、报告、记忆工程可以在这三个基础上增量交付

**如果触发：**
- 方案A：大幅缩减P0范围——六月版本仅保留授课+对练+辅导三栏基础版，砍掉测评/报告/记忆闭环/看板双向联动/R3补充功能，全部延期到v0.4
- 方案B：延期里程碑——六月底改为7月底
- 不应做的事：在质量不达标的情况下强行上线——天使客户信任是不可再生资源

### KC9: KGP内容制作能力不足（v0.3.3新增）

> **如果截至5月中旬，KGP团队无法完成横向协作Course的完整内容包（包含：全部SCO脚本+至少1个ASSESSMENT_TAG的tagConfig配置+至少1组闭环反馈脚本FEEDBACK_COLLECT+FEEDBACK_REVIEW+至少1个对练剧本含心理画像/行为模式/成功标准），则"内容驱动行为"的产品架构不可行。**

| 维度 | 说明 |
|------|------|
| 对应假设 | TM3: KGP内容生产能力；PA8(假设): 对练剧本质量 |
| 对应实验 | KGP内容生产进度追踪（内部） |
| 测量方式 | 横向协作Course内容包的完整性和质量评审 |
| 判定时间 | 5月中旬 |
| 标签 | [Assumption — v0.3.3新增，因为"内容驱动行为"架构的可行性取决于KGP能力] |

**为什么这是新的Kill Criteria：** v0.3.3的核心设计哲学是"内容驱动行为"（ADR-004）——平台是引擎，内容是燃料。如果KGP无法按v0.3.3的规格制作内容，再好的引擎也跑不起来。横向协作是MVP第一个Course，如果第一个Course的内容都无法完成，后续7个Course更不可能。

**如果触发：**
- 方案A：简化内容规格——降低v0.3.3对内容的要求，砍掉打标SCO、多版本Segment、闭环反馈SCO，回退到"PPT+Quiz+基础对练"模式
- 方案B：增加KGP资源——外聘培训内容专家，紧急补充KGP团队能力
- 方案C：调整"内容驱动行为"的设计哲学——将部分行为控制从内容层面移回平台层面（如对练情绪天花板由平台硬编码而非剧本控制），降低对KGP的依赖
- 不应做的事：让技术团队"临时凑"内容——内容质量决定产品体验的天花板，技术团队做出的内容在培训专业性上无法保证

## 与v1版本的关键变化

### 新增Kill Criteria
- **KC5（天使客户完成完整Course）**：v0.3.3定义了完整的Course交付流程（辅导→授课→对练→测评→报告），需要验证学员能否走完全程
- **KC9（KGP内容制作能力）**：v0.3.3的"内容驱动行为"设计哲学引入了KGP作为关键依赖

### 更新的Kill Criteria
- **KC4**：新增三栏布局满意度、辅导教练引用率、闭环反馈完整度作为评判维度
- **KC8（原KC7）**：从"授课+测评"扩展为"授课+对练+三栏布局"三个必达模块

### 删除的Kill Criteria
- 无删除，所有v1标准仍然有效

## 执行承诺

**建议创始团队在启动验证实验前，签署以下承诺：**

1. 我们同意以上9条Kill Criteria是预先定义的、不可事后修改的终止标准
2. 如果任何一条被触发，我们承诺在72小时内召开紧急决策会议
3. 决策会议的选项仅限于：(a) 转向具体新方向 (b) 终止项目——不允许"再试试看"
4. 所有实验数据将如实记录，不因结果不利而选择性忽略
5. **新增承诺：KC9（KGP内容能力）的判定由产品负责人和内容专家共同执行，不由KGP团队自评**

## Flags

**Red Flags:**
- KC1和KC3是"致命级"Kill Criteria，但它们最早要到Week 3-4才能判定——而MVP开发可能在Week 1就已全面启动。建议在验证结果出来之前，开发优先做"无论假设成立与否都需要的基础设施"（如三栏布局框架、数据模型、对话引擎），延后"仅在假设成立时才需要的功能"（如复杂报告模板、看板双向联动）。 [Opinion]
- **KC9（KGP内容能力）是v0.3.3引入的新时间敏感性风险**——如果5月中旬KGP无法完成横向协作Course的内容包，5月底的内测和6月底的MVP都将延误。建议本周就开始KGP内容制作的能力评估。 [Opinion]
- KC7（活跃客户数）是本周就能完成的验证，且成本为零。如果这一条被触发，其他所有实验的意义都需要重新评估。 [Opinion]

**Yellow Flags:**
- Kill Criteria的样本量较小（15-20家访谈、5位HR评估），在统计学意义上不够严谨。但在种子阶段，方向性信号比统计显著性更重要。 [Opinion]
- KC5（完成完整Course）的验证需要4-5周的实际学习周期，是所有Kill Criteria中验证周期最长的。如果等到Week 7才判定，可能已经错过了调整窗口。建议在Week 4-5就开始监控学员的Course进度数据，提前发现趋势。 [Opinion]
- 部分Kill Criteria之间存在依赖关系（如KC1触发后KC2-KC6变得无意义；KC9触发后KC4-KC5无法执行），执行时需注意序列逻辑。 [Opinion]
