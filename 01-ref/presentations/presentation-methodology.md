# 会议录音 → 汇报演示文稿：全流程方法论

> 本文档记录了一次完整的「会议录音转汇报材料」生产过程，覆盖需求澄清、内容结构化、视觉设计、HTML 技术实现、人机协作模式五个维度。目标读者：团队成员（可复制操作）和 AI Agent（可程序化复用）。

---

## 一、任务概述

**输入**: 一份 113 行的会议录音转写文本（说话人A独白，约 12 分钟）
**输出**: 10 页科技感 PPT 风格 HTML 演示文稿，支持键盘翻页、触屏滑动
**用时**: 约 1 小时（含多轮需求确认和迭代修改）
**汇报结果**: 用户反馈"非常满意，汇报效果很好"

---

## 二、内容策略：从非结构化录音到结构化叙事

### 2.1 录音文本的特征分析

会议录音转写文本有几个典型特征，直接影响信息提取策略：

| 特征 | 表现 | 处理策略 |
|------|------|----------|
| **口语化** | "就是""然后""那么"等填充词频繁出现 | 剥离语气词，提取核心语义 |
| **非线性** | 说话人在不同主题间跳跃、回溯 | 按主题聚类，而非按时间顺序 |
| **模糊数字** | "大概""约""1600到1800" | 保留区间表达，不擅自取中间值 |
| **隐含逻辑** | 产品线之间的关系未显式说明 | 推断并向用户确认 |
| **专有名词不确定** | "划斜"（横向协同）、"KGP" | 保持原文表述，除非用户修正 |

### 2.2 信息提取的四层模型

从录音文本中提取信息时，按以下四层递进：

```
第一层：事实提取    → 有哪些产品？各自什么时间节点？定价多少？
第二层：关系梳理    → 产品之间是什么关系？版本之间如何演进？
第三层：逻辑重构    → 对特定受众，什么叙事顺序最有说服力？
第四层：信息分级    → 哪些是必须展示的？哪些口头补充即可？
```

**本次案例的具体提取结果：**

**第一层 — 事实清单：**
- 两大产品方向：培训项目 (NeoLearning) 和智能工具 (Neo Course)
- 培训项目包含：基层管理者（V1标准版/V2高级版/销售场景版）、AI组织
- 工具方向包含：AI伴学（视频老课升级）、新产品赋能（微型项目）
- 彩蛋：沙盘模拟与沙盘教练（9月底）
- 里程碑：5月中→6月初→6月底→7月中→7月底→8月初~中旬
- 定价和业绩目标的完整数字

**第二层 — 关系图谱：**
```
NeoLearning (培训项目)
├── 基层管理者
│   ├── V1 标准版 → Online → 混合式
│   ├── V2 高级版（中层）
│   └── 销售场景版 → Online → 混合式
└── AI 组织（仅 Online）

Neo Course (生产平台)
├── AI伴学（4门老课升级）
└── 新产品赋能 Mini Project（萃取→制课→学习→报告）

彩蛋：沙盘模拟与沙盘教练（增购服务）
```

**第三层 — 叙事重构（面向高层领导）：**

原始录音的叙述顺序是：项目方向 → 时间线穿插 → 工具方向 → 业绩数字。这种顺序适合团队内部讨论，但面向高层汇报时需要重构：

最终采用的叙事线：
```
封面（定调）
→ 全景（两大交付形态，建立认知框架）
→ 深入产品线A（NeoLearning：基管 → AI组织）
→ 深入产品线B（Neo Course：工具 → 应用场景）
→ 彩蛋（沙盘，制造期待感）
→ 时间线（里程碑汇总，建立信心）
→ 业绩（数字收尾，给出承诺）
→ Q&A
```

**关键决策**：里程碑页放在业绩页前面，而非原稿中穿插在产品介绍之间。原因：先让领导看到"我们要做什么"和"什么时候做完"，再看"能赚多少钱"，说服力更强。这个页面顺序经过用户确认后调整。

**第四层 — 信息分级：**

| 级别 | 内容 | 处理方式 |
|------|------|----------|
| 必须展示 | 产品方向、里程碑、业绩目标 | 独立成页 |
| 辅助展示 | 定价细节、版本差异 | 放在表格中 |
| 口头补充 | 马森老师、具体技术实现 | 不放入材料 |
| 刻意隐藏 | 沙盘还在调研 | 用"彩蛋"定性，降低承诺感 |

### 2.3 录音中的"坑"和处理方式

| 录音原文 | 问题 | 处理 |
|----------|------|------|
| "经济协同" | 实为"积极协同"，录音转写错误 | 用户在迭代中修正 |
| "KGP制课" | 专有名词，外部读者可能不懂 | 用户后来统一为"Neo Course 生产平台" |
| "3门自评课" | 后续用户补充了第4门"王牌销售" | 迭代中追加 |
| "5月底天使客户" | 用户最终不想放这个节点 | 多轮确认后删除 |
| "600到800万" | 录音中的总数，但各产品线加总应为1300~1600万 | 按产品线分别计算后汇总，数字更准确 |

### 2.4 录音之外的信息：用户的隐性知识

录音转写文本不是唯一的信息源。在迭代过程中，用户多次补充了录音中没有明确说出的信息：

| 用户补充 | 录音中的线索 | 说明 |
|----------|-------------|------|
| "王牌销售"是第4门老课 | 录音只提了3门 | 用户掌握最新规划，录音可能是旧版 |
| 基管V1 Online 应归6月初而非6月底 | 录音说"6月底之前完成" | "之前"的模糊性 → 实际节奏更紧 |
| 新产品赋能是6月底的 mini project | 录音中没有明确说这个时间 | 用户有会后的决策更新 |
| "本质上是一个产品团队" | 录音按"两大方向"分别讲述 | 组织架构认知比录音更新 |

**核心经验：录音是信息基线，不是信息全集。** AI 必须意识到用户脑中有比录音更新、更准确的信息。当两者冲突时，以用户当前确认为准。处理方式：先基于录音生成初稿，再通过迭代让用户补齐隐性知识。

### 2.5 时间粒度的选择：不要过度聚合

初版里程碑将6月的所有事项归为"6月底"一个节点。用户指出"有一些六月底和六月初要做出区别"。

**错误做法**：把同一个月的事项都堆在月底。
**正确做法**：根据实际交付节奏拆分时间节点。

时间粒度的选择取决于受众：
- **高层领导**：关心"哪个月有什么成果"，需要半月级粒度（月初/月中/月底）
- **项目管理**：需要周级粒度
- **团队执行**：需要日级粒度

本次面向高层汇报，最终采用了6个时间节点而非按月聚合的4个，信息密度更合理。

### 2.6 "彩蛋"框架：如何展示未确定的事项

沙盘模拟处于调研阶段，放入正式里程碑会变成"承诺"，但完全不提又浪费了一个展示团队野心的机会。

解决方案：用**"彩蛋"框架**包装：
- 视觉上独立成页，用特殊光效（琥珀色光晕）制造仪式感
- 用"彩蛋产品"标签定性，暗示"这是 bonus，不是 KPI"
- 描述中用"计划""调研中"等词降低承诺感
- 不放入里程碑时间轴，避免被当作硬节点追踪

这个框架适用于所有"想秀但不想被追责"的内容。

### 2.7 命名变更的连锁反应

本次经历了多轮命名变更，每次都必须全局同步：

```
Project → NeoLearning        （影响：封面、总览卡片、第3/4页标签、业绩表格、甘特图）
KGP制课 → Neo Course 生产平台 （影响：封面、总览卡片、第5/6页标题标签、甘特图）
沙盘推演 → 沙盘模拟与沙盘教练 （影响：第7页标题、描述、服务商描述）
两大产品方向 → 两大交付形态   （影响：第2页标题和副标题）
经济协同 → 积极协同            （影响：第5页标签、第9页表格）
```

**经验**：每次命名变更后必须用 Grep 搜索旧名称，确认返回 0 结果。封面页是"命名契约"——任何改名都必须回查封面是否已同步。

### 2.8 删减的判断：less is more

用户主动要求删掉"下半年收入目标汇总"页（大字展示 1,300~1,600 万的总览页）。原因：业绩目标拆解表已经包含了所有数字信息，总览页与之重复。

**经验**：当两页传递相同信息、只是形式不同时，保留信息密度更高的那页。面向高层的汇报，精炼比华丽更有说服力。10 页比 11 页好。

## 三、需求澄清：多轮提问的设计方法

### 3.1 为什么要多轮提问而非一次性询问

一次性抛出 10+ 个问题会让用户认知过载。将问题分为 3 轮，每轮 3-4 个问题，每轮聚焦一个层次：

```
第一轮（骨架层）：呈现形式、受众、数据深度、视觉风格
第二轮（内容层）：内容板块、产品命名、篇幅
第三轮（细节层）：团队名称、标题文案、人物信息
```

### 3.2 提问设计的原则

**原则一：每个问题必须附带选项和解释**

不要问开放式问题"你希望什么风格？"，而是给出 2-4 个具体选项，每个选项带一句解释。原因：
- 降低用户决策成本
- 避免用户给出模糊回答（如"好看一点"）
- 选项本身就是在展示你的专业判断

**原则二：第一个选项放推荐项**

用户倾向于选择第一个选项。把你认为最合适的选项放在首位并标注"（推荐）"，同时在描述中说明推荐理由。

**原则三：关键信息不靠猜测**

以下信息必须向用户确认，不能自行推断：
- 公司/团队名称
- 汇报对象（决定信息分级策略）
- 是否展示敏感数字（如具体定价和业绩目标）
- 特定人名是否需要出现

**原则四：预留 "Other" 出口**

每个问题自动提供 "Other" 选项，让用户可以跳出你预设的框架。本次案例中，多个关键决策来自用户在 Other 中的自由输入：
- "用会议中的名称" → 不做品牌美化
- "改成丰富项目交付的形态类型" → 替换 ARPU 术语
- "标准化的基层管理者online培训" → 补充6月初的里程碑内容

### 3.3 用户反向提问 AI 的处理

本次出现了一个特殊场景：用户问 AI "你还记得是什么吗"——用户在测试 AI 是否理解了源材料。

这种情况的正确处理方式：
1. **坦诚回答你记住了什么**：列出你从录音中提取的相关信息
2. **明确你的不确定性**：如果信息模糊，说"录音中这部分指的是 X，但你提到的6月底 mini project 我不确定是否同一个"
3. **用提问工具给出选项让用户确认**：不要猜测后直接执行

**错误做法**：假装记住了随便填一个。
**正确做法**：展示你的理解 + 承认边界 + 结构化确认。

这种模式在制课 Harness 中会频繁出现——用户给出不完整指令时，AI 需要表达"我知道 A，但不确定 B，请确认"。

### 3.4 提问的时机：不仅在开始时

除了开始时的三轮需求确认，在以下时机也应主动提问：

| 时机 | 示例 | 原因 |
|------|------|------|
| 发现录音信息模糊时 | "6月底的 mini project 具体指什么？" | 避免错误假设 |
| 用户提出修改但指向不明确时 | "这个和横向协同是同一个吗？" | 确保理解一致 |
| 涉及专业术语时 | "ARPU 这个词要换成更直白的表述吗？" | 替用户考虑受众理解度 |
| 用户要求确认内容完整性时 | 用多选列表让用户勾选保留项 | 结构化确认，避免遗漏 |

### 3.4 AskUserQuestion 工具的使用策略

```yaml
设计选项时:
  - 选项数量: 2-4 个，不超过 4 个
  - 选项措辞: 简短标签（1-5词）+ 一句描述
  - multiSelect: 当选项不互斥时开启（如"保留哪些里程碑"）
  
分轮策略:
  - 每轮 3-4 个问题（1-4 个是工具限制）
  - 同一轮内的问题应属于同一决策层级
  - 前一轮的答案可能影响后续轮的问题设计
  
何时不用选项而用自由文本:
  - 需要用户提供具体文字内容（标题、名称）
  - 选项无法覆盖所有可能性
  - 此时仍可提供选项作为参考起点，用户通过 Other 自定义
```

## 四、视觉设计：科技感暗色主题的设计系统

### 4.1 设计决策的依据

用户选择了"科技感"风格，且产品本身是 AI 教育方向。设计决策：

```
基调：深色背景 + 发光效果 = 科技感/未来感
配色：多色系渐变（蓝、紫、青、绿、琥珀）= 区分不同产品线/时间节点
字体：Inter（英文）= 现代感、高可读性
布局：大量留白 + 卡片 = 信息分区清晰，投屏可读
```

### 4.2 色彩系统设计

核心原则：**每种颜色对应一个语义，全局一致**。

```css
--accent-blue:   #60a5fa  /* NeoLearning 培训项目 */
--accent-purple: #a78bfa  /* 扩展版本、高级功能 */
--accent-cyan:   #22d3ee  /* 平台交付、6月里程碑 */
--accent-green:  #34d399  /* 混合式学习、7月里程碑 */
--accent-amber:  #fbbf24  /* 彩蛋/特别关注、Neo Course工具 */
--accent-rose:   #fb7185  /* 销售场景（甘特图中使用） */
```

背景层次：
```css
--bg-deep:     #0a0e1a           /* 页面底色 - 深海蓝黑 */
--bg-card:     rgba(255,255,255,0.04)  /* 卡片 - 微透明 */
--bg-card-hover: rgba(255,255,255,0.08) /* 卡片悬停 - 稍亮 */
```

文字层次（三级灰度）：
```css
--text-primary:   #f1f5f9  /* 标题、重要信息 */
--text-secondary: #94a3b8  /* 正文、说明 */
--text-muted:     #64748b  /* 提示、注脚 */
```

### 4.3 光效与氛围

科技感的核心不是颜色深，而是**光与影的对比**：

```css
/* 每个 slide 的背景使用径向渐变模拟光晕 */
radial-gradient(ellipse at 30% 20%, rgba(96,165,250,0.06) 0%, transparent 50%)

/* 封面页使用 3 个大尺寸模糊光球 */
.orb { filter: blur(80px); opacity: 0.4; }

/* 卡片 hover 时产生辉光 */
box-shadow: 0 0 40px rgba(96,165,250,0.15);

/* 网格线增加纵深感 */
background-image: linear-gradient(rgba(255,255,255,0.02) 1px, transparent 1px);
background-size: 60px 60px;
```

### 4.4 标签(Tag)系统

用彩色标签区分产品线和阶段，设计要点：
- 背景色用对应颜色的 15% 透明度
- 文字用对应颜色的实色
- 圆角 20px，字号 12px，font-weight 600

```css
.tag-blue { background: rgba(96,165,250,0.15); color: #60a5fa; }
```

这种低饱和背景 + 高饱和文字的组合在深色背景上可读性极好。

## 五、HTML 技术实现：PPT 化的关键方案

### 5.1 架构选择：纯 HTML + CSS + Vanilla JS

**不使用任何框架和库**。原因：
- 单文件交付，用浏览器直接打开，零依赖
- 无需构建步骤，非技术人员也能使用
- 文件体积小（< 50KB），可邮件直接发送
- 唯一外部依赖：Google Fonts（Inter），离线时会 fallback 到系统字体

### 5.2 翻页系统实现

核心机制：所有 slide 使用 `position: absolute` 堆叠，通过 CSS class 切换控制显隐和动画方向。

```
所有 slide 默认状态：opacity: 0; transform: translateX(60px);  （右侧待命）
激活状态 .active：opacity: 1; transform: translateX(0);        （居中显示）
退出动画分方向：
  .exit-left：transform: translateX(-60px)  （向左退出 = 前进时的旧页面）
  .exit-right：transform: translateX(60px)  （向右退出 = 后退时的旧页面）
```

**踩坑记录：翻页方向 Bug**

初版实现中，`goTo()` 函数没有区分前进和后退方向，导致向左翻页时新页面仍从右侧滑入（方向反了）。

修复方案：
```javascript
function goTo(idx) {
  const forward = idx > current;
  
  // 旧页面：前进时向左退出，后退时向右退出
  slides[current].classList.add(forward ? 'exit-left' : 'exit-right');
  
  // 新页面：临时禁用 transition，设置初始位置，再启用 transition 触发动画
  const newSlide = slides[idx];
  newSlide.style.transition = 'none';
  newSlide.style.transform = forward ? 'translateX(60px)' : 'translateX(-60px)';
  void newSlide.offsetWidth;  // 强制 reflow
  newSlide.style.transition = '';
  newSlide.style.transform = '';
  newSlide.classList.add('active');
}
```

关键技巧：`void newSlide.offsetWidth` 强制浏览器 reflow，确保 `transition: none` 和后续的 `transition: ''` 不会被浏览器合并优化掉。

### 5.3 多种交互方式支持

```javascript
// 键盘：方向键、空格、回车、Backspace、Home、End
document.addEventListener('keydown', handler);

// 鼠标：底部导航圆点 + 左右箭头按钮
navDots.forEach(dot => dot.onclick = () => goTo(i));

// 触屏：滑动手势（50px 阈值）
document.addEventListener('touchstart/touchend', handler);
```

### 5.4 动画系统

入场动画使用 `.fade-up` class + CSS animation：

```css
@keyframes fadeUp {
  from { opacity: 0; transform: translateY(20px); }
  to { opacity: 1; transform: translateY(0); }
}

/* 子元素依次延迟入场 */
.slide.active .fade-up:nth-child(2) { animation-delay: 0.1s; }
.slide.active .fade-up:nth-child(3) { animation-delay: 0.2s; }
```

**设计要点**：动画延迟不超过 0.4s，否则观众会感觉"在等页面加载"。

### 5.5 各类页面的布局模式

本次共使用了以下布局模式：

| 页面类型 | 布局 | 适用场景 |
|----------|------|----------|
| 封面/结尾 | 居中 + 光球背景 | 开场定调、收尾 |
| 双卡片 | `.card-grid.cols-2` | 两个并列概念的对比 |
| 三卡片 | `.card-grid.cols-3` | 多版本并列展示 |
| 流程图 | `display:flex` + 箭头 | 萃取→制课→学习→报告 |
| 里程碑 | `position:absolute` 按时间比例定位 | 时间轴 + 里程碑卡片 |
| 数据表格 | `<table>` + 圆角行 | 业绩目标拆解 |
| 聚焦卡 | 居中 + 光晕背景 | 彩蛋页面，制造仪式感 |

### 5.6 里程碑页面的布局演进

这个页面经历了三次重大重构：

**V1 甘特图式（条形图）：**
```
每行一个 version-row，用 left% 和 width% 控制条形位置和长度
```
问题：看起来像项目管理工具，不像汇报材料。用户反馈"不一定是甘特图，是里程碑"。

**V2 Grid 布局里程碑：**
```
6列 grid（对应6个月），里程碑卡片用 grid-column 定位
```
问题：grid 的空占位 `<div></div>` 导致布局不稳定，部分卡片显示异常。

**V3 绝对定位里程碑（最终版）：**
```
父容器 position:relative + min-height
每个里程碑 position:absolute + left:百分比 + top 错落
```
优势：精确控制位置，卡片上下错落避免重叠，视觉层次清晰。

**经验总结**：对于非均匀分布的时间轴元素，绝对定位比 grid 更可控。grid 适合均匀栅格，不适合按实际时间比例放置的里程碑。

## 六、人机协作模式：迭代式交付

### 6.1 协作节奏

整个过程的协作模式可以抽象为：

```
需求澄清（3轮提问）→ 一次性生成完整初稿 → 多轮定向修改
```

**不是**逐页确认后再做下一页，而是先出完整版，再迭代。原因：
- 用户需要看到全貌才能做出局部判断
- 全局一致性（配色、命名、口径）只有在完整版中才能校验
- 减少来回轮次，提高效率

### 6.2 修改请求的类型分类

用户的修改请求可以分为以下类型：

| 类型 | 示例 | 处理方式 |
|------|------|----------|
| Bug 修复 | "向左翻页有bug" | 立即定位修复，无需确认 |
| 内容修正 | "经济协同写错了，是积极协同" | 全局替换，无需确认 |
| 结构调整 | "甘特图调整到业绩页之前" | 移动 slide，更新 id 序号 |
| 内容增删 | "增加一个王牌销售""删掉这一页" | 局部编辑，注意连带影响 |
| 措辞优化 | "提升客户ARPU是什么意思" | 解释后提供替代选项 |
| 概念重定义 | "两大产品方向改成两大交付形态" | 理解背后意图，调整措辞 |
| 完整性质疑 | "里程碑那一页好像丢内容了" | 核对源材料，用提问工具逐项确认 |

### 6.3 截图驱动的修改模式

用户多次通过截图指定修改区域，这是一种高效的沟通方式：
- 截图明确了修改范围（不需要说"第几页第几行"）
- AI 通过截图中的文字内容定位到代码位置
- 减少歧义，提高修改准确率

**处理流程**：
1. 识别截图中的关键文字（如标题、标签文字）
2. 用 Grep 在文件中搜索这些文字
3. 读取上下文确认位置
4. 执行 Edit

### 6.4 概念重定义：识别措辞背后的战略意图

用户说"两大产品方向换个表述，变成两大交付形态，本质上是一个产品团队"。

表面上是改 4 个字，实际上是**组织认知的重新框架**：
- "两大方向" → 暗示两个独立团队，各走各的
- "两大形态" → 暗示同一个团队，两种输出方式

AI 不能只改文字。要理解改动背后的意图，并调整副标题从"以 X 打标杆、以 Y 做规模"变为"一个产品团队，两种交付模式"——把"一个团队"这个战略信号放进去。

**经验**：当用户修改的是标题级别的措辞时，通常不只是文字偏好，而是认知框架的调整。要问自己：这个改动是否影响副标题、正文描述、甚至页面结构？

### 6.5 "内容丢失"可能是渲染问题

用户反馈"里程碑那一页好像丢内容了"。排查后发现内容完整存在于 HTML 中，问题出在 CSS Grid 布局导致部分卡片没有正确渲染。

**处理流程**：
1. 先 Read 文件确认数据是否真的丢了
2. 如果数据存在，判断是布局/渲染问题
3. 不要急于补数据，而是修复布局

这次经历直接导致了里程碑页面从 Grid 布局重构为绝对定位（V2 → V3），从根本上解决了渲染稳定性问题。

**推广经验**：用户说"少了东西"时，有三种可能性要依次排查：
1. 数据真的没写进去（代码级问题）
2. 数据写了但没渲染出来（CSS/布局问题）
3. 数据展示了但用户没注意到（信息层级问题）

### 6.6 修改的连锁效应

一次内容调整往往触发多步连锁操作。本次的典型案例：

```
用户："Neo Course 产品上线提前到七月末"
  → 步骤1：从8月里程碑中删除 "Neo Course 新产品赋能交付"
  → 步骤2：在7月里程碑中添加 "Neo Course 生产平台上线"
  → 步骤3：7月节点文案变成 "7月中旬 — 7月底"
  
用户："七月中旬和七月底拆成两个里程碑"
  → 步骤4：拆分为两个独立的绝对定位卡片
  → 步骤5：调整两个卡片的 left% 和 top 值避免重叠
```

一个"提前到七月末"的需求，最终触发了 5 步修改。**经验**：执行修改前，先在脑中推演完整的影响链，一次性规划好所有步骤，而不是改一步看一步。

### 6.7 人物信息的政治敏感性

用户选择"马森老师不需要出现"，用"线下讲师"替代。在商业汇报中，人名的出现与否往往有组织政治含义：
- 提人名 = 绑定此人的信用，如果此人离职或关系变化会很尴尬
- 不提人名 = 保持灵活性，讲师可替换

**经验**：在汇报材料中涉及具体人名时，必须向用户确认。默认用角色（"线下讲师"）而非人名。

### 6.8 何时主动提问 vs 直接执行

```
直接执行（无需确认）：
  - 用户指令明确且无歧义："经济协同改成积极协同"
  - Bug 修复：翻页方向错误
  - 纯技术操作：slide id 重新编号

主动提问（需要确认）：
  - 内容完整性："里程碑有哪些节点？"
  - 概念理解："6月底的 mini project 具体指什么？"
  - 用户偏好："ARPU 要换成什么表述？"
  - 超出源材料的信息："这个内容录音中没提到"
```

## 七、工具调用模式（面向 Harness 产品化）

### 7.1 工具使用全景

本次任务使用的核心工具和调用策略：

| 工具 | 使用场景 | 调用次数 | 策略要点 |
|------|----------|----------|----------|
| **Read** | 读取录音源文件、读取HTML定位修改区域 | 多次 | 首次完整读取提取信息，后续按行号范围精确读取 |
| **Write** | 生成完整HTML初稿 | 1次 | 一次性写入完整文件，不分段写入 |
| **Edit** | 所有迭代修改 | 约20次 | 精确匹配旧文本替换，`replace_all` 用于全局重命名 |
| **Grep** | 定位修改位置、检查遗漏 | 约15次 | 修改前定位、修改后验证，确保无遗漏 |
| **AskUserQuestion** | 需求澄清和内容确认 | 6次 | 分层提问，选项+描述+Other |

### 7.2 关键工具链模式

**模式一：初始生成**
```
Read(源文件) → AskUserQuestion(3轮) → Write(完整HTML)
```

**模式二：定向修改**
```
Grep(定位关键词) → Read(确认上下文) → Edit(替换)
```

**模式三：全局重命名**
```
Grep(搜索旧名称,确认所有出处) → Edit(replace_all=true)
```

**模式四：结构调整（移动页面）**
```
Read(确认页面边界) → Edit(删除原位置) → Edit(插入新位置) → Grep(验证id连续性) → Edit(修复id)
```

**模式五：内容确认**
```
Read(重读源材料) → AskUserQuestion(列出选项让用户勾选) → Edit(按确认结果修改)
```

### 7.3 Grep 的战术价值

Grep 在迭代修改中承担三个关键角色：

1. **修改前定位**：用户说"删掉这一页"时，通过截图中的文字搜索定位代码位置
2. **修改后验证**：全局替换后，搜索旧词确认"0 matches"证明替换完整
3. **一致性检查**：修改一个名称后，搜索相关变体确保全局一致

### 7.4 截图作为定位输入的处理链

用户发送截图指定修改区域时，AI 的完整处理链：

```
1. 视觉识别：从截图中提取关键文字（标题、标签、数据）
2. Grep 定位：用截图中的独特文字在文件中搜索
3. 上下文确认：Read 该位置前后 20-30 行，确认是目标区域
4. 执行修改：Edit 精确替换
```

本次用户发送了 4 次截图，每次都成功通过截图文字完成定位。截图中最有定位价值的元素是：
- 页面标题（如 "6月底实战目标"）
- 标签文字（如 "启航""经济协同"）
- 独特的数据（如 "200万""500万"）

**面向 Harness 产品化**：如果制课 Harness 支持截图输入，应将"截图 → 文字提取 → 文件定位 → 修改"封装为标准能力。

### 7.5 Edit 工具的使用经验

**经验一：old_string 必须在文件中唯一**
- 如果目标文本不唯一，需要包含更多上下文使其唯一
- 对于 HTML，通常用注释或父级标签作为定位锚点

**经验二：replace_all 仅用于确定性全局替换**
- 适用场景：产品重命名（"KGP" → "Neo Course"）
- 不适用：相同文本在不同上下文中含义不同

**经验三：复杂结构变更用"删后插"**
- 移动一个 slide 时，先 Edit 删除原位置内容，再 Edit 在新位置插入
- 不要尝试一次 Edit 完成跨区域移动

**经验四：old_string 会因前序修改而过时**
- 在一轮对话中多次修改同一文件时，前几次修改会改变文件内容
- 后续 Edit 的 old_string 必须基于当前文件状态，不能基于记忆中的旧版本
- 本次踩坑案例：修改7月里程碑时，old_string 用了修改前的注释文本（"7月中旬：两项"），但注释已在上一步被改为"7月中旬 (位置 ≈ 58%)"，导致 Edit 失败
- **应对策略**：如果不确定当前文件状态，先 Grep 或 Read 获取最新内容再 Edit

**经验五：删除 slide 后必须修复后续 id**
- 删除第10页后，原第11页（Q&A）的 id 从 s11 变为 s10
- 遗漏 id 修复不会导致显示错误（JS 按 DOM 顺序遍历），但会导致代码可维护性下降
- 移动 slide 时同理：本次移动甘特图后出现了两个 id="s8" 的 bug，通过 Grep 检查 `id="s\d+"` 发现并修复

## 八、可复用的经验法则

### 8.1 内容层

1. **录音转写 ≠ 真相**：转写可能有错别字（经济→积极），专有名词可能拼写错误，数字可能口误。始终向用户确认关键事实。

2. **信息分级决定页面结构**：不是所有录音内容都要放进汇报。根据受众分级：必须展示 → 独立成页，辅助展示 → 放入表格/列表，口头补充 → 不放入材料。

3. **叙事顺序 ≠ 录音顺序**：录音是发散的讨论，汇报要重构为有说服力的线性叙事。"是什么→要做什么→什么时候做完→能赚多少" 是面向高层的经典结构。

4. **数字要可追溯**：业绩目标不能只给总数。"200万"要能拆解到"1600元/账号 × 30人 × 20个项目"，让数字有底气。

### 8.2 视觉层

5. **颜色即语义**：在整份材料中，一种颜色只代表一个含义。不要因为"好看"而在不相关的地方使用同一颜色。

6. **暗色主题的可读性陷阱**：深色背景上的文字不能用纯白（#fff），用 #f1f5f9 更柔和。次要信息用更暗的灰度，形成层次。

7. **光效要克制**：blur 光球、glow shadow 营造氛围，但过多会显得杂乱。封面页可以多一些，内容页要收敛。

### 8.3 技术层

8. **单文件优先**：汇报材料追求"打开即用"，不要引入构建步骤。一个 .html 文件胜过一个需要 npm install 的项目。

9. **翻页动画必须区分方向**：前进（向右翻）和后退（向左翻）的入场方向必须相反，否则直觉上会很违和。

10. **绝对定位 > Grid 用于非均匀时间轴**：里程碑在时间轴上的分布是不均匀的，绝对定位按百分比放置更精确。

### 8.4 协作层

11. **先出完整版，再迭代**：不要逐页确认。全貌 → 修改比逐步构建更高效。

12. **截图是最好的需求文档**：用户发截图说"这个改一下"，比用文字描述"第X页第X行"更快更准。

13. **主动暴露不确定性**：当录音信息模糊时，不要猜测，直接带着你的理解去问用户确认。用选项展示你的理解，让用户做选择题而非填空题。

14. **术语要替用户想一步**：用户可能没意识到某个术语（如 ARPU）对汇报受众来说不够直白。主动提出，让用户决定。本次 ARPU 事件的完整过程：AI 写了"提升客户ARPU" → 用户问"ARPU是什么意思" → AI 解释并主动建议换表述 → 用多选题让用户选择 → 用户在 Other 中给出了比任何选项都好的表述"丰富项目交付的形态类型"。这说明 AI 的选项不是要替用户决定，而是要激发用户给出更好的答案。

15. **措辞变更要理解层级**：区分"文字修正"和"概念重定义"。"积极协同"替换"经济协同"是文字修正（全局替换即可）；"两大交付形态"替换"两大产品方向"是概念重定义（需要同步调整副标题的框架表述）。判断标准：如果修改的是标题级别文字，大概率是概念重定义。

16. **修改后的 Grep 验证不可省略**：每次全局重命名（如 Project→NeoLearning）后，必须 Grep 搜索旧名称确认返回 0 条。本次 "Project→NeoLearning" 和 "KGP→Neo Course" 两轮替换都在最后做了验证，确保了 HTML、注释、标签中的所有出现都被替换。

17. **录音转写错误是系统性风险**："经济协同"→"积极协同"这类错误在语音转写中很常见（同音/近音词混淆）。对于关键产品名、人名、术语，应在初稿交付时标注"来自录音转写，请确认"。

## 九、面向产品化的抽象

### 9.1 可提取为 Skill 的流程模板

```yaml
presentation-from-transcript:
  inputs:
    - transcript: 会议录音转写文本
    - context: 汇报场景描述（可选）
  
  phases:
    1_clarify:
      - round_1: [形式, 受众, 数据深度, 视觉风格]
      - round_2: [内容板块, 命名规范, 篇幅]
      - round_3: [标题文案, 组织信息, 细节确认]
    
    2_extract:
      - facts: 提取事实清单（产品/时间/数字）
      - relations: 梳理实体关系图谱
      - narrative: 根据受众重构叙事线
      - priority: 信息分级（展示/辅助/省略）
    
    3_generate:
      - template: 根据视觉风格选择 HTML 模板
      - content: 填充结构化内容
      - output: 一次性输出完整文件
    
    4_iterate:
      - loop: 接收修改指令 → 定位 → 修改 → 验证
      - confirm: 遇到不确定时主动提问确认
  
  outputs:
    - presentation.html: 可直接打开的演示文稿
```

### 9.2 迭代阶段的修改分类与处理状态机

产品化时，迭代阶段的修改请求需要分类路由到不同的处理流程：

```
用户输入 → 分类器
  │
  ├─ [截图 + 文字指令] → 截图定位链 → Grep → Read → Edit
  │
  ├─ [纯文字，指令明确] → 直接执行
  │   ├─ 文字修正型（"A改成B"）→ Grep 全局搜索 → Edit(replace_all)  → Grep 验证
  │   ├─ 内容增删型（"加一个X"）→ Grep 定位目标区域 → Read 上下文 → Edit
  │   ├─ 结构调整型（"X移到Y前"）→ Read 确认边界 → Edit删 → Edit插 → Grep 验证id
  │   └─ 页面删除型（"删掉这一页"）→ Grep SLIDE注释 → Edit 删除 → 修复后续id
  │
  ├─ [纯文字，指向模糊] → 需要确认
  │   ├─ 概念重定义型 → 理解意图 → 调整措辞 + 可能影响的关联页面
  │   ├─ 内容完整性质疑 → Read 文件检查数据 → 若数据在则检查渲染 → AskUserQuestion 逐项确认
  │   └─ 新增源材料外内容 → 回查源材料 → AskUserQuestion 确认
  │
  └─ [用户反向提问] → 坦诚回答 + 展示理解 + AskUserQuestion 确认
```

### 9.3 可参数化的设计变量

以下变量可在产品化时抽象为可配置项：

```yaml
visual_theme:
  tech_dark:    # 本次使用的科技感暗色主题
    bg: "#0a0e1a"
    accent_colors: [blue, purple, cyan, green, amber]
    font: "Inter"
```
