## 一、program config 详细设计 backlog

### 1.1 项目里程碑（Milestone）详细配置

#### 里程碑分类详细规则

**3 类里程碑**：

| 类型 | 含义 | 数量约束 |
|------|------|---------|
| **定时里程碑** | 按周期重复生成的里程碑，仅 3 种模板：单周 / 双周 / 单月 | 模板触发后系统按周期自动生成，数量取决于项目周期长度 |
| **自定义里程碑** | admin / 项目运营自由插入的任意日期 + 名称里程碑 | **上限 10 个** |
| **阶段里程碑** | 项目阶段性质的里程碑——本期仅 2 种：中期 + 结营 | 每种各 1 个 |

#### 阶段里程碑细则

| 阶段里程碑 | 必选 / 可选 | 是否可改 | 默认时机 |
|-----------|-----------|---------|---------|
| **结营** | 必选（不可删）| 可改名 + 调时间 | 项目周期结束日 |
| **中期** | 可选（创建项目时由 admin 勾选）| 创建后可改名 + 调时间 + 删除 | 项目周期中点 |

> 未来扩展更多阶段里程碑（开营 / 半程 / 收官等），本期仅做中期 + 结营。

#### 约束规则

- 里程碑只能在项目周期内（不能在剩余服务期）
- 里程碑分布完全自由（不要求均匀）
- 同一天**可以**有多个里程碑（未来考虑合并 / 去重逻辑）
- 不设禁用状态——要么有要么删
- 不设"完成"状态——只有"是否到期"两态
- 数量无上限（除自定义里程碑上限 10 外）
- 结营外都可配置

#### 里程碑关联动作 · 管理员侧

| 动作 | 触发时机 | 说明 |
|------|---------|------|
| **生成 team report 草稿** | 里程碑当天 | 默认生成草稿，prompt 由该里程碑配置时填的"关联动作 / 评估要点"驱动生成 |
| **每个管理员一份副本** | 同上 | 每个管理员独立报告库 + 独立草稿副本，本期不做协同分享 / 不做实时多人编辑——需要协同时由管理员导出线下同步 |
| **Ora 提醒管理员** | 草稿生成后立即 | Ora 推送站内通知，含 team report 副本跳转链接 |
| **站内消息预提醒** | 里程碑前 3 天每日 | 与 Ora 提醒为两套不同机制——站内消息提前提醒，Ora 在草稿生成后追加 |
| **接收人** | 项目全部管理员（admin / HR / 项目运营）| 不区分谁负责，运营侧自行协调 |
| **过期未处理** | 提醒保留在列表，不堆累、不重复推送、不升级渠道 | 管理员任意时间打开 team report 副本继续处理 |

#### 里程碑关联动作 · 学员侧（轻量）

| 动作 | 触发时机 | 说明 |
|------|---------|------|
| 大厅倒计时 | 全程 | 学员在大厅看到"距下个里程碑 X 天"（明文可见的里程碑）|
| 里程碑列表 | 全程 | 学员可在大厅进度区看到项目时间轴上明文可见的里程碑 |
| 里程碑到期推送 | 当天 | 仅站内消息（产品化语言："项目过半啦"等，不出现"里程碑"运营术语）|
| Neo 督学话术调整 | —— | **本期 Neo 不感知里程碑**（Neo 督学功能后期再做）|
| 学员主动触发 | —— | 学员**不能**主动触发里程碑动作（运营控制）|

### 1.2 催学规则详细配置

#### 对象

- **本期只做"全员催学"**——所有学员都收到
- **异常学员**：系统标识但不自动催，由项目运营**手动选人**发送站内消息

#### 异常学员判定（系统标识）

两个信号并行（任一触发即标识为异常）：

| 信号 | 默认值 | program config 可调 |
|------|--------|---------------------|
| **脱训信号** | 连续 X 天未登录 | X 可调 |
| **进度落后信号** | 落后项目平均进度 X%（按 Activity 粒度计算）| X 可调 |

#### 渠道

- **本期仅站内消息**——邮件 / 钉钉 / 企业微信 / 飞书 / SMS 全部本期不做（看后续渠道集成成本逐步加）

#### 文案模板

- 支持**模板变量**：`{学员姓名}` / `{个人进度}` / `{项目平均进度}` / `{下一里程碑名}` / `{距里程碑天数}` / `{距结营天数}` 等
- 推送时自动填充

#### 触发类型

| 触发类型 | 说明 |
|---------|------|
| **里程碑绑定型** | 选某个里程碑当天触发（与里程碑关联）|
| **脱训信号型** | 学员被标识为脱训时触发 |
| **进度落后信号型** | 学员被标识为进度落后时触发 |
| 手动触发型 | 运营手动选人发送（异常学员场景）|

#### 调性

- **仅"提醒 / 警示"** —— 鼓励 / 表扬不做催学（这部分由 Neo 在大厅交互中自然表达）

#### 频率上限

- **不设上限**——运营自负责防骚扰

#### 投递日志

- **完整记录**：谁收到 / 是否查看 / 是否点击链接
- **本期不支持回应**（学员无法在催学消息上回复）

### 1.3 平台个性化详细字段

#### 学习期可改动项

| 字段 | 学习期可改？ | 备注 |

- 字体定制（之后也不做）
- 证书 / 徽章定制（本期不做证书 / 徽章）

- **"powered by NeoLearning"** 品牌声明（页脚 / 关键位置保留）
- **大模型安全声明**——如"内容由 NeoLearning AI 提供，使用前请核对"等合规声明 不可替换 / 不可隐藏

### 1.4 program config 全阶段功能矩阵

> 此矩阵汇总每阶段需要 program config 提供的具体功能，作为 § 8.3 program config 详细设计的输入清单。

| 阶段 | program config 功能 / 字段 |
|------|---------------------------|
| **1 商机识别** | **公司画像 context 字段**（可空置）|
| **2 试运行体验** | 试运行项目创建（独立项目实例）/ 简化 schema / 数据隔离 |
| **3 签单** | 项目实例初始化（学员上限 / 内容范围 签单即锁定 + 服务周期默认 1 年待开营锁定）/ **创建 admin 账号**触发 |
| **4 开营组织** | 项目信息编辑（项目名默认自动生成 / **服务周期默认 1 年合同锁定** / 项目周期默认 = 服务周期 - 1 周 / 项目描述 / 客户名 / 公司画像 context 全部选填）/ **批量导入人员名单**（必填，默认学员角色）/ **角色分配**（学员 / 管理员 多选 checkbox）/ 催学规则配置（选填，默认站内）/ 平台个性化（选填）/ 内容预览（仅查看）/ Token 用量统计（后台默认开启，无开关）/ **❌ 不做学员分组标签** / **❌ 预览模式本期不做** / **"确认开营"按钮**（连锁触发：账号开放 + 系统消息 + 状态切换 + 轻量化开营）|
| **5 学习期** | Ora 看板 + Ora 问答 / 异常学员发现（→ 线下跟进）/ **里程碑配置**（详见 § 6.6.3）/ **学员名单 / 项目周期变更**（严格确认）/ **催学规则**（详见 § 6.6.5）/ **平台个性化**（详见 § 6.6.6）/ **team report 生成入口**（每管理员独立报告库 / 不分享）/ **小结 Activity 由 AOM 编排** / ❌ 内容范围不可改 / ❌ 变更日志本期不做 |
| **6 结营** | 结营触发器 / **team report 结项报告生成**（基于结项 prompt 模板）/ "已结营"状态切换 |
| **7 剩余服务期** | 状态显示"剩余服务期内" / 配置只读 / 学员端完整继续学习能力 / **AOM 学习实例 phase 标记**（区分结营前 / 后记录）/ team report 生成入口持续可用 / **服务到期前 1 个月自动提醒** |
| **8 归档** | 归档状态展示 / 产品运营端手工归档接口 / **数据交付包导出工具**（含管理端产物 / 不含原始学习记录）|

**关键设计原则**：

1. **服务周期开营后锁定**——开营前 admin 可通过特殊安全措施修改服务周期；**确认开营后任何阶段都不可改**
2. **学员上限 / 内容范围签单即锁定**——签单后这两项不可改（区别于服务周期 / 项目周期开营后才锁定）
3. **项目周期可调但严格确认**——变更需要 HR 确认 + 销售签字 + 二次弹窗
4. **里程碑驱动**——项目里程碑（默认必选结营 + 可选中期 + 自由插入上限 10）触发 team report 草稿生成 + 站内消息提醒（前 3 天）+ Ora 当天提醒（详见 § 6.6.3）
5. **"确认开营"是分水岭**——之前 program config 全量可编辑；之后进入只读 + 变更管理状态；连锁动作（账号开放 + 系统消息 + 状态切换）
6. **admin 账号 vs 管理员账号 严格区分**——admin 是项目根账号（阶段 3 创建）；管理员账号是 admin 在阶段 4 创建的子账号（HR / 项目运营）
7. **team report 是核心交付引擎**——中期反馈 / 半程 review / 结项报告 / 剩余服务期内的新报告 全部通过 team report 自定义功能实现，本期优先做"结项报告 prompt 模板"
8. **个人结营报告本期不做**——学员个人小结全部在阶段 5 学习期内通过 Course 个人小结 Activity 完成
9. **变更日志本期不做**——未来再加（变更操作的可追溯性后续支持）
10. **数据交付分层**——管理端产物可导出；原始学习记录不导出（防反推产品功能）

## 二、§ 8 管理端 详细设计输入

### 2.1 team report 引擎

- **每个管理员独立报告库 + 独立草稿副本**
- 不做协同分享 / 不做实时多人编辑
- 需要协同时由管理员导出线下同步
- prompt 由里程碑配置时填的"关联动作 / 评估要点"驱动生成（不是选模板 / 不手写）

### 2.2 Ora 双交互模式

- **看板被动呈现**——三维度 + 热力图 + Ora 自动生成的健康度解读（本期不做"看板点击 → Ora 实时解释"）
- **问答主动对话**——管理员主动向 Ora 提问（"为什么 W4 脱训突增？"）

### 2.3 异常学员处置流程

- 平台层：Ora 看板 / 问答 中发现异常学员
- 线下层：项目运营线下跟进（电话 / 微信 / 1on1 沟通）
- 平台不做学员个体的强制干预动作

### 2.4 投递日志查询入口

- 谁收到 / 是否查看 / 是否点击链接
- 本期不支持学员在催学消息上回复
