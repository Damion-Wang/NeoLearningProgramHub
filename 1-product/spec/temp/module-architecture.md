### 1.1 分层原则

- **应用层**：用户直接看到和操作的功能模块。按端划分（学员/HR/运营），加上跨端通用模块（账户权限）
- **平台层**：多个应用层模块共用的能力。不直接面向用户，通过 API 被应用层调用
- **基础设施**：技术组件，不含业务逻辑。可替换、可独立演进

### 1.2 调用关系原则

- 应用层 → 平台层 → 基础设施（单向依赖，不能反向调用）
- 同一层内的模块可以互相通信，但通过平台层中转（如学员端辅导模块通过记忆工程读取授课模块产生的数据）
- 三端通过统一 Database 实现数据共享，不做模块间直接数据传递

## 2. 第一层：应用层

### 2.1 学员端模块

#### 2.1.1 辅导模块（大厅）

**定位**：学员的"家"，1v1 AI 教练的持续陪伴界面

**核心功能**：
- 辅导对话（右侧 Chat UI）：回答真实工作问题、推荐课程、接收实践反馈
- 左侧看板：学习进度 / 课程结构 / 消息中心 / 笔记资料 / 工具卡片 / 待办任务
- 首次进入引导（LUI方式，教练自我介绍）
- 每次回到大厅教练主动提一句刚学完的内容

**依赖的平台层能力**：记忆工程（引用课堂记忆）、Agent引擎（辅导教练运行时）、通知系统（消息中心）

**关键设计决策**：
- [ADR-003] 辅导模块是学员端主入口
- [ADR-006] 知识库用渐进式披露而非RAG
- 辅导教练角色优先级：辅导 > 督学 > 工具引导
- 人格本期固定，架构上按 Context 包隔离设计

#### 2.1.2 授课模块（教室一）

**定位**：AI 老师的结构化知识演绎

**核心功能**：
- AOM 驱动的 SCO 序列执行（PPT讲解/案例视频/小测评/小解答/Quiz/闭环反馈）
- AI 自主发挥讲解（根据提示和学员状态，非照念脚本）
- 插嘴提问+探讨（Activity内简单答疑，非辅导级陪伴）
- 主动回顾（学员发起，AI补充后拉回）
- L2 自适应（打标SCO + Segment版本切换）
- 工具卡片/表格产出（预制+AI生成）
- 左侧：Tab1=SCO列表（类似PPT缩略图）/ Tab2=Project目录
- 中间：课件展示（PPT/视频/Quiz题干+辅助文件）
- 右侧：AI老师对话

**依赖的平台层能力**：自适应引擎（打标+版本切换）、记忆工程（小解答引用记忆）、报告数据引擎（碎片生产）

**关键设计决策**：
- [ADR-004] 内容驱动架构，教学行为由AOM脚本控制
- SCO与Segment一对一，Segment用作版本管理
- 打标SCO独立于互动SCO，影响范围限Activity内
- 闭环反馈本期做完整版（FEEDBACK_COLLECT + FEEDBACK_REVIEW）
- 纯讲述SCO需加互动但要找平衡点（与教学内容相关，不泛化）

#### 2.1.3 对练模块（教室二）

**定位**：基于内容的沉浸式角色扮演训练

**核心功能**：
- 四阶段：情景导入 → Roleplay演绎 → 反思复盘 → 对练报告（可选）
- AI角色行为完全由对练剧本控制（70+字段角色画像/行为反应规则/评估标准）
- 反思阶段AI退出角色，苏格拉底追问
- 对练可多版本（基于打标匹配不同剧本版本）
- 可完整重练（重练=重练Roleplay阶段，不含前导）
- 左侧：Project目录
- 中间：场景描述 + 角色信息 + 角色形象（文生图，未来可切换表情）
- 右侧：Roleplay对话 / 反思对话
- 反思复盘时中间看板展示"对话回看"组件

**依赖的平台层能力**：Agent引擎（导演-演员分离模式）、报告数据引擎（能力评估碎片）、记忆工程

**关键设计决策**：
- 本期仅Roleplay一种玩法，角色互换/费曼/辩论/多人为迭代方向
- 情绪天花板由剧本定义，不是平台硬编码
- 对练结束机制由剧本控制
- 无旁白（区别于代码中的Narrator设计）
- excludeReport机制（对练报告可选）

#### 2.1.4 测评模块（教室三）

**定位**：AI通过对话/量表获取学员真实信息并评判

**核心功能**：
- 两种类型：对话式测评（BEI/学员画像/困惑收集/高光萃取）+ GUI量表（类MBTI）
- 两段结构：测（信息采集）+ 评（解读反馈，可选）
- 测评类型和策略完全由测评剧本控制
- BEI是否使用由内容决定
- 不考虑编造真假——AI不做判官
- 左侧：量表类有题目目录Tab / 对话类仅Project目录
- 中间：量表类→量表GUI交互 / 对话类→背景信息展示
- 右侧：测评对话

**依赖的平台层能力**：Agent引擎、报告数据引擎、记忆工程

#### 2.1.5 个人报告模块（教室四）

**定位**：学员的学习成果可视化+AI解读

**核心功能**：
- 每个Course结束后固定生成（未来可从辅导随时进入）
- 两段结构：报告生成（渐进式加载+情感化文案）+ 报告解析（学员与AI讨论）
- 报告内容：高光时刻/能力成长/认知转化/学习表现/技能提升/成长表现/综合评估/行动建议（无基础信息）
- 学员端展示：个人成长时间线+匿名分位区间（不展示名次）
- BI图表：雷达图/折线图/热力图/饼图/柱状图
- 左侧：Project目录
- 中间：报告内容展示（含BI图表）
- 右侧：与AI讨论报告

**依赖的平台层能力**：报告数据引擎（碎片汇聚+模板+计算）、Agent引擎（报告解析对话）

### 2.2 HR 端模块

#### 2.2.1 进度看板

- 查看所有学员整体完成进度（Activity级粒度）
- 按Course维度查看细分进度
- 按学员维度查看个人进度详情

#### 2.2.2 数据看板

- 学习进度维度（必须有，个人角度）
- 学习效果维度（待定具体指标）
- 数据呈现为主，不做智能评价
- 项目健康度指标待细化设计

#### 2.2.3 报告查看

- 查看个人学习报告（与学员看到的同一份）
- 查看项目整体报告
- 进度排名+互动排名（不做能力排名）

#### 2.2.4 HR Agent

- 与报告智能体共享底层能力，面向HR的交互模式
- 自然语言提问："本周谁的进度最快？"
- 生成汇报报告（PDF格式，记忆HR模板偏好）
- 解读数据（可逐层钻取到原始数据）
- 贯穿学习期和结项期
- 本期运营端Agent与HR Agent一致

### 2.3 运营端模块

#### 2.3.1 项目配置

- 导入学员名单（Excel/CSV）
- 配置项目基础参数（名称/时间/课包绑定）
- 设置催学规则（可配置，不硬编码）

#### 2.3.2 学员管理

- 查看学员列表、状态、进度

#### 2.3.3 催学管理

- 可配置的催学规则（HR端和运营端都可配置）
- 系统自动执行，重要情况人工确认
- 效果追踪漏斗（发送→查看→登录→完成活动）

#### 2.3.4 数据展示

- 数据呈现为主，运营自行判断什么指标不健康
- 可配置的自动提醒功能（不做智能预警评价）

#### 2.3.5 报告工厂

- 选择报告类型、范围、模板
- 批量生成→审核队列→导出PDF
- Agent辅助审核（解释数据来源和计算逻辑）

#### 2.3.6 运营Agent

- 本期与HR Agent一致（同一Agent实例，不同权限视图）
- 未来迭代为独立Agent

## 3. 第二层：平台层

### 3.1 记忆工程 [ADR-005]

**职责**：管理跨模块、跨Course的学员记忆数据

**核心能力**：
- 课堂内记忆采集（授课对话/对练表现/测评结果）→ 写入统一Database
- 辅导引用（读取课堂记忆，回到大厅时必然引用一次+话题条件触发）
- 跨Course记忆打通（引用优先级：当前Course > 相关Course > 远期Course）
- 课前采集记忆存储（structured_tags + raw_dialogue 双层格式）
- 被动信号采集（停留时长/是否离开/是否回翻/是否点击）

**被调用方**：辅导模块、授课模块、对练模块、测评模块、报告数据引擎、HR Agent

### 3.2 自适应引擎

**职责**：执行内容驱动的学习路径个性化

**核心能力**：
- L1：LLM实时响应（每次交互自适应）
- L2：打标SCO判定 → Segment版本切换（标签严格限Activity内）
- 低置信度降级（<0.6时使用默认版本）
- 断点续播遇后台打标SCO时自动跳过

**被调用方**：授课模块、对练模块（多版本剧本匹配）

### 3.3 Agent引擎

**职责**：提供AI对话运行时环境

**核心能力**：
- 辅导教练Agent运行时（Persona + Skill组合，基于coaching-skills架构）
- 授课AI运行时（AOM脚本驱动的演绎引擎，基于Prosona）
- 对练AI运行时（导演-演员分离模式，基于SIDE/drill-mate）
- 测评AI运行时
- HR/运营Agent运行时
- Context包加载与隔离

**被调用方**：所有包含AI对话的应用层模块

### 3.4 通知系统

**职责**：消息触达

**核心能力**：
- 产品内消息（辅导对话顶部/消息中心）
- 可配置的催学规则自动执行
- 渠道：V1=产品内+Email；V1可选=企微/钉钉（需OAuth）

**被调用方**：催学管理、运营端

### 3.5 报告数据引擎

**职责**：碎片汇聚、计算、模板填充

**核心能力**：
- 碎片生产规则：Activity级学习特质碎片 / Quiz级认知转化碎片 / 对练级能力评估碎片
- 碎片汇聚 → 按能力维度组织 → 行为改变证据链
- 报告模板引擎（V1至少3套：项目整体/个人学习/CEO汇报PDF）
- 成长幅度计算（BEI基线优先；无基线用前1/3 vs 后1/3均值）
- 进度排名+互动排名计算

**被调用方**：个人报告模块、HR报告查看、报告工厂、HR/运营Agent

## 4. 第三层：基础设施

### 4.1 统一Database

- 所有端按权限读取，不做方向性数据流转
- 授课对话记录用半结构化格式（含passive_signals字段）
- 数据隔离：不同Project的数据隔离

### 4.2 LLM接入

- 优先国产大模型（DeepSeek/Qwen等）
- 不同模块可能使用不同模型（如打标用轻量模型，辅导用旗舰模型）
- AI质量控制：基础版，后台查找溯源即可

### 4.3 AOM解析引擎

- 解析AOM数据包（Project→Module→Activity→SCO→Segment五层）
- SCO类型路由（NARRATION/CHAT/ASSESSMENT_TAG/FEEDBACK_COLLECT/FEEDBACK_REVIEW/quiz/roleplay）
- Segment版本选择（基于打标结果）

### 4.4 BI图表组件库

- 通用化——不仅用于报告，也用于HR端和运营端看板
- 必需类型：雷达图/热力图/饼图/折线图/柱状图
- 可交互——点击图表元素联动Agent对话区
- 本期必须有但优先级为优化项

### 4.5 三栏布局框架

- 三端统一三栏：左侧目录（可折叠）| 中间看板（GUI，默认宽）| 右侧对话（Agent，宽度可调）[ADR-007]
- 看板区和对话区双向联动
- 11种场景的默认比例（大厅15:50:35、授课PPT 10:55:35、对练演绎10:35:55等）
- 切换场景恢复默认比例
- 学员端大厅→教室切换时三栏内容变化（不是全屏切换）
